# 2026-06-11 작업 내역

## 오늘 반영된 주요 작업 요약

로그인 유지 체크박스 동작을 실제로 구현하고, Axios 401 인터셉터에서 토큰 자동 갱신 후 원본 요청을 재시도하는 로직을 추가했습니다. 4차 코드 리뷰 피드백을 반영해 결제 API 라우트 세 곳에 인증 가드를 적용했습니다.

---

## 세부 작업

### 1. 로그인 유지 체크박스 구현 (`store/authStore.ts`, `hooks/auth/useLoginForm.ts`)

- **원인:** `keepLogin` 상태가 UI에만 존재하고 실제 스토리지 분기 처리가 없었음
- Zustand `persist` 미들웨어 제거 → 직접 `localStorage` / `sessionStorage` 분기 처리로 교체
  - 체크 시: `localStorage`에 저장 (브라우저 재시작 후에도 유지)
  - 미체크 시: `sessionStorage`에 저장 (탭/브라우저 종료 시 삭제)
- `setAuth(accessToken, refreshToken, user, keepLogin?)` 시그니처 변경
  - `keepLogin` 파라미터 없으면 현재 스토어의 `keepLogin` 값을 유지 (토큰 갱신 시 재사용)
- `rehydrate()` 메서드 추가 — 앱 시작 시 `localStorage` → `sessionStorage` 순으로 탐색해 복원
  - 구형 Zustand persist 포맷(`{ state: { ... } }`) 하위 호환 처리 포함
- `patchUser()` / `clearAuth()` 도 스토리지와 동기화
- `useLoginForm.ts` — `setAuth` 호출에 `keepLogin` 인수 전달

---

### 2. Axios 401 인터셉터 토큰 자동 갱신 (`lib/api.ts`)

- 401 응답 시 `supabase.auth.refreshSession()` 호출
  - 성공: 스토어 업데이트 후 원본 요청에 새 토큰 주입해 재시도
  - 실패: `clearAuth()` 후 `/login`으로 리다이렉트
- `isRefreshing` 플래그 + `refreshQueue` 패턴으로 동시 401 요청 처리
  - 첫 요청이 갱신 진행 중일 때 추가 401이 오면 큐에 쌓고, 갱신 완료 후 일괄 재시도
- `refreshToken`이 없을 경우 갱신 시도 없이 일반 에러로 통과 (로그인 실패 등 정상 401과 구분)

---

### 3. 결제 API 라우트 인증 가드 적용 (`app/api/payment/`)

- **원인:** 4차 리뷰에서 `serverAuth`의 `requireAuth`가 결제 라우트에 적용되지 않아 비인증 요청 통과 가능 지적
- `POST /api/payment/kakao/ready` — `requireAuth` 추가, `partner_user_id`를 클라이언트 전달 `userId` 대신 인증된 `authed.userId`로 교체
- `POST /api/payment/kakao/approve` — `requireAuth` 추가, 동일하게 `partner_user_id` 교체
- `POST /api/payment/toss/confirm` — `requireAuth` 추가

---

### 4. AuthInitializer 수정 (`components/(auth)/AuthInitializer.tsx`)

- Zustand `onRehydrateStorage` 콜백 의존 제거 → 마운트 시 `rehydrate()` 직접 호출
- `TOKEN_REFRESHED` 이벤트 핸들러는 그대로 유지 (Supabase 내부 갱신 시 스토어 동기화)
