# 2026-07-06 작업 내역

## 오늘 반영된 주요 작업 요약

메인 홈 "오늘의 추천 레시피" 카드 이동 링크를 고치는 것으로 시작해, 주문 취소/환불 흐름을 하루 종일 집중적으로 점검·보강했습니다. 멀티셀러 주문에서 일부 상품이 이미 배송 시작됐어도 전체 취소 신청이 가능했던 문제와 판매자가 환불 승인을 재시도하면 PG사에 이중 취소 요청이 나갈 수 있던 문제를 고쳤고, 사용자 쪽에 없던 "배송완료 후 환불 신청" 기능을 신설해 판매자 환불요청 탭이 항상 비어있던 gap을 메웠습니다. 이어서 레시피 재료를 실제 장바구니에 연결해 레시피 경유 구매 시 추천 포인트가 지급되도록 만들었고, 그 과정에서 드러난 취소/환불 동시성·중복 신청 버그를 `lib/orderClaims.ts`로 로직을 공용화하며 함께 정리했습니다. 마지막으로 죽어있던 주문 상세 스텁 라우트를 제거하고, 관리자가 회원을 정지시켜도 로그인이 그대로 허용되던 보안 gap을 막았습니다.

---

## 세부 작업

### 1. 오늘의 추천 레시피 카드 상세페이지 이동 링크 추가

- `TodayRecipeCard.tsx`: 카드 전체에 `cursor-pointer` 스타일만 있고 실제 클릭 핸들러/링크가 없어 "더보기"와 썸네일·제목을 클릭해도 이동하지 않던 문제 수정
- 인기 레시피 섹션(`PopularRecipeSection`)과 동일하게 "더보기"는 `/recipes`(추천 레시피가 없을 때는 기본값)로, 카드 본문은 `/recipes/[recipeId]`로 이동하도록 `next/link` 연결
- 기능 없이 이미지 위에 떠 있던 좋아요(하트) 버튼 제거

### 2. 주문 취소 신청 시 아이템별 배송 상태 검증 (`POST /api/orders/[orderId]/cancel`)

- 기존엔 `orders.status`(전체 주문 상태)만 보고 취소 가능 여부를 판단했는데, 멀티셀러 주문의 `orders.status`는 "가장 덜 진행된 아이템" 기준으로 집계되어 A셀러 상품이 배송중이어도 B셀러 상품이 결제완료 상태면 전체가 취소 가능한 것처럼 보였음
- `order_items.shipping_status`가 `결제완료`인 상품만 취소 신청 대상으로 필터링하도록 변경, 이미 배송 시작된 상품은 자동 제외

### 3. 환불 승인/처리 시 이중 취소 방지 및 원자성 보강 (`approveRefund`, `processRefund`)

- PG 결제 취소(`cancelPayment`) 직전에 `order_items.shipping_status`가 이미 `취소`/`환불`인지 검증하는 가드 추가
- PG 취소 성공 이후 후속 DB 반영(상태 업데이트·이력 기록·포인트 환급)을 하나의 트랜잭션 블록으로 묶고, 이 단계에서 실패하면 상태를 되돌리는 대신(재승인 시 동일 `payment_key`로 재취소되는 위험 방지) 에러 로그를 남기고 "다시 승인하지 말 것"을 안내하는 메시지로 처리하도록 변경
- 취소 금액 계산에서 쿠폰 할인분(`allocated_coupon_discount`)을 차감하지 않던 문제 수정

### 4. 배송완료 주문 환불 신청 기능 추가

- 기존엔 결제완료/주문확인 상태의 취소 신청만 있고, 배송완료 후 하자·단순변심으로 환불을 신청하는 사용자 액션이 없어 판매자 쪽 "환불요청" 탭이 항상 비어있었음
- `POST /api/orders/[orderId]/refund` 신설: 배송완료 아이템만 대상으로 하고, 구매확정 완료·중복 신청 건은 가드 처리
- 마이페이지 주문 카드에 환불 신청 버튼/모달 추가, 신청 후 중복 신청 방지 UI 반영
- 주문 목록/상세 API가 `hasPendingRefundRequest`를 함께 내려주도록 확장(기존 취소 신청 처리 방식과 동일하게 통일)

### 5. 레시피 재료 장바구니 연동 및 구매확정 시 추천 포인트 지급

- `recipe_order_referrals`에 INSERT하는 코드가 어디에도 없어 관리자 통계 화면이 항상 빈 데이터였고, 레시피 재료의 "장바구니 담기" 버튼도 `onClick` 없이 장식용으로만 존재해 레시피 경유 구매 자체가 발생할 수 없었음
- 레시피 재료 개별/전체 담기를 실제 장바구니(`recipeId` 포함)에 연결
- 장바구니 → 주문 생성 시 `recipeId`를 `order_items`에 태깅
- 배송완료 후 지연 구매확정 시점(`ensurePurchaseConfirmations`)에 `recipe_order_referrals` insert + 레시피 작성자에게 포인트 지급하는 로직 추가(`lib/purchaseConfirmation.ts`)

### 6. 취소/환불 신청 동시성·상태 처리 버그 수정 및 로직 공용화

- 취소/환불 신청 로직을 `lib/orderClaims.ts`로 공용화하고 DB unique index로 더블클릭/다중 탭으로 인한 중복 신청을 차단
- 레시피 경유 구매확정 포인트가 중복 지급되지 않도록 unique 제약 + 충돌 시 스킵 처리
- 장바구니 병합 시 `recipeId`가 유실되던 문제 수정
- 주문 생성 후 `recipe_id` 백필 실패가 조용히 무시되던 문제 수정
- 판매자 승인 후 "환불진행중" 상태를 구매자 화면이 인식하지 못하던 로직 수정

### 7. 미사용 주문 상세 스텁 라우트 제거

- `app/(main)/mypage/orders/[orderId]/page.tsx`는 "주문 상세 페이지 준비 중입니다."만 렌더링하는 스텁이었고, 실제 상세는 목록 페이지의 `OrderDetailModal`로 대체 구현되어 있어 어디서도 링크되지 않던 죽은 라우트 — 삭제

### 8. 정지된 계정 로그인 차단

- 관리자가 회원/판매자를 정지(`users.status = 'suspended'`)시켜도 `app/api/auth/login/route.ts`가 이를 검사하지 않아 정지 계정이 그대로 로그인되던 문제 수정
- 로그인 시 `status`를 함께 조회해 `suspended`면 세션을 무효화하고 403 + 안내 메시지 반환
- `useLoginForm`이 서버가 내려준 한글 에러 메시지를 무시하고 항상 일반 문구로 뭉개던 부분도 함께 수정해 정지 안내가 실제로 노출되도록 함
