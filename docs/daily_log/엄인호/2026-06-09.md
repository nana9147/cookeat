## 세부 작업

### 1. sellers 테이블 스키마 수정

- **원인:** `is_approved (BOOLEAN)`과 `approve_status (ENUM)` 두 컬럼이 중복 역할을 하고 있었고, 거절 사유를 저장할 컬럼이 없었음
- `is_approved` 컬럼 제거
- `rejected_reason VARCHAR(200) NULL` 컬럼 추가 — `rejected` 상태일 때만 값 존재
- `business_address VARCHAR(300) NULL` 컬럼 추가 — 사업장 주소

```sql
ALTER TABLE public.sellers DROP COLUMN IF EXISTS is_approved;
ALTER TABLE public.sellers ADD COLUMN IF NOT EXISTS rejected_reason VARCHAR(200);
ALTER TABLE public.sellers ADD COLUMN IF NOT EXISTS business_address VARCHAR(300);
```

### 5. 로그아웃 시 MypageGuard 리다이렉트 충돌 방지

- **원인:** `clearAuth()` 호출 시 `MypageGuard` useEffect가 `accessToken = null`을 감지해 `router.replace('/login')`을 실행, 로그아웃 핸들러의 `router.replace('/')` 보다 나중에 실행되어 `/login`으로 이동
- `MypageGuard.tsx` — `wasAuthed` ref 추가: 처음부터 미인증 상태로 접근한 경우에만 `/login` 리다이렉트, 로그아웃으로 토큰이 사라진 경우는 가드가 개입하지 않음
- `MypageCategory.tsx` — `router.push('/')` → `router.replace('/')` 로 변경 (back 스택 정리)

### 9. 역할 기반 인증 공통 유틸 추가

- **원인:** seller·admin API 라우트마다 토큰 파싱·유저 조회·role 확인 로직이 중복될 수 있어 공통 유틸로 분리
- `lib/serverAuth.ts` 생성
  - `getAuthedUser(token)` — 토큰으로 Supabase auth 유저 조회 후 `users` 테이블에서 `user_id`·`role` 반환
  - `requireAuth(req)` — 로그인 여부만 확인, 미인증 시 401
  - `requireSeller(req)` — `role: seller | admin` 만 허용, 미인증 401 / 권한 없음 403
  - `requireAdmin(req)` — `role: admin` 만 허용
- **사용 패턴:** `const authed = await requireAuth(req); if (authed instanceof NextResponse) return authed;`
- `app/api/seller/apply/db.ts` — 중복된 `getUserId` 함수 제거 (`lib/serverAuth.ts`로 통합)
- `app/api/seller/apply/route.ts` — 직접 토큰 파싱 제거, `requireAuth` 사용으로 교체
