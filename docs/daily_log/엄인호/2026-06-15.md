# 2026-06-15 작업 내역

## 오늘 반영된 주요 작업 요약

장바구니 플로우(장바구니 → 주문서 작성 → 결제 완료)의 마지막 단계인 결제 완료 페이지를 구현했습니다. `components/order` 폴더를 `components/cart`로 이동하고, `PaymentSummary`를 공용 컴포넌트로 확장해 cart·checkout·complete 세 페이지에서 재사용하도록 정리했습니다.

---

## 세부 작업

### 1. components/order → components/cart 폴더 이동

- **원인:** 컴포넌트 폴더명이 라우트 구조(`app/(main)/cart/`)와 일치하지 않아 혼란 유발
- `components/(auth)/order/` 하위 전체를 `components/(auth)/cart/`로 이동
- `app/(main)/cart/page.tsx`, `checkout/page.tsx`, `complete/page.tsx`의 import 경로 일괄 수정

---

### 2. PaymentSummary 공용 컴포넌트 확장

- **원인:** checkout 전용으로 작성된 컴포넌트를 complete 페이지에서도 재사용하기 위해 확장 필요
- `noCard?: boolean` 추가 — `true`일 때 카드 래퍼·제목·안내박스 없이 금액 rows만 렌더링
  - complete 페이지 왼쪽 카드(배송정보+결제정보) 내부에 임베드할 때 사용
- `paymentMethodLabel?: string` 추가 — 결제 수단 행을 동적으로 표시
- `장바구니로 돌아가기` 링크를 `mode === 'checkout'` 조건으로 이동 (complete에서 미노출)

---

### 3. 결제 완료 페이지 구현 (`cart/complete/`)

- **구성:** 파일당 80줄 이하로 컴포넌트 분리
  - `completeData.ts` — 타입, 상수(DELIVERY, ORDER_INFO, ITEMS, FINAL_AMOUNT), PAYMENT_LABEL
  - `CompleteHeader.tsx` — 카드1: 체크마크 + 완료 메시지 + 주문번호/일시 + 주문내역 링크
  - `CompleteOrderInfo.tsx` — 카드2(좌): 배송 정보 + `PaymentSummary(noCard)` 결제 정보
  - `CompleteOrderItems.tsx` — 카드3(우): 주문 상품 목록 + 결제금액 합계 + CTA 버튼
  - `CompleteNotice.tsx` — 카드4: 안내사항
  - `PaymentResult.tsx` — 오케스트레이터 (loading/fail/success 분기 후 카드 조합)
- **레이아웃:** 카드2·3은 `flex-1` + `desktop:items-stretch`로 1:1 동일 높이 유지
- **반응형:** mobile/tablet 세로 스택, desktop 가로 2컬럼

- **원인:** 빌드가 5회 연속 깨진 채 develop에 머지됨. 사람이 매번 확인하기 어렵다고 판단
- `.github/workflows/ci.yml` 생성 — develop 대상 PR에서 `npm ci` → `npx tsc --noEmit` → `npm run build` 자동 실행
- 타입 오류가 있는 PR은 머지 버튼이 비활성화됨

### 5. useKakaoPostcode 전역 타입 수정 (빌드 복구)

- **원인:** `hooks/user/useKakaoPostcode.ts:7`의 `Window.kakao` 선언이 필수(`kakao:`)로 되어 있어 TS2687 발생, 5회째 빌드 실패
- `kakao:` → `kakao?:` 수정 — Kakao SDK 로드 전에는 `window.kakao`가 실제로 `undefined`이므로 선택이 맞음
- SDK 미로드 상태에서 `openPostcode` 호출 시 조기 반환 guard(`if (!window.kakao) return;`) 추가

---

### 6. 결제 금액 위조방지

- **원인:** `FINAL_AMOUNT = 35410`이 클라이언트에 하드코딩되어 있고, 서버가 금액을 검증하지 않아 요청 본문 변조 시 임의 금액으로 결제 가능
- `app/api/order/route.ts` 신규 생성 — 결제 전 서버가 주문을 DB(`orders` 테이블)에 `status: 결제전`으로 생성하고 `final_amount`를 저장
- `useCheckoutPayment.ts` — `FINAL_AMOUNT` 상수 제거, 결제 전 `/api/order` 호출 후 서버 반환 금액(`confirmedAmount`)을 PG에 전달
- `PaymentSummary.tsx` — `onAmountChange` 콜백 추가, `Checkout`이 계산된 금액을 상태로 관리
- `Complete.tsx` — `userId: 'test_user'` 제거

---

### 7. 주문 확정 단계 구현

- **원인:** 결제 성공 후 주문이 DB에 저장되지 않아 주문 내역 조회 불가, 중복 결제 방지 로직 없음
- `payment/toss/confirm/route.ts` — DB에서 주문 조회 → `final_amount`와 클라이언트 `amount` 대조 → Toss confirm 호출 → `status: 결제완료` 업데이트. `status === 결제완료`면 즉시 반환(idempotent)
- `payment/kakao/approve/route.ts` — DB 주문 확인 → 카카오 승인 → `status: 결제완료` 업데이트. 중복 방지 동일 적용

---

### 8. 배송지 정보 주문 DB 저장 연동

- **원인:** 주문 생성 시 `recipient`, `phone`, `address`가 빈 값으로 저장되어 배송 정보 없는 주문 생성
- `DeliveryInfo.tsx` — `onAddressSelect` 콜백 추가, 초기 선택 주소를 마운트 시 즉시 부모로 올림. 주소 변경 시에도 콜백 호출
- `Checkout.tsx` — `deliveryInfo` state 추가, `DeliveryInfo`에서 받은 주소를 `useCheckoutPayment`으로 전달
- `useCheckoutPayment.ts` — `deliveryInfo` 파라미터 추가, `/api/order` 호출 시 `recipient`, `phone`, `address` 포함
- `app/api/order/route.ts` — 요청 본문에서 배송 정보 수신 후 orders 테이블에 저장

---

### 9. Complete 페이지 useSearchParams Suspense 경계 추가

- **원인:** `Complete` 컴포넌트에서 `useSearchParams()`를 사용하나 Suspense 경계 없이 렌더링되어, Next.js 정적 빌드 중 `/cart/complete` 프리렌더링 실패
- `app/(main)/cart/complete/page.tsx`에서 `<Complete />`를 `<Suspense>`로 감쌈
- Suspense 경계가 생기면 해당 컴포넌트를 클라이언트 사이드 렌더링으로 위임해 빌드 통과

---

### 10. Supabase 클라이언트 지연 초기화로 CI 빌드 오류 해소

- **원인:** `lib/supabase.ts`, `lib/supabaseAdmin.ts`에서 `createClient`가 모듈 평가 시점에 즉시 실행되어, CI 빌드 중 Next.js가 API 라우트를 임포트할 때 `supabaseUrl is required` 오류 발생
- `Proxy` 기반 lazy init으로 변경 — 첫 번째 메서드 접근 시 인스턴스를 생성하도록 미룸
- 기존 `supabaseAdmin.xxx` / `supabase.xxx` 호출 코드는 수정 없이 그대로 동작
