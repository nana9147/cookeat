# 2026-06-16 작업 내역

## 오늘 반영된 주요 작업 요약

멘토 피드백을 반영해 Next.js 16 proxy 방식으로 인증 체계를 전면 재구성했습니다. 기존 클라이언트 기반 권한 체크를 서버 proxy에서 JWT 직접 검증하는 방식으로 전환했고, 소셜 로그인(OAuth)의 세션 동기화 누락도 함께 해결했습니다. Google 프로필 이미지 렌더링 허용도 추가했습니다.

---

## 세부 작업

### 1. Next.js 16 proxy 방식 권한 체크 구현

- **원인:** seller/admin 페이지에 권한 없이 접근 가능했고, 클라이언트 기반 권한 체크는 우회 위험 존재
- `proxy.ts` 신규 생성
  - `/seller/:path*`, `/admin/:path*` 경로에서 `sb-access-token` httpOnly 쿠키로 JWT 직접 검증
  - 토큰 만료 시 `sb-refresh-token`으로 자동 갱신 후 새 쿠키 재발급
  - role 불일치(`admin`·`seller`) 시 `/`로 리다이렉트
  - `/admin/login`은 인증 없이 통과 허용

### 2. 로그인/로그아웃 httpOnly 쿠키 처리

- **원인:** proxy.ts는 JS 접근 불가 httpOnly 쿠키만 읽을 수 있어 기존 클라이언트 스토리지 방식으로는 작동 불가
- `app/api/auth/login/route.ts` 수정
  - 로그인 성공 시 `sb-access-token`, `sb-refresh-token`, `sb-role` httpOnly 쿠키 발급
  - `error || !data.user || !data.session` 조건 강화
- `app/api/auth/logout/route.ts` 수정
  - supabase admin으로 서버 측 세션 무효화 후 세 쿠키 전부 삭제
- `lib/cookieOptions.ts` 신규 생성 — `httpOnly`, `secure`, `sameSite: lax`, `path: /` 공통 옵션

### 3. 토큰 갱신 API 분리 (`/api/auth/refresh`)

- **원인:** 클라이언트에서 supabase SDK로 갱신하면 httpOnly 쿠키가 갱신되지 않아 proxy가 만료 토큰으로 계속 차단
- `app/api/auth/refresh/route.ts` 신규 생성
  - `refreshToken`으로 supabase admin에서 새 세션 발급
  - DB에서 role 재조회 후 응답에 포함
  - 새 토큰을 httpOnly 쿠키로 재발급
- `lib/api.ts` 수정 — 401 인터셉터에서 supabase SDK 직접 호출 → `/api/auth/refresh` 경유로 변경

### 4. OAuth 콜백 세션 동기화 (`/api/auth/sync-session`)

- **원인:** 소셜 로그인(OAuth)은 클라이언트에서 토큰을 받기 때문에 httpOnly 쿠키 미설정 → proxy에서 미인증으로 판단
- `app/api/auth/sync-session/route.ts` 신규 생성
  - 클라이언트에서 받은 `accessToken`·`refreshToken`을 서버에서 검증 후 httpOnly 쿠키로 동기화
  - role DB 조회 후 `sb-role` 쿠키도 함께 발급
- `app/auth/callback/page.tsx` 수정 — 소셜 로그인 콜백 직후 `/api/auth/sync-session` 호출 추가

### 5. Google 프로필 이미지 렌더링 허용

- **원인:** 구글 소셜 로그인 후 프로필 이미지가 깨지는 문제
- `next.config.ts` 수정 — `lh3.googleusercontent.com` http/https 패턴 추가
