주문 관리 상세정보 다이얼로그의 주문일시가 `2026-07-01T07:22:07.395393+00:00` 같은 원본 ISO 문자열로 그대로 표시되던 문제. 목록에서는 `formatDate`를 거치는데 상세 다이얼로그만 누락되어 있었습니다. `lib/utils.ts`에 이미 있던 `formatDateTime`(seller 쪽에서 쓰던 것과 동일)을 재사용해 `yyyy-MM-dd HH:mm` 형식으로 통일했습니다.

### 3. 정산·1:1 문의 목록 잘림 문제 해결

**파일**: `app/admin/settlements/page.tsx`, `app/admin/support/inquiry/page.tsx`

- 정산 관리: 파라미터 없이 API를 한 번만 호출해 기본값(`limit=20`)에 걸려 최신 20건만 표시되고, 탭 필터(대기/완료)도 클라이언트에서만 걸러지던 문제. 탭·페이지 상태에 맞춰 `status`/`page`/`limit`을 서버에 전달하고 `Pagination` 컴포넌트를 연동. "일괄 정산"도 화면에 보이는 20건이 아니라 대기 중인 전체 건을 별도 조회해 처리하도록 수정.
- 1:1 문의 관리: `{ limit: 100 }`으로 한 번만 조회해 100건을 넘으면 오래된 미답변 문의를 열 수 없던 문제. `page`/`limit=20` 페이지네이션과 `keyword` 서버 검색을 연동하고, 답변대기/완료 통계는 잘린 목록이 아닌 별도 카운트 조회로 정확히 표시하도록 수정.

### 4. 판매자 상태 변경 시 정지 해제 미반영

**파일**: `app/admin/sellers/page.tsx`

'정지' 상태 판매자를 '대기'/'거절'로 변경해도 클라이언트가 `suspend: false`를 서버에 전달하지 않아 `users.status`가 계속 `suspended`로 남던 문제. 저장 시엔 반영된 것처럼 보이지만 새로고침하면 다시 '정지'로 표시됨. 편집 시작 시점의 원래 상태를 별도로 저장해두고, 정지에서 벗어나는 편집일 때만 해제 여부를 명시적으로 전달하도록 수정.

### 5. 판매자 승인 시 계정 활성화 실패 무시

**파일**: `app/api/admin/sellers/[sellerId]/approve/route.ts`

승인 처리 중 `users.status`를 `active`로 바꾸는 업데이트의 에러를 체크하지 않아, DB 오류가 나도 API가 `{ok:true}`를 반환하던 문제. 에러 체크를 추가해 실패 시 500을 반환하도록 수정.

### 6. 회원 관리 등급(VIP) 필터 미동작

**파일**: `app/api/admin/users/route.ts`

클라이언트가 `grade` 쿼리 파라미터를 보내지만 서버가 읽지 않아 필터가 조용히 무시되던 문제. PostgREST 집계 함수(`count()`)가 프로젝트 DB에서 비활성화되어 있는 걸 실제 조회로 확인한 뒤, `orders.user_id`를 가져와 앱 코드에서 사용자별 주문 수를 집계 → 주문 10건 이상인 `user_id` 목록을 구해 `.in()`(VIP)/`.not(...'in')`(일반)으로 필터링하도록 구현. 실제 DB로 VIP 1명 + 일반 7명 = 전체 8명임을 검증.

### 7. 대시보드 취소/환불 통계 누락

**파일**: `app/api/admin/dashboard/route.ts`

"취소/환불" 카드가 `status='취소'`만 세고 '환불' 상태 주문은 카운트에서 빠지던 문제. 오늘 매출 계산도 '환불' 주문을 제외하지 않아 매출이 과대 표시되던 문제. 카운트 쿼리를 `status in ('취소','환불')`로 확장하고, 매출 계산 필터에도 '환불'을 함께 제외하도록 수정.
