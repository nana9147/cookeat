## 세부 작업

### 1. 배송 상태 전환에서 취소/환불 직행 경로 제거

- **작업**: `app/api/seller/shipping/orders/db.ts`
- **문제**: `SHIPPING_STATUS_TRANSITIONS`가 결제완료→취소, 배송중/배송완료→환불을 직행 가능하도록 열어둔 상태라, `refund_requests` 승인 절차 없이 배송 상태만 임의로 취소/환불로 바꿀 수 있는 구멍이 있었음
- **수정 내용**: 해당 전환 경로 제거. 이제 취소·환불은 오직 승인 절차(`approveRefund`/`processRefund`)를 통해서만 가능

### 2. 환불진행중 단계 도입 및 취소·환불 처리 플로우 개편

- **작업**: `types/seller/order.ts`, `types/seller/status.ts`, `app/api/seller/orders/refunds/db.ts`, `app/api/seller/orders/refunds/[refundId]/route.ts`, `app/api/seller/orders/refunds/route.ts`, `app/seller/components/OrderList/RefundTable.tsx`, `app/seller/orders/cancel-refund/page.tsx`
- **내용**:
  - 취소요청(결제완료/배송준비 단계)은 승인 시 즉시 `취소`로 확정
  - 환불요청(배송중/배송완료 단계)은 승인 시 `환불진행중`으로 이동, 셀러가 반송 운송장 확인 후 "환불처리" 버튼으로 `환불` 최종 확정
  - 반송 택배사/운송장번호 입력 기능 추가 (선택 사항, 미입력 상태에서도 환불처리 가능)
  - 승인/처리 롤백 시 `order_items.shipping_status` 동기화가 누락되어 있던 버그 수정
  - 취소·환불 상태 카드 카운트가 `counts.x = 1`로 매번 덮어써지던 것을 `+= 1`로 수정 (실제 건수가 아닌 0/1로만 표시되던 버그)
  - 취소·환불 확정 시 포인트/쿠폰 자동 환급 로직 추가: 부분 취소·환불은 남은 결제금액이 사용 포인트보다 작아지는 시점부터 차액만 순차 환급, 주문에 속한 상품이 전부 취소·환불되면 쿠폰도 재사용 가능 상태로 복원

### 3. 주문 상세 조회 500 에러 수정 및 상태·환불금액 계산 정확도 개선

- **작업**: `app/api/seller/orders/[orderId]/db.ts`, `app/seller/components/OrderDetail/OrderProductSection.tsx`, `app/seller/orders/[orderId]/page.tsx`, `app/seller/components/OrderDetail/OrderPaymentSection.tsx`
- **문제**:
  - `shippings` 테이블 UNIQUE 제약이 `item_id` 기준으로 바뀐 뒤에도 `.maybeSingle()` 조회가 그대로 남아, 상품 여러 개인 주문에서 "Results contain N rows" 에러(PGRST116) 발생
  - `orders.coupon_id`가 `user_coupon_id`로 스키마 변경된 뒤에도 예전 관계(`coupons` 직접 조인)로 조회해 관계를 못 찾는 에러(PGRST200) 발생
  - 상품별 상태가 환불요청 등 일부 상태만 반영되고, 정상 배송상태(결제완료/배송중 등)는 표시 안 되던 문제
- **수정 내용**:
  - `shippings` 조회에 `.limit(1)` 추가
  - `orders` 조회를 `user_coupon_id → user_coupons → coupons` 관계로 수정
  - 상품별 상태를 승인 대기 중인 클레임은 `refund_requests`, 그 외에는 `order_items.shipping_status`를 우선하도록 통합
  - 주문 상세 화면 구조를 "간결한 요약 + 취소·환불 관리로 링크"로 정리 (상세 계산은 클레임 관리 화면 역할로 분리)

### 4. 결제 시 상품할인이 실제로 반영되지 않던 문제 수정

- **작업**: `lib/productPricing.ts`, `app/api/order/route.ts`, `create_order` RPC
- **문제**: `order/route.ts`가 상품 조회 시 `discount_type`/`discount_value` 컬럼 자체를 select하지 않아, 상품관리에서 할인을 설정해도 결제 시 항상 정가로 계산되던 버그 발견 (표시 문제가 아니라 실제 결제 금액 자체가 틀리던 문제)
- **수정 내용**:
  - `lib/productPricing.ts`에 `calcDiscountedPrice` 함수 추가 (상품 등록 폼의 할인 계산 로직과 동일 공식 재사용)
  - `order/route.ts`에서 할인 적용 단가(`unit_price`)와 정상가(`original_unit_price`)를 함께 계산해 `create_order`로 전달
  - `order_items`에 `original_unit_price` 컬럼 추가, `create_order` RPC가 주문 생성 시점에 상품별 정상가/할인가/배분된 포인트·쿠폰 할인을 확정 저장하도록 수정
  - 배송비/쿠폰 최소주문금액/최종결제금액 계산 기준을 상품할인 반영 후 금액으로 통일

- **내용**:

- **작업**: `app/api/seller/orders/refunds/db.ts` (`restoreOrderBenefitsForItem`)
- **내용**: "포인트 사용금액을 먼저 결제에서 소진시키고, 남은 결제금액이 사용 포인트보다 작아지는 시점부터 차액만 즉시 환급"하는 방식으로 정책 확정. `orders.refunded_point` 컬럼으로 지금까지 환급된 누계를 추적해 중복 환급 방지

## 특이사항

- `order_items.allocated_point`/`allocated_coupon_discount`는 주문 생성 시점에 1회 확정되는 값으로, 화면 표시(결제금액 계산)와 실제 포인트 환급 로직(`restoreOrderBenefitsForItem`) 양쪽에서 동일하게 참조하도록 통일함 — 각자 다른 방식으로 재계산하던 것을 한 곳(주문 생성 시점)으로 합침
- `create_order` RPC가 DB에 이름이 같은 오버로드로 3개 남아있음을 발견. 실제로 호출되는 건 `p_used_point`/`p_user_coupon_id` 파라미터가 있는 버전이고, 나머지 2개는 죽은 코드로 추정
- `orders.refunded_point` 컬럼: 순차 소진 방식 확정
