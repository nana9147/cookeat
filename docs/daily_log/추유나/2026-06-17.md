### 1. StatusBadge 타입 통합 및 승인 상태 추가

  - 영문 status 값을 한글로 표시하는 `labels` 맵 추가

### 2. 판매자 정보 관리 페이지 구현

- **작업**: `app/seller/info/page.tsx` 및 하위 섹션 컴포넌트 구현
- **내용**:
  - `SellerBasicInfo`, `SellerBusinessInfo`, `SellerBankInfo` 3개 섹션 컴포넌트 구현 (조회/수정 모드 전환)
  - `page.tsx`에서 `isEditing`, `localData` 상태를 중앙 관리하여 세 섹션 동시 편집 가능하도록 구성

### 3. SellerApplication 타입 실제 DB 스키마와 일치시키기

- **작업**: `types/seller/seller.ts` 재정의
- **내용**:
  - 실제 `sellers` 테이블 기준으로 `email`/`phone` 제거, `business_address`/`representative_name`/`cs_phone`/`is_co_representative`/`commission_rate`/`user_id` 추가
  - 세 섹션 컴포넌트에 대표자명, CS연락처, 공동대표 여부, 수수료율 필드 반영

### 4. 판매자 정보 GET/PATCH API 라우트 구현

- **작업**: `app/api/seller/me/route.ts`, `db.ts` 신규 구현
- **내용**:
  - `requireSeller` 가드로 인증/권한 확인
  - GET — `getSellerByUserId`로 판매자 정보 조회
  - PATCH — `updateSeller`로 상점명/대표자명/CS연락처/사업장주소/계좌정보 수정

### 5. 판매자 정보 페이지 API 연동

- **작업**: `page.tsx` mock 데이터 제거 및 실제 API 연동
- **내용**:
  - `useEffect`(cancelled 패턴)로 GET 호출, `localData`/`savedData` 분리로 저장·취소 로직 정확하게 처리
  - `handleSave`에서 PATCH 호출 시 snake_case → camelCase 변환

### 6. 사업장 주소 빈 문자열 표시 처리

- **작업**: `business_address` NOT NULL 제약 대응
- **내용**:
  - ERD 문서상 NULL 허용으로 적혀 있었으나 실제 DB는 NOT NULL 제약이 걸려 있어 빈 값 저장 시 500 에러 발생하던 것을 확인
  - `db.ts`에서 빈 문자열로 저장하도록 수정, 조회 시 빈 문자열도 `'-'`로 표시되도록 수정

### 7. sonner 토스트 추가

- **작업**: `sonner` 설치 및 적용
- **내용**:
  - `app/layout.tsx`에 `Toaster` 전역 추가 (가운데 상단, 성공/실패 색상 구분)
  - 판매자 정보 저장 성공/실패 시 토스트 안내

### 8. 상품 관리 페이지 작업을 위한 사전 점검

- **작업**: `types/seller/product.ts`를 ERD 및 `docs/api.md` 명세와 대조
- **내용**:
  - `ProductStatus` 불일치 확인 (`판매대기`/`판매종료` → `숨김`으로 변경 필요)
  - `manufacturer` 필드 제거, 명세 기준 `brand`로 대체 결정
  - `linkedRecipeCount`는 별도 컬럼 캐싱 대신 `recipe_ingredients` 기준 COUNT 계산 방식으로 결정
  - `shippingInfo`/`returnPolicy`가 대응 DB 테이블 없이 mock 상태로만 존재함을 확인
  - 반품/교환 신청 시 귀책(판매자/구매자)에 따른 배송비·환불액 계산 로직이 ERD에 전혀 반영되어 있지 않음을 발견
