  - 최소/최대 주문수량 필드는 제한 로직이 없어 프론트/타입에서 제거
  - 상품 수정 페이지(`ProductEditPage`)를 서버 컴포넌트에서 클라이언트 컴포넌트로 전환 (인증이 `Authorization` 헤더 기반 localStorage 토큰 방식이라 서버 컴포넌트에서 접근 불가) — `useEffect` + `api.get`으로 실제 데이터 연동, mock 제거
  - 상품 목록 삭제 버튼에 `DELETE` API 연동
  - 배송 템플릿/반품정책 미선택 시 등록·수정 차단 검증 추가 (프론트 toast + 백엔드 양쪽)
  - **버그 수정**: 상품 목록 조회(`getSellerProducts`) 시 Supabase raw row를 그대로 펼쳐 써서 `productId`/`categoryId`/`createdAt`이 `undefined`로 전달되던 문제 발견 — 명시적 camelCase 매핑으로 수정 (수정/삭제 버튼이 동작하지 않던 원인)

### 2. 상품 상세 미리보기 모달 추가

- **작업**: `app/seller-preview/[productId]/page.tsx` 신규, `ProductTable.tsx` 수정
- **내용**:
  - 상품 목록의 상세보기(Eye) 버튼 클릭 시 소비자 상품 상세 페이지를 모달(Dialog + iframe)로 미리보기
  - 소비자 페이지(`app/(main)/shopping/[id]/page.tsx`)가 `app/(main)/layout.tsx`(Header/Footer), `app/seller/layout.tsx`(Header/Sidebar) 양쪽 레이아웃 영향을 받지 않도록, `app/seller-preview/[productId]/page.tsx`를 최상위 독립 경로로 구성
  - 소비자 페이지의 `page.tsx`를 직접 import하여 호출하는 방식으로 구현 — 컴포넌트 복사 없이 원본을 그대로 재사용해 추후 API 연동 등 변경 시 별도 동기화 불필요
  - shadcn `Dialog`의 기본 클래스에 `sm:max-w-sm`이 포함되어 있어 `tailwind-merge`가 커스텀 `max-w-6xl`을 덮어쓰지 못하던 문제 확인, `sm:max-w-6xl`을 명시해 해결

### 3. 배송 템플릿 CRUD API 연동

- **작업**: `app/api/seller/shipping/templates/` 하위 라우트 신규, `TemplateList.tsx`/`ShippingTemplateTable.tsx`/`ShippingTemplateForm.tsx` 수정
- **내용**:
  - 기존 `shipping-templates` 경로를 `shipping/templates`로 이동 (반품정책과 도메인 구조 통일, 추후 배송 처리 기능도 `shipping` 하위로 편입 예정)
  - `GET/POST /api/seller/shipping/templates`, `PATCH/DELETE /api/seller/shipping/templates/[templateId]` 구현
  - "기본 템플릿 설정"을 일반 필드 수정과 분리해 `PATCH /api/seller/shipping/templates/[templateId]/default` 전용 엔드포인트로 구현 (UI 흐름이 폼 수정과 라디오 선택+저장으로 원래 분리되어 있던 것에 맞춤)
  - 첫 템플릿 등록 시 입력값과 무관하게 자동으로 기본 템플릿 지정
  - 기본 템플릿 삭제 시 남은 템플릿 중 최신순으로 자동 승격, 응답의 `newDefaultTemplateId`로 프론트에서 별도 안내 toast 표시
  - `TemplateList.tsx`의 Mock 데이터를 실제 API 연동으로 전환
  - `ShippingTemplateForm`이 모달 재오픈 시 이전 입력값이 남아있던 버그 수정 (`useEffect` 의존성에 `isOpen` 추가)

### 4. 반품정책 템플릿 CRUD API 연동

- **작업**: `app/api/seller/return-policy/templates/` 하위 라우트 신규, `TemplateList.tsx` 수정, `types/seller/shipping.ts` 반품 불가 사유 교체
- **내용**:
  - 배송 템플릿과 동일한 패턴으로 `return-policy-templates` → `return-policy/templates` 경로 이동, GET/POST/PATCH/DELETE + 기본 설정 전용 엔드포인트 구현
  - API 응답은 배송 템플릿과 동일하게 flat 구조 유지, 프론트 컴포넌트(`ReturnPolicy.content`)가 기대하는 nested 구조로의 변환은 `mapReturnPolicyResponse` 헬퍼로 화면 레이어에서만 처리
  - 반품 불가 사유(`NonReturnReason`)가 가전제품 기준 문구(`개봉/사용/설치 완료`, `디지털 콘텐츠` 등)로 되어 있던 것을 식재료 플랫폼에 맞게 전면 교체 (신선식품 단순변심 반품불가, 포장 개봉/사용 후, 소비기한 경과, 보관방법 미준수로 인한 손상/변질, 주문제작(정육손질/소분 등))
  - 배송/반품정책 템플릿 모두 Mock 데이터 완전 제거, 초기값을 빈 배열로 변경
