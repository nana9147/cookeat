  - `approveRefund`에 `faultType` 파라미터 추가, **승인 시점**에 잔여 상품금액(셀러 단위)이 무료배송 기준 미달이면 배송비를 재계산해 `shippings.shipping_fee`에 반영 (최초에는 최종처리(`processRefund`) 시점에 재계산하도록 했다가, 판매자가 승인 전에 배송비 발생 여부를 미리 알 수 있어야 한다는 판단으로 승인 시점으로 이동)
  - 구매자귀책이면 환급액에서 배송비 차감, 판매자귀책이면 구매자에게 청구하지 않음
  - 최초엔 "판매자귀책 시 정산금액에서 차감"으로 안내했으나, 기존에 확정된 정산 설계 원칙("배송비는 판매자가 직접 귀속받는 구조, 플랫폼 정산 계산에 미포함")과 모순됨을 확인 → 정산 연동 없이 "구매자에게 청구하지 않았다"는 안내로만 수정
  - 환불 상세 페이지에 상품 개별 카드 대신 페이지 상단 전체 요약 카드(정상가/할인/배송비차감/포인트/최종환불액) 신설, 승인 다이얼로그에 `RadioGroup` 기반 귀책구분 선택 UI 추가

  ### 5. DB 스키마 전수 점검 및 shippings 배송비 갱신 범위 버그 수정

- **작업**: `app/api/seller/orders/refunds/db.ts`
- **내용**:
  - 전체 스키마를 다시 점검하던 중 `shippings`에 `item_id`(nullable, unique) 컬럼이 있는 것을 확인 — `recalcShippingFeeForOrder`가 "주문+셀러 = shippings row 1개"로 잘못 가정하고 짜여 있던 것을 발견
  - `TEST-MANY-ITEMS-001`(실제 create_order RPC로 생성된 11개 상품 주문)으로 검증한 결과, `shippings`는 **상품(order_item) 단위로 row가 생성**되며, 같은 셀러의 같은 주문에 속한 상품들은 운송장 등록 시 전부 동일한 `carrier`/`tracking_number`를 공유하는 구조임을 확인 — 기존에 정한 "운송장 등록은 주문 단위, 배송완료는 상품 단위" 원칙과 일치
  - 기존 `recalcShippingFeeForOrder`가 `.limit(1)`로 여러 row 중 대표 하나만 갱신하고 있어, 상품이 2개 이상인 주문에서 배송비 재계산 시 일부 row만 반영되고 나머지는 0원으로 방치되는 버그 발견
  - 조회는 대표 1건으로 유지하되, 갱신은 `order_id`+`seller_id` 조건의 전체 row에 적용하도록 수정
  - 실제 데이터에 이 버그로 인한 불일치(같은 order_id+seller_id인데 shipping_fee 값이 서로 다른 경우)가 있었는지 전수 조회했으나 0건으로, 아직 실사용 흔적은 없었음을 확인

## 특이사항

- **RPC 오버로드 관리의 위험성 재확인**: 같은 이름의 함수가 여러 개 존재하면 Postgres가 파라미터 이름으로 암묵적으로 하나를 골라 실행하기 때문에, 스키마 변경 시 예전 버전을 반드시 삭제해야 함. 이번엔 다행히 죽은 함수들이 이미 삭제된 컬럼(`orders.coupon_id`)을 참조하고 있어 호출 시 즉시 에러가 나는 구조였지만, 만약 그렇지 않았다면 어떤 버전이 실행될지 예측 불가능한 상태로 오래 방치될 뻔함
- **`item_total_price` 컬럼 미스매치**: RPC 코드상 컬럼명과 실제 테이블 스키마가 어긋난 채로 배포되어 있었음 — 이번 세션 전까지 실제 체크아웃 결제가 항상 실패했을 가능성이 있어, 프로덕션 영향 여부 확인 필요
- **배송비-정산 연동 여부 재확인**: 배송비를 정산 로직에 끼워 넣으려던 최초 설계가 6/29에 이미 확정된 "배송비는 판매자 직접 귀속, 정산 계산 미포함" 원칙과 충돌했던 사례 — 새 기능 설계 시 기존에 확정된 원칙을 먼저 확인하는 절차 필요
