## 세부 작업

### 1. GitHub 레포 상태 점검

- `feat/seller-order` 브랜치에 전날 커밋 4개(구매확정 카드, 환불 상세페이지 분리, 귀책구분·배송비, shippings 버그 수정)가 정상적으로 push되어 있는 것을 확인

- **`supabase/migrations/` 폴더가 레포에 아예 없다는 것을 확인** — 이 프로젝트는 DB 스키마 변경을 git으로 트래킹하지 않고 Supabase SQL 에디터에서 직접 실행하는 방식으로 운영되고 있음. 이전에 마이그레이션 파일 커밋을 제안했던 건 프로젝트 컨벤션과 안 맞는 제안이었음을 확인하고 철회

### 2. 옵션·SKU 관리 — 범위 검토 후 보류

- 소비자 도메인(엄인호) 쪽 처리 현황을 직접 코드로 확인
  - `components/product/ProductPurchasePanel.tsx`에 옵션 선택 UI는 있으나, 실제 호출부(`app/(main)/shopping/[id]/page.tsx`)에서 `options = [{ label: '${product.name} (기본)', price: product.price }]`로 **가짜 옵션 1개만 항상 생성**하고 있음을 확인
  - `store/cartStore.ts`도 `productId`+`quantity`만 있고 옵션 구분 필드 자체가 없음
  - DB에도 옵션/SKU 관련 테이블 전혀 없음
- 결론: 셀러 어드민에서 옵션 등록 기능만 먼저 만들어도, 소비자가 실제로 옵션을 선택해 구매할 경로가 없어 당장 가치가 없음. DB·장바구니·결제·상품 상세까지 전 영역이 얽혀 있어 **금번 사이클에서는 제외**, 추후 도메인 담당자 전체와 범위 조율 후 진행하기로 함

### 3. 판매자 상품문의 응대 페이지 — 검토 후 보류

- 포털 간 점검 리포트에서 지적된 "상품문의 응대가 판매자 도메인에 없음" 이슈 대응 시도
- `inquiries` 테이블에 `product_id`(FK, ON DELETE SET NULL) 컬럼 추가 실행 — 실행 중 이미 컬럼이 존재하는 것을 발견(원인 불명, 이전 작업 흔적으로 추정)
- `order_id` 컬럼 추가 시도 중, 실제로는 사용자/관리자 양쪽 문의 기능이 이미 완성되어 운영 중인 상태에서 **카테고리 재설계·소비자 UI 진입점 추가·관리자 권한 범위 조정**까지 다 걸리는 작업임을 재확인
- 판단: 세 도메인(판매자/구매자/관리자)이 다 얽힌 작업이라 혼자 판단해서 진행하기엔 리스크가 큼 → **`product_id`, `order_id` 컬럼 전부 롤백(`DROP COLUMN`)하여 원상복구**
- 논의 요청 메시지를 정리해 팀에 전달 (문의-상품/주문 연결 방식, 카테고리별 응대 주체 구분, 멀티셀러 주문 시 배송문의 라우팅 3가지 안건)

- **내용**:

  - 상품별 매출 TOP 10, 카테고리별 매출 비중(대카테고리 기준, 파이차트)
  - 기간별 매출 추이(에리어차트) — 기존 대시보드 차트 스타일(`ChartContainer`, CSS 변수 색상) 그대로 재사용
  - 취소·환불율을 귀책구분(구매자귀책/판매자귀책)별로 나눠 집계 — 지난 사이클에 추가한 `refund_requests.fault_type` 활용
  - 신규·재구매 고객 비율 — 셀러 전체 주문 이력에서 유저별 최초 구매일을 구해 조회 기간 시작일과 비교하는 방식으로 판정
  - 오픈마켓 판매자센터 통계 메뉴 사례를 참고해 스코프 선정(조회수/퍼널/유입경로 등은 트래킹 인프라 부재로 제외)
- **작업 중 발견한 실수**: `addDays`/`startOfDay`가 공용 유틸(`lib/utils.ts`)에 있다고 착각하고 import 코드를 작성함 — 실제로는 `app/api/seller/dashboard/db.ts` 파일 내부에 로컬로만 정의되어 있었음(프로젝트가 이런 날짜 헬퍼를 공용화하지 않고 파일별로 두는 컨벤션). `toDateStr`만 `lib/dateRange.ts`에 실존. 사용자가 실제 `lib/utils.ts` 내용을 공유해줘서 발견, 즉시 로컬 헬퍼로 교체
- **확인된 지표 특이사항**: 신규·재구매 고객 비율이 조회 기간을 90일로 늘리면 전원 "신규"로 집계됨 — 버그가 아니라 서비스 자체의 데이터 역사(6월 초 시작, 약 32일)가 90일보다 짧아서 발생하는 자연스러운 현상. UI 안내 문구 추가는 보류하고 그대로 두기로 결정
