### 6. 판매자 주문 상태가 배송관리와 어긋나던 문제 수정 (`fix/seller-order-status-sync`)

- **증상**: 배송관리에선 상품 상태가 `배송준비`인데 주문관리에선 해당 주문이 `결제완료`로 남아있어 화면 간 상태가 안 맞음
- **원인 1 (동기화 누락)**: 상품 상태가 바뀔 때마다 `syncOrderStatus()`가 그 주문의 모든 상품 상태를 재계산해 `orders.status`(주문 전체 공유 컬럼)에 반영하는 구조인데, 이 동기화가 부가 기록인 이력 로그(`logOrderItemStatusHistory`) 호출 *다음에* 실행되도록 되어 있었음. 이력 로그 insert가 실패하면 상품 자체 상태 변경은 이미 성공했음에도 동기화 재계산은 실행되지 못한 채 함수가 끝나버림 — 특히 여러 상품을 한 번에 "발주확인" 처리할 때 마지막 상품에서 이 문제가 나면 증상과 정확히 일치
  - `updateShippingStatus`/`updateShippingTracking`/`bulkUpdateShippingTracking`의 이력 로그 호출을 `try/catch`로 분리해 로그 실패가 동기화를 막지 못하도록 수정
  - `bulkUpdateShippingStatus`에 안전장치 추가 — 배치 처리 후 이번에 건드린 주문들을 한 번 더 `syncOrderStatus`로 재확인
  - 겸사겸사 프론트엔드 `handleConfirmOrder`(발주확인)의 관련 버그도 수정 — 일부 상품이 처리 실패해도 무조건 주문 전체를 화면 목록에서 지워버려 실패를 숨기던 것을, 실제 성공한 상품만 지우도록 수정
- **원인 2 (설계 문제, 멀티셀러 주문)**: 위 수정 후에도 실사용 중 재현 — 확인해보니 문제의 주문이 멀티셀러 주문이었고, 다른 판매자의 상품은 아직 `결제완료` 상태라 `orders.status`(주문 전체 공유 상태)가 `결제완료`로 남는 것 자체는 정상 계산 결과였음. 다만 판매자 화면(주문관리)은 애초에 "내 상품 상태"를 보여줘야 하는데 공유 컬럼(`orders.status`)을 그대로 표시하고 있던 게 진짜 문제
  - `get_seller_orders` RPC: 표시/필터 값을 `o.status`(주문 전체 공유 상태) → `oi.shipping_status`(내 상품 자체 상태)로 변경. 다만 `oi.shipping_status`가 `order_status` enum이 아니라 `character varying`이라 `$2::order_status` 캐스팅이 타입 불일치 에러를 내서, 캐스팅 없이 비교하도록 수정
  - `app/api/seller/orders/db.ts`의 `getSellerOrderCounts`(상태카드 건수)와 `app/api/seller/orders/export/db.ts`(엑셀 다운로드)도 동일하게 주문 단위 집계 → 상품(아이템) 단위 집계로 전환
  - 부수 효과: export에서 기존엔 목록에 없던 상태값 목록 때문에 `구매확정` 상태 주문이 빠져있었는데, 이번 정리로 포함되도록 바뀜(확인 후 의도된 동작으로 확정)

### 7. 운송장 양식 다운로드가 멀티셀러 주문의 내 상품을 누락하던 문제 수정 (`fix/tracking-export-multiseller`)
