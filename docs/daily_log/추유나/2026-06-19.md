## 세부 작업

### 1. 상품 등록 폼 카테고리 2단 선택 구현

- **작업**: `types/seller/product.ts`, `BasicInfoField.tsx` 리팩터링
- **내용**:
  - 기존엔 부모 카테고리(CategoryName 하드코딩 배열)만 단일 선택하던 구조였는데, 실제 `products.category_id`는 자식 카테고리를 가리키는 FK라서 구조가 맞지 않음을 확인
  - `BasicInfoData`에 `parentCategoryId`/`categoryId`를 분리하고, `categories` 테이블의 부모-자식(`parent_id`) 구조에 맞춰 부모 선택 → 자식 선택 2단 Select로 변경
  - 부모 변경 시 자식 선택값 초기화 처리, 자식 Select는 부모 미선택 시 disabled
  - `manufacturer` 필드를 실제 DB 컬럼명인 `brand`로 통일
  - 등록 단계 판매 상태 옵션은 `판매중`/`판매종료`만 노출하도록 `BasicInfoField` 내 `STATUS_OPTIONS`로 제한 (DB enum 자체는 기존 4종 유지)

### 2. 배송/반품정책 템플릿 목록 조회 API 구현

- **작업**: `app/api/seller/shipping-templates/route.ts`, `app/api/seller/return-policy-templates/route.ts` 신규 구현
- **내용**:
  - `requireSellerContext` 가드 적용
  - `shipping_templates`/`return_policy_templates` 테이블에서 `seller_id` 기준 조회, `is_default` 내림차순 → `created_at` 내림차순 정렬
  - snake_case(DB 컬럼) → camelCase(API 응답) 변환
  - 기존엔 두 템플릿을 조회하는 API가 전혀 없었고, 배송관리 페이지(`TemplateList.tsx`)의 템플릿 목록도 Mock 데이터로만 동작 중이었음을 확인
  - 쿼리 로직이 짧아 `db.ts` 분리 없이 `route.ts` 단일 파일로 작성 (팀 컨벤션: 쿼리 길어지면 추후 분리)

### 3. 배송/반품정책 섹션을 템플릿 선택 UI로 변경

- **작업**: `types/seller/shipping.ts`, `ShippingSection.tsx`, `ReturnPolicyField.tsx`, `ProductForm/index.tsx` 리팩터링
- **내용**:
  - `products.shipping_template_id`/`return_policy_template_id`가 템플릿 테이블을 가리키는 FK 구조임에 맞춰, 배송비·반품정책을 직접 입력하던 자유입력 폼을 템플릿 선택 방식으로 전환
  - `ShippingTemplateOption`/`ReturnPolicyTemplateOption`, 새 `ShippingSectionProps`/`ReturnPolicyFieldProps`(`templates`/`value`/`onChange(templateId)`) 타입 정의
  - 모달 + 카드 클릭으로 템플릿 선택, 선택된 템플릿 상세는 `dl`/`dt`/`dd`로 읽기 전용 표시
  - `index.tsx`에서 템플릿 목록 fetch 및 props 연결
  - 템플릿이 0개인 경우 "배송 관리에서 먼저 만들어주세요" 안내 문구만 표시 (모달 내 신규 템플릿 등록은 보류 — `ShippingTemplateForm`이 로컬 state만 갱신하고 실제 POST API가 없어서, 배송관리 페이지 템플릿 CRUD가 API 연동될 때 함께 처리 예정)

### 4. 상품 등록 API 연동

- **작업**: `ProductForm/index.tsx`(handleSubmit), `app/api/seller/products/route.ts`(POST), `db.ts`(createSellerProduct) 신규/리팩터링
- **내용**:
  - `handleSubmit`에서 필수값 검증 후 `FormData` 구성(텍스트 필드 + 이미지 파일) → `POST /api/seller/products` 호출, 성공 시 `toast` + `router.push`로 목록 페이지 이동
  - 기존엔 해당 엔드포인트에 GET만 존재해 등록 기능 자체가 동작하지 않는 상태였음
  - Supabase Storage에 `product-images` 버킷(public) 신규 생성
  - 대표 이미지(이미지 목록 0번째) → `products.image`, 나머지 → `product_images` 테이블에 `sort_order`(1번부터)와 함께 저장
  - `products` row를 먼저 INSERT해 `product_id` 확보 후 이미지 업로드 → `image` 컬럼 UPDATE 순서로 처리 (Storage 경로에 productId 필요해서)
  - 이미지 업로드/등록 중 실패 시 생성된 `products` row를 삭제 처리 (고아 row 방지)
  - `category_id` 존재 여부를 이미지 업로드 전에 서버 측에서 먼저 검증, 없으면 400 응답으로 구분 처리 (그 외 에러는 500)
  - 실제 상품 등록 시도 → `products`/`product_images` 정상 저장 확인
