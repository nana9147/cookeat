## 세부 작업

### 1. 배송 처리 페이지 4단계 상태 모델 및 API 구현

- **작업**: `types/seller/shipping.ts`, `app/api/seller/shipping/orders/db.ts`, `route.ts` 신규
- **내용**:
  - 배송 상태를 `결제완료 → 배송준비 → 배송중 → 배송완료`(취소/환불 포함) 4단계로 재정의, `SHIPPING_STATUS_TRANSITIONS` 전환 규칙 맵으로 상태 전환 검증
  - `updateShippingStatus`(단순 상태 전환), `updateShippingTracking`(운송장 등록 + 배송중 자동 전환) 분리 구현
  - `PATCH /api/seller/shipping/orders/[orderId]`(운송장 등록), `PATCH /api/seller/shipping/orders/[orderId]/status`(상태 전환) 신규
  - `shippings.shipped_at`/`delivered_at`을 각 전환 시점에 자동 기록

### 2. 배송 처리 페이지 5개 탭 구조 구현

- **작업**: `PaymentInfoTable.tsx`, `TrackingTable.tsx`, `AllOrdersTable.tsx`, `ShippingPage.tsx` 신규/리팩터링
- **내용**:
  - `PaymentInfoTable`(결제완료 — 배송지/수령인 정보 + 발주확인), `TrackingTable`(배송준비/배송중/배송완료 — 운송장 입력 또는 읽기전용)로 역할 분리
  - `AllOrdersTable` 신규 — 전체 탭에서 압축 컬럼(주문번호/주문일/주문자/상품명/상태/택배사·운송장/관리)으로 조회, 행별 상태에 따라 발주확인/운송장입력/배송완료처리 액션 분기
  - `StatusCards`에 전체/결제완료/배송준비/배송중/배송완료 5개 카드 구성
  - "전체" 탭에서 상태 변경 시 행이 목록에서 사라지지 않고 그 자리에서 갱신되도록 `handleStatusChangeInAllView`/`handleUpdateInAllView` 별도 분리 (단일 상태 탭은 처리 후 목록에서 제거, 전체 탭은 상태만 갱신)

### 3. 기간별 필터 추가

  - 프리셋(전체/오늘/1주일/1개월/3개월/직접입력) 선택 시 계산된 날짜를 input에 표시, 직접 날짜를 수정하면 프리셋이 자동으로 '직접입력'으로 전환

## 특이사항

- 결제 도메인 담당자가 `payment_method`를 영문 코드(`PaymentType: 'card' | 'kakao' | 'toss' | 'bankbook' | 'mobile'`)로 표준화하기로 확정. 해당 타입/라벨이 결제 담당자의 mock 파일에 같이 정의되어 있어, 추후 그 파일이 삭제될 위험에 대비해 동일한 타입/라벨을 `types/seller/order.ts`에 독립적으로 복사해서 사용하기로 결정 (담당자에게 별도 공유)
- 배송 상태 enum 실제 DB 값(`결제전`, `주문확인` 포함 8종) 중 판매자 입장에서 불필요한 `결제전`/`주문확인`은 셀러 화면(`OrderStatus`)에서 제외하기로 결정 — 결제 담당자에게는 전체 enum이 필요하지만 셀러 도메인에서는 6종만 사용
- 엑셀 다운로드는 화면 표시용 컬럼이 아니라 회계/정산 참고용 원장 형태(`OrderExportRow`)로 설계, 선택 건수가 많을 경우(1000건 초과) 배치 단위로 나눠 순차 요청 후 하나의 파일로 합치는 방식으로 설계 합의
