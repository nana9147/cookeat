### 9. 멀티셀러 주문 운송장 충돌 문제 해결 (shippings 스키마 변경)

- **작업**: `shippings` 테이블 마이그레이션, `app/api/seller/shipping/orders/db.ts`, `docs/08-db-schema.md`
- **내용**:
  - **문제 발견**: 재료 주문은 장바구니에 여러 셀러 상품이 섞일 수 있는데, `shippings.order_id`가 `UNIQUE` 제약이라 한 주문에 운송장이 1개만 허용됨 — 멀티셀러 주문 시 셀러 간 운송장 입력이 충돌(덮어쓰기)하는 구조적 문제 확인
  - **사전 확인**: `order_items`에서 `order_id`별 `seller_id` 종류를 집계해, 기존에 멀티셀러 주문이 실제로 존재하는지 확인 (결과 없음 확인 후 진행)
  - **마이그레이션**: `shippings`에 `seller_id` 컬럼 추가 → 기존 데이터는 `order_items` 기준으로 채움 → `NOT NULL` 적용 → 기존 `UNIQUE(order_id)` 제약 제거 → `UNIQUE(order_id, seller_id)`로 변경
  - **검증**: 멀티셀러 테스트 주문(`TEST-MULTI-001`, 셀러 5+6 상품 혼합)에 양쪽 셀러가 각자 운송장을 입력해 충돌 없이 독립적으로 저장되는 것 확인
  - `getSellerShippingOrders`의 `shippings` 조회에 `seller_id` 필터 추가 — 다른 셀러의 운송장 정보가 섞여서 조회되던 문제 방지
  - DB 스키마 문서(`08-db-schema.md`)를 실제 Supabase 스키마 기준으로 전체 갱신 (`products.category_id` 정정, `orders.status` enum 8종 명시, `shippings.seller_id` 변경 내용, `addresses`/`faqs` 테이블 추가 등 반영)
