## 세부 작업

### 1. 운송장 엑셀 일괄등록 및 배송완료 일괄처리

- **작업**: `app/api/seller/shipping/orders/bulk-tracking/`, `app/api/seller/shipping/orders/export-tracking/`, `TrackingTable.tsx`
- **내용**:
  - 배송준비 탭에 운송장 엑셀 일괄등록 추가 — 양식 다운로드(주문번호/수령인/상품명/수량 + 빈 택배사·운송장번호 칸) 후 업로드하면 일괄 반영
  - 양식 다운로드는 현재 화면(페이지 단위)이 아닌, 필터 조건에 해당하는 전체 배송준비 건을 대상으로 함
  - 업로드 시 검증 2단계: ① 같은 엑셀 내 동일 운송장번호가 다른 주문에 중복 입력된 경우 차단(같은 주문 내 여러 상품에 동일 운송장번호는 허용) ② 이미 DB에 등록된 운송장번호와 충돌하는 경우 그 건만 실패 처리
  - 배송중 탭에 체크박스(상품 단위 개별 선택) + 일괄 배송완료 처리 버튼 추가, 기존 개별 처리 버튼도 유지

### 2. 배송 상태를 상품(order_item) 단위로 전면 재구조화

- **작업**: DB 스키마 변경, `app/api/seller/shipping/orders/db.ts`, `[itemId]/route.ts`, `[itemId]/status/route.ts`, `bulk-status/route.ts`
- **내용**:
  - 기존에는 운송장(`shippings`)이 `(order_id, seller_id)` 단위 1 row, 배송 상태도 `orders.status` 하나로 관리되어 한 주문 내 상품들이 항상 같은 배송 단계를 가져야 했음
  - `shippings`에 `item_id` 컬럼 추가(유니크 제약을 `order_id+seller_id`에서 `item_id`로 변경), 기존 데이터는 상품 개수만큼 row를 복제하여 마이그레이션
  - `order_items`에 `shipping_status` 컬럼 추가, 상품마다 독립적인 배송 진행 상태(결제완료/배송준비/배송중/배송완료)를 가질 수 있도록 변경
  - 운송장 등록, 배송완료 처리는 상품(item) 단위 액션으로 전환. 발주확인(결제완료→배송준비)은 운송장이 아직 없는 단계라 주문 단위 처리를 유지(한 주문의 상품들을 모아 일괄 처리)
  - API 라우트 `[orderId]` → `[itemId]`로 전환
  - 한 주문 내 모든 상품이 배송완료되어야 주문 전체가 배송완료로 표시되는 자동 동기화 로직을 검토했으나, 목록/상세가 이미 상품 단위로 노출되는 구조라 불필요하다고 판단하여 적용하지 않음

### 3. 주문관리/배송관리 카드·목록 단위 정리 (스마트스토어 방식)

- **작업**: `app/api/seller/orders/db.ts`, `app/api/seller/shipping/orders/db.ts`, `OrderTable.tsx`, `page.tsx` 각 2곳
- **내용**:
  - 상품 단위로 화면을 구성하면서 상태 카드 숫자(상품 개수)와 실제 "신규 주문이 몇 건인지"가 안 맞아 혼란이 발생하는 문제 발견
  - 스마트스토어 실제 사례를 참고해 원칙을 재정립: **주문 진행 상태 카드(전체/결제완료/배송준비/배송중/배송완료)는 순수하게 주문(orders.status) 단위로 집계**하고, **목록의 각 행은 그대로 상품 단위**로 유지. 클레임(취소/환불)은 더 이상 이 카드에 섞지 않고 완전히 별도 페이지(취소·환불 관리)에서만 다룸
  - 주문관리 목록에서 환불요청 진행 중인 상품이 있어도 상태값 자체는 그 주문의 배송 단계를 그대로 보여주되, 보조적으로 "클레임" 뱃지를 추가해 한눈에 확인 가능하도록 처리
  - `StatusCards` 컴포넌트의 `grid-cols-${cols}` 동적 클래스가 Tailwind 빌드 시 인식되지 않아 레이아웃이 깨지는 버그 발견 및 정적 매핑 객체로 수정

## 특이사항

- **카드 단위와 목록 단위를 다르게 가져가는 이유**: 처음에는 카드도 상품 단위로 맞추려 했으나, "신규주문 몇 건"이라는 직관적 정보가 사라지는 문제가 있어 스마트스토어 사례를 다시 참고함. 결과적으로 "주문 단위 카드 + 상품 단위 목록 + 클레임 완전 분리"라는 원칙으로 정리
- **운송장/배송완료를 상품 단위로 바꾸면서, 부분 처리 시 체크박스 동작 방식**도 같이 정리함: 운송장 등록처럼 한 주문이 항상 같이 발송되는 액션은 주문 단위로 묶고, 배송완료처럼 상품마다 독립적으로 도착할 수 있는 액션은 상품 단위로 분리
- **"구매확정 전" 취소/환불과 "구매확정 후" 환불의 차이를 명확히 정리**: 구매확정 후에는 환불이 불가능한 정책이므로, 정산 금액에서 별도로 차감하는 로직은 불필요. 다만 구매확정 전(배송준비/배송중 단계)에 취소·환불된 건은 셀러가 참고할 수 있도록 정산 상세에 별도 섹션으로 노출
- **부가세/물류비를 정산 계산에서 제외한 이유**: 배송비는 판매자가 직접 귀속받는 구조이고, 부가세는 신고/납부를 판매자가 별도로 처리하는 영역이라 플랫폼이 정산 시점에 선차감하지 않는 것이 스마트스토어 등 오픈마켓의 일반적인 방식임을 확인하고 동일하게 적용
- **PostgREST `.or()` 필터의 한계**: date 타입 컬럼에 `ilike`나 `::text` 캐스팅을 시도했으나 모두 파싱 에러가 발생, 결국 DB 쿼리 단계의 텍스트 검색을 포기하고 애플리케이션 레벨 필터링으로 전환
