## 세부 작업

### 1. 주문 관리 상품 단위 전환

- **작업**: `types/seller/order.ts`, `app/api/seller/orders/db.ts`, `OrderTable.tsx`, `page.tsx`
- **내용**:
  - 목록/카운트/export를 주문 단위가 아닌 상품(order_item) 단위 행으로 재설계 — 한 주문에 상품이 여러 개면 그만큼의 행으로 분리되어 표시
  - 정렬은 주문일시/주문번호 기준으로, 같은 주문의 상품 행들이 흩어지지 않도록 그룹 정렬 유지(금액 정렬은 상품 단위에서 의미가 약해 제거)
  - 부분 취소/환불이 발생해도 나머지 정상 상품은 원래 배송 상태로, 해당 상품만 취소/환불 상태로 정확히 카운트되도록 로직 수정
  - 날짜를 일자만이 아닌 시:분까지 표시(`formatDateTime` 공용 함수 추가)
  - 목록 페이지의 페이지/필터/정렬 상태를 URL 쿼리스트링과 동기화 — 상세 페이지에서 뒤로가기 시 이전 위치/필터로 정확히 복원
  - `OrderRow` 타입명이 다른 도메인(소비자 페이지)의 동일 이름 타입과 충돌해 `SellerOrderRow`로 변경
  - 상태카드 숫자에 천단위 구분자(,) 적용
  - `환불요청`, `주문확인` 등 일부 상태가 카운트 집계에서 누락되던 버그 수정

### 3. 취소·환불 관리 신규 구현

- **작업**: `refund_requests` 테이블, `app/api/seller/orders/refunds/**`, `RefundTable.tsx`, `cancel-refund/page.tsx`, 사이드바
- **내용**:
  - 부분 취소/환불을 지원하기 위해 환불 관련 컬럼을 `order_items`에서 분리해 별도 테이블(`refund_requests`)로 구성, RLS 활성화
  - `order_status` enum에 `환불요청` 추가, 이후 배송 전 단계의 취소 요청을 구분하기 위해 `취소요청`도 추가
  - 셀러가 승인하면 환불요청→환불, 취소요청→취소로 각각 다른 최종 상태로 전환되도록 처리
  - 거부 시 원본 상태(주문의 배송 상태)는 건드리지 않고, 거부 사유만 별도 기록 — 화면에는 `취소반려`/`환불반려`로 표시
  - 상태 카드(전체/취소요청/환불요청/처리완료) 및 탭 필터, 검색(주문번호/상품명/구매자명), 기간 필터 추가
  - 사이드바에 "취소·환불 관리" 서브메뉴 추가
  - 한 상품에 여러 번 요청 이력이 있어도 최신 건만 집계하도록 처리
  - 페이지네이션을 주문 단위에서 상품(claim) 단위로 수정 — 카드 합계와 실제 목록 건수가 일치하도록 보정
  - 배송상태 컬럼 추가(그 주문의 현재 배송 진행 상태 표시), 관리 컬럼에 처리 대기 시 승인/거부 버튼, 처리완료 시 결과 뱃지를 통합 표시
  - 엑셀 다운로드 기능 추가(주문 관리와 동일한 공용 훅 재사용)

## 특이사항

- **부분 환불 시 주문 상태 처리 원칙 재확인**: 주문 전체 상태(orders.status)는 부분 취소/환불이 발생해도 변경하지 않고, 해당 상품의 환불 진행 상태만 별도로 추적. 단, 주문에 속한 모든 상품이 환불 완료되면 주문 상태도 자동으로 환불로 전환되도록 처리
- **신선식품 vs 가공식품 반품 정책 차이 논의**: 신선식품은 환불요청→환불완료로 즉시 처리하지만, 가공식품(소스 등)은 실물 회수/검수가 가능해 중간 단계(반품중→반품확인중→환불완료)가 필요할 수 있음. 상품의 반품정책 템플릿에 따라 처리 흐름을 분기하는 방안은 추후 작업으로 보류
