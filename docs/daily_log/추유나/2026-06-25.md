## 세부 작업

### 1. 주문 관리 API 구현

- **작업**: `app/api/seller/orders/**` 신규
- **내용**:
  - `getSellerOrders`: 주문번호/주문자명/연락처/수령인명 통합 검색(닉네임 매칭은 `users` 서브쿼리로 선행 조회 후 결합), 정렬(화이트리스트 검증으로 SQL 인젝션 방지), 기간필터, 페이지네이션
  - `getSellerOrderCounts`: 상태별 건수 (결제완료/배송준비/배송중/배송완료/취소/환불 개별 집계, 화면에서는 취소+환불 합산해 표시)
  - `getSellerOrderDetail`: 멀티셀러 주문 대응 — 셀러 몫 상품만 필터링해서 노출, 쿠폰/포인트는 이 셀러 상품금액이 전체 주문에서 차지하는 비율로 분배 계산
  - 배송비는 `shippings.shipping_fee`에서 직접 조회 (셀러별 독립 배송비 — 결제 도메인이 주문 생성 시점에 미리 채워준다는 전제로 설계, 관련 스키마 변경 요청은 특이사항 참고)
  - 모든 조회에 `order_items.seller_id` 필터를 우선 적용해 다른 셀러 데이터 노출 차단
  - `getSellerOrdersForExport`: export 전용, offset/limit 배치 조회 지원

- **작업**: SQL
- **내용**:
  - 멀티 상품 + 할인 상품 혼합 주문 5건(`TEST-DISCOUNT-001~005`), 11종 상품 대량 주문 1건(`TEST-MANY-ITEMS-001`)
  - 상품 3종에 할인 적용(`고등어` rate 20%, `사과` amount 2000원, `새우` rate 15%)
  - 쿠폰(`WELCOME10`, rate 10%) 및 포인트 사용 케이스 검증용 데이터 추가
  - 신규 주문들에 대해 `shippings` row가 누락되어 배송비 0원으로 표시되던 문제 발견 및 보정

## 특이사항

- 멀티셀러 주문에서 배송비를 셀러별로 정확히 분리할 수 있는 컬럼이 기존에 없었음(`orders.shipping_fee`만 존재).
  `shippings` 테이블에 `shipping_fee` 컬럼을 추가하고, 한 셀러 내에서도 상품별로 배송 템플릿이 다를 수 있다는 점까지 포함해 결제 담당자에게 스키마/로직 변경 요청.
  정식 반영 전까지는 셀러 어드민 쪽 코드는 "결제 도메인이 주문 생성 시점에 `shippings` row를 미리 만들어준다"는 전제로 작성해두고, 검증용 데이터는 SQL로 직접 보정
- 주문 상품의 정상가/할인가를 같이 표시하는 기능은 `order_items`에 할인 전 원가를 저장하는 컬럼이 없어 보류.
  결제 담당자에게 `original_unit_price` 컬럼 추가 및 주문 생성 시점 가격 캡처 방식으로 변경 요청해둠
- 엑셀 export 설계 시 "1주문 1행 vs 1상품 1행" 고민 — 오픈마켓 표준(상품 단위 행 분리)을 따르기로 결정
