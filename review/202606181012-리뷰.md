# Cookeat 8차 코드 리뷰 (2026-06-18)

지난 7차(2026-06-17) 이후 `develop`에 들어온 변경을 봤습니다. 75커밋짜리 대형 라운드였던 7차에 비하면 이번은
그 위에서 **다듬고 연결하는** 회차네요. 판매자 정보 페이지를 실제 DB에 붙이고, 재료 쇼핑을 mock에서 실데이터로 바꾸고,
지난 회차에 짚었던 proxy 주석까지 손봤습니다. 리뷰는 앱을 먼저 띄워 사용자→판매자 흐름을 순서대로 돌려 보고(아래 "E2E 결과"),
거기서 드러난 동작을 근거로 코드를 봤습니다.

이번에 고친 것들이 먼저 눈에 들어옵니다. 다만 7차에 [필수]로 짚은 정산 상세 라우팅 문제는 아직 그대로라, 이번에도 1순위입니다.

proxy 주석을 정확히 고친 점이 좋습니다. "실제 게이트는 API의 `requireSeller`/`requireAdmin`이고 미들웨어는 낙관적 1차 거름망"이라는
역할 분담이 주석에 또렷이 드러나니, 다음 사람이 보안 경계를 오해할 일이 줄어듭니다.

## 2. 이번에도 1순위 — 정산 상세 라우팅 [필수]

E2E로 판매자 권한으로 들어가 `/seller/settlement/SET-001`과 `/seller/settlement/SET-999`를 차례로 열어 봤더니,
URL만 다를 뿐 화면 내용은 완전히 같았습니다(둘 다 6월 3주차·총 4,850,000원·ORD-2026-001234…). 원인은 7차와 동일합니다.

```tsx
// settlement/[id]/page.tsx — 'use client' 인데 useParams 로 id 를 읽지 않음
const MOCK_SETTLEMENT_DETAIL: SettlementDetail = {
  id: 'SET-001',   // 어떤 id로 들어와도 이 상수만 렌더
  ...
};
// ...
<SettlementBasicInfo detail={MOCK_SETTLEMENT_DETAIL} />
```

지금은 어떤 정산 건을 눌러도 항상 SET-001 화면이 나옵니다. 다른 정산을 눌렀는데 같은 내용이 뜨면 사용자 입장에선 신뢰가 깨지는 부분이죠.
판매자 정보 페이지(`/seller/info`)는 이번에 실DB로 잘 붙였으니, 그와 똑같이 `useParams()`로 id를 받아 해당 정산을 조회하면 됩니다.
구체적인 단계는 가이드에 적어 두었습니다.

## 3. 새 엔드포인트 리뷰 (이번 회차 핵심)

### 3-1. 판매자 정보 — `/seller/info` · `/api/seller/me` (추유나)

이번 회차에서 제일 깔끔하게 마무리된 부분이라 생각합니다. 판매자 권한으로 `/seller/info`에 들어가니 상점 정보·사업자 정보·정산 정보가
실제 DB 값으로 정확히 떴습니다(E2E C8a). API 쪽도 잘 짜였습니다.

- `GET`/`PATCH` 모두 `requireSeller(req)`로 먼저 권한을 확인하고, `instanceof NextResponse`로 가드 결과를 깔끔하게 분기합니다.
- `PATCH`에서 `storeName`·`representativeName`·`csPhone`·`bankName`·`bankAccount` 필수값을 검증하고, `businessAddress`는 선택값으로 `?? ''` 처리합니다.
- 응답을 `{success, data}` 공통 형태로 맞춘 것도 좋습니다.

작은 제안 하나만 드립니다.
- [제안] `PATCH`의 마지막에서 DB 에러를 `error.message`로 그대로 내려보냅니다. Supabase 원문 대신 우리 코드(예: `'update_failed'`)로 응답하고
  원문은 서버 로그에만 남기면, 내부 컬럼명 같은 정보가 클라이언트로 새는 걸 막을 수 있습니다. (다른 조에도 공통으로 드리는 이야기입니다.)

### 3-2. 재료 쇼핑 실DB 연동 (엄인호)

`/shopping` 목록과 `/shopping/8` 상세가 mock 없이 실데이터로 떴고, 없는 상품(`/shopping/99999999`)은 404로 정확히 떨어졌습니다(E2E C7a·C7b).
지난 회차에 남아 있던 mockProducts 잔재를 걷어내고 실DB로 옮긴 진전이네요. `FilterSheet`·`ShoppingClient`를 서브컴포넌트로 80줄 이하로 쪼갠 리팩터도 좋았습니다.

### 3-3. 토큰 갱신 동시성 버그 수정 (칭찬)

커밋 b38d432에서 "토큰 갱신 실패 시 대기 중인 요청의 Promise가 영원히 pending으로 남던" 버그를 고친 게 눈에 띕니다.
Axios 인터셉터에서 401 갱신을 큐로 모았다가 갱신이 실패하면 큐에 쌓인 요청들을 reject 해줘야 하는데, 이걸 놓치면 화면이 영원히 로딩에서 멈춰 버립니다.
이런 비동기 경합은 초보 단계에서 놓치기 쉬운 부분인데 직접 찾아서 고쳤네요. 칭찬하고 싶습니다.

## 4. 보안 — 두 가지

### 4-1. admin keyword 검색의 필터 인젝션 [제안] (이월)

`app/api/admin/products/route.ts:61`이 검색어를 PostgREST 필터 문자열에 직접 끼웁니다.

```ts
query = query.or(`name.ilike.%${keyword}%,seller_id.in.(${matchingSellerIds.join(',')})`);
```

`.or()`에 들어가는 문자열은 PostgREST가 필터 문법으로 해석합니다. 그래서 keyword에 쉼표나 괄호가 섞여 들어오면 필터가 깨지거나
의도치 않은 조건이 끼어들 여지가 있습니다(바로 위 27행의 `.ilike('store_name', \`%${keyword}%\`)`는 값으로 전달되니 안전한 형태입니다).
keyword에서 PostgREST 특수문자를 걸러내거나, 가능하면 값 바인딩 방식으로 바꿔 주면 좋겠습니다. 가이드에 예시를 넣어 뒀습니다.

### 4-2. RLS 정책 무한 재귀 [제안]

비로그인(anon) 키로 `users`·`sellers`·`orders`를 읽어 봤더니 데이터 대신 `42P17 infinite recursion detected in policy`가 돌아왔습니다.
데이터가 새는 건 아니니 당장의 보안 사고는 아닙니다. 오히려 막혀 있는 셈이죠. 다만 7차에는 같은 요청이 빈 배열(`[]`)로 깔끔히 막혔던 걸 보면,
그 사이 RLS 정책이 자기 테이블을 다시 참조하는 형태로 바뀌면서 재귀에 빠진 것으로 보입니다.

지금은 API가 전부 service role(`supabaseAdmin`)로 도는 덕에 이 문제가 가려져 있습니다. 그런데 RLS가 깨진 상태라,
나중에 클라이언트에서 publishable 키로 이 테이블을 직접 읽으려 하면 그대로 막혀 버립니다. 정책 정의를 한 번 점검해서
재귀를 끊어 두면 RLS가 의도대로 동작합니다. 보통 한 정책이 같은 테이블을 `select`로 다시 거는 데서 이런 재귀가 생깁니다.

## 5. 팀 공통 / 협업

- 역할 분담이 코드에 잘 드러납니다. 판매자 도메인은 추유나, 쇼핑·인프라는 엄인호, 7차의 admin API는 최유종까지 각자 영역이 또렷하네요.
  팀장 공백이 있었던 걸 생각하면 팀이 잘 굴러가고 있다고 느꼈습니다.
- `npm run build` 게이트(CI)가 있고 tsc도 0건이라 빌드 건강 상태는 좋습니다.
- "고친 것은 고치고, 못 고친 것은 다음으로" 하는 흐름이 회차마다 쌓이고 있습니다. 정산 상세 params처럼 [필수]로 두 회차 이어지는 항목은
  다음 PR에서 먼저 처리하는 식으로 우선순위를 매겨 두면 좋겠습니다.
