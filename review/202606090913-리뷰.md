## 2. `[필수]` 빌드가 타입 체크에서 실패합니다 (실측)

`npm run build` — 컴파일은 되지만 **타입 체크에서 멈춥니다.**

```
✓ Compiled successfully
Failed to type check.
Type error: Property '판매대기' is missing in type
'{ 판매중; 품절; 판매종료 }' but required in type 'Record<ProductStatus, string>'.
  → app/seller/components/StatusBadge.tsx:4
```

`ProductStatus`는 `'판매대기' | '판매중' | '품절' | '판매종료'` 4개인데, `StatusBadge`의 색상 맵엔 3개만 있습니다. 게다가 mock 데이터(`types/seller/product.ts:118`)가 실제로 `status: '판매대기'`를 쓰고 있어, 그 상품의 뱃지는 색 클래스가 `undefined`로 나옵니다. `판매대기` 항목만 추가하면 빌드도 살아나고 화면도 맞습니다.

> tip: `Record<ProductStatus, string>`로 선언하면 이렇게 enum 값이 빠졌을 때 컴파일러가 바로 잡아줍니다. 좋은 타입 사용이고, 누락만 채우면 됩니다.

추가로 `tsc --noEmit`에서 잡힌 타입 에러 2건이 더 있습니다(빌드는 StatusBadge에서 먼저 멈춰 안 보이지만, 고치면 이어서 드러납니다):

- **`components/(auth)/AuthInitializer.tsx:20`** — 토큰 갱신 시 `setAuth`에 넘기는 user 객체에 `email`·`isSocial`이 빠져 있습니다(`AuthUser` 타입 불충족). 토큰이 갱신되면 스토어의 email/isSocial이 사라집니다. 이미 `authTypes.ts`에 완성된 `toAuthUser()`가 있으니 그걸 재사용하면 됩니다.
- **`services/auth/authTypes.ts:31`** — `user_metadata.custom_avatar_url`을 읽는데 타입엔 그 필드가 없습니다. 타입에 `custom_avatar_url?: string`을 추가하세요(안 그러면 항상 `undefined`로 빠집니다).

## 3. `[필수]` 권한(authorization) — 인증은 됐는데 "권한"은 안 봅니다

`AdminAuthGuard`(파일 내 컴포넌트명은 `RoleGuard`)를 보면:

```ts
const token = useAuthStore((s) => s.accessToken);
useEffect(() => { if (hydrated && !token) router.replace('/admin/login'); }, ...);
if (!token) return null;
```

- **이름은 RoleGuard인데 role(관리자 권한)을 확인하지 않습니다.** 토큰이 있기만 하면 통과해서, **일반 회원도 로그인만 하면 `/admin/*`에 들어갑니다.** "로그인했다(인증)"와 "관리자다(인가)"는 다른 문제예요.
- 또 이건 **클라이언트 전용 가드**라, 화면만 가릴 뿐 데이터를 막지 못합니다. 지금은 어드민이 mock이라 실피해는 없지만, 실데이터가 붙으면 **API 라우트에서 서버사이드로 role을 확인**해야 합니다.
- **셀러(`/seller/*`)는 가드가 아예 없습니다.** E2E에서 로그인 없이 `/seller/products/new`가 그대로 열렸습니다. admin엔 게이트를 붙였으니 seller에도 같은 보호가 필요합니다.

→ `users` 테이블(또는 메타데이터)에 `role`을 두고, 가드에서 `role === 'admin'`을 확인 + 어드민/셀러 API는 서버에서 role 재확인. (가이드에 예시)

## 4. 보안 — RLS 실측

anon(publishable) 키로 직접 호출해 봤습니다.

```
users:  anon=0행 / service=7행  → RLS가 anon을 막고 있음 ✓ (잘됨)
products / orders / reviews: anon=0 / service=0 → 빈 테이블이라 RLS 판정 보류
```

`users`가 anon으로 안 읽히는 건 좋습니다(1차의 service role 격리에 이어 일관된 보안 태도). products/orders 등은 아직 데이터가 없어(프론트는 mock 단계) RLS가 켜졌는지 확정은 못 했습니다 — **실데이터 붙이기 전에 모든 테이블에 RLS를 켜고 정책을 정의**해 두세요.

### 추유나 (nana9147) — 셀러 상품 등록 폼 리팩토링

**아주 잘했습니다.** 이번 라운드 하이라이트라 생각합니다.
- `ProductFormData`를 `basicInfo`/`pricingInfo`/`shippingInfo`/`productInfo`/`returnPolicy` 섹션으로 **중첩 구조화**하고, 각 섹션을 독립 컴포넌트(`BasicInfoField` 등)로 분리했습니다.
- 특히 `handleChange<S extends keyof ProductFormData, K extends keyof ProductFormData[S]>` — **제네릭으로 섹션·필드 타입을 안전하게** 잡은 부분은 초보 단계에서 보기 드문 좋은 코드입니다. 잘못된 필드명을 쓰면 컴파일러가 막아줍니다.
- E2E로 띄운 상품 등록 폼(`images/2nd-01-seller-product-form.png`)도 기본정보·이미지·리치텍스트 설명·가격(할인 토글)·배송·상품정보제공고시·반품정책까지 굉장히 완성도 높게 나왔습니다.

**`[제안]`** `ImageUploadField`와 `DescriptionEditor`는 `form` 상태와 연결돼 있지 않습니다(둘 다 props 없이 자체 동작). 그래서 `handleSubmit`의 `form`엔 이미지·설명이 안 담깁니다. 다른 필드처럼 `data`/`onChange`로 묶어 주면 제출 데이터가 완성됩니다.

**`[필수]`** 위 2번 StatusBadge `판매대기` 누락(셀러 영역). 작은 수정이지만 빌드를 막고 있어 우선순위가 높습니다.
