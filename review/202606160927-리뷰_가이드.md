## 1. [필수] 셀러 화면 가드 — admin 가드를 그대로 본뜨면 됩니다

이미 `app/admin/components/AdminAuthGuard.tsx`라는 모범 답안이 있습니다. role만 `seller`로 바꿔 같은 걸 하나 더 만들면 됩니다.

먼저 `app/seller/components/SellerAuthGuard.tsx`를 새로 만듭니다.

```tsx
'use client';

import { useEffect } from 'react';
import { useRouter } from 'next/navigation';
import { useAuthStore } from '@/store/authStore';

export default function SellerAuthGuard({ children }: { children: React.ReactNode }) {
  const router = useRouter();
  const hydrated = useAuthStore((s) => s._hydrated);
  const token = useAuthStore((s) => s.accessToken);
  const user = useAuthStore((s) => s.user);
  const isSeller = user?.role === 'seller'; // admin도 허용하려면: user?.role === 'seller' || user?.role === 'admin'

  useEffect(() => {
    if (!hydrated) return;          // 세션 복원 끝날 때까지 판단 보류 (깜빡임 방지)
    if (!token) {
      router.replace('/login');     // 비로그인 → 로그인으로
      return;
    }
    if (!isSeller) {
      alert('판매자만 접근할 수 있는 페이지입니다.');
      router.replace('/');
    }
  }, [hydrated, token, isSeller, router]);

  if (!hydrated) return null;       // 판단 전엔 아무것도 안 그림
  if (!token || !isSeller) return null;
  return <>{children}</>;
}
```

그리고 `app/seller/layout.tsx`를 이 가드로 감쌉니다.

```tsx
'use client';

import { SidebarProvider } from '@/components/ui/sidebar';
import SellerAuthGuard from './components/SellerAuthGuard';   // 추가
import Header from './components/Header/Header';
import SellerSidebar from './components/Sidebar/SellerSidebar';
import SellerMobileNav from './components/Sidebar/SellerMobileNav';

export default function SellerLayout({ children }: { children: React.ReactNode }) {
  return (
    <SellerAuthGuard>   {/* ← 이걸로 감쌉니다 */}
      <Header />
      <div className="flex flex-1">
        <SidebarProvider style={{ minHeight: '6.25rem' }}>
          <SellerSidebar />
          <div className="flex flex-1 flex-col">
            <SellerMobileNav />
            <main className="flex flex-1 flex-col">{children}</main>
          </div>
        </SidebarProvider>
      </div>
    </SellerAuthGuard>
  );
}
```

이렇게만 하면 admin과 똑같이 비로그인·권한 없는 사용자가 차단됩니다.

**왜 `_hydrated`를 먼저 보나요?** zustand persist는 새로고침 직후 잠깐 저장소에서 상태를 복원하는 시간이 있습니다. 이때 `token`이 아직 안 들어와서, 가드가 성급하게 "로그인 안 됨"으로 판단하면 로그인한 사용자가 매번 튕깁니다. `_hydrated`가 true가 될 때까지 판단을 미루는 게 그래서 중요합니다(admin 가드도 같은 이유로 그렇게 돼 있습니다).

**한 가지 짚을 점.** 이 가드는 클라이언트 화면 가드라 "화면이 안 보이게" 막습니다. 데이터를 실제로 바꾸는 건 결국 API 라우트라, 셀러 전용 API에는 `lib/serverAuth.ts`의 `requireSeller(req)`를 함께 걸어 주세요. 화면 가드(보이게/안 보이게)와 서버 가드(쓰게/못 쓰게)는 역할이 다릅니다. 결제 라우트에 `requireAuth`를 건 것과 같은 식입니다.

## 2. [제안] 주문 생성 시 서버측 합계 계산

지금 `/api/order`는 클라가 보낸 `finalAmount`를 그대로 주문 금액으로 저장합니다. confirm 단계가 그 주문과 대조하니 위조 자체는 막히지만, **주문이 만들어지는 그 순간의 금액 기준점이 아직 클라**에 있습니다. 클라가 처음부터 5만원짜리 주문을 만들면 confirm도 5만원으로 통과합니다.

다음 단계는 클라가 **품목 id와 수량만** 보내고, 서버가 가격을 DB에서 읽어 합치는 것입니다.

```ts
// app/api/order/route.ts (개념)
const { items, paymentMethod, recipient, phone, address } = await req.json();
// items: [{ productId, quantity }, ...]  ← 가격은 안 받음

// 1) 서버가 진짜 가격을 DB에서 조회
const ids = items.map((i) => i.productId);
const { data: products } = await supabaseAdmin
  .from('products')
  .select('id, price, stock')
  .in('id', ids);

// 2) 서버가 합계를 직접 계산 (클라 금액은 쓰지 않음)
let totalAmount = 0;
for (const item of items) {
  const p = products?.find((x) => x.id === item.productId);
  if (!p) return NextResponse.json({ error: '없는 상품' }, { status: 400 });
  if (p.stock < item.quantity) return NextResponse.json({ error: '재고 부족' }, { status: 400 });
  totalAmount += p.price * item.quantity;
}
const shippingFee = totalAmount >= 30000 ? 0 : 3000; // 규칙도 서버에서
const finalAmount = totalAmount + shippingFee - couponDiscount;

// 3) 서버가 계산한 finalAmount로 주문 생성
```

핵심은 "돈에 관한 숫자는 전부 서버가 계산한다"입니다. 가격·배송비·할인·합계 어느 하나라도 클라 값을 그대로 저장하면 거기가 구멍이 됩니다. 지금 구조가 이미 주문-확정으로 잘 나뉘어 있어서, 합계 계산만 서버로 옮기면 끝입니다.

## 3. [제안] 메인 본문 채우기

`app/(main)/page.tsx`가 지금 이렇습니다.

```tsx
export default function Home() {
  return <main className="flex-1" />;   // 빈 화면
}
```

레이아웃(Header/Footer)은 잘 잡혀 있으니, 이 안에 섹션을 채우면 됩니다. 이미 만들어 둔 자산을 재활용하면 빠릅니다.

```tsx
export default function Home() {
  return (
    <main className="flex-1">
      <HeroBanner />            {/* shopping의 ShoppingBanners를 참고 */}
      <PopularRecipes />        {/* 레시피 카드 그리드 */}
      <RecommendedProducts />   {/* shopping의 ShoppingGrid 일부 재사용 */}

}
```

재료 쇼핑 페이지에 이미 그리드·배너 컴포넌트가 있으니, 그걸 메인에서도 부분 재사용하면 적은 코드로 첫 화면을 채울 수 있습니다.

## 4. [제안] seller StatusBadge 공통으로 합치기

admin은 `@/components/common/StatusBadge`로 통일됐는데 seller만 자기 것(`app/seller/components/StatusBadge.tsx`)을 따로 씁니다. 합치면 배지가 완전히 한 벌이 됩니다.

다만 seller 것은 union 타입(`ProductStatus | OrderStatus | ...`)이라 오타를 컴파일 단계에서 잡아 줍니다. 공통 것은 `status: string`이라 그 안전망이 없고요. 그래서 단순히 seller import만 바꾸기보다, **공통 컴포넌트를 유니온도 받게 다듬는** 방향이 좋습니다.

```tsx
// components/common/StatusBadge.tsx
type KnownStatus = ProductStatus | OrderStatus | /* ... 어드민 상태들 */;

export default function StatusBadge({ status }: { status: KnownStatus | string }) {
  return (
    <span className={`px-3 py-1 rounded-full text-xs font-medium ${styles[status] ?? 'bg-gray-100 text-gray-600'}`}>
      {status}
    </span>
  );
}
```

이렇게 하면 seller도 공통을 쓰면서 타입 안전성을 유지합니다. 급한 일은 아니니, 셀러 가드(1번)부터 끝낸 다음에 정리해도 됩니다.
