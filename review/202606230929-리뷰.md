# Cookeat 코드 리뷰 — 2026-06-23

오늘은 API가 실제로 쓰는 데이터베이스 테이블(ERD)을 한 번 정리하고, 그 위에서 마켓 도메인의 **느려질 만한 쿼리**를 집중해서 봤습니다. 마켓은 주문·정산·목록처럼 데이터가 쌓이면 가장 먼저 느려지는 곳이라, 어디가 위험한지 미리 짚어두면 좋겠다 싶었습니다. 정확성과 보안도 같이 봤고요.

> 참고: 리포지토리에 SQL 마이그레이션 파일이 없다 보니(`*.sql` 0건), 스키마는 `docs/api/08-schema.md`와 실제 코드의 `.from('...')`를 보고 재구성했습니다. 그래서 "인덱스가 없다"는 어디까지나 문서·코드 기준이고, 실제로 걸려 있는 인덱스는 Supabase SQL Editor에서 `select * from pg_indexes where schemaname='public';`로 한 번 확인해 주세요.

## 1. API가 쓰는 테이블 정리 (ERD) — 문서와 코드 차이

실제 코드 기준으로 정리하다 보니 문서(`08-schema.md`)와 어긋나는 부분이 보였습니다. 문서를 코드에 맞춰 고쳐두면 다음 사람이 헷갈리지 않습니다.

- **`products`의 카테고리 컬럼이 문서와 코드에서 다릅니다.** 문서는 `products.ingredient_id → ingredients`(대카테고리)라고 돼 있는데, 실제 코드는 `products.category_id → categories`(소카테고리)를 씁니다(`app/api/products/route.ts:72,79,132-133`, `app/api/seller/products/db.ts:13,21,97`). 대카테고리명은 `categories.parent_id → ingredients.category` 2홉 조인으로 끌어오고요(`products/route.ts:124-135`). → 문서를 `category_id` 기준으로 수정해 주세요.
- **문서에 없는 컬럼이 INSERT됩니다.** `seller/products/db.ts:104-105`가 `discount_type`, `discount_value`를 넣는데 문서 products 정의에는 빠져 있습니다. 실제 테이블엔 있다는 뜻이니 문서에 추가해 주세요.

핵심 관계는 `products.seller_id → sellers`, `order_items.order_id → orders`·`product_id → products`, `settlements.seller_id → sellers`, `reviews.product_id → products`입니다. 주문은 `orders` ↔ `order_items`(1:N) 구조이고, 주문 시점 단가를 `order_items.unit_price`에 저장합니다(이건 잘 돼 있습니다 — 아래 4번).

## 2. 느려질 만한 쿼리 (마켓 집중)

### [필수] 상품 목록/검색이 한 요청에 풀스캔을 여러 번 합니다 — `app/api/products/route.ts:69-107`
여기에 느려지는 요소가 한꺼번에 모여 있습니다.

1. **ILIKE 선두 와일드카드 검색** `ilike('name', '%키워드%')`(`:78`). 앞에 `%`가 붙으면 B-tree 인덱스를 못 써서 상품 테이블을 통째로 스캔합니다. B-tree는 정렬된 값의 앞부분부터 찾아 내려가는데, 시작 글자를 모르면 어디서부터 뒤져야 할지 잡을 수가 없거든요.
2. **별점순 정렬 시 500개를 통째로 로딩**(`:89-91,194-197`). 별점은 DB 컬럼이 아니라 리뷰를 집계해야 나오는 값이라, 500개 상품과 그 리뷰를 전부 받아와 JS에서 정렬합니다. 1페이지(보통 20개)를 보여주려고 500개와 그 리뷰를 다 전송하는 셈입니다.
3. **`count: 'exact'`**(`:74`). 매 요청마다 조건에 맞는 전체 행을 한 번 더 셉니다. 위 ILIKE와 겹치면 사실상 풀스캔을 두 번 도는 꼴이 됩니다.
4. **OFFSET 페이지네이션**(`:106`). `range`는 OFFSET 방식이라 뒤 페이지로 갈수록 앞 페이지 행을 다 읽고 버립니다(100페이지면 앞 99페이지를 읽고 폐기).

지금은 상품이 적어 체감이 없지만, 인기 키워드 검색에 별점순, 깊은 페이지까지 겹치면 단일 요청 중에 가장 무거워지는 지점입니다. 해결 방법은 가이드에 정리해 뒀습니다(검색 인덱스, 정렬·페이지네이션을 DB로 내리기).

### [필수] 정산 통계를 테이블 전체를 받아와 JS에서 더합니다 — `app/api/admin/settlements/route.ts:48-66`
`from('settlements').select('fee')`로 **정산 전체 행의 fee를 다 가져와** JS `reduce`로 합산합니다(`:51,65`). `pendingAmount`/`completedAmount`도 상태별 전체를 받아 JS로 더하고요(`:62,64`). 이런 합계는 원래 `SELECT status, SUM(amount), SUM(fee) ... GROUP BY status` 한 방이면 DB가 인덱스만 읽고 끝낼 수 있는 일입니다. 지금처럼 모든 행을 네트워크로 빼서 앱에서 더하면 정산 건수가 늘어날수록 전송량·메모리·지연이 같이 불어납니다. RPC(SQL 집계)로 바꿔주는 게 맞습니다.

### [제안] 관리자 판매자 평점을 상품·리뷰 통째로 받아 JS로 계산 — `app/api/admin/sellers/route.ts:64-85`
현재 페이지의 판매자들에 대해 `products.select('seller_id, reviews(rating)')`로 **각 판매자의 모든 상품과 모든 리뷰 rating을 통째로** 가져온 뒤 JS 이중 루프로 평균을 냅니다. 판매자가 상품·리뷰를 많이 가질수록 전송량이 확 늘어납니다. 평점도 DB 집계 뷰(또는 RPC)로 미리 계산해 두는 편이 좋습니다.

### [제안] 관리자 상품/주문 목록도 같은 ILIKE + count + OFFSET 패턴 — `admin/products/route.ts:48-73`, `admin/orders/route.ts:20-39`
관리자 상품 검색은 키워드로 sellers를 ILIKE로 한 번 풀스캔하고(`:21-28`), 이어서 products를 `or(name ILIKE ... OR seller_id IN (...))`로 또 풀스캔합니다(`:61`). 관리자 주문은 `or('order_id.ilike.%kw%,recipient.ilike.%kw%')`(`:38`)인데 두 컬럼 모두 선두 와일드카드라 인덱스를 못 탑니다. 처방은 위 1번과 같습니다.

### [제안] 대시보드 카테고리 매출도 오늘 주문 전체를 받아 JS 합산 — `admin/dashboard/route.ts:74-114`
`count head:true`로 던지는 카드들(`:30-60`)은 가벼워서 좋습니다. 다만 카테고리 매출은 오늘 주문의 모든 아이템에 2홉 조인 결과까지 받아 JS로 더합니다. 주문이 많은 날엔 부담이 되니 GROUP BY로 내리면 좋습니다. 그리고 인기상품 매출을 `sales_count * price`(현재가)로 계산하는데(`:99-100`), 이건 과거 판매가와 무관해서 부정확합니다 — 매출은 `order_items.unit_price` 기준이 맞습니다.

## 3. 검색어 이스케이프 (보안 + 정확성) — `[제안]`

`admin/orders:38`, `admin/products:61,64`, `admin/sellers:41`, `products:78`이 사용자 keyword를 `%${keyword}%`로 그대로 끼워 넣습니다. Supabase 체이닝이라 SQL 인젝션 자체는 막히지만, 두 가지가 남습니다. 먼저 keyword에 `%`나 `_`가 들어가면 의도와 다른 와일드카드 매칭이 일어나 전체가 매칭될 수 있습니다(풀스캔에 결과 오염까지). 또 하나는 `.or('a.ilike.%kw%,b.ilike.%kw%')` 형태에서 keyword에 `,`나 `)`가 끼면 PostgREST의 or 필터 구문 자체가 깨질 수 있다는 점입니다. 최소한 `%`, `_`, `,`, `(`, `)`는 검증·이스케이프해 주세요.

## 4. ERD로 본 비즈니스 관점

### [잘함] 주문 시점 가격 스냅샷이 있습니다
`order_items.unit_price`에 주문 당시 단가를 저장하고, 주문 생성 때도 검증된 `product.price`를 unit_price로 기록합니다(`order/route.ts:104-108,179-186`). 그래서 나중에 상품 가격이 바뀌어도 과거 주문 금액이 틀어지지 않습니다. 마켓에서 자주 놓치는 부분인데 잘 잡아 두셨습니다.

### [필수] 주문 생성이 한 트랜잭션이 아닙니다 — `app/api/order/route.ts`
주문은 재고 차감(`:120-135`) → 품절 전환(`:141-147`) → orders insert(`:155-171`) → order_items insert(`:179-187`)가 **개별 쿼리들의 연쇄**로 돌아갑니다. 코드가 직접 보상(restoreStock, order delete)으로 롤백을 흉내 내고 있지만, 그 보상 자체가 실패할 수도 있고("rollback failed" 로그 `:195,197`) 중간에 프로세스가 죽으면 재고와 주문이 어긋나 버립니다. 재고 차감을 `update({stock: prev-qty}).eq('stock', prev)` CAS로 막아둔 건(`:116-135`) 좋은 시도인데, 여러 상품을 순차로 차감하다 중간에 실패하면 결국 보상에 기대게 됩니다. 정석은 plpgsql RPC 하나로 묶어 DB 트랜잭션 안에서 처리하는 것입니다(예시는 가이드에 있습니다).

### [제안] 정산 "생성" 로직이 안 보입니다 — settlements
`settlements`는 admin 조회와 단건 수정만 있고, **판매액에서 수수료·환불을 빼서 정산액을 만들어내는 생성 파이프라인이 코드에 없습니다.** `amount`/`fee`가 어떻게 채워지는지 불명확하고, `sellers.commission_rate`나 `return_requests.refund_amount`를 반영하는 곳도 없어 정합성을 검증할 방법이 없습니다. 정산은 돈이 걸린 부분이니, 어떤 입력으로 어떻게 산출되는지부터 한 번 정리해 보면 좋겠습니다.

### [제안] 쿠폰·포인트가 주문에 반영 안 됩니다
스키마엔 `orders.coupon_id`/`coupon_discount`/`used_point`와 `point_history`가 있는데, 정작 주문 생성은 `coupon_discount: 0` 하드코딩이고 포인트 적립·차감도 없습니다(`order/route.ts:162`). 계획만 잡아두고 아직 구현은 안 한 상태로 보입니다 — 우선순위는 낮지만 스키마와 코드가 어긋나는 항목이라 적어 둡니다.

## 6. 팀 공통

- **인덱스 점검 한 번.** 오늘 [제안]의 상당수가 인덱스가 있느냐 없느냐에 달려 있습니다. `products(status, category_id, seller_id, created_at)`, `orders(user_id, created_at, status)`, `order_items(order_id, product_id)`, `reviews(product_id)`, `settlements(status)`가 있는지 `pg_indexes`로 확인하고, 없으면 추가해 주세요.

- 문서(`08-schema.md`)와 코드의 컬럼 차이(category_id, discount 컬럼)를 맞춰 두면 다음 작업자가 헷갈리지 않습니다.
