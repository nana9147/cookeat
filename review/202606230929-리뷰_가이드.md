## 0. 인덱스 확인 (코드는 안 고치고도 효과가 큰 부분)

Supabase SQL Editor에서 아래를 돌려봅니다.

```sql
select tablename, indexname, indexdef
from pg_indexes
where schemaname='public'
  and tablename in ('products','orders','order_items','reviews','settlements')
order by tablename;
```

없으면 추가:

```sql
create index if not exists idx_products_status_created on products (status, created_at desc);
create index if not exists idx_products_category on products (category_id);
create index if not exists idx_orders_user_created on orders (user_id, created_at desc);
create index if not exists idx_order_items_order on order_items (order_id);
create index if not exists idx_reviews_product on reviews (product_id);
create index if not exists idx_settlements_status on settlements (status);

-- 한글 부분일치 검색용(상품명). 선두 와일드카드 ILIKE를 인덱스로 가속
create extension if not exists pg_trgm;
create index if not exists idx_products_name_trgm on products using gin (name gin_trgm_ops);
```

## 1. 정산 합계를 DB 집계로 — `admin/settlements/route.ts`

지금처럼 전체를 받아와 JS로 더하지 말고, RPC 하나로 끝내면 됩니다.

```sql
create or replace function settlement_summary()
returns table(status text, total_amount bigint, total_fee bigint)
language sql security definer as $$
  select status, coalesce(sum(amount),0), coalesce(sum(fee),0)
  from settlements group by status;
$$;
```

```ts
const { data } = await supabaseAdmin.rpc("settlement_summary");
// data: [{status:'pending', total_amount, total_fee}, ...] 한 번에
```

판매자 평점(`admin/sellers`)이나 대시보드 카테고리 매출(`admin/dashboard`)도 같은 식으로 GROUP BY RPC로 옮겨주면 됩니다.

## 2. 상품 목록/검색을 DB로 내리기 — `products/route.ts`

- **검색**: 위 0번의 `pg_trgm` 인덱스를 만들어 두면 `ilike('%kw%')`도 인덱스를 타게 됩니다.
- **별점순**: 지금은 500개를 받아와 JS로 정렬하고 있는데요, 평점을 미리 계산해 두면 DB에서 바로 정렬하고 페이지네이션까지 할 수 있습니다.

```sql
-- 상품별 평균 평점/리뷰수 집계 뷰
create or replace view product_ratings as
  select product_id, avg(rating)::numeric(3,2) as avg_rating, count(*) as review_count
  from reviews group by product_id;
```

이걸 products와 조인하면 `order('avg_rating', desc).range(...)`를 DB에서 바로 처리할 수 있어서, 500개를 통째로 불러오던 부분이 사라집니다.

- **count**: 첫 페이지에서만 `count: 'exact'`를 쓰고, 이후 페이지에서는 생략하거나 `count: 'estimated'`로 두면 됩니다.

## 3. 주문 생성을 한 트랜잭션으로 — `order/route.ts`

쿼리를 여러 번 날리고 보상 코드를 붙이는 대신, plpgsql 함수 하나로 묶어두면 중간에 실패하더라도 DB가 알아서 전부 되돌려 줍니다.

```sql
create or replace function create_order(p_user_id bigint, p_items jsonb, ...)
returns bigint language plpgsql security definer as $$
declare v_order_id bigint;
begin
  -- 재고 차감 (음수면 예외 → 자동 롤백)
  -- orders insert → order_items insert
  -- 모두 한 트랜잭션. 예외 발생 시 전체 롤백.
  return v_order_id;
end; $$;
```

```ts
const { data, error } = await supabaseAdmin.rpc("create_order", { ... });
```

함수 안에서 예외(`raise exception`)가 발생하면 그 함수 전체가 한 트랜잭션으로 묶여 있어서 재고 차감과 주문 삽입이 통째로 취소됩니다. 지금처럼 restoreStock이나 order delete를 손으로 일일이 불러줄 필요가 없어집니다.

## 4. 검색어 이스케이프 — 공통

```ts
function escapeLike(s: string) {
  return s.replace(/[%_,()\\]/g, "\\$&");
}
const kw = escapeLike(keyword);
query = query.ilike("name", `%${kw}%`);
```

`.or('a.ilike...,b.ilike...')`에 사용자 입력을 그대로 넣을 때는 특히 `,`와 `)`를 막아줘야 필터 구문이 깨지지 않습니다.

## 5. is_default 템플릿 — `return-policy/templates/db.ts`

부분 UNIQUE 인덱스를 걸어서 "기본값은 판매자당 1개"라는 규칙을 DB가 직접 강제하게 만들어 주세요.

```sql
create unique index if not exists uq_return_policy_default
  on return_policy_templates (seller_id) where is_default = true;
```

이렇게 해두면 동시에 두 번 호출되더라도 두 번째 호출이 에러로 막혀서 기본값이 2개가 되는 일이 없습니다. 배송 템플릿도 같은 방식으로 처리하면 됩니다.

---

손대는 순서는 **0(인덱스) → 1(정산 집계) → 3(주문 트랜잭션)** 이 효과도 크고 중요도도 높습니다. 화이팅입니다.
