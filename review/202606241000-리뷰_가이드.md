# Cookeat 해결 가이드 (2026-06-24)

이번 [필수]/[제안]을 어떻게 풀면 되는지 정리해 봤습니다. 손대는 순서는 주문 트랜잭션부터 잡고, 그다음 정산 합산, 나머지는 그 뒤로 가면 됩니다.

## 1. 주문 생성을 하나의 트랜잭션으로 묶기

### 왜 필요한가

지금은 재고 차감과 주문 INSERT, 포인트 차감이 각각 따로따로 네트워크를 한 번씩 왕복하며 실행되는데, 이 중간 어딘가에서 하나라도 실패하거나 함수가 끊겨 버리면 "재고는 깎였는데 주문은 없는" 식의 어긋난 상태가 그대로 남습니다. `restoreStock` 같은 보상 코드를 넣어 뒀어도 그건 catch가 실행될 때만 도는 거라, 프로세스가 그냥 툭 죽어 버리는 경우까지는 못 막아 줍니다. 이럴 때 딱 맞는 게 DB 트랜잭션인데, "전부 성공 아니면 전부 취소"를 DB가 직접 보장해 주거든요.

### Postgres 함수(RPC)로 묶기

여러 쓰기를 한 함수 안에 몰아넣으면 그 함수 전체가 통째로 하나의 트랜잭션으로 돕니다.

```sql
create or replace function public.create_order(
  p_user_id uuid,
  p_items jsonb            -- [{product_id, qty, unit_price}, ...]
)
returns uuid                -- 생성된 order_id
language plpgsql
security definer
as $$
declare
  v_order_id uuid;
  v_item jsonb;
begin
  -- 1) 주문 생성
  insert into orders (user_id, status) values (p_user_id, 'paid')
  returning id into v_order_id;

  -- 2) 항목별 재고 차감 + order_items
  for v_item in select * from jsonb_array_elements(p_items)
  loop
    update products
      set stock = stock - (v_item->>'qty')::int
      where id = (v_item->>'product_id')::uuid
        and stock >= (v_item->>'qty')::int;     -- 재고 부족이면 0 rows
    if not found then
      raise exception '재고 부족: %', v_item->>'product_id';  -- 전체 롤백
    end if;

    insert into order_items (order_id, product_id, quantity, unit_price)
    values (v_order_id, (v_item->>'product_id')::uuid,
            (v_item->>'qty')::int, (v_item->>'unit_price')::int);
  end loop;

  return v_order_id;
end;
$$;
```

라우트에서는 이 함수 한 번만 호출하면 됩니다.

```ts
// app/api/order/route.ts
const { data: orderId, error } = await supabaseAdmin
  .rpc("create_order", { p_user_id: userId, p_items: items });
if (error) {
  // 재고 부족·기타 실패 모두 여기로. 이미 DB가 전부 롤백한 상태라 보상 코드 불필요.
  return NextResponse.json({ success: false, message: error.message }, { status: 409 });
}
```

이렇게 바꾸면 `restoreStock` 보상 로직을 통째로 들어낼 수 있고, 동시 주문이 몰려서 재고가 음수로 빠지는 경합도 `stock >= qty` 조건 한 줄로 막을 수 있습니다.

## 2. 정산 통계를 DB 합산으로

### 지금 (`admin/settlements/route.ts:48-66`)

```ts
const { data: pendingData } = await supabaseAdmin.from("settlements").select("amount").eq("status", "pending");
const pendingSum = pendingData.reduce((a, b) => a + b.amount, 0);   // 전체 행을 메모리로
// completedData, allFeeData 도 같은 식 ...
```

### 바꾸기 — 합계는 DB가

```sql
create or replace function public.settlement_summary()
returns table (status text, total bigint, fee_total bigint)
language sql stable
as $$
  select status, sum(amount)::bigint, sum(fee)::bigint
  from settlements
  group by status;
$$;
```

```ts
const { data } = await supabaseAdmin.rpc("settlement_summary");
// data: [{status:'pending', total, fee_total}, {status:'completed', ...}]
```

행을 통째로 끌어오지 않으니, 정산이 아무리 쌓여도 응답 시간은 그대로 일정하게 유지됩니다.

## 3. 문의 `answered` 필터를 DB에서 처리 (`admin/inquiries/route.ts:34-51`)

전체를 다 가져온 뒤 JS로 거르면 페이지네이션이 어긋나 버립니다. 그냥 답변 여부를 컬럼으로 하나 두는 게 제일 단순합니다.

```sql
alter table inquiries add column answered boolean not null default false;
-- 답변 INSERT 시 트리거나 코드에서 inquiries.answered = true 로 갱신
```

```ts
let q = supabaseAdmin.from("inquiries").select("*", { count: "exact" });
if (answered === "true")  q = q.eq("answered", true);
if (answered === "false") q = q.eq("answered", false);
q = q.range(from, to);   // DB가 필터+페이지네이션을 함께 처리
```

## 4. 배송 템플릿 `is_default` 중복 막기 (`templates/[templateId]/db.ts:56-66`)

두 쿼리(해제 → 설정)를 인덱스로 보강하거나 트랜잭션으로 묶어 주면 되는데, 그중에서도 제일 깔끔한 건 부분 UNIQUE 인덱스입니다.

```sql
-- 셀러별로 기본 템플릿은 하나만 존재하도록 DB가 강제
create unique index uniq_default_template_per_seller
  on return_policy_templates (seller_id)
  where is_default = true;
```

이러면 동시에 두 개를 기본값으로 세우려는 시도 자체가 DB 선에서 막힙니다. 전환 로직도 "해제 → 설정"을 한 트랜잭션(RPC)으로 묶어 주면 한결 더 안전해집니다.

## 5. 검색 키워드 `.or()` 이스케이프 (`admin/products/route.ts:61`, `admin/orders/route.ts:38`)

PostgREST `.or()`에 사용자 입력을 손대지 않고 그대로 흘려 넣으면 `,`·`)` 같은 문자가 구문으로 해석돼서 쿼리가 깨져 버립니다. 키워드에서 이 문자들을 미리 제거하거나 다른 문자로 치환해 주고, 여유가 되면 `.or()` 문자열을 직접 조립하는 대신 안전한 빌더 패턴을 쓰는 쪽이 좋습니다.

```ts
const safe = keyword.replace(/[(),*]/g, " ").trim();
query = query.or(`name.ilike.%${safe}%,description.ilike.%${safe}%`);
```

---

순서는 1(주문 트랜잭션)부터 잡고 2(정산 합산)로 넘어가면 되고, 3~5는 데이터가 본격적으로 쌓이기 전에 틈틈이 정리해 두면 충분합니다. 이번 회차는 관리자 영역 완성도가 꽤 좋았다고 봅니다. 화이팅입니다.
