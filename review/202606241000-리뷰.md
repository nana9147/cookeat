# Cookeat 코드 리뷰 (2026-06-24)

이번 회차는 관리자 페이지가 중심입니다. 대시보드·회원·주문·정산·고객센터·FAQ·상품·레시피·판매자·리뷰·analytics까지 한 영역이 통째로 실데이터로 붙었고, 여기에 판매자 배송 처리와 레시피 목록·상세 실데이터 연동도 같이 들어왔네요. 새 화면이 많아서 사용자 동선(홈 → 레시피/상품 → 로그인 → 마이페이지)부터 판매자, 관리자까지 순서대로 띄워 봤습니다.

관리자는 `/admin/login`으로 따로 들어가고 `requireAdmin`이 `users.role === 'admin'`을 확인하는 구조라, 리뷰용 계정(`cookeat-review@example.com`)을 잠깐 admin으로 올려 전 영역을 돌아본 뒤 원래 권한으로 되돌려 놨습니다.

## 3. 팀 공통 — 이월 [필수] 둘

### 3-1. [필수] 주문 생성이 트랜잭션으로 묶여 있지 않습니다

`app/api/order/route.ts`는 재고 차감·주문 생성·포인트 차감이 각각 별도 쿼리로 흩어져 있습니다(`:26`, `:117`, `:174`, `:192` 부근). 지금도 실패 시 `restoreStock`으로 되돌리는 보상 코드가 있긴 한데, 이 보상 로직은 중간에 프로세스 자체가 죽어 버리면 실행된다는 보장이 없습니다. 가령 재고는 이미 깎였는데 주문 INSERT 직전에 함수가 끊기면 재고만 줄고 주문은 남지 않는 불일치가 생기는 식이지요. 결제·재고가 걸린 곳이니만큼 **여러 쓰기를 하나의 DB 트랜잭션(RPC)으로 묶어서** "전부 되거나 전부 안 되거나"를 보장하는 게 맞습니다. 가이드에 Postgres 함수로 묶는 예시를 적어 둘게요.

### 3-2. [필수→제안 경계] 정산 통계를 DB 합산으로

`app/api/admin/settlements/route.ts:48-66`은 `pendingData`·`completedData`·`allFeeData`를 전부 메모리로 가져와 JS `reduce`로 더합니다. 정산 건수가 쌓일수록 응답이 점점 느려지고 메모리도 그만큼 더 쓰게 됩니다. 이런 합계 계산은 DB가 가장 잘하는 일이라, `select status, sum(amount) ... group by status`를 RPC로 한 번에 받아오는 쪽으로 옮기면 훨씬 깔끔해집니다. 당장 터지는 버그는 아니지만 데이터가 늘면 체감되는 부분이라 이참에 함께 정리해 두면 좋겠습니다.

### 최유종 (@jjong0) — 관리자 페이지 전체, 정산, FAQ, 고객센터

관리자 영역을 통째로 올리면서 모든 API에 `requireAdmin`을 하나도 빠뜨리지 않고 붙인 게 이번 회차에서 가장 잘한 부분이라 생각합니다. 고객센터(문의) 실데이터 연동과 `inquiry_replies` 오작동 수정도 좋았고요. 두 가지만 더 봐주세요.

- **[제안] 문의 목록 `answered` 필터가 DB 페이지네이션을 무력화합니다.** `app/api/admin/inquiries/route.ts:34-51`에서 `?answered=`가 들어오면 전체 행을 가져와 JS로 거른 뒤 `slice` 하고 있어요. 문의가 쌓이면 2페이지 이후 결과가 어긋날 수 있습니다. `inquiry_replies` 존재 여부를 PostgREST 필터로 처리하거나, 아예 답변 여부를 컬럼(`answered`)으로 두고 그걸로 필터·페이지네이션하면 정확해집니다.
- **[제안] 관리자 검색 키워드 `.or()` 미이스케이프**가 남아 있습니다(`admin/products/route.ts:61`, `admin/orders/route.ts:38`). 관리자 게이트 뒤라 외부에서 악용될 위험은 낮지만, `,`나 `)`가 섞인 검색어가 들어오면 PostgREST 구문이 깨져 내부 에러가 납니다. 키워드를 이스케이프하거나 안전한 필터로 바꿔 주세요.
