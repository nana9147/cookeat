## 1. [필수·보안] 서버 입력 검증 추가

**(1) 문제**: `app/api/auth/register/route.ts`, `login/route.ts`가 클라이언트가 보낸 값을 검증 없이 그대로 Supabase에 넘깁니다.

**(2) 왜**: 지금 검증은 전부 브라우저(`hooks/auth/useRegisterFields.ts`)에서만 합니다. 그런데 브라우저 검증은 사용자 편의용일 뿐, 누군가 개발자도구 콘솔이나 Postman으로 라우트를 직접 호출하면 그냥 통과합니다. 진짜 방어선은 항상 서버라 생각합니다. `docs/api/01-auth.md`에 "비밀번호 8자 이상, 전화번호 010-0000-0000 형식"이라고 적어 뒀으니, 그 규칙을 서버에서도 한 번 더 확인하면 됩니다.

**(3) 해결 단계**
1. 검증 함수를 모아 둘 작은 파일을 하나 만듭니다 (`lib/validators.ts`).
2. `register` 라우트에서 `signUp` 호출 전에 그 함수로 입력을 검사하고, 틀리면 400으로 돌려보냅니다.
3. `login` 라우트에도 이메일/비밀번호가 비었는지 정도만 가볍게 추가합니다.

**(4) Before / After**

`lib/validators.ts` — 새로 만들기

```ts
// 서버·클라이언트 어디서든 재사용할 수 있는 입력 검증 모음
const EMAIL_RE = /^[^\s@]+@[^\s@]+\.[^\s@]+$/;
const PHONE_RE = /^010-\d{4}-\d{4}$/;

export function validateSignup(input: {
  email?: string; password?: string; nickname?: string; phone?: string;
}): string | null {
  if (!input.email || !EMAIL_RE.test(input.email)) return '이메일 형식이 올바르지 않습니다.';
  if (!input.password || input.password.length < 8) return '비밀번호는 8자 이상이어야 합니다.';
  if (!input.nickname?.trim()) return '닉네임을 입력해주세요.';
  if (!input.phone || !PHONE_RE.test(input.phone)) return '전화번호 형식(010-0000-0000)이 올바르지 않습니다.';
  return null; // null이면 통과
}
```

`app/api/auth/register/route.ts:4~11` — Before

```ts
export async function POST(req: NextRequest) {
  const { email, password, nickname, phone } = await req.json();

  const { data, error } = await supabase.auth.signUp({
    email,
    password,
    options: { data: { nickname, phone } },
  });
```

After

```ts
import { validateSignup } from '@/lib/validators';

export async function POST(req: NextRequest) {
  const { email, password, nickname, phone } = await req.json();

  // 서버에서 한 번 더 검증 — 브라우저를 우회한 요청도 여기서 막힌다
  const invalid = validateSignup({ email, password, nickname, phone });
  if (invalid) return NextResponse.json({ error: invalid }, { status: 400 });

  const { data, error } = await supabase.auth.signUp({
    email,
    password,
    options: { data: { nickname, phone } },
  });
```

`app/api/auth/login/route.ts:5` — After (가볍게)

```ts
export async function POST(req: NextRequest) {
  const { email, password } = await req.json();

  // 빈 값으로 들어오는 요청 차단
  if (!email?.trim() || !password) {
    return NextResponse.json({ error: '이메일과 비밀번호를 입력해주세요.' }, { status: 400 });
  }

  const { data, error } = await supabase.auth.signInWithPassword({ email, password });
```

## 2. [필수·보안] admin / seller 페이지에 접근 권한 게이트 추가

**(1) 문제**: `app/admin/layout.tsx`, `app/seller/layout.tsx` 어디에도 "로그인했는지 / 관리자·판매자인지" 확인이 없습니다.

**(2) 왜**: 지금은 화면이 전부 mock 데이터라 새는 게 없지만, 실제 회원/주문 데이터를 붙이는 순간 누구나 주소창에 `/admin`을 쳐서 들어올 수 있게 됩니다. "데이터를 붙이기 전에 게이트를 단다"를 팀 규칙으로 정해 두면 깜빡할 일이 없습니다.

**(3) 해결 단계**

지금 단계에서 할 수 있는 건 두 단계입니다. (a) 우선 로그인 여부만 막는 클라이언트 가드를 붙이고, (b) 실데이터 연동 시점에 역할(role) 확인을 서버에서 추가합니다.

1. 역할 확인을 재사용할 수 있게 작은 클라이언트 가드 컴포넌트를 만듭니다 (`components/auth/RoleGuard.tsx`).
2. admin/seller 레이아웃에서 children을 이 가드로 감쌉니다.
3. (다음 단계) 실데이터를 다룰 때는 API 라우트에서도 `getUser(token)` 후 role을 확인합니다. 화면 가드만으로는 부족하기 때문입니다.

**(4) Before / After**

`components/auth/RoleGuard.tsx` — 새로 만들기

```tsx
'use client';

import { useEffect } from 'react';
import { useRouter } from 'next/navigation';
import { useAuthStore } from '@/store/authStore';

// 로그인 안 했으면 홈으로 돌려보내는 1차 가드 (역할 확인은 추후 서버에서 보강)
export default function RoleGuard({ children }: { children: React.ReactNode }) {
  const router = useRouter();
  const hydrated = useAuthStore((s) => s._hydrated);
  const token = useAuthStore((s) => s.accessToken);

  useEffect(() => {
    if (hydrated && !token) router.replace('/login');
  }, [hydrated, token, router]);

  if (!hydrated) return null;        // 로그인 상태 복원 전에는 깜빡임 방지
  if (!token) return null;           // 비로그인 → 위 useEffect가 리다이렉트
  return <>{children}</>;
}
```

`app/admin/layout.tsx:13~15` — Before

```tsx

  <AdminMobileNav />
  <main className="flex flex-1 flex-col">{children}</main>
</div>
```

After

```tsx
import RoleGuard from '@/components/auth/RoleGuard';
// ...
<div className="flex flex-1 flex-col">
  <AdminMobileNav />
  <main className="flex flex-1 flex-col">
    <RoleGuard>{children}</RoleGuard>
  </main>
</div>
```

`app/seller/layout.tsx`도 똑같이 `{children}`을 `<RoleGuard>{children}</RoleGuard>`로 감싸 주세요.

> 중요: 위 가드는 화면을 막을 뿐, 데이터 자체를 막는 건 아닙니다. 실제 데이터를 다루는 API 라우트에서는 `app/api/auth/profile/route.ts`처럼 `supabaseAdmin.auth.getUser(token)`로 본인을 확인하고, 거기에 더해 이 사람이 admin/seller가 맞는지(예: `users` 테이블의 `role` 컬럼)를 확인해야 완성됩니다. 이건 마켓 API를 만들 때 같이 하면 됩니다.

## 3. [필수·정확성] `Product` 타입 ↔ mock 데이터 불일치

**(1) 문제**: `types/seller/product.ts:24`에 `store: string`이 필수로 있는데, `app/seller/products/page.tsx:13~58`의 mock 데이터 4개에는 `store`가 없습니다.

**(2) 왜**: 타입이 "이 필드는 반드시 있다"고 약속했는데 실제 데이터엔 없으니, TypeScript가 거짓말을 하게 됩니다. `npm run build`에서 막히거나, 나중에 `product.store`를 썼을 때 `undefined`로 조용히 터집니다. 타입과 데이터는 항상 한 몸이어야 합니다.

**(3) 해결 단계**: 둘 중 하나만 고르면 됩니다.
- 판매처 정보가 아직 필요 없다면 → 타입에서 `store`를 빼거나 선택(`store?: string`)으로.
- 필요하다면 → mock 데이터 4개에 `store` 값을 채웁니다.

여기서는 "아직 필요 없다" 쪽 예시를 보여드리겠습니다(지금 화면에서 안 쓰고 있으니까요).

**(4) Before / After**

## 4. [제안] 닉네임 중복확인의 빈틈 — DB UNIQUE 제약으로 마무리

**(1) 문제**: 닉네임 중복확인(`check-nickname`)과 실제 가입(`register`)이 두 번에 나뉘어 있어서, 그 사이에 다른 사람이 같은 닉네임으로 먼저 가입하면 중복이 생길 수 있습니다(race condition).

**(2) 왜**: 확인한 시점과 저장하는 시점이 다르면, 그 틈에 상황이 바뀔 수 있습니다. 이런 건 애플리케이션 코드로 100% 막기 어렵고, DB가 최종 심판을 보게 하는 게 정석입니다. DB에 "닉네임은 중복 불가"라고 못을 박아 두면, 누가 어떤 경로로 들어와도 중복은 생기지 않습니다.

**(3) 해결 단계**
1. Supabase SQL Editor에서 `users.nickname`에 UNIQUE 제약을 겁니다.
2. 가입/프로필 저장 시 중복 에러가 나면 친절한 메시지로 바꿔 줍니다.

**(4) 예시**

Supabase SQL Editor에서 한 번 실행:

```sql
-- 닉네임 중복을 DB 차원에서 원천 차단
ALTER TABLE public.users ADD CONSTRAINT users_nickname_unique UNIQUE (nickname);
```

`app/api/auth/profile/route.ts:31~36` 부근 — 중복 에러를 친절하게 (After)

```ts
const { error: updateError } = await supabaseAdmin
  .from('users')
  .update({ nickname: nickname.trim(), phone: phone.trim() })
  .eq('email', user.email);

if (updateError) {
  // UNIQUE 제약 위반(Postgres 코드 23505)이면 중복 안내
  if (updateError.code === '23505') {
    return NextResponse.json({ error: '이미 사용 중인 닉네임입니다.' }, { status: 409 });
  }
  return NextResponse.json({ error: updateError.message }, { status: 500 });
}
```

이렇게 하면 `check-nickname`은 사용자 편의용 미리보기, DB UNIQUE는 최종 방어선 역할로 깔끔하게 나뉩니다.

> 참고: `check-nickname`이 인증 없이 열려 있어 닉네임을 긁어 갈 수 있는 점은, 지금 당장은 신경 쓰지 않아도 되는 [제안]입니다. 나중에 여유가 생기면 호출 횟수 제한(rate limit)을 떠올려 보세요.

## 5. [사소] 정리하면 좋은 작은 것들

동작에는 문제없으니 여유 될 때 하나씩 보면 됩니다.

- **Prettier 자동 포맷 켜기**: 파일마다 세미콜론(`;`)이 있다 없다 합니다. `.prettierrc`가 이미 있으니, VSCode에서 셋 다 아래 설정만 켜면 정리됩니다. 저장만 하면 자동 정렬되어 git diff가 깨끗해집니다.
  - `.vscode/settings.json`에 추가:
    ```json
    {
      "editor.formatOnSave": true,
      "editor.defaultFormatter": "esbenp.prettier-vscode"
    }
    ```
- **색상 토큰 통일**: `text-red-500`(기본 Tailwind)과 `bg-green-700` 같은 직접 색상이 군데군데 섞여 있습니다. 팀 디자인 토큰(`text-red`, `bg-primary`)으로 통일하면 나중에 메인 컬러를 바꿀 때 한 곳만 고치면 됩니다. (해당 위치: `app/admin/members/page.tsx:151`, `app/seller/components/ProductForm/ImageUploadField.tsx:39,201~209`)
- **닉네임 중복확인 로직 중복 제거**: `app/auth/profile/page.tsx:29~40`에 손으로 짠 중복확인 로직이 있는데, 이미 `hooks/auth/useNicknameCheck.ts`에 같은 게 있습니다. 훅을 재사용하면 메시지 문구를 한 곳에서만 고치면 됩니다.

## 6. (조언) 공통 코드를 함께 나누는 일

이건 고칠 코드라기보다 팀 운영 이야기입니다. 리뷰에서 적었듯, 공통 토대(`lib/`·`store/`·인증)가 한쪽으로 모인 상황인지 먼저 확인하고 싶습니다. 만약 그렇다면, 한 사람에게만 토대가 몰려 있을 때 그 사람이 바쁠 때 팀 전체가 멈출 수 있다는 점에서 아래 두 가지를 권합니다. 이미 함께 정하며 진행 중이라면 그대로 가도 좋습니다.

1. **다음 작업부터 마켓 데이터 연동을 나눠 맡기**: 지금 최유종 님(어드민)·추유나 님(셀러) 화면이 mock 배열입니다. 그걸 실제 API로 바꾸는 작업을, 엄인호 님의 `app/api/auth/*` 라우트를 참고서 삼아 두 분이 하나씩 맡아 보세요. 자연스럽게 백엔드(API 라우트) 경험이 분산됩니다.
2. **공통 파일 같이 읽기**: `lib/api.ts`, `store/authStore.ts` 같은 핵심 파일을 15분만 셋이 같이 읽어 보는 시간을 가져 보세요. 작성한 사람이 설명하면 본인 정리도 되고, 다른 분들도 나중에 손댈 수 있게 됩니다.

서두를 필요는 없습니다. 위 [필수] 세 가지부터 차근차근 처리하고 다음에 또 보겠습니다.
