# Cookeat 4차 리뷰 — 해결 가이드 (2026-06-11)

## 1. [필수] 빌드 복구 — useKakaoPostcode 전역 kakao 선언 충돌

`tsc`가 막는 곳:
```
hooks/user/useKakaoPostcode.ts(7,5): error TS2687:
  All declarations of 'kakao' must have identical modifiers.
```
전역 `kakao`(또는 `window.kakao`)를 두 곳에서 다르게 선언해서 납니다(예: 한 곳은 `readonly`, 한 곳은 아님 / 한 곳은 `declare global`, 한 곳은 모듈 안). **전역 타입 선언은 한 파일로 모으세요.**

```ts
// src/types/global.d.ts  (한 곳에서만 선언)
export {};
declare global {
  interface Window {
    kakao: any;     // 또는 정확한 타입
    daum: any;
  }
}
```
그리고 `useKakaoPostcode.ts` 등 개별 파일의 `declare global { ... kakao ... }`는 지웁니다. 같은 전역을 두 번 선언하지 않게 하는 게 핵심입니다.

## 2. [필수·보안] 결제 — 서버가 금액의 주인이 되게

핵심 원칙: **금액·사용자·주문은 서버가 DB로부터 만든다. 클라가 보낸 값은 신뢰하지 않는다.**

권장 흐름:
```ts
// app/api/payment/prepare/route.ts (결제 시작 전, 주문을 서버에서 생성)
import { requireAuth } from '@/lib/serverAuth';

export async function POST(req: NextRequest) {
  const user = await requireAuth(req);
  if (user instanceof NextResponse) return user;   // 미인증 차단

  const { cartId } = await req.json();
  // 1) 장바구니를 서버에서 다시 읽어 금액을 직접 계산
  const items = await getCartItems(cartId, user.id);
  const amount = items.reduce((s, it) => s + it.price * it.qty, 0); // 서버 계산값

  // 2) orders 테이블에 'pending'으로 주문 생성 (amount를 DB에 박아둠)
  const order = await createOrder({ userId: user.id, amount, status: 'pending' });

  return NextResponse.json({ orderId: order.id, amount }); // 클라엔 서버가 정한 값만
}
```
```ts
// confirm 단계: 클라가 보낸 amount를 쓰지 말고, DB의 주문 금액과 대조
export async function POST(req: NextRequest) {
  const user = await requireAuth(req);
  if (user instanceof NextResponse) return user;

  const { paymentKey, orderId } = await req.json();
  const order = await getOrder(orderId, user.id);          // 서버에서 주문 조회
  if (!order || order.status !== 'pending') return NextResponse.json({ error: 'invalid order' }, { status: 400 });

  const res = await fetch('https://api.tosspayments.com/v1/payments/confirm', {
    method: 'POST',
    headers: { Authorization: `Basic ${Buffer.from(`${process.env.TOSS_SECRET_KEY}:`).toString('base64')}`, 'Content-Type': 'application/json' },
    body: JSON.stringify({ paymentKey, orderId, amount: order.amount }),  // ★ DB 금액 사용
  });
  if (!res.ok) return NextResponse.json({ error: await res.json() }, { status: res.status });

  await markOrderPaid(orderId);   // 성공 시에만 'paid'로 (중복 승인 방지: 이미 paid면 skip)
  return NextResponse.json(await res.json());
}
```
카카오도 동일: `ready`에서 `total_amount`를 클라가 아니라 서버 주문 금액으로, `partner_user_id`는 `user.id`로.

## 3. [필수] 셀러 화면 가드

레이아웃을 서버 컴포넌트로 만들어 입구에서 한 번 막습니다.
```tsx
// app/seller/layout.tsx
import { redirect } from 'next/navigation';
import { createClient } from '@/lib/supabase/server';

export default async function SellerLayout({ children }: { children: React.ReactNode }) {
  const supabase = await createClient();
  const { data: { user } } = await supabase.auth.getUser();
  if (!user) redirect('/login');

  // role까지 확인 (users 또는 sellers 테이블에서)
  const { data: profile } = await supabase.from('users').select('role').eq('auth_id', user.id).single();
  if (profile?.role !== 'seller' && profile?.role !== 'admin') redirect('/');

  return ( /* 기존 헤더/사이드바 그대로 */ );
}
```
admin 레이아웃도 같은 패턴으로 `role !== 'admin'`이면 redirect.

## 5. [제안] 공개 데이터(상품) 읽기 정책 정하기

지금 anon으로 `products`가 `[]`라, 스토어프론트는 service-client API 라우트로 우회해야 동작합니다. 두 길 중 하나를 팀에서 정하세요.
- (A) 상품 같은 공개 데이터는 **anon select 정책을 따로 허용**(`for select to anon using (true)`) — 클라에서 바로 읽기. 간단.
- (B) 전부 API 라우트(service client) 경유 — 일관되지만 라우트가 많아짐.

상품/리뷰처럼 "누구나 봐도 되는" 데이터는 (A)가 보통 편합니다. 단 sellers의 계좌·사업자번호 같은 민감 컬럼은 절대 anon에 열지 마세요(뷰로 분리).
