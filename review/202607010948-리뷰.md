## 2. 결제 서버검증 — 잘한 것과 남은 것

### 잘한 것: 토스 경로 + 주문 생성이 서버 진실

`order/route.ts:47-92`가 상품 가격·재고·상태를 DB에서 다시 조회해 서버가 `totalAmount`를 재계산하고, 포인트(96-113)·쿠폰(115-145)도 서버가 재검증합니다. 재고 차감·주문·포인트 차감·쿠폰 카운트 증가는 `create_order` RPC로 **하나의 트랜잭션**에 묶었는데, 이게 정석입니다. 실제로 anon 토큰으로 아래 시나리오를 다 찔러 봤는데 전부 막히더군요.

| 공격 시나리오 | 결과 |
| --- | --- |
| `usePoint=-99999` | 400 "포인트는 0 이상" |
| `usePoint=99999999` (잔액 초과) | 400 "보유 포인트가 부족" |

`toss/confirm/route.ts:22`는 `order.final_amount !== amount`로 금액까지 대조합니다. 엄인호님, 4차에 짚었던 문제를 이렇게 닫아 주셔서 좋았습니다.

### [필수] 남은 것: 카카오페이만 금액 검증이 없습니다

토스는 막았는데 카카오 경로만 뚫려 있습니다.

- `payment/kakao/ready/route.ts:12`가 클라이언트가 보낸 `totalAmount`를 그대로 KakaoPay `ready`에 넘깁니다. 주문 DB의 `final_amount`를 조회하지 않아요.
- `kakao/approve`는 소유권·상태만 확인하고 **금액을 전혀 대조하지 않습니다.**

그래서 10만원짜리 주문을 `totalAmount:100`으로 ready 호출하면 100원만 결제하고 주문이 완료됩니다. 토스에 이미 `final_amount` 대조라는 좋은 패턴이 있으니, 카카오도 `ready`에서 orderId로 `orders.final_amount`를 조회해 서버 값으로 강제하고 클라가 보낸 `totalAmount`는 무시하면 됩니다. 코드는 가이드에 담았습니다.

## 3. [필수] 쿠폰 %할인이 전량 오작동 중입니다

이건 실측하다 발견한 건데, **모든 % 할인 쿠폰이 정액(원)으로 잘못 적용되고 있습니다.**

DB는 쿠폰 타입을 `discount_type='rate'`로 저장합니다(anon으로 `WELCOME10`을 읽어 확인했습니다). 상품 폼도 `'rate'/'amount'/'none'`을 씁니다(`validators.ts:6`, `PricingField.tsx:18`). 그런데 쿠폰을 계산하는 세 곳이 `=== '%'`로 비교합니다.

- `order/route.ts:142`
- `coupons/validate/route.ts:46`
- `DiscountSection.tsx:151`

`'rate' === '%'`가 항상 false라서 `else` 분기로 빠지고, `discount_value`(10)를 **정액 10원**으로 적용합니다. 실제로 `GET /coupons/validate?code=WELCOME10&amount=999999`를 부르면 원래 99,999쯤 깎여야 하는데 `discountAmount:10`이 나오네요. 그나마 "덜 깎이는" 방향이라 금전 손해는 안 나지만 기능 자체가 완전히 죽은 상태입니다. 세 곳 모두 `'%'` → `'rate'`로 바꾸면 됩니다.

이건 컨벤션 관점에서도 교훈이 있는데, 상품 도메인은 `'rate'`로 잘 통일해 둔 반면 쿠폰 도메인만 `'%'`로 엇나가면서 버그가 생긴 겁니다. 이런 리터럴은 상수 하나로 모아 두면 애초에 이렇게 표류할 일이 없습니다(가이드에 예시).

## 4. 보안 실증 (RLS)

anon(publishable) 키로 민감 테이블을 직접 읽어 보고 service-role로 실제 행 수를 대조했습니다. **PII 유출 없이 잘 막혀 있습니다.**

| 테이블 | 실제 행 | anon read | 판정 |
| --- | --- | --- | --- |
| users | 10 | `[]` | 차단 O |
| sellers (PII) | 4 | `[]` | 차단 O (15차 fix 유지) |
| orders / order_items | 84 / 100 | `[]` | 차단 O |
| settlements | 5 | `[]` | 차단 O |
| user_shipping_addresses | 2 | `[]` | 차단 O |
| coupons | 1 | **1행 노출** | 아래 [제안] |

- **[제안]** `coupons`가 anon에 통째로 읽힙니다. `WELCOME10` 코드·할인율·사용 횟수까지 다 보이네요. 쿠폰 코드는 원래 사용자에게 배포하는 값이라 치명적이진 않지만, 아직 배포 안 한 프로모 코드까지 긁힐 수 있으니 anon select는 RLS로 막고 검증은 서버 라우트(service-role)로만 태우는 게 안전합니다.
- **[사소]** 라우트들이 `await req.json()`을 try/catch로 감싸지 않다 보니, 깨진 JSON 바디를 보내면 500(uncaught SyntaxError)이 납니다. 실제로 `-d '{bad json'`를 보내면 500이 나오는데, 400으로 감싸 주면 깔끔합니다.

## 7. 팀 공통

- **빌드 초록불 복귀**가 이번 회차 최대 성과입니다. CI에 `npm run build` 게이트가 있는지 확인하고 없으면 추가해서 이 상태를 유지합시다.
- **결제 두 경로(토스·카카오)의 방어 수준이 비대칭**인 게 전형적인 함정입니다. 토스만 금액을 검증하고 카카오는 안 하는 상태죠. 결제 확정 로직을 공통 헬퍼로 뽑아 두 경로가 같은 검증을 타게 하면 이런 비대칭은 재발하지 않습니다.
- **리터럴 표류(`'rate'` vs `'%'`)**가 쿠폰 버그의 뿌리입니다. 도메인 상수를 한 곳(SSOT)에 두고 공유하면 타입이 오타·표류를 대신 잡아 줍니다.
- RLS·소유권·입력검증 성숙도는 부트캠프 평균 이상입니다. 잘 하고 있습니다.
