리뷰에서 짚은 것들을 어떻게 고치는지 단계별로 적어 둡니다. 위로 갈수록 급한 순서입니다.

## 1. 정산 상세 params 연결 [필수]

증상: `/seller/settlement/SET-001`과 `/seller/settlement/SET-999`가 같은 화면입니다.
원인: `settlement/[id]/page.tsx`가 URL의 id를 읽지 않고 `MOCK_SETTLEMENT_DETAIL` 상수만 그립니다.

이 페이지는 `'use client'`라 서버 컴포넌트의 `params` props 대신 `useParams()` 훅으로 id를 읽으면 됩니다. 판매자 정보 페이지(`/seller/info`)에서 이미 잘 쓰고 있는 "api 클라이언트로 내 데이터 fetch" 패턴을 그대로 가져오면 됩니다.

```tsx
'use client';
import { useParams } from 'next/navigation';
import { useEffect, useState } from 'react';
import api from '@/lib/api';
import { SettlementDetail } from '@/types/seller/settlement';

export default function SettlementDetailPage() {
  const { id } = useParams<{ id: string }>();   // URL 의 [id]
  const [detail, setDetail] = useState<SettlementDetail | null>(null);
  const [loading, setLoading] = useState(true);

  useEffect(() => {
    let cancelled = false;
    (async () => {
      try {
        const res = await api.get(`/seller/settlements/${id}`); // 정산 단건 API
        if (!cancelled) setDetail(res.data.data);
      } finally {
        if (!cancelled) setLoading(false);
      }
    })();
    return () => { cancelled = true; };
  }, [id]);

  if (loading) return <div>불러오는 중…</div>;
  if (!detail) return <div>정산 내역을 찾을 수 없습니다.</div>;

  return (
    <>
      <SettlementBasicInfo detail={detail} />
      <SettlementAmountDetail detail={detail} />
      {/* ... */}
    </>
  );
}
```

정산 단건 API(`/api/seller/settlements/[id]`)가 아직 없다면 `api/seller/me`를 만든 방식 그대로 `requireSeller` 가드를 붙이고 본인 정산만 조회하는 라우트를 하나 추가하면 됩니다. 당장 API까지 손대기 어렵다면, 최소한 `MOCK` 목록에서 `id`로 찾아 보여주게만 바꿔도 "다른 id면 다른 화면"은 맞출 수 있습니다.

```tsx
const detail = MOCK_LIST.find((s) => s.id === id) ?? null;
```

## 2. admin keyword 필터 인젝션 방어 [제안] (이월)

`app/api/admin/products/route.ts:61`의 `.or()`에 검색어를 직접 끼우는 부분입니다.

```ts
// 지금 — keyword 가 PostgREST 필터 문법으로 해석됨(쉼표·괄호 위험)
query = query.or(`name.ilike.%${keyword}%,seller_id.in.(${matchingSellerIds.join(',')})`);
```

PostgREST 필터에서 의미를 갖는 문자(`,` `(` `)` `.`)를 미리 걸러 주면 됩니다.

```ts
const safe = keyword.replace(/[,()*]/g, ' ').trim();   // 특수문자 제거
if (safe) {
  if (matchingSellerIds.length > 0) {
    query = query.or(`name.ilike.%${safe}%,seller_id.in.(${matchingSellerIds.join(',')})`);
  } else {
    query = query.ilike('name', `%${safe}%`);   // 이 형태는 값으로 전달돼 안전
  }
}
```

제일 안전한 건 위 27행처럼 `.ilike('컬럼', 값)` 형태로 값을 그대로 전달하는 방식입니다. OR 조건이 꼭 필요해서 `.or()`를 써야 하는 경우에만 특수문자를 걸러 쓰면 됩니다.

## 3. RLS 정책 무한 재귀 정리 [제안]

증상: anon(publishable) 키로 `users`/`sellers`/`orders`를 읽으면 `42P17 infinite recursion detected in policy`가 납니다.
원인: 한 테이블의 RLS 정책이 **자기 테이블을 다시 select** 하는 서브쿼리를 둘 때 흔히 생깁니다. 가령 `users` 정책 안에서 다시 `users`를 조회해 role을 확인하면, 그 조회가 또 정책을 타고, 그 정책이 또 조회를 부르는 식으로 무한히 돕니다.

해결은 두 갈래로 잡으면 됩니다.
- role 같은 값을 확인할 때는 `auth.jwt()` 클레임이나 비재귀 함수(`SECURITY DEFINER` 함수)를 쓰고, 정책 안에서 같은 테이블을 다시 select 하지 않게 합니다.
- Supabase 대시보드 → Authentication → Policies에서 각 테이블 정책을 열어, `USING`/`WITH CHECK` 식에 같은 테이블 조회가 들어간 곳을 찾아 들어냅니다.

지금이야 API가 service role로 돌아 가려져 있지만, RLS는 클라이언트가 직접 읽어도 안전한지를 지키는 마지막 방어선이라 제대로 동작하게 두는 편이 낫습니다.
