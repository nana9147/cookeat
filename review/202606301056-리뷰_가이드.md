이번 리뷰에서 짚은 것들을 어떻게 고치는지 단계별로 정리했습니다. 우선순위는 카카오페이 금액 검증 → SEO → `fetchInquiries` 순입니다.

## 1. [필수/보안] 카카오페이 결제 금액 서버 검증

핵심은 단순합니다. **금액은 클라이언트를 절대 믿지 않고, 항상 DB의 `final_amount`를 기준으로 삼는다.** 토스(`toss/confirm/route.ts:22, 30`)가 이미 이렇게 돼 있으니 같은 모양으로 맞춰주면 됩니다.


### ready — DB 금액으로 PG 요청을 만든다

클라이언트가 보낸 `totalAmount`를 PG에 넘기지 말고, 주문을 조회해 `final_amount`를 직접 읽어 씁니다.

```ts
// app/api/payment/kakao/ready/route.ts
import { supabaseAdmin } from '@/lib/supabaseAdmin';

export async function POST(req: NextRequest) {
  const authed = await requireAuth(req);
  if (authed instanceof NextResponse) return authed;

  // totalAmount는 더 이상 신뢰하지 않는다
  const { orderId, itemName, quantity } = await req.json();

  // 금액은 DB에서 직접 읽는다
  const { data: order, error } = await supabaseAdmin
    .from('orders')
    .select('final_amount, user_id, status')
    .eq('order_id', orderId)
    .single();

  if (error || !order)
    return NextResponse.json({ error: '주문을 찾을 수 없습니다.' }, { status: 404 });
  if (order.user_id !== authed.userId)
    return new NextResponse('Forbidden', { status: 403 });
  if (order.status === '결제완료')
    return NextResponse.json({ error: '이미 결제된 주문입니다.' }, { status: 400 });

  const amount = order.final_amount; // 단일 진실 원천

  const res = await fetch('https://open-api.kakaopay.com/online/v1/payment/ready', {
    method: 'POST',
    headers: {
      Authorization: `SECRET_KEY ${SECRET_KEY}`,
      'Content-Type': 'application/json',
    },
    body: JSON.stringify({
      cid: CID,
      partner_order_id: orderId,
      partner_user_id: String(authed.userId),
      item_name: itemName,
      quantity,
      total_amount: amount,            // 클라가 아니라 DB 값
      vat_amount: Math.floor(amount / 11),
      tax_free_amount: 0,
      approval_url: `${BASE_URL}/cart/complete`,
      cancel_url: `${BASE_URL}/cart/checkout`,
      fail_url: `${BASE_URL}/cart/checkout`,
    }),
  });
  // ... 이하 동일
}
```

### approve — 승인 응답 금액을 DB 금액과 대조한다

approve는 두 가지를 더 합니다. 주문 조회에 `final_amount`를 포함하고, KakaoPay 승인 응답의 `amount.total`이 DB 금액과 같은지 확인합니다.

```ts
// app/api/payment/kakao/approve/route.ts
const { data: order, error } = await supabaseAdmin
  .from('orders')
  .select('final_amount, user_id, status')   // final_amount 추가
  .eq('order_id', orderId)
  .single();

if (error || !order)
  return NextResponse.json({ error: '주문을 찾을 수 없습니다.' }, { status: 404 });
if (order.user_id !== authed.userId)
  return new NextResponse('Forbidden', { status: 403 });
if (order.status === '결제완료') return NextResponse.json({ tid });

const res = await fetch('https://open-api.kakaopay.com/online/v1/payment/approve', {
  /* ... 기존 그대로 ... */
});

if (!res.ok) {
  const kakaoError = await res.json();
  return NextResponse.json({ error: kakaoError }, { status: res.status });
}

const data = await res.json();

// 승인된 실제 금액이 DB 금액과 일치하는지 확인
if (data.amount?.total !== order.final_amount) {
  return NextResponse.json(
    { error: '결제 금액이 일치하지 않습니다.' },
    { status: 400 },
  );
}

await supabaseAdmin
  .from('orders')
  .update({ status: '결제완료', payment_method: '카카오페이', payment_key: tid, updated_at: new Date().toISOString() })
  .eq('order_id', orderId);

return NextResponse.json(data);
```

> 왜 두 곳 다 막느냐면, ready에서 막아도 ready를 안 거치고 approve만 직접 호출하는 경로를 닫아야 하기 때문입니다. ready는 "잘못된 금액으로 PG 창을 못 열게", approve는 "혹시 열렸어도 잘못된 금액으로 확정되지 않게" — 이중으로 잠그는 셈입니다.

## 2. [필수/SEO] 메타데이터 · sitemap · robots · 목록 SSR화

### 2-1. 상세 페이지에 `generateMetadata`

상세는 이미 서버에서 데이터를 들고 있으니(`recipes/[id]/page.tsx`, `shopping/[id]/page.tsx`) 메타만 붙이면 됩니다.

```ts
// app/(main)/recipes/[id]/page.tsx
import type { Metadata } from 'next';

export async function generateMetadata(
  { params }: { params: Promise<{ id: string }> },
): Promise<Metadata> {
  const { id } = await params;
  const recipe = await fetchRecipeDetail(Number(id));
  if (!recipe) return { title: '레시피를 찾을 수 없습니다 | Cookeat' };

  return {
    title: `${recipe.title} | Cookeat`,
    description: recipe.summary?.slice(0, 120) ?? `${recipe.title} 레시피`,
    openGraph: {
      title: recipe.title,
      images: recipe.thumbnailUrl ? [recipe.thumbnailUrl] : [],
    },
  };
}
```

상품 상세도 같은 패턴으로 `${product.name} | Cookeat`를 넣어주세요.

### 2-2. `metadataBase`

`layout.tsx`의 정적 `metadata`에 `metadataBase`를 추가합니다. OG 이미지·canonical이 절대 URL로 만들어지게 하는 기준점입니다.

```ts
// app/layout.tsx
export const metadata: Metadata = {
  metadataBase: new URL(process.env.NEXT_PUBLIC_BASE_URL ?? 'https://cookeat.app'),
  title: { default: 'Cookeat', template: '%s | Cookeat' },
  description: '레시피 공유와 식재료 마켓을 한곳에서, Cookeat',
};
```

### 2-3. `sitemap.ts` · `robots.ts`

`app/sitemap.ts`와 `app/robots.ts`를 추가하면 `/sitemap.xml`, `/robots.txt`가 자동 생성됩니다(현재 sitemap은 404).

```ts
// app/sitemap.ts
import type { MetadataRoute } from 'next';

export default async function sitemap(): Promise<MetadataRoute.Sitemap> {
  const base = process.env.NEXT_PUBLIC_BASE_URL ?? 'https://cookeat.app';
  // 레시피·상품 id 목록을 DB에서 조회
  const recipes = await fetchRecipeSitemapEntries(); // [{ id, updatedAt }]
  const products = await fetchProductSitemapEntries();

  return [
    { url: base, changeFrequency: 'daily', priority: 1 },
    { url: `${base}/recipes`, changeFrequency: 'daily', priority: 0.8 },
    { url: `${base}/shopping`, changeFrequency: 'daily', priority: 0.8 },
    ...recipes.map((r) => ({ url: `${base}/recipes/${r.id}`, lastModified: r.updatedAt })),
    ...products.map((p) => ({ url: `${base}/shopping/${p.id}`, lastModified: p.updatedAt })),
  ];
}
```

```ts
// app/robots.ts
import type { MetadataRoute } from 'next';

export default function robots(): MetadataRoute.Robots {
  const base = process.env.NEXT_PUBLIC_BASE_URL ?? 'https://cookeat.app';
  return {
    rules: { userAgent: '*', allow: '/', disallow: ['/admin', '/seller', '/api'] },
    sitemap: `${base}/sitemap.xml`,
  };
}
```

### 2-4. 목록 SSR화

`/recipes`와 `/shopping`이 첫 HTML에 콘텐츠를 실으려면, 데이터 fetch를 서버컴포넌트로 끌어올리고 인터랙션(필터·정렬 UI)만 클라이언트 자식으로 남깁니다.

```tsx
// app/(main)/recipes/page.tsx  — 서버컴포넌트
export default async function RecipesPage({
  searchParams,
}: { searchParams: Promise<{ sort?: string; category?: string }> }) {
  const sp = await searchParams;
  const recipes = await fetchRecipeList(sp);   // 서버에서 직접 조회 (api.get 아님)

  return (
    <RecipeListView
      initialRecipes={recipes}   // 첫 HTML에 콘텐츠가 실린다
      initialSort={sp.sort}
    />
  );
}
```

`RecipeListView`는 `'use client'`로 두되, `initialRecipes`를 props로 받아 첫 렌더에 그대로 그리고 필터를 바꿀 때만 클라이언트에서 다시 fetch하면 됩니다. 이렇게 하면 크롤러는 콘텐츠가 채워진 HTML을 받고, 사용자는 인터랙션을 그대로 누립니다. `useShoppingFilter` 쪽도 같은 방식으로 초기 데이터를 서버에서 주입해주세요.

## 3. [필수] `fetchInquiries` 잔류 정리

`load`로 통일하면 됩니다. `fetchInquiries`를 지우고, `sendReply`에서 재조회가 필요하면 `load`를 호출하도록 바꿉니다.

```tsx
// app/admin/support/inquiry/page.tsx
useEffect(() => {
  let cancelled = false;
  async function load() {
    try {
      const { data } = await api.get('/admin/inquiries', { params: { limit: 100 } });
      if (!cancelled) setInquiries(data.inquiries ?? []);
    } catch (err) {
      console.error(err);
      if (!cancelled) setError(true);
    } finally {
      if (!cancelled) setLoading(false);
    }
  }
  load();
  loadRef.current = load;        // sendReply에서 재사용
  return () => { cancelled = true; };
}, []);

// sendReply 안에서
await api.post(`/admin/inquiries/${selectedId}/reply`, { content: text });
setInputText('');
await loadRef.current?.();       // fetchInquiries() 대신
```

`loadRef`는 `const loadRef = useRef<() => Promise<void>>()` 한 줄이면 됩니다. 굳이 ref가 부담스러우면 `load`를 `useCallback`으로 컴포넌트 스코프에 올려 `useEffect`와 `sendReply` 양쪽에서 같은 함수를 쓰는 방법도 괜찮고요. 핵심은 **cancellation이 있는 한 벌만 남기는 것**입니다.

## 4. [제안] `ensureSettlements` 원자화

`settlements/db.ts:242-253`의 조회-후-insert를 `upsert`로 바꾸면 동시 호출에서도 누락이 안 생깁니다.

```ts
const { data: row, error } = await supabaseAdmin
  .from('settlements')
  .upsert(settlementPayload, { onConflict: 'seller_id,settlement_week' })  // 실제 유니크 키로
  .select('settlement_id')
  .single();

if (error) throw error;
const settlementId = row.settlement_id;
```

`onConflict`에는 그 테이블의 실제 유니크 제약 컬럼을 넣어주세요. 이렇게 하면 `23505`를 `continue`로 흡수하던 분기 자체가 사라지고, 같은 주차를 두 요청이 동시에 만들어도 한 행으로 수렴합니다. 더 강하게 묶고 싶다면 정산 생성 구간 전체를 `pg_advisory_xact_lock(seller_id)`로 감싸는 방법도 있는데, 우선은 `upsert`만으로도 충분하다고 봅니다.
