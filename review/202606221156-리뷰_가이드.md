## 1. [필수·보안] `sellers` 테이블 anon 접근 차단

### 왜 닫아도 되는가

상품 상세에서 판매자 정보를 읽는 `getProductDetail`(`lib/products.ts`)이 `supabaseAdmin`(service-role)으로 조회하고 있는데, service-role은 RLS를 우회하니 **anon(브라우저 publishable 키)이 sellers를 직접 못 읽게 막아도 상세 화면은 그대로 잘 뜹니다.** anon 쪽은 닫아버리고, 화면에 필요한 판매자 정보는 지금처럼 서버 라우트가 책임지는 구조로 가면 됩니다.

### 방법 — RLS로 anon SELECT 차단

Supabase SQL Editor에서 `sellers`에 RLS를 켜고, 공개 읽기 정책이 있다면 지웁니다.

```sql
-- 1) RLS 활성화 (이미 켜져 있으면 생략)
alter table public.sellers enable row level security;

-- 2) anon/public 에게 열려 있던 SELECT 정책이 있으면 제거
--    (정책 이름은 대시보드 Authentication > Policies 에서 확인)
drop policy if exists "sellers anon read" on public.sellers;
drop policy if exists "Enable read access for all users" on public.sellers;

-- 3) (선택) 본인 판매자 행만 읽게 하려면 — 로그인 판매자 본인 조회용
create policy "sellers self read"
on public.sellers for select
to authenticated
using ( user_id = auth.uid() );   -- 컬럼명은 실제 스키마에 맞게(예: auth_id)
```

핵심은 **anon에게는 SELECT 정책을 하나도 주지 않는 것**입니다. RLS가 켜진 테이블은 정책이 없으면 기본적으로 막히니까요.

### 9차 무한재귀(42P17)를 피하면서 막기

지난번 sellers 정책에서 무한재귀가 났던 건 정책 조건이 다시 같은 테이블(또는 서로 참조하는 테이블)을 조회하면서 순환이 생긴 탓이었습니다. 위 "self read"처럼 **`auth.uid()`와 자기 컬럼만 비교**하는 단순한 조건은 다른 테이블을 타지 않아서 재귀가 생길 일이 없습니다. 관리자 조회가 필요하더라도 정책 안에서 `users` 역할을 조회하지 말고 그 경로마저 service-role 서버 라우트(`requireAdmin`)로 처리하면, 정책 자체는 계속 단순하게 둘 수 있습니다.

### 확인 방법

정책을 적용한 뒤 비로그인 상태에서 직접 찔러서 0행이 나오는지 보면 됩니다.

```bash
curl "$NEXT_PUBLIC_SUPABASE_URL/rest/v1/sellers?select=*" \
  -H "apikey: <publishable-key>" -H "Authorization: Bearer <publishable-key>"
# 기대: []  (지금은 사업자번호·계좌가 그대로 나옵니다)
```

그다음 상품 상세 `/shopping/4`가 여전히 200으로 잘 뜨는지도 한 번 더 봐주세요. 판매자 정보는 서버가 읽으니 정상이어야 합니다.

## 2. (참고) 상품 상세 404는 해결됐습니다

`getProductDetail`의 조인을 `categories`로 바꾼 뒤로 `/shopping/4`가 200으로 잘 뜨고 있어서, 여기는 더 손볼 게 없습니다. 다만 존재하지 않는 조인이나 컬럼 하나가 쿼리를 통째로 깨뜨리는 패턴이라, 레시피 상세에도 같은 게 숨어 있진 않은지 한 번 점검해 두면 같은 종류의 404를 미리 막을 수 있습니다.
