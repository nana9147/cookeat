## 1. [필수] 정산 상세가 항상 SET-001을 보여주는 문제

`/seller/settlement/[id]/page.tsx`가 라우트 파라미터 `id`를 안 받고 mock에서 늘 같은 정산만 꺼내고 있습니다. 그래서 어떤 행을 눌러도 같은 화면이 뜹니다.

Next.js 동적 라우트는 `params`로 id를 넘겨줍니다. 이 프로젝트는 Next 16이라 `params`가 Promise입니다.

```tsx
// app/seller/settlement/[id]/page.tsx
export default async function SettlementDetailPage({
  params,
}: {
  params: Promise<{ id: string }>;
}) {
  const { id } = await params;                 // ← 라우트의 id를 읽는다
  const settlement = await getSettlement(id);  // id로 해당 정산 조회(실 API 또는 mock에서 find)
  if (!settlement) notFound();
  return <SettlementDetail data={settlement} />;
}
```
아직 mock 단계라면 `MOCK.find((s) => s.id === id)`로 id에 맞는 항목만 꺼내도 "다른 정산을 누르면 다른 화면"이 됩니다. 목록(Table)에서 상세로 가는 링크가 `/seller/settlement/${row.id}`로 제대로 이어졌는지도 같이 봐주세요.

## 2. [필수/주의] proxy.ts 주석과 실제 일치시키기

`decodeJwtPayload`는 서명 검증 없이 base64 디코드만 하고, role이 없으면 위조 가능한 `sb-role` 쿠키를 씁니다. proxy는 "빠른 1차 체크"일 뿐 진짜 보안선은 아닙니다. 실제 게이트는 각 API의 `requireAdmin`/`requireSeller`입니다.

보안은 이미 API에서 닫혀 있으니 코드를 바꿀 필요는 없습니다. 다만 **주석만 사실대로** 고쳐서 다음 사람이 오해하지 않게 해주세요.

```ts
// proxy.ts
// ⚠️ 낙관적(optimistic) 체크일 뿐입니다. JWT 서명을 검증하지 않고,
//    role 은 토큰 또는 sb-role 쿠키에서 읽으므로 위조 가능합니다.
//    실제 인가는 각 API 라우트의 requireAdmin/requireSeller(getUser 재검증)가 담당합니다.
function decodeJwtPayload(token: string) { ... }
```
이 한 줄만 남겨둬도 "proxy에서 막혔으니 API 가드는 생략하자" 같은 실수를 막을 수 있습니다.

## 3. [제안] admin 검색 keyword sanitize (PostgREST 필터 인젝션)

```ts
// 사용자 keyword가 .or() 필터 문자열에 그대로 들어가면
// ',' ')' '.' 같은 문자로 필터가 깨지거나 의도치 않은 조건이 될 수 있다.
function sanitizeKeyword(raw: string) {
  // PostgREST 필터에서 의미를 갖는 문자 제거(또는 이스케이프)
  return raw.replace(/[(),.*:]/g, "").trim();
}

const keyword = sanitizeKeyword(searchParams.get("keyword") ?? "");
// ...
query = query.or(`name.ilike.%${keyword}%,description.ilike.%${keyword}%`);
```
admin 전용이라 위험도는 낮습니다. 그래도 사용자 입력을 쿼리 DSL에 그대로 넣는 자리는 한 겹 걸러주는 습관을 들여두면 좋습니다.

## 4. [제안] seller settlement/reviews 실 DB 연동

admin은 이미 실 API로 옮겼으니, seller도 같은 패턴을 따라가면 됩니다.
- 정산 목록/상세/리뷰를 `GET /api/seller/settlements` 등으로 빼고, 페이지는 서버에서 fetch.
- API route는 `requireSeller(req)` 가드 + 본인(seller) 데이터만 조회하도록 `userId`로 필터.
mock에서 실 API로 넘어갈 때, 이미 만들어 둔 컴포넌트(HeroCard·Table·Detail)는 그대로 두고 데이터 소스만 바꾸면 됩니다. 컴포넌트를 잘 쪼개둔 게 여기서 빛을 봅니다.
