# Cookeat 17차 코드 리뷰 (2026-07-02)

마감(7/8)까지 나흘 남았습니다. 어제(16차) 오전 이후 develop에 커밋이 53개 올라왔는데, 어드민 쿠폰 관리 페이지 신설, 판매자 상품관리 확장(엑셀 일괄등록·재고 알림·정렬·정책 템플릿), 쿠폰 방식 전환(코드 입력에서 계정 지급으로), 카카오·토스 결제 손질이 핵심이었습니다. 이번엔 결제와 쿠폰 흐름을 특히 깊게 따라가 봤습니다.

순서는 지난 반영 현황, E2E 결과, 결제·쿠폰 흐름 정밀 점검, 조원별, 팀 공통 순입니다.

## 3. 결제·쿠폰 흐름 정밀 점검

### [필수] 쿠폰 방식 전환이 서버만 되고 프론트가 안 따라와, 쿠폰이 주문에 전혀 안 붙음

커밋 `ecf02cf`("계정 지급 방식(userCouponId)으로 전환")이 실제로 건드린 파일은 **`app/api/order/route.ts` 하나뿐**입니다. 서버는 이제 body에서 `userCouponId`만 읽는데(`route.ts:23,37`), 정작 결제 프론트는 옛 코드 그대로입니다.

- `Checkout.tsx:56` → `appliedCoupon?.code`(즉 `couponCode`)를 넘기고
- `useCheckoutPayment.ts:69` → `{ couponCode }`를 `/order`로 보냅니다.
- 서버는 `couponCode`를 안 읽으니 `resolvedUserCouponId = null` → `couponDiscount = 0`.

게다가 `CouponModal.tsx:44`는 새 엔드포인트 `/coupons/mine`(userCouponId 반환)가 아니라 옛 `/coupons`를 부르고 있고, `DiscountSection.tsx`의 `AppliedCoupon` 타입엔 `userCouponId` 필드 자체가 없습니다. 그러다 보니 **화면 요약(PaymentSummary)에는 할인이 보이더라도, 서버가 확정하는 `finalAmount`에는 할인이 안 빠져 정가로 결제**되고, 쿠폰의 `used_at`도 안 찍힙니다. 여기에 앞서 E2E에서 본 "쿠폰 목록을 불러오지 못했습니다"까지 겹쳐서, 지금은 쿠폰 기능이 사실상 멈춰 있는 셈입니다.

### [필수] `orders.coupon_id` 컬럼 이름을 바꾼 뒤 옛 소비처들이 안 고쳐짐

커밋 `7551d10`이 orders의 쿠폰 컬럼을 `coupon_id → user_coupon_id`로 바꿨습니다(anon으로 `orders?select=coupon_id`를 찔러 보면 `column orders.coupon_id does not exist` 400이 납니다). 문제는 그 컬럼을 아직 조회하는 곳이 남아 있다는 점입니다.

- `app/api/seller/orders/[orderId]/db.ts:58`가 여전히 `...coupon_id, coupons(code, ...)`를 select → 판매자 주문 상세 API 500.
- 옛 `coupons/route.ts:29-33`, `coupons/validate/route.ts:38-43`도 `orders.coupon_id`로 "이미 사용" 여부를 판정 → 쿼리가 조용히 죽어 중복 사용 방지 필터가 무력화.

컬럼 이름을 바꿀 때는 그 컬럼을 읽는 모든 곳을 전역 검색으로 한 번 훑어 함께 고쳐야 하는데, 그 단계가 빠진 겁니다.

### [필수] 카카오페이 1원 결제 가능 (3회 연속)

토스와 주문 생성은 모범적입니다. `toss/confirm/route.ts`는 서버가 `orders.final_amount`를 다시 읽어 클라 금액과 불일치하면 400을 내고, 토스 승인에도 서버 금액을 보냅니다. `order/route.ts`도 DB에서 실제 가격·재고를 읽어 총액을 재계산하고 `create_order` RPC로 트랜잭션 처리하고요. 잘 짜였습니다.

문제는 카카오에만 이 원칙이 빠져 있다는 점입니다.

- `kakao/ready/route.ts:18,32` — 클라가 보낸 `totalAmount`를 그대로 카카오 `total_amount`로 넘깁니다(주문 재조회·대조 없음).
- `kakao/approve/route.ts:20-28` — 주문을 로드하지만 `user_id, status`만 select하고, 카카오 승인 금액과 `order.final_amount`를 비교하지 않은 채 결제완료 처리합니다.

그래서 악성 클라가 `/payment/kakao/ready`를 `orderId` + `totalAmount: 1`로 직접 호출하면 카카오 1원 결제가 열리고, `/approve`도 그대로 통과해 5만 원 주문이 1원으로 완료됩니다. 토스에 이미 들어가 있는 검증(서버가 `order.final_amount`로 재계산·대조)을 카카오에도 그대로 옮겨 오면 됩니다.

### [제안] `coupons` 테이블 anon 노출

anon(publishable) 키로 `GET /rest/v1/coupons`를 해보면 `WELCOME10(rate 10, 최소 20000)`·`SUMMER10(fixed 5000, 최소 25000)`이 코드·할인값까지 그대로 읽힙니다. 쿠폰 코드가 공개되면 누구나 가져다 쓸 수 있으니, RLS로 anon SELECT를 막고 필요한 조회는 서버 라우트를 거치게 하는 편이 좋습니다.

### 칭찬하고 싶은 부분

- 토스·주문 생성의 서버측 금액·재고·포인트 재계산은 정확히 원칙대로 가고 있습니다.
- 정산 자동생성을 JS 인메모리 집계에서 SQL RPC(`ensure_seller_settlements`)로 옮기고 대시보드의 중복 호출을 제거한 건 눈에 띄는 성능 개선입니다.
- 판매자 상품 일괄 상태변경도 `.eq('seller_id')`로 소유권을 확인한 뒤 단일 UPDATE로 처리해 안전하고 효율적입니다.
- 어드민 쿠폰 API 3종이 전부 `requireAdmin`을 통과시키고 사용된 쿠폰 삭제를 막는 등, 인가·검증이 탄탄합니다.

## 4. 조원별

### 엄인호 (djsy01) — 결제·쿠폰 프론트/주문 라우트
잘한 점: `order/route.ts`의 서버 금액·재고·쿠폰·포인트 재계산(53–168행), `req.json()` 파싱 실패를 500이 아니라 400으로 처리한 점이 좋습니다. 개선할 점은 두 가지입니다. 하나는 체크아웃 프론트가 새 `userCouponId` 흐름을 안 따라가 쿠폰이 주문에 안 붙는 부분(`useCheckoutPayment.ts:69`, `CouponModal.tsx:44`, `DiscountSection.tsx`), 다른 하나는 카카오 금액 서버검증이 아직 없는 부분(`kakao/ready`·`approve`)입니다.

### 최유종 (jjong0) — 어드민 쿠폰 API·타입·서버 전환
잘한 점: 어드민 쿠폰 API 3종의 `requireAdmin` 인가와 입력 검증, 사용된 쿠폰 삭제 방지, 발급 최대수량 체크, `/coupons/mine` 신설이 꼼꼼합니다. 개선할 점은 두 가지입니다. `orders.coupon_id → user_coupon_id` rename(`7551d10`) 시 소비처(`seller/orders/[orderId]/db.ts:58`, 옛 `/coupons`·`/validate`)를 함께 못 고쳐 판매자 주문상세가 500이 나는 부분, 그리고 서버만 `userCouponId`로 전환하고 프론트 연결이 미완인 부분(`ecf02cf`가 order 라우트 1파일만 건드림)입니다.

### 추유나 (nana9147) — 판매자 상품관리·정산 리팩토링
잘한 점: 정산 RPC 전환과 중복 호출 제거, 일괄 상태변경의 소유권 검증에 이은 단일 UPDATE, 재고 부족 알림·엑셀 export/import·상태 요약 카드까지 기능 폭이 넓고 인가도 일관됩니다. 개선할 점은 엑셀 일괄등록이 최대 500행을 for 루프 안에서 한 건씩 insert해 최악의 경우 500번을 왕복한다는 점입니다(검증 통과분을 모아 batch insert 하는 걸 권합니다). 입력 검증도 `!row.price` 수준이라 음수나 문자열 가격이 들어갈 수 있고요. 참고로 본인 파일 `seller/orders/[orderId]/db.ts:58`이 다른 분의 컬럼 rename 때문에 깨진 상태라, 이 부분은 팀에서 함께 봐야 합니다.

## 5. 팀 공통

- 이번 [필수] 세 개(쿠폰 프론트 미연결 / `coupon_id` rename 소비처 / 카카오 금액검증)는 모두 **"한쪽만 바꾸고 반대쪽을 안 맞춘"** 형태입니다. 스키마 컬럼명이나 API 계약을 바꿀 때, 그 이름을 쓰는 곳을 전역 검색으로 한 번 훑고 프론트–서버가 주고받는 필드명(`couponCode` vs `userCouponId`)을 먼저 합의해 두면 마감 막판 사고를 크게 줄일 수 있습니다.
- **결제는 서버를 신뢰의 기준으로 둬야 합니다.** 토스에서 이미 잘 하고 있는 "서버가 주문 금액을 다시 읽어 대조" 원칙을 카카오에도 그대로 적용하면 됩니다. 클라가 보낸 금액은 참고용일 뿐이라는 관점만 카카오로 옮겨 오면 1원 결제는 닫힙니다.
- `coupons`처럼 민감하거나 비공개인 테이블은 anon RLS를 기본 차단으로 두고, 공개가 필요한 것만 열어 주세요.

우선순위는 카카오 1원 결제, 쿠폰 프론트 연결(+`coupon_id` 소비처), coupons RLS 순으로 보면 됩니다. 해결 방법은 가이드에 코드로 적어 뒀습니다. 화이팅입니다.
