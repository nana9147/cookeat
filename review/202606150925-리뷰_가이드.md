## 1. [필수] 빌드 살리기 — `useKakaoPostcode` 전역 타입 (한 줄)

`hooks/user/useKakaoPostcode.ts`의 전역 선언이 `Window.kakao`를 **필수**로 잡아서, 다른 곳에서 병합되는 선언과 modifier가 안 맞습니다(TS2687). `?`(선택) 하나만 붙이면 풀립니다.

```ts
// before
declare global {
  interface Window {
    kakao: {        // 필수
      maps: { ... };
    };
  }
}

// after
declare global {
  interface Window {
    kakao?: {       // 선택 — SDK 로드 전에는 없을 수도 있으니 이게 실제로도 맞습니다
      maps: { ... };
    };
  }
}
```

왜 `?`가 맞냐면, Kakao SDK 스크립트가 로드되기 전에는 `window.kakao`가 실제로 `undefined`입니다. 선택으로 두는 게 타입상으로도, 런타임상으로도 정확합니다.

더 깔끔하게 가려면, 이런 전역 선언은 컴포넌트 파일이 아니라 **한 곳**에 모으세요. `types/global.d.ts`를 하나 만들어 거기에만 `Window` 확장을 두면, 앞으로 누가 또 전역을 선언해 충돌 낼 일이 없어집니다.

```ts
// types/global.d.ts
export {};
declare global {
  interface Window {
    kakao?: { maps: { /* ... */ } };
  }
}
```

고친 뒤 `npx tsc --noEmit && npm run build`가 통과하는지 꼭 확인하고 머지하세요.

## 2. [필수] 셀러 화면 가드 — admin에서 한 걸 그대로

admin은 이미 잘 막아 뒀습니다(`app/admin/layout.tsx:18`의 `AdminAuthGuard`). seller도 같은 방식이면 됩니다. `app/seller/layout.tsx`가 지금은 헤더·사이드바만 그립니다. 여기서 로그인·role을 한 번 확인하세요.

```tsx
// app/seller/layout.tsx — admin 패턴을 그대로
export default function SellerLayout({ children }: { children: React.ReactNode }) {
  return (
    <SellerAuthGuard>      {/* admin의 AdminAuthGuard와 같은 모양 */}
      <SellerHeader />
      <SellerSidebar />
      <main>{children}</main>
    </SellerAuthGuard>
  );
}
```

`SellerAuthGuard`는 `AdminAuthGuard`를 거의 복사해서, 확인하는 role만 `seller`로 바꾸면 됩니다. 왜 화면에서도 막아야 하냐면, API는 `requireSeller`로 막혀 있어도 **화면이 열리면 사용자에겐 "되는 것처럼" 보이고** 빈 폼이나 에러만 노출됩니다. 비로그인 사용자가 `/seller/products/new`에 들어와 폼을 다 채운 뒤 저장 단계에서야 막히면 경험이 나쁩니다.

## 3. [필수] 결제 금액을 서버가 다시 계산하게

지금은 클라이언트가 `FINAL_AMOUNT = 35410`을 정하고, 서버는 받은 `amount`를 그대로 PG에 넘깁니다. 사용자가 요청 본문을 바꾸면 금액이 위조됩니다. 흐름을 이렇게 바꾸세요.

**(1) 주문을 먼저 DB에 만든다 (서버가 금액 확정).**

```ts
// app/api/order/route.ts (새로, 혹은 체크아웃 시작 라우트)
const authed = await requireAuth(req);
const { items } = await req.json();   // 클라는 "무엇을 몇 개" 만 보냄. 금액은 안 보냄

// 서버가 상품 테이블에서 진짜 가격을 읽어 합계를 계산
const products = await supabaseAdmin
  .from('products').select('id, price').in('id', items.map(i => i.productId));
const amount = items.reduce((sum, it) => {
  const p = products.find(p => p.id === it.productId);
  return sum + p.price * it.quantity;
}, 0);

const { data: order } = await supabaseAdmin.from('orders')
  .insert({ user_id: authed.userId, amount, status: 'pending' })
  .select().single();

return Response.json({ orderId: order.id, amount });   // 서버가 정한 금액을 돌려줌
```

**(2) 결제 확정 때 서버가 주문에서 금액을 읽어 대조한다.**

```ts
// app/api/payment/toss/confirm/route.ts
const authed = await requireAuth(req);
const { paymentKey, orderId, amount } = await req.json();

// 클라가 보낸 amount를 믿지 말고, 주문 DB의 진짜 금액과 대조
const { data: order } = await supabaseAdmin
  .from('orders').select('amount, user_id, status')
  .eq('id', orderId).single();

if (!order || order.user_id !== authed.userId) return new Response('Forbidden', { status: 403 });
if (order.amount !== amount) return new Response('Amount mismatch', { status: 400 });
if (order.status === 'paid') return Response.json(order);  // 중복 승인 방지(idempotent)

// 여기서야 토스 confirm 호출 (서버 금액 기준)
// 성공하면 주문을 'paid'로 확정
await supabaseAdmin.from('orders').update({ status: 'paid' }).eq('id', orderId);
```

핵심은 **"금액의 진짜 출처는 서버의 주문 테이블 하나뿐"** 으로 만드는 겁니다. 클라가 보낸 금액은 참고용일 뿐, 매번 DB와 대조합니다. `status === 'paid'` 체크는 같은 결제가 두 번 들어와도 한 번만 처리하게 해줍니다(idempotent).

그리고 `userId: 'test_user'`(`useCheckoutPayment.ts:35`, `Complete.tsx:30`)는 이제 지워주세요. 서버가 토큰으로 사용자를 알기 때문에 클라가 보낼 필요가 없고, 남아 있으면 읽는 사람이 헷갈립니다.

## 4. [제안] 공통 컴포넌트/훅을 admin도 쓰게 (사일로 풀기)

추유나님이 만든 `StatusBadge`·`useFilter`·`usePagination`·`StatusCards`가 지금은 `app/seller/` 안에만 있습니다. 어드민은 같은 일을 인라인으로 다시 합니다. 한곳으로 올려 둘이 같이 쓰면, 색이나 정책을 바꿀 때 한 곳만 고치면 됩니다.

**(1) 옮기기.** `app/seller/components/StatusBadge.tsx` → `components/common/StatusBadge.tsx`. 훅들도 `hooks/common/`으로.

**(2) seller import 경로만 수정.**

```ts
// before
import StatusBadge from '../components/StatusBadge';
// after
import StatusBadge from '@/components/common/StatusBadge';
```

**(3) admin의 인라인 배지를 교체.** `app/admin/members/page.tsx:76-84`의 `gradeBadge`/`statusBadge` 리터럴을 지우고 공통 `StatusBadge`로 바꿉니다. `members/page.tsx:87-108`의 `useState` + 인라인 `.filter()`도 `useFilter`로 갈아끼웁니다.

이렇게 하면 admin과 seller의 표·필터·배지가 한 코드에서 나옵니다. 리뷰 5장에서 말한 "한 사람처럼"이 코드로 드러나는 가장 빠른 길입니다.

## 5. [제안] CI 빌드 게이트 하나 — 빨간불 5회 패턴 끝내기

빌드가 다섯 번째 깨진 채로 머지됐습니다. 사람이 매번 기억하긴 어려우니 기계가 막게 하면 됩니다. `.github/workflows/ci.yml`을 하나 만드세요.

```yaml
name: CI
on:
  pull_request:
    branches: [develop]
jobs:
  build:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-node@v4
        with:
          node-version: 20
      - run: npm ci
      - run: npx tsc --noEmit
      - run: npm run build
```

이러면 타입이 깨진 PR은 머지 버튼이 막힙니다. develop을 보호 브랜치로 두고 이 체크를 필수(required)로 걸면, "머지 전에 빌드 한 번"을 사람이 기억하지 않아도 됩니다. 더해서 커밋 메시지 컨벤션(`feat: ` 콜론 뒤 공백)도 README나 PR 템플릿에 한 줄 적어두면, 셋의 히스토리가 한 사람이 쓴 것처럼 정렬됩니다.
