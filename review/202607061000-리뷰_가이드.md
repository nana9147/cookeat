## 1. [제안] 레시피 수정을 원자적으로 — 중간 실패 시 데이터 손실 막기

### 지금 무슨 일이 벌어지나

`app/api/recipes/db.ts`의 `updateRecipe`는 이렇게 흘러갑니다.

```ts
// 1) 기존 조리순서 전부 삭제
await supabaseAdmin.from('recipe_steps').delete().eq('recipe_id', recipeId);
// 2) 새 조리순서 삽입   ← 여기서 에러가 나면?
await supabaseAdmin.from('recipe_steps').insert(stepRows);
```

1번(삭제)은 성공했는데 2번(삽입)이 실패하면, `catch` 블록은 업로드한 이미지만 지우고 끝납니다.
방금 지워버린 조리순서는 되돌아오지 않아서, **레시피는 남았는데 조리순서가 0개인 깨진 레시피**가 됩니다.
재료(`recipe_ingredients`)도 똑같은 구조라 같은 위험이 있습니다.

### 가장 안전한 해법 — RPC 트랜잭션 (주문에서 이미 해봤죠)

주문 생성 때 `create_order` RPC로 "여러 테이블 변경을 전부 성공 아니면 전부 롤백"으로 묶었던 것과 같은 방식입니다.
Supabase SQL 편집기에서 함수 하나를 만들고, 라우트에서는 그 함수만 호출하면 됩니다.

```sql
create or replace function update_recipe_tx(
  p_recipe_id   bigint,
  p_recipe      jsonb,   -- title, difficulty, cooking_time, ...
  p_steps       jsonb,   -- [{ step_order, title, description, tip, image }, ...]
  p_ingredients jsonb    -- [{ ingredient_id, product_id, name, amount, unit }, ...]
) returns void
language plpgsql
as $$
begin
  update recipes set
    recipe_category_id = (p_recipe->>'recipe_category_id')::int,
    title       = p_recipe->>'title',
    difficulty  = p_recipe->>'difficulty',
    cooking_time= (p_recipe->>'cooking_time')::int,
    servings    = (p_recipe->>'servings')::int,
    description = p_recipe->>'description',
    thumbnail   = p_recipe->>'thumbnail'
  where recipe_id = p_recipe_id;

  delete from recipe_steps where recipe_id = p_recipe_id;
  insert into recipe_steps (recipe_id, step_order, title, description, tip, image)
  select p_recipe_id, (s->>'step_order')::int, s->>'title', s->>'description', s->>'tip', s->>'image'
  from jsonb_array_elements(p_steps) s;

  delete from recipe_ingredients where recipe_id = p_recipe_id;
  insert into recipe_ingredients (recipe_id, ingredient_id, product_id, name, amount, unit)
  select p_recipe_id, (i->>'ingredient_id')::int, (i->>'product_id')::int, i->>'name',
         (i->>'amount')::numeric, i->>'unit'
  from jsonb_array_elements(p_ingredients) i;
end;
$$;
```

plpgsql 함수는 통째로 한 트랜잭션이라, 중간에 어느 한 줄이라도 실패하면 앞의 삭제까지 자동으로 되돌아갑니다.
이미지 업로드만 함수 밖(라우트)에서 먼저 처리하고, URL 문자열을 `p_steps`에 담아 넘기면 됩니다.

### 시간이 없다면 — 순서만 바꾸는 임시 방편

RPC까지 만들 여유가 없으면, **"새로 넣고 나서 옛것을 지우는" 순서**로만 바꿔도 최악(스텝 0개)은 피합니다.
단 이건 완전한 해결은 아니라(옛것 삭제가 실패하면 중복이 남을 수 있음), 발표 후엔 위 RPC로 정리하는 걸 권합니다.

작성 쪽(`createRecipe`)의 `catch`는 레시피 행을 지워 스텝·재료가 FK로 함께 지워지는 걸 기대하는데,
스키마에 그 CASCADE가 실제로 걸려 있는지 한 번 확인해두세요.

```sql
-- recipe_steps, recipe_ingredients의 recipe_id FK에 ON DELETE CASCADE가 있는지 확인
select tc.constraint_name, rc.delete_rule
from information_schema.referential_constraints rc
join information_schema.table_constraints tc using (constraint_name)
where tc.table_name in ('recipe_steps', 'recipe_ingredients');
```

`delete_rule`이 `CASCADE`면 지금 로직이 안전하고, `NO ACTION`이면 보상 삭제가 실패해 고아 데이터가 남습니다.

## 2. [제안] eslint 빨간불 정리 + CI lint 여부 결정

### 지금 상태

`npm run lint`를 돌리면 학생 소스에서 39개 에러가 나고, 대부분(35건)이 이 규칙입니다.

```
react-hooks/set-state-in-effect
  → useEffect 안에서 곧바로 setState 하는 패턴을 새 규칙이 에러로 올림
```

대표 예시(`hooks/usePagination.ts:8`):

```ts
useEffect(() => {
  setCurrentPage(1);        // ← 여기가 걸림
}, [items.length]);
```

### 왜 규칙이 이걸 싫어하나

useEffect 안에서 바로 setState 하면, 렌더 → effect → setState → 다시 렌더가 연쇄로 일어나
한 번 그릴 걸 두 번 그리게 됩니다. React 공식 문서도 "렌더 중에 계산할 수 있으면 effect 대신 그렇게 하라"고 안내합니다.

### 어떻게 고치나 — 두 가지 방향

**(a) "값이 바뀌면 초기화" 류는 렌더 중 파생값으로.** 페이지네이션의 "목록이 바뀌면 1페이지로"는
effect 대신 "현재 페이지가 총 페이지를 넘으면 조정"으로 바꿀 수 있습니다.

```ts
const totalPages = Math.max(1, Math.ceil(items.length / pageSize));
const page = Math.min(currentPage, totalPages);   // 렌더 중 보정 — setState 불필요
```

**(b) 데이터 로딩 후 setState 하는 건 규칙 대상이 아닙니다.** `fetch → setData` 같은 비동기 뒤의
setState는 이 규칙에 걸리지 않으니 그대로 둬도 됩니다. 지금 걸린 35건 중 상당수는 (a)처럼 "동기 초기화" 패턴이라,
그 몇 개만 손봐도 숫자가 크게 줍니다.

**CI에 넣을지 결정**: 발표 전이라 전부 고치기 부담되면, 우선 `next.config`나 CI에서 lint를 "경고까지는 통과,
에러는 차단"으로 두고 대표 몇 개만 정리하는 절충도 좋습니다. 취업 레포 관점에선 `npm run lint`가 초록불인 게
가장 깔끔하니, 여유가 되는 만큼 줄여두세요.

## 3. [제안] 데모 이미지 시드

레시피 상세·수정, 상품 목록에 600×400 회색 플레이스홀더가 그대로 보입니다. 발표 첫인상을 좌우하는 부분이라
대표 레시피·상품 몇 개만이라도 실제 사진으로 채워두면 화면이 확 살아납니다.

  해당 레시피의 `thumbnail`을 그 public URL로 업데이트하면 됩니다.

## 4. [제안] 판매 통계 고객 집계 쿼리 좁히기

`app/api/seller/statistics/db.ts`의 신규/재구매 고객 계산이 판매자의 `order_items` 전량을 다시 읽어
"고객별 첫 주문일"을 구합니다. 지금 규모에선 문제없지만, 필요한 건 `user_id`별 최소 주문일뿐이라
나중엔 이렇게 좁히면 좋습니다.

```sql
-- 예: 판매자의 고객별 첫 주문일만 반환하는 RPC
create or replace function seller_customer_first_order(p_seller_id bigint)
returns table(user_id bigint, first_order_at timestamptz)
language sql stable as $$
  select o.user_id, min(o.created_at)
  from order_items oi
  join orders o on o.order_id = oi.order_id
  where oi.seller_id = p_seller_id
  group by o.user_id;
$$;
```

전건을 앱 메모리로 끌고 와 JS로 훑는 대신, DB가 집계해 필요한 것만 돌려주니 데이터가 커져도 가볍습니다.
성능 항목이라 급하지 않습니다 — 발표 후 정리해도 됩니다.
