# 프론트엔드 테스트 가이드 — Cookeat

작성일: 2026-06-30

이번 글은 우리 프로젝트에 테스트를 어떻게 들이면 좋을지 정리한 안내입니다. 개념은 어느 조나 똑같이 적용되는 이야기라 공통으로 적었고, 예제만큼은 우리 코드에 맞춰 실제로 돌아가는 것으로 넣어 뒀습니다. 리뷰 폴더에 동작하는 예제를 함께 넣어 뒀으니, 읽고 나서 직접 한 번 돌려보면 가장 빠릅니다.

---

## 1. 왜 테스트를 하는가

지금까지는 기능을 만들면 브라우저를 띄워 직접 클릭해 보고 "되네" 하고 넘어갔을 겁니다. 처음엔 그게 제일 빠릅니다. 그런데 화면이 스무 개를 넘어가고, 한 군데를 고치면 엉뚱한 다른 화면이 같이 깨지기 시작하면 이야기가 달라집니다. 매번 모든 화면을 손으로 다시 눌러볼 수는 없으니까요.

테스트는 그 "손으로 다시 눌러보는 일"을 코드로 적어 두고 기계가 대신 반복하게 만드는 겁니다. 한 번 적어 두면, 코드를 고칠 때마다 명령 한 줄로 "예전에 되던 게 지금도 되는지"를 몇 초 만에 확인할 수 있습니다. 테스트가 주는 건 결국 **고칠 용기**입니다. 리팩토링을 하거나 기능을 갈아끼울 때, 테스트가 초록불이면 "적어도 내가 확인해 둔 동작은 안 깨졌다"는 안전망이 생기거든요. 파이널 프로젝트처럼 여럿이 같은 코드를 동시에 만지는 상황에서는, 이 안전망이 있고 없고가 협업 속도를 꽤 가릅니다.

테스트는 크게 두 종류만 알면 충분합니다. 작은 함수·컴포넌트 하나가 제대로 도는지 보는 **단위 테스트**, 그리고 사용자가 실제로 페이지를 오가며 겪는 흐름이 끊기지 않는지 보는 **E2E 테스트**입니다. 둘은 역할이 다르니 같이 씁니다.

---

## 2. 단위 테스트 — 작은 조각을 못 박는다

### 무엇을, 왜

단위 테스트는 함수 하나, 컴포넌트 하나처럼 작은 단위가 "이 입력을 주면 이 결과가 나와야 한다"를 검증합니다. 특히 **순수 함수**(같은 입력이면 항상 같은 출력, 바깥 상태를 건드리지 않는 함수)는 테스트하기도 쉽고 효과도 큽니다. 가격·날짜 포맷, 거리 계산, 검색어 처리 같은 게 그렇죠. 이런 함수는 화면 곳곳에서 공유되기 때문에 한 군데가 틀어지면 여러 화면이 동시에 어긋납니다. 그러니 여기에 테스트를 깔아 두면 비용 대비 효과가 가장 좋습니다.

### 도구 — Vitest + React Testing Library

우리는 Next.js 16 + React 19 + TypeScript를 쓰니, 같은 Vite 계열이라 설정이 가벼운 **Vitest**를 권합니다. 컴포넌트를 "사용자가 보는 관점"에서 테스트하도록 도와주는 **React Testing Library(RTL)** 를 같이 씁니다.

설정 파일은 세 개면 됩니다. (리뷰 폴더 `review/unit/`에 그대로 넣어 뒀습니다.)

```ts
// vitest.config.ts
import { defineConfig } from "vitest/config";
import react from "@vitejs/plugin-react";
import { resolve } from "node:path";

export default defineConfig({
  plugins: [react()],
  resolve: {
    alias: { "@": resolve(__dirname, ".") }, // 이 조는 @/* → ./* 이므로 루트로 맞춤
    // 앱과 테스트가 서로 다른 React 복사본을 쓰면 "Invalid hook call"이 납니다.
    dedupe: ["react", "react-dom"],
  },
  test: {
    environment: "jsdom",     // 브라우저 DOM을 흉내내는 환경(컴포넌트 렌더용)
    globals: true,            // describe/it/expect를 import 없이 사용
    setupFiles: ["./vitest.setup.ts"],
  },
});
```

```ts
// vitest.setup.ts — toBeInTheDocument 같은 DOM 매처를 더해 줍니다.
import "@testing-library/jest-dom/vitest";
```

```jsonc
// package.json (필요 패키지)
{
  "scripts": { "test": "vitest run", "watch": "vitest" },
  "devDependencies": {
    "vitest": "^4", "jsdom": "^29", "@vitejs/plugin-react": "^6",
    "@testing-library/react": "^16", "@testing-library/dom": "^10",
    "@testing-library/jest-dom": "^6", "@testing-library/user-event": "^14"
  }
}
```

> 한 가지 함정을 미리 적어 둡니다. 앱과 테스트가 **서로 다른 React 복사본**을 쓰면 `Invalid hook call`이라는 에러가 납니다. 본인 프로젝트 안에 직접 설치하면 React가 하나뿐이라 이 문제가 없습니다. 리뷰 폴더의 예제는 앱과 분리돼 있어, `dedupe` 설정과 `.npmrc`(`legacy-peer-deps=true`)로 React를 따로 받지 않고 앱의 한 copy를 쓰도록 맞춰 뒀습니다.

### 우리 코드로 본 예제 — 가격·날짜 포맷과 StatusBadge

상품·주문·정산 화면이 공통으로 쓰는 `formatWon`/`formatDate`(`lib/format.ts`)는 순수 함수라, 한 군데가 틀어지면 여러 화면이 같이 어긋납니다. 그래서 여기에 테스트를 깔아 두는 게 효과가 가장 좋습니다.

```ts
// review/unit/tests/format.test.ts
import { describe, it, expect } from "vitest";
import { formatWon, formatDate } from "@/lib/format";

describe("formatWon", () => {
  it("값이 없으면(null/undefined) 대시(-)로 막는다", () => {
    expect(formatWon(null)).toBe("-");
  });
  it("숫자는 천 단위 콤마와 '원'을 붙인다", () => {
    expect(formatWon(12000)).toBe("12,000원");
  });
  it("0원은 대시가 아니라 '0원'으로 구분한다", () => {
    expect(formatWon(0)).toBe("0원");
  });
});
```

`null`과 `0`을 갈라서 검증한 부분을 눈여겨봐 주세요. `n == null`로 막는 코드는 `0`을 실수로 같이 걸러내기 쉬운데, 그런 경계값을 테스트로 박아 두면 나중에 조건을 잘못 바꿔도 바로 빨간불이 켜집니다.

컴포넌트도 같은 식으로 "이 props를 주면 화면에 무엇이 보이는가"를 검증합니다. 상태에 맞는 색을 입히는 `StatusBadge`가 그렇습니다.

```tsx
// review/unit/tests/StatusBadge.test.tsx
import { render, screen } from "@testing-library/react";
import StatusBadge from "@/components/common/StatusBadge";

it("상태 문자열을 그대로 텍스트로 보여준다", () => {
  render(<StatusBadge status="판매중" />);
  expect(screen.getByText("판매중")).toBeInTheDocument();
});

it("'품절'은 빨강 계열 색 클래스를 입는다", () => {
  render(<StatusBadge status="품절" />);
  expect(screen.getByText("품절").className).toContain("bg-red-100");
});
```

상태별로 색이 갈리는 매핑은 항목이 많아 실수가 끼기 쉬운데, 대표 상태 몇 개를 테스트로 박아 두면 매핑을 잘못 건드렸을 때 바로 드러납니다.

### 돌려보기

```bash
cd review/unit
npm install
npm test
```

지금 이 예제는 8개 테스트가 전부 통과합니다. `npm run watch`로 띄워 두면 파일을 고칠 때마다 관련 테스트만 다시 돌아서, 고치는 즉시 결과를 확인할 수 있습니다.

## 3. E2E 테스트 — 사용자처럼 끝까지 눌러본다

### 무엇을, 왜

단위 테스트가 부품 하나하나를 보는 거라면, E2E(End-to-End) 테스트는 **사용자가 실제로 겪는 흐름 전체**를 봅니다. 진짜 브라우저를 띄워 메인 진입부터 목록·상세, 로그인, 글쓰기까지 사람이 하는 행동을 그대로 따라 하는 거죠. 부품은 다 멀쩡한데 막상 화면을 연결해 보면 버튼이 안 눌리거나 페이지가 끊기는 일이 있는데, 그런 건 단위 테스트로는 못 잡고 E2E라야 잡힙니다.

도구는 **Playwright**를 씁니다. 크롬 같은 실제 브라우저를 코드로 조종해 주는 도구이고, 우리 리뷰 폴더(`review/e2e/`)에 이미 이걸로 만든 E2E가 들어 있습니다.

### E2E 프로젝트를 처음부터 만드는 법

E2E는 보통 앱과 섞지 않고 **별도의 작은 프로젝트**로 둡니다. 앱 코드에 테스트 의존성이 끼지 않아 깔끔하거든요. 우리 `review/e2e/`도 그렇게 만들어진 겁니다. 순서는 이렇습니다.

```bash
mkdir e2e && cd e2e
npm init -y
npm i -D @playwright/test       # Playwright 설치
npx playwright install chromium  # 테스트용 크롬 브라우저 내려받기
```

그다음 설정 파일 하나를 둡니다. 핵심은 `webServer`인데, 테스트를 돌리기 직전에 개발 서버를 자동으로 띄워 주는 부분입니다.

```ts
// playwright.config.ts
import { defineConfig, devices } from "@playwright/test";

export default defineConfig({
  testDir: "./tests",
  use: {
    baseURL: "http://localhost:3300",
    headless: true,                 // 평소엔 브라우저를 숨기고 빠르게(아래 4절 참고)
    screenshot: "only-on-failure",  // 실패한 순간만 화면을 캡처해 둠
  },
  projects: [{ name: "chromium", use: { ...devices["Desktop Chrome"] } }],
  webServer: {
    command: "PORT=3300 npm run dev", // 테스트 직전에 개발 서버를 자동 기동
    url: "http://localhost:3300",
    cwd: "../../",                    // 앱 루트에서 dev 서버를 띄움
    reuseExistingServer: true,        // 이미 떠 있으면 그대로 재사용
    timeout: 120_000,
  },
});
```

테스트 파일은 `tests/` 아래에 둡니다. 한 시나리오는 "어디로 가서 → 무엇을 누르고 → 무엇이 보여야 한다"의 흐름으로 적습니다.

```ts
// tests/smoke.spec.ts
import { test, expect } from "@playwright/test";

test("메인이 정상 응답한다", async ({ page }) => {
  const res = await page.goto("/");
  expect(res?.status()).toBeLessThan(400);
});
```

```bash
npx playwright test            # 전체 실행(설정의 webServer가 dev 서버를 알아서 띄움)
npx playwright show-report     # 끝나고 HTML 리포트 보기
```

### 우리 프로젝트의 E2E

우리 `review/e2e/`에는 위 구조가 이미 갖춰져 있고, 사용자 여정을 따라가는 시나리오를 `review/e2e/scenarios.md`에 살아 있는 문서로 정리해 두었습니다. 레시피 공유와 마켓을 함께 묶은 서비스이니, 일반 사용자 흐름이라면 이런 순서가 자연스럽습니다.

```
메인(/) → 레시피 목록(/recipes) → 레시피 상세(/recipes/{id})
        → 상품 목록(/shopping) → 상품 상세(/shopping/{id}) → 장바구니 → 주문
```

E2E다운 검증은 이런 겁니다. 상세 페이지가 제목·재료와 함께 잘 뜨는지, 장바구니에 담고 주문까지 흐름이 끊기지 않는지, 비로그인으로 마이페이지에 들어가면 막히는지, 이런 건 실제로 페이지를 오갈 때만 드러납니다. 코드로는 이렇게 적습니다.

```ts
// review/e2e/tests/cookeat-daily.spec.ts (발췌)
test("상품 목록에서 첫 상품으로 들어가면 상세가 뜬다", async ({ page }) => {
  await page.goto("/shopping");
  await page.getByRole("link").first().click();
  await expect(page.getByRole("heading", { level: 1 })).toBeVisible();
});
```

상세 페이지(레시피·상품)는 서버에서 렌더되니 로그인 없이도 잘 잡힙니다. 로그인이 필요한 마이페이지·판매자·관리자 흐름은 테스트 전용 세션으로 확인합니다(앱 코드는 건드리지 않습니다).

실행은 이렇게 합니다.

```bash
cd review/e2e
npm install
npx playwright install chromium   # 처음 한 번만
npx playwright test tests/cookeat-daily.spec.ts
```

### 브라우저를 보이게 하고 테스트하기

평소 E2E는 브라우저를 화면에 띄우지 않고(headless) 빠르게 돌립니다. 그런데 "왜 이 테스트가 실패하지?"를 눈으로 보고 싶을 때가 있죠. 그럴 땐 브라우저를 보이게 하면 됩니다.

```bash
npx playwright test --headed         # 실제 브라우저 창을 띄워 동작을 눈으로 보며 실행
npx playwright test --ui             # 타임라인·각 단계 스냅샷을 보는 UI 모드(가장 추천)
npx playwright test --debug          # 한 단계씩 멈춰가며 디버깅(Inspector)
npx playwright test --headed --workers=1 --trace on  # 보이게 + 추적 기록 남기기
```

설정에서 항상 보이게 하려면 `playwright.config.ts`의 `use.headless`를 `false`로 두면 됩니다. 처음 시나리오를 짤 때는 `--ui` 모드로 켜 두고 단계별 스냅샷을 보며 만드는 게 제일 편합니다. 실패가 났을 때는 `show-report`로 열리는 리포트에 실패 직전 스크린샷이 남아 있어, 어디서 어긋났는지 바로 보입니다.

## 4. 면접에 자주 나오는 테스트 질문과 답

테스트는 신입 프론트엔드 면접에서 단골 주제입니다. 자주 나오는 것만 추려 짧게 정리합니다.

**Q. 단위 테스트와 통합·E2E 테스트의 차이는?**
보는 범위가 다릅니다. 단위 테스트는 함수·컴포넌트 같은 작은 조각을 따로 떼어 검증합니다. 통합 테스트는 그 조각들이 함께 맞물려 도는지를 보고, E2E는 실제 브라우저에서 사용자 흐름 전체를 봅니다. 아래로 갈수록 현실에 가깝지만 그만큼 느리고 작성·유지비가 큽니다.

**Q. 테스트 피라미드가 무엇인가요?**
테스트를 어떤 비율로 둘지에 대한 이야기입니다. 빠르고 싼 단위 테스트를 가장 많이 두고, 통합 테스트는 그보다 적게, 느리고 비싼 E2E는 핵심 흐름만 얇게 덮자는 거죠. 거꾸로 E2E만 잔뜩 만들면 느리고 잘 깨져 유지가 안 됩니다.

**Q. React Testing Library는 왜 "구현이 아니라 사용자 관점"으로 테스트하라고 하나요?**
버튼의 내부 state 값 같은 구현 세부에 테스트를 묶으면, 리팩토링만 해도 테스트가 깨집니다. 대신 "화면에 이 텍스트가 보인다, 이 버튼을 누르면 이렇게 바뀐다"처럼 사용자가 겪는 결과로 검증하면, 내부를 바꿔도 동작이 같으면 테스트가 통과해 더 튼튼합니다. 그래서 `getByRole`·`getByText`처럼 사용자가 인지하는 방식으로 요소를 찾습니다.

**Q. 좋은 테스트의 조건은?**
빠르게 돌고, 같은 코드면 항상 같은 결과가 나와야 합니다. 거기에 서로 독립적이어야 하고, 실패했을 때 원인이 한눈에 보여야 하고요. 시간·랜덤·네트워크처럼 매번 달라지는 것에 의존하면 "그냥 가끔 실패하는" 플래키(flaky) 테스트가 되니, 그런 건 고정값으로 바꾸거나 모킹합니다.

**Q. 모킹(mocking)은 언제 하나요?**
네트워크 요청, 시간(`Date.now`), 외부 SDK처럼 테스트 대상 바깥의 의존성을 가짜로 바꿔, 내가 검증하려는 로직만 떼어 보고 싶을 때 합니다. 다만 너무 많이 모킹하면 "실제로는 안 되는데 테스트만 통과"하는 함정에 빠지니, 순수 로직은 모킹 없이 그대로 테스트하는 편이 좋습니다.

**Q. 커버리지 100%면 충분한가요?**
아닙니다. 커버리지가 알려주는 건 "코드가 실행됐는지"까지입니다. 정작 "결과가 맞는지"는 거기서 보장되지 않거든요. 숫자를 채우기보다 중요한 로직·엣지 케이스를 제대로 검증하는 게 먼저입니다.

**Q. TDD가 무엇인가요?**
실패하는 테스트를 먼저 쓰고(Red), 통과시키는 최소 코드를 짠 뒤(Green), 정리(Refactor)하는 순서로 개발하는 방식입니다. 항상 이렇게 할 필요는 없지만, 요구사항이 분명한 순수 로직에는 잘 맞습니다.

---

## 5. 정리

작은 순수 함수부터 단위 테스트로 못 박고(우리는 `review/unit/`에 동작하는 예제를 넣어 뒀습니다), 사용자 여정의 핵심 흐름은 E2E로 한 겹 덮는 것 — 이 두 가지면 "고쳐도 안 깨진다"를 확인하며 갈 수 있습니다. 처음부터 많이 만들 필요는 없습니다. 가장 자주 쓰이고 가장 자주 깨지는 곳부터 하나씩 늘려가면 됩니다. 먼저 `review/unit`과 `review/e2e`를 한 번씩 돌려보는 것에서 시작하면 좋겠습니다.
