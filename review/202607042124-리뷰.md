# Cookeat 코드 리뷰 (2026-07-04) — 취업 레포 관점

마감(7/8)까지 나흘, 발표는 다음 주 수요일이니 실제 개발은 화요일까지로 보고 있습니다. 지난 17차 이후로 코드 쪽 [필수]는 크게 정리가 된 상태라, 이번에는 관점을 하나 더 얹어서 봤습니다. 이 레포는 발표로 끝나는 게 아니라 앞으로 이력서에 링크로 걸릴 물건이니, 코드 안쪽 정확성뿐 아니라 "레포를 처음 열었을 때 보이는 것"까지 함께 짚었습니다.

## 한 줄 총평

지난 17차에서 남겨둔 [필수] 네 건이 전부 실측으로 확인됐고, 그중 결제·쿠폰 관련 네 건은 리뷰 당일 오후 한 커밋(`3967df3`, 엄인호)에서 한 번에 정리된 걸 보면 반영 회전이 꽤 빠른 편입니다. 이제 코드 안의 버그보다는 레포를 처음 여는 사람이 받는 인상 쪽에 손볼 게 남아 있습니다.

네 건 모두 `3967df3` 한 커밋에서 처리됐습니다. 쿠폰 재사용 방지 로직과 카카오 결제 검증을 같이 정리한 커밋인데, 결제·쿠폰처럼 서로 얽힌 이슈를 따로따로 땜질하지 않고 한 번에 정리한 점이 좋았습니다.

다만 한 가지는 코드만으로 끝까지 확인이 안 됐습니다. 쿠폰의 `used_at` 마킹이 `create_order` RPC 안에서 이뤄지는 것으로 보이는데, 이 저장소에는 `supabase/migrations`가 비어 있어 RPC 본문을 직접 읽을 방법이 없었습니다. 실제로 결제 완료 시점에 `used_at`이 정확히 찍히는지는 이번에 단정하지 못했습니다. 다음 회차에 실제 주문 하나를 넣어 확인해보면 좋겠습니다.

## 2. 지금 먼저 손볼 세 가지 (취업 레포 관점)

코드 쪽 [필수]가 거의 정리된 지금 시점에는, 아래 세 가지가 오히려 첫인상에 더 크게 영향을 줍니다. 공통으로 챙길 항목은 함께 전달하는 `취업레포-가이드.md`에 정리해뒀고, 여기서는 Cookeat에서 실제로 확인된 것만 짚습니다.

### [필수] README 클론 명령이 개인 fork를 가리키고 있음

`README.md:155`

```md
git clone https://github.com/djsy01/Cookeat.git
```

`git remote -v`로 확인해보면 이 저장소의 실제 origin은 `https://github.com/FRONTENDBOOTCAMP-17th/Cookeat.git`(조직 저장소)입니다. README만 아직 엄인호 님 개인 fork 주소로 남아 있는 건데, 채용담당자가 이 명령을 그대로 복사해서 clone하면 팀 전체가 아니라 개인 fork로 들어가게 됩니다. 조직 저장소 URL로 한 줄만 바꾸면 됩니다.

### [제안] 팀 표에 중도 하차 이력이 그대로 남아 있음

`README.md:199-204`

최유종 님이 팀장으로 재지정된 건 반영이 됐는데, 홍정빈 님이 여전히 "팀원"으로 표에 남아 있습니다. 이 부분은 팀에서 실제로 어떻게 표기하기로 정했는지가 먼저인 것 같습니다. 그대로 두고 각주로 사정을 적어둘지, 표에서 빼고 기여한 커밋만 기록으로 남길지, 팀이 정한 방향이 있나요? 어느 쪽이든 지금처럼 아무 설명 없이 팀원으로만 적혀 있으면 보는 사람 입장에서는 그냥 4인 팀으로 오해하게 됩니다.

README 어디에도 지금 눌러볼 수 있는 배포 링크가 없습니다. Vercel + Supabase 조합이라 배포 자체는 크게 어렵지 않을 텐데, 지금은 코드를 내려받아 직접 띄워야만 서비스를 볼 수 있습니다. 기술 스택 표(README:69-95)도 로고 나열에 가까운데, `docs/api/`에 도메인별 API 명세를 이렇게 잘 정리해둔 걸 보면 인증 흐름이나 결제 흐름 다이어그램 한 장 정도는 어렵지 않게 뽑아낼 수 있어 보입니다. 여기에 홈 화면(`public/screenshots/main/home.webp`)의 인기 레시피 카드가 전부 `600 × 400` 회색 플레이스홀더인 것도 같이 걸립니다. 기능은 이미 다 되어 있으니, 레시피 몇 개에 실제 사진만 채워 넣으면 캡처 인상이 확 달라집니다.

### 엄인호 (djsy01) — 인증·주문/결제·마이페이지

`app/api/order/route.ts:65-227`에서 클라이언트가 보낸 금액을 신뢰하지 않고 상품가·재고·쿠폰·포인트를 서버가 전부 다시 조회한 뒤 `create_order` RPC로 트랜잭션 처리하는 구조는, 그대로 포트폴리오나 면접 답변에 써도 될 만한 완성도입니다. 카카오 1원 결제도 이번에 정면으로 막았고요. 한 가지 참고로 말씀드리면, git 커밋 로그를 보니 같은 분이 `djsy01`과 `Inho`라는 두 이름으로 나뉘어 커밋하고 계십니다(각각 260건·81건). GitHub Contributors 그래프에서는 이게 서로 다른 사람으로 집계되기 때문에, 실제 기여량이 그래프상으로는 적어 보일 수 있습니다. `git config user.name`을 하나로 통일해두시는 걸 권합니다.

## 4. 팀 공통

- **위생 상태는 전반적으로 양호합니다.** `console.log`·`TODO`는 소스 전체에서 0건이었고, `.env.local`은 `.gitignore:35`(`.env*`)로 보호돼 있어 `git ls-files`에도 잡히지 않습니다. `LICENSE.md`(MIT)도 있습니다.
- **폴더·네이밍·커밋 컨벤션은 세 분이 일관되게 따르고 있습니다.** `app/api` 구조, 컴포넌트 위치 규칙(`components/` vs `app/seller/components/`) 모두 `CLAUDE.md`에 적힌 대로 지켜지고 있어 세 분이 각자 작업해도 코드가 낯설지 않습니다.
- **[제안] `npx eslint .`가 42건 에러 + 33건 경고를 냅니다.** 대부분 최신 Next.js가 새로 추가한 `react-hooks/set-state-in-effect` 규칙(예: `hooks/usePagination.ts:8` — `useEffect` 안에서 `setCurrentPage`를 동기 호출)입니다. 지금 당장 버그를 일으키는 건 아니지만, CI(`.github/workflows/ci.yml`)가 `npm ci` → `tsc --noEmit` → `next build`만 게이트하고 lint는 검사하지 않아서 이 상태로 계속 쌓여왔습니다. `npm run lint -- --fix`로 정리하거나, 최소한 CI에 lint 스텝 하나만 추가해도 "빌드·타입·린트 다 초록불"이라는 인상을 줄 수 있습니다.
- **컬럼명·API 계약을 바꿀 때 전역 검색이 습관화되면 좋겠습니다.** 이번 17차의 [필수] 세 건(쿠폰 프론트 미연결, `coupon_id` rename 소비처 누락, 그리고 이번에 함께 해결된 것들)이 전부 "한쪽만 고치고 반대쪽을 안 맞춘" 형태였습니다. 지금은 다 정리됐지만, 스키마 컬럼명이나 프론트-서버 필드명(`couponCode` vs `userCouponId`)을 바꿀 때 이름을 쓰는 곳을 한 번씩 전역 검색하는 습관을 들이면 막판 사고를 더 줄일 수 있습니다.
