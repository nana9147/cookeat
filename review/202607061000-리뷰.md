이번 delta는 커밋 기준 추유나(nana9147) 10, 엄인호(djsy01) 6, 최유종(jjong0) 0입니다.
최유종님은 그동안 관리자 영역을 거의 마무리해 왔으니 이번엔 자연스럽게 손이 덜 간 걸로 보이는데,
혹시 남은 작업 분담을 어떻게 나눴는지 한 번 팀에서 맞춰보면 좋겠습니다.

### 엄인호 (djsy01) — 레시피 작성/수정 신규, 주문 취소·환불 분리

이번의 대형 신규 기능입니다. 레시피 작성(`app/api/recipes/route.ts` POST)과 수정(`[recipeId]/route.ts` PATCH)을
붙였는데, 완성도가 상당히 좋습니다.

**잘한 점**
- **인증·인가를 서버에서 제대로 막았습니다.** 작성은 `requireAuth`, 수정은 `requireAuth` + `updateRecipe`
  안에서 `recipeRow.user_id !== userId → '본인 레시피만 수정할 수 있습니다.'`(db.ts:210)로 소유권을 서버에서
  검증합니다. 화면 쪽 `RecipeEditGuard`(클라)와 합쳐 이중 방어라, 클라 가드를 우회해도 서버가 403으로 막습니다.
  실제로 남의 레시피(6번) 수정 URL로 들어가니 화면은 상세로 튕기고, 서버 PATCH도 막히는 걸 확인했습니다.
- **입력 검증이 꼼꼼합니다.** 제목·카테고리·난이도·조리시간·인분·설명은 `validateRecipeFields`로, 이미지는
  `validateImageFile`(타입 jpg/png/webp·5MB)로, 조리순서는 "1개 이상 + 설명 필수", 재료는 `parseIngredientsMeta`에서
  대분류·용량·단위·재료명 길이까지 막습니다. 초보 단계에서 이 정도 방어는 흔치 않습니다.
- **실패 시 뒷정리(보상 삭제)를 챙겼습니다.** 작성 중간에 에러가 나면 이미 올린 이미지들을 `deleteRecipeImageFile`로
  지우고 레시피 행도 삭제합니다(db.ts:178-182). Storage 누수를 막으려는 습관이 좋습니다.
- 작성 폼(`RecipeWriteForm`)을 수정 화면에서 `initialRecipe`만 넘겨 재사용한 구조도 깔끔합니다.
  실제로 수정 화면에서 "불고기/한식/40분/3인분/보통"이 그대로 프리필됐습니다.

**[제안] 레시피 수정이 원자적이지 않아, 중간 실패 시 조리순서·재료가 통째로 사라질 수 있습니다.**
`updateRecipe`(db.ts:267~288)는 기존 조리순서를 `delete` 한 뒤(:267) 새 것을 `insert`(:273) 합니다. 재료도
`delete`(:276) 후 `insert`(:282)이고요. 그런데 삭제는 성공하고 삽입이 실패하면, `catch`(db.ts:293-296)는 **업로드한
이미지만 정리할 뿐 방금 지운 조리순서를 되돌리지 못합니다.** 결과적으로 레시피는 남았는데 조리순서가 0개인 깨진 상태가
됩니다. 주문 쪽에서 이미 `create_order` RPC로 트랜잭션을 잡아둔 경험이 있으니, 같은 결로 `update_recipe` RPC 하나에
"레시피 수정 + 스텝 교체 + 재료 교체"를 묶으면 전부 성공하거나 전부 롤백되어 안전합니다. 발표까지 시간이 빠듯하면,
최소한 "새 스텝을 먼저 insert 하고, 성공한 뒤에 옛 스텝을 지우는" 순서 조정만으로도 데이터가 사라지는 최악은 피할 수 있습니다.
(작성 쪽 `createRecipe`도 catch에서 레시피 행 삭제로 스텝·재료가 FK CASCADE로 함께 지워진다는 가정에 기대고 있으니,
스키마에 `ON DELETE CASCADE`가 실제로 걸려 있는지 한 번 확인해두면 좋습니다.)

**[사소]** `app/api/orders/route.ts`의 취소·환불 분리(f4b9d81)는 로직 자체는 정확합니다. 다만 취소요청 목록을 구하는
`refund_requests` 조회를 이제 탭과 무관하게 매 요청마다 돌리게 됐는데(예전엔 '취소' 탭에서만), 주문 목록 조회 때마다
쿼리 한 번이 더 붙습니다. 성능상 크지 않으니 지금은 두어도 되고, 신경 쓰이면 '취소' 탭이 아닐 때만 실행하도록 되돌려도 됩니다.

### 추유나 (nana9147) — 판매 통계 신규, 환불 귀책구분·배송비 재계산

이번 delta의 최다 커밋입니다. 판매 통계 페이지(`app/api/seller/statistics/*`)를 새로 붙였고, 환불 처리의
디테일을 여럿 다듬었습니다.

**잘한 점**
- **판매 통계 인가를 `requireSellerContext`로 잡았습니다.** 판매자 본인은 물론 관리자가 `?sellerId=`로 대리 조회하는
  경로까지 한 함수로 처리해, 팀이 정해둔 판매자 API 규칙을 그대로 따랐습니다. user 롤로 `/seller/statistics`에 들어가니
  RoleGuard가 홈으로 돌려보내는 것도 확인했습니다.
- **통계 로직이 성실합니다.** 취소·환불 건은 매출에서 제외(`EXCLUDED_STATUSES`), 카테고리는 부모 카테고리명으로 롤업,
  신규/재구매 고객은 "판매자의 첫 주문일이 기간 시작 전인지"로 판정합니다. 클레임률에 귀책구분(구매자/판매자)까지 나눈 건
  꽤 꼼꼼한 설계입니다.
- 배송비 재계산 버그 수정(2979abd)이 정확합니다. `shippings`가 상품(item) 단위 row라는 걸 파악하고,
  `.limit(1)`로 대표 row 하나만 갱신하던 걸 `order_id + seller_id`의 모든 row로 바꿨습니다. 커밋 메시지에
  "실제 데이터 불일치는 아직 발생하지 않았다"고 정직하게 적은 것도 좋습니다.

**[제안] 신규/재구매 고객 계산이 판매자의 주문 전체를 매번 다시 읽습니다.**
`getSellerStatistics`(db.ts:200-204)는 "각 고객의 첫 주문일"을 구하려고 기간과 무관하게 해당 판매자의
`order_items` 전량을 한 번 더 조회합니다. 판매가 쌓일수록 이 조회가 계속 커지고, 앞의 메인 조회까지 합치면 한 페이지에서
전건 스캔이 두 번입니다. 지금 데이터 규모에선 문제없지만, "고객별 첫 주문일"만 필요하니 나중엔
`min(created_at) group by user_id` 같은 집계 쿼리(또는 RPC)로 좁히는 편이 좋습니다. 급하지 않은 성능 항목입니다.

### 최유종 (jjong0) — 이번 delta 커밋 없음

eslint 정리 등)을 누가 무얼 맡을지 팀에서 정리해두면 좋겠습니다.

## 3. 팀 공통

- **eslint 게이트는 여전히 숙제입니다.** 학생 소스 기준 39개 에러가 남아 있고 대부분(35건)이
  `react-hooks/set-state-in-effect`입니다. Next 16 + 새 react-hooks 규칙이 "useEffect 안에서 곧바로 setState 하는 패턴"을
  에러로 올린 건데, 당장 동작을 깨는 버그는 아니지만 취업용 공개 레포에서 `npm run lint`가 빨간불이면 인상이 좋지 않습니다.
  발표 전에 CI에 lint를 넣을지 말지를 팀에서 한 번 정하고, 넣는다면 대표적인 몇 개(`usePagination`의 `setCurrentPage(1)` 같은)만
  손봐도 숫자가 확 줍니다. (가이드에 패턴 정리해뒀습니다.)
- **데모 이미지 시드**: 레시피 상세·수정, 상품 목록에 600×400 회색 플레이스홀더가 그대로 보입니다. 발표 때 첫인상을 좌우하는 부분이라,
  대표 레시피 몇 개만이라도 실제 사진으로 채워두면 화면이 훨씬 살아납니다.
- 신규 두 기능 모두 서버 인증/인가·입력검증을 팀 공통 패턴(`requireAuth`/`requireSellerContext`/`validators`)으로 맞춰
  들어온 점은 "한 사람이 짠 것처럼" 일관돼서 좋습니다. 이 결을 발표 때 아키텍처 설명 포인트로 써도 됩니다.

## 5. 우리 조 마무리 우선순위 (발표까지 이틀)

1. **[제안] 레시피 수정 원자성** — 스텝/재료 delete→insert 사이 실패 시 데이터 손실. `update_recipe` RPC로 묶거나,
   최소한 "새로 insert 먼저 → 옛것 delete" 순서 조정. 데모에서 깨질 확률은 낮으니 시간 남을 때.
2. **[제안] eslint 빨간불 정리 + CI lint 여부 결정** — 취업 레포 인상. 대표 패턴 몇 개만 손봐도 숫자 급감.
3. **[제안] 데모 이미지 시드** — 레시피/상품 대표 몇 개 실제 사진으로. 발표 첫인상.
4. **[제안] 판매 통계 고객 집계 쿼리 좁히기** — 성능, 급하지 않음.

큰 구멍은 없습니다. 남은 이틀은 새 기능을 더 늘리기보다, 데모 동선(회원가입→레시피 작성→상세→장바구니→결제,
판매자 통계) 한 바퀴를 실제로 돌려보며 끊기는 데 없는지 확인하는 데 쓰는 걸 권합니다. 고생 많았습니다. 화이팅입니다.
