### 1-1. StatusBadge `취소`/`환불` (OrderStatus 타입)

`app/seller/components/StatusBadge.tsx`가 `취소`/`환불` 배지를 쓰는데 `OrderStatus` 타입엔 두 상태가 없습니다. **주문에 취소·환불 상태가 실제로 있다면** 타입에 추가하는 게 맞습니다.

```ts
// types/seller/order.ts
export type OrderStatus =
  | '결제완료'
  | '배송준비중'
  | '배송중'
  | '배송완료'
  | '취소'    // 추가
  | '환불';   // 추가
```

> 만약 취소/환불을 아직 안 쓸 거라면, 반대로 StatusBadge에서 그 두 줄을 빼면 됩니다. **타입과 배지 키가 정확히 일치**하기만 하면 됩니다.

### 1-2. `app/seller/page.tsx` mock과 `Order` 타입 불일치

타입은 `price: string`인데 mock은 숫자고, mock엔 `orderDate`가 있는데 타입엔 없습니다. 둘을 맞춥니다. 화면에서 가격을 `15,900원`처럼 보여주려면 타입을 string으로 두고 mock도 문자열로:

```ts
// types/seller/order.ts
export interface Order {
  id: string;
  customer: string;
  product: string;
  price: string;       // '15,900원' 같은 표시용 문자열
  status: OrderStatus;
  orderDate: string;   // mock이 쓰고 있으니 타입에 추가
}
```

```ts
// app/seller/page.tsx — mock도 문자열로
{ id: 'ORD-2024-001', customer: '김가을', product: '유기농 토마토 500g',
  price: '15,900원', status: '배송준비중', orderDate: '2024-06-09 14:16:22' },
```

> 팁: 화면 표시용(`'15,900원'`)과 계산용(`15900`)은 성격이 다릅니다. 나중에 정렬·합계가 필요하면 숫자 필드를 따로 두고(`price: number`) 화면에서 `price.toLocaleString()`으로 포맷하는 쪽이 깔끔합니다. 지금은 우선 타입만 일치시키면 빌드가 풀립니다.

### 1-4. 재발 방지 — CI에 build 한 줄

머지 전에 빌드가 깨졌는지 자동으로 잡으려면, GitHub Actions에 체크 하나만 걸면 됩니다.

```yaml
# .github/workflows/ci.yml
name: ci
on: { pull_request: { branches: [develop] } }
jobs:
  build:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-node@v4
        with: { node-version: 20 }
      - run: npm ci
      - run: npx tsc --noEmit
```

이러면 "타입 깨진 PR"이 빨간불로 표시돼 머지 전에 보입니다.

## 2. 셀러 화면 가드 (`/seller/*`)

어드민엔 `AdminAuthGuard`가 있는데 셀러 레이아웃엔 없습니다. 같은 방식으로 셀러용 가드를 만들고 레이아웃에 감싸면 됩니다.

```tsx
// app/seller/components/SellerAuthGuard.tsx
'use client';
import { useEffect } from 'react';
import { useRouter } from 'next/navigation';
import { useAuthStore } from '@/store/authStore';

export default function SellerAuthGuard({ children }: { children: React.ReactNode }) {
  const router = useRouter();
  const hydrated = useAuthStore((s) => s._hydrated);
  const token = useAuthStore((s) => s.accessToken);

  useEffect(() => {
    if (hydrated && !token) router.replace('/login');
  }, [hydrated, token, router]);

  if (!hydrated || !token) return null;
  return <>{children}</>;
}
```

```tsx
// app/seller/layout.tsx — 가드로 감싸기
import SellerAuthGuard from './components/SellerAuthGuard';
// ...
export default function SellerLayout({ children }: { children: React.ReactNode }) {
  return (
    <SellerAuthGuard>
      <Header />
      {/* 기존 내용 그대로 */}
    </SellerAuthGuard>
  );
}
```

이러면 비로그인으로 `/seller/products/new`에 들어가면 `/login`으로 보내집니다.

### 더 중요한 것 — 셀러 API에 `requireSeller`

화면 가드는 UX고, 진짜 방어선은 서버입니다. 상품 등록 API를 붙일 때 이렇게:

```ts
// app/api/seller/products/route.ts (예시)
import { requireSeller } from '@/lib/serverAuth';

export async function POST(req: NextRequest) {
  const authed = await requireSeller(req);   // seller 또는 admin만 통과
  if (authed instanceof NextResponse) return authed;
  // authed.userId 로 본인 상품만 등록
}
```

이미 `requireSeller`를 만들어 두셨으니 호출만 하면 됩니다.

## 3. RoleGuard가 role까지 보게 하기

`AdminAuthGuard.tsx`(=RoleGuard)는 지금 토큰만 봅니다. 로그인한 일반 회원도 `/admin`에 들어옵니다. role을 확인하려면 store에 user의 role이 있어야 합니다. 두 가지 방법:

- **간단**: 어드민 데이터 API마다 서버에서 `requireAdmin`을 호출(이게 진짜 방어선). 화면 가드는 토큰만 봐도, API가 막으면 일반 회원은 아무 데이터도 못 봅니다.
- **화면도 막고 싶으면**: 로그인 시 `authStore`에 `role`을 저장해두고 가드에서 확인.

```tsx
// 예: store에 role이 있다고 가정
const role = useAuthStore((s) => s.user?.role);
useEffect(() => {
  if (hydrated && !token) return router.replace('/admin/login');
  if (hydrated && token && role !== 'admin') router.replace('/');  // 권한 없음
}, [hydrated, token, role, router]);
```

우선순위는 **API의 `requireAdmin`이 먼저**입니다. 화면 가드만 채우고 API가 무방비면, API를 직접 호출하는 요청은 못 막습니다.

## 4. (제안) useSellerApply 에러 처리

GET이 `res.ok`를 안 봐서 401/500도 그냥 `null`로 처리됩니다. "신청 안 함"과 "에러"를 구분합시다.

```ts
fetch('/api/seller/apply', { headers: { Authorization: `Bearer ${accessToken}` } })
  .then(async (r) => {
    if (!r.ok) throw new Error('조회 실패');
    return r.json();
  })
  .then((json) => setApplication(json.data ?? null))
  .catch(() => setLoadError('신청 내역을 불러오지 못했습니다.'));  // 에러 상태 따로
```

---

정리하면 **① 빌드(타입 6건) → ② 셀러 가드 → ③ role 확인** 순서입니다. ①만 끝내도 배포가 다시 되니 거기부터 갑시다. serverAuth·판매자 신청은 토대가 좋아서, 이 위에 가드만 마저 채우면 됩니다. 화이팅입니다.
