# Cookeat 리뷰 해결 가이드 (2026-07-04)

이번 리뷰에서 짚은 항목을 어떻게 고치면 되는지 단계별로 정리했습니다. 공통으로 챙길 포트폴리오 항목은 함께 전달하는 `취업레포-가이드.md`에 더 자세히 있으니, 여기서는 Cookeat 코드에 맞춰 바로 적용할 수 있는 부분 위주로 적습니다.

잘 잡아둔 것부터 짚겠습니다. `app/api/order/route.ts`에서 클라이언트가 보낸 금액을 안 믿고 서버가 상품가·재고·쿠폰·포인트를 전부 다시 계산한 뒤 `create_order` RPC로 트랜잭션 처리하는 구조는, 결제를 다루는 프로젝트에서 꼭 지켜야 할 원칙을 정확히 지킨 사례입니다. 카카오페이도 이번에 같은 원칙(`kakao/ready`가 서버에서 `final_amount`를 재조회하고, `kakao/approve`가 카카오 승인 금액과 대조)으로 정리됐고요. 이 원칙만 흔들지 않으면 되고, 아래는 그 위에 얹어서 마저 손볼 것들입니다.

## 1. README 클론 URL을 조직 저장소로 정정하기

`README.md:155`가 지금 이렇게 되어 있습니다.

```md
git clone https://github.com/djsy01/Cookeat.git
```

`git remote -v`로 실제 origin을 확인해보면 조직 저장소 주소가 나옵니다.

```
origin  https://github.com/FRONTENDBOOTCAMP-17th/Cookeat.git
```

README를 아래처럼 한 줄만 바꾸면 됩니다.

```md
git clone https://github.com/FRONTENDBOOTCAMP-17th/Cookeat.git
```

개인 fork 주소가 README에 남아 있으면, 이 프로젝트를 처음 보는 사람이 그 명령을 그대로 복사했을 때 팀 저장소가 아니라 개인 fork로 들어가게 됩니다. 사소해 보여도 "설치 및 실행" 섹션은 방문자가 실제로 손으로 따라 치는 부분이라 정확한 게 중요합니다.

## 5. CI에 lint 게이트 추가하기 (여유 되면)

지금 `.github/workflows/ci.yml`은 이렇게 `tsc`와 `build`만 검사합니다.

```yaml
- run: npx tsc --noEmit
- run: npm run build
```

여기에 한 줄만 추가하면 됩니다.

```yaml
- run: npx tsc --noEmit
- run: npm run lint
- run: npm run build
```

다만 지금 상태로 바로 추가하면 42건의 에러 때문에 CI가 바로 빨간불이 됩니다. 두 가지 중 하나를 고르면 됩니다.

**옵션 A — 시간 있으면 먼저 정리**

대부분 `react-hooks/set-state-in-effect` 규칙이라, `hooks/usePagination.ts:8` 같은 곳은 이렇게 고치면 규칙을 만족합니다.

```ts
// 기존
export function usePagination<T>(items: T[], pageSize: number) {
  const [currentPage, setCurrentPage] = useState(1);

  useEffect(() => {
    setCurrentPage(1);
  }, [items.length]);
  // ...
}

// 수정 — 렌더 중 파생 상태로 처리 (effect 자체를 없앰)
export function usePagination<T>(items: T[], pageSize: number) {
  const [currentPage, setCurrentPageRaw] = useState(1);
  const [prevLength, setPrevLength] = useState(items.length);

  if (items.length !== prevLength) {
    setPrevLength(items.length);
    setCurrentPageRaw(1);
  }
  // ...
}
```

(React 공식 문서의 "You Might Not Need an Effect" 패턴입니다. 리스트 길이가 바뀔 때 페이지를 1로 되돌리는 로직을 effect 없이 렌더 중 비교로 처리하는 방식입니다.)

**옵션 B — 지금은 시간이 없으니 lint만 먼저 게이트에 추가하고 기존 에러는 `--fix` 가능한 것만 정리**

```bash
npm run lint -- --fix
```

`--fix`로 안 잡히는 나머지는 `next.config.ts`나 `eslint.config.mjs`에서 해당 규칙만 일시적으로 warn으로 낮추고, 새로 짜는 코드부터는 규칙을 지키는 것도 현실적인 절충안입니다.

## 6. git 커밋 author 이름 통일 (엄인호·추유나)

`git log --format='%an <%ae>'`로 확인해보면 이렇게 나옵니다.

```
260 djsy01 <dlsgh8884@naver.com>
 81 Inho <164964087+djsy01@users.noreply.github.com>
211 nana9147 <yuna.chu9147@gmail.com>
 19 yuna <yuna.chu9147@gmail.com>
```

이메일로 보면 `djsy01`/`Inho`는 같은 분, `nana9147`/`yuna`는 같은 분인데, GitHub Contributors 그래프는 커밋 author 이름(정확히는 이메일)이 갈리면 별도 인물로 집계합니다. 즉 실제 기여량보다 그래프에는 적게 잡힙니다. 지금부터라도 이렇게 통일해두면 앞으로의 커밋은 하나로 잡힙니다.

```bash
git config user.name "djsy01"
git config user.email "dlsgh8884@naver.com"
```

과거 커밋까지 소급해서 합치고 싶다면 `git filter-repo`로 author 이메일을 재작성하는 방법도 있지만, 이건 히스토리를 다시 쓰는 작업이라 팀 전체와 상의 없이 혼자 하면 안 됩니다. 마감이 얼마 안 남았으니 이번엔 앞으로의 커밋만 통일해두는 정도로 충분합니다.
