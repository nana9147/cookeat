# Cookeat 17차 리뷰 — 해결 가이드 (2026-07-02)

우선순위는 카카오 1원 결제, 쿠폰 프론트 연결 + `coupon_id` 소비처, coupons RLS, 엑셀 일괄등록 배치화 순입니다.

## 1. [필수] 카카오페이 금액 서버검증

토스는 이미 이렇게 하고 있으니, 카카오도 같은 원칙만 옮겨 오면 됩니다. 요지는 **클라가 보낸 금액을 믿지 말고, 서버가 주문을 다시 읽어 그 금액으로 결제한다**는 겁니다.

### ready 단계 — 클라 금액 대신 서버 금액 사용

```ts
// app/api/payment/kakao/ready/route.ts
const { orderId } = await req.json();               // ← totalAmount는 클라에서 받지 않음

// 서버가 주문을 다시 읽어 실제 금액을 구한다
const { data: order, error } = await supabaseAdmin
  .from("orders")
  .select("final_amount, status, user_id")
  .eq("id", orderId)
  .single();
if (error || !order) return NextResponse.json({ message: "주문을 찾을 수 없습니다" }, { status: 404 });
if (order.status !== "pending") return NextResponse.json({ message: "이미 처리된 주문" }, { status: 400 });

// 카카오에 넘기는 total_amount = 서버가 계산한 order.final_amount
const kakaoBody = {
  cid: process.env.KAKAO_CID,
  partner_order_id: orderId,
  total_amount: order.final_amount,   // ← 클라값이 아니라 서버값
  // ...
};
```

### approve 단계 — 승인 금액을 서버 주문금액과 대조

```ts
// app/api/payment/kakao/approve/route.ts
const { data: order } = await supabaseAdmin
  .from("orders")
  .select("final_amount, status, user_id")   // ← final_amount도 함께
  .eq("id", orderId)
  .single();

const kakaoRes = await approveKakao(...);      // 카카오 승인 응답
if (kakaoRes.amount.total !== order.final_amount) {
  // 승인 금액이 주문 금액과 다르면 결제 무효 처리
  return NextResponse.json({ message: "결제 금액 불일치" }, { status: 400 });
}
// 여기까지 통과해야 '결제완료' 처리
```

이렇게 해두면 `/kakao/ready`를 `totalAmount: 1`로 직접 찔러도 서버가 실제 주문 금액으로 결제를 열기 때문에, 1원 결제가 성립하지 않습니다.

## 2. [필수] 쿠폰 프론트 연결 + `coupon_id` 소비처

서버는 이미 `userCouponId` 기준으로 바뀌었으니, 프론트를 거기에 맞춰 주면 됩니다.

### (1) 쿠폰 모달이 `/coupons/mine`를 부르게

```ts
// components/(auth)/cart/checkout/CouponModal.tsx
// 변경 전: api.get("/coupons")            (사용 가능한 전체 쿠폰 — 옛 방식)
// 변경 후: 내 계정에 지급된 쿠폰 목록
const { data } = await api.get("/coupons/mine");
// data 항목에 userCouponId, code, discountType, discountValue ... 가 담김
```

### (2) AppliedCoupon 타입에 userCouponId 추가

```ts
// components/(auth)/cart/DiscountSection.tsx
export type AppliedCoupon = {
  userCouponId: number;   // ← 서버가 실제로 쓰는 식별자
  code: string;
  discountType: "rate" | "fixed";
  discountValue: number;
};
```

### (3) 주문 요청에 userCouponId를 실어 보내기

```ts
// components/(auth)/cart/complete/usePaymentConfirm.ts (또는 useCheckoutPayment.ts)
await api.post("/order", {
  // couponCode  ← 이건 서버가 더 이상 안 읽음, 제거
  userCouponId: appliedCoupon?.userCouponId ?? null,
  // ...
});
```

이 세 곳을 맞춰 두면 화면 요약의 할인과 서버가 확정하는 `finalAmount`가 일치하고, 쿠폰 `used_at`도 정상적으로 찍힙니다.

### (4) rename된 컬럼을 아직 읽는 곳 고치기

`orders.coupon_id`는 이제 없습니다(`user_coupon_id`로 바뀌었죠). 아래를 전역 검색으로 찾아 바꿔 주세요.

```bash
# 프로젝트에서 남은 옛 컬럼 참조 찾기
grep -rn "coupon_id" app/api | grep -v "user_coupon_id"
```

- `app/api/seller/orders/[orderId]/db.ts:58` — select의 `coupon_id`, 임베드 `coupons(...)`를 `user_coupon_id` + `user_coupons(...)` 관계로 교체.
- `coupons/route.ts`·`coupons/validate/route.ts`의 "이미 사용" 판정도 `user_coupons`(또는 orders의 `user_coupon_id`) 기준으로.

## 3. [제안] coupons 테이블 anon 차단 (RLS)

Supabase SQL Editor에서:

```sql
alter table coupons enable row level security;
-- anon에게 부여됐던 select 권한 회수 (있다면)
revoke select on coupons from anon;
-- 공개가 꼭 필요하면 서버(service_role) 라우트로만 조회하도록 두고,
-- 클라가 직접 봐야 하는 항목만 별도 정책으로 최소 노출
```

확인:

```bash
curl -s "$SUPABASE_URL/rest/v1/coupons?select=*&limit=1" \
  -H "apikey: $ANON_KEY" -H "Authorization: Bearer $ANON_KEY"
# 빈 배열([]) 또는 permission denied 가 나와야 정상
```

## 4. [제안] 엑셀 일괄등록 배치 insert

지금은 500행이면 insert를 500번 부릅니다. 검증을 먼저 다 돌린 뒤, 통과한 행만 모아서 한 번에 넣으면 됩니다.

```ts
// app/seller/products/db.ts (개념)
const valid: ProductRow[] = [];
const errors: { row: number; reason: string }[] = [];

rows.forEach((row, i) => {
  if (typeof row.price !== "number" || row.price <= 0) {
    errors.push({ row: i + 1, reason: "가격이 올바르지 않습니다" });
    return;
  }
  // ...다른 검증
  valid.push(row);
});

// 통과분 한 번에 insert (Postgres는 배열 insert 지원)
if (valid.length) {
  const { error } = await supabaseAdmin.from("products").insert(valid);
  if (error) throw error;
}
return { inserted: valid.length, errors };
```

이렇게 하면 왕복이 500번에서 1번으로 줄고, 어느 행이 왜 빠졌는지도 `errors`에 담아 사용자에게 돌려줄 수 있습니다.
