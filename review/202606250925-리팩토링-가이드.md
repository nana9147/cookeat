# 중복 코드, 언제 지우고 언제 두는가 — 초보 React·Next.js 개발자를 위한 리팩토링 안내

파이널 프로젝트를 보다 보면 어느 조에서나 똑같은 장면이 나옵니다. 날짜를 보기 좋게 바꾸는 함수가 화면마다 따로 적혀 있고, 가격에 콤마를 찍어 "원"을 붙이는 코드는 수십 군데에 흩어져 있죠. 로그인 여부를 확인하는 네 줄짜리 코드는 서버 액션마다 복사돼 있습니다. 이걸 보면 누구나 "중복은 나쁘니까 다 합쳐야지"라고 생각하게 되는데, 저는 여기서 한 박자 멈추자고 말하고 싶습니다.

중복을 무조건 없애는 게 항상 좋은 건 아니기 때문입니다. 어설프게 합치면 코드가 오히려 더 어려워지고, 나중엔 그 "공통 함수" 하나를 건드릴 때마다 온 화면이 흔들려서 손대기가 무서워집니다. 그래서 이 글은 "중복을 다 지우는 법"이 아니라 **언제 지우고 언제 그냥 두는지, 그리고 지운다면 무엇부터 어떻게 지우는지**를 이야기합니다. 막 React와 Next.js로 첫 프로젝트를 만들고 있는 분들을 염두에 두고 썼습니다.

---

## "중복은 나쁘다"는 말부터 의심하자

DRY(Don't Repeat Yourself)라는 말을 한 번쯤 들어봤을 겁니다. 반복하지 마라. 좋은 원칙입니다. 그런데 이 말이 너무 유명해지다 보니, 많은 사람이 "똑같이 생긴 코드가 두 번 보이면 무조건 합쳐야 한다"로 받아들입니다. 정작 DRY를 처음 말한 책(『The Pragmatic Programmer』)이 가리킨 건 코드의 생김새가 아니라 **지식(knowledge)** 입니다. 같은 *지식*이 두 군데에 적혀 있으면 안 된다는 뜻이지, 우연히 비슷하게 생긴 코드를 다 합치라는 뜻이 아니었어요.

이 차이가 초보 단계에서 특히 중요합니다. 여러분은 지금 만드는 것 자체가 어렵습니다. 화면이 의도대로 그려지고, 데이터가 제대로 들어오고, 버튼이 동작하게 만드는 게 최우선입니다. 이 단계에서 "코드가 예쁘지 않다"는 이유로 자꾸 추상화를 시도하면, 아직 어디가 진짜 공통이고 어디가 우연인지 분간이 안 된 상태에서 잘못된 틀을 만들어 버리기 쉽습니다. **일단 동작하게 만들고, 패턴이 충분히 드러난 뒤에 리팩토링한다** — 이 순서가 초보자에게는 거의 항상 맞습니다.

그러니 코드가 좀 반복돼 보여도 죄책감을 가질 필요 없습니다. 중복은 죄가 아닙니다. 다만 어떤 중복은 그냥 두면 시간이 지나면서 조용히 버그가 되고, 어떤 중복은 끝까지 둬도 아무 문제가 없습니다. 그 둘을 구분하는 눈을 기르는 게 이 글의 목표입니다.

## 1. 중복에도 종류가 있다 — 진짜 중복과 우연한 중복

같아 보이는 코드라고 다 같은 게 아닙니다. 두 종류로 나눠 봅시다.

**진짜 중복(real duplication)** 은 같은 하나의 사실이 여러 곳에 적힌 경우입니다. 예를 들어 "게시글 카테고리 `rental`은 화면에 '방 렌탈'이라고 표시한다"는 규칙이 있다고 합시다. 이 규칙이 세 군데에 따로 적혀 있다면, 그건 같은 지식이 세 번 복사된 진짜 중복입니다. 규칙이 바뀌면 세 곳을 다 고쳐야 하고, 한 곳이라도 빠뜨리면 화면마다 다른 라벨이 뜹니다.

이건 가상의 이야기가 아닙니다. 실제로 한 조의 코드에서 이런 일이 있었어요. 카테고리 라벨 맵이 세 파일에 복사돼 있었는데, 시간이 지나면서 한 곳만 누군가 손을 대는 바람에 이렇게 갈라져 있었습니다.

```ts
// 어떤 파일에서는
rental: "방 렌탈",
// 또 다른 파일에서는
rental: "방 렌트",
```

같은 카테고리인데 어느 화면에서 들어왔느냐에 따라 "방 렌탈"이 보이기도 하고 "방 렌트"가 보이기도 합니다. 이게 바로 중복이 버그로 변하는 순간입니다. 복사할 땐 분명 똑같았는데, 코드는 시간이 지나면 반드시 한쪽만 수정되거든요. 진짜 중복은 이렇게 "언젠가 갈라진다"가 전제입니다.

**우연한 중복(incidental duplication)** 은 지금은 코드가 비슷하게 생겼지만, 그 둘이 같은 이유로 같은 게 아닌 경우입니다. 예를 들어 회원가입 폼과 상품 등록 폼이 둘 다 "입력값을 state에 담고, 제출하면 검증하고, 서버로 보낸다"는 비슷한 골격을 가질 수 있습니다. 생김새는 닮았지만, 회원가입의 규칙과 상품 등록의 규칙은 서로 아무 상관이 없습니다. 한쪽이 바뀐다고 다른 쪽이 따라 바뀌지 않아요. 이런 걸 억지로 하나의 `useForm` 같은 걸로 합치면, 두 폼의 서로 다른 요구사항이 한 함수에 욱여넣어지면서 조건 분기가 끝없이 늘어납니다.

> **핵심 질문 하나** — 이 두 코드는 "**항상 함께 바뀌어야** 하는가?"
> 함께 바뀌어야 하면 진짜 중복이니 합칠 후보입니다. 따로 바뀔 수 있으면 우연한 중복이니 그냥 두세요. 지금 똑같이 생겼다는 사실만으로 합치면, 나중에 둘이 서로 다른 방향으로 가고 싶어질 때 발목을 잡습니다.

## 2. 성급한 추상화는 중복보다 비싸다

여기서 꼭 새기고 갈 문장이 하나 있습니다. 미국의 유명한 개발자 Sandi Metz가 한 말인데요.

> "중복은 잘못된 추상화보다 훨씬 싸다(Duplication is far cheaper than the wrong abstraction)."

무슨 뜻이냐면, 코드를 두 번 복사해 두는 비용보다, 어설프게 만든 공통 함수를 끌고 가는 비용이 훨씬 크다는 겁니다. 잘못된 추상화는 어떻게 만들어질까요. 보통 이런 식입니다. 비슷한 코드 두 개를 하나로 합칩니다. 그런데 곧 세 번째 비슷한 경우가 나타나는데, 살짝 다릅니다. 그래서 공통 함수에 `if (isSpecialCase)` 같은 분기를 하나 추가합니다. 네 번째가 나타나면 또 옵션 파라미터를 추가합니다. 이게 몇 번 반복되면, 그 함수는 온갖 플래그와 분기로 뒤덮여서 아무도 손대기 싫은 괴물이 됩니다. 정작 그 함수를 부르는 쪽은 `doThing(data, true, false, null, "mode2")` 처럼 무슨 뜻인지 알 수 없는 인자들을 넘기게 되고요.

이 괴물은 두 번 복사된 코드보다 읽기 어렵고, 고치기 무섭고, 지우기도 까다롭습니다. 차라리 처음부터 복사해 뒀다면, 어느 날 둘이 달라져야 할 때 그냥 각자 고치면 끝이었을 텐데 말이죠.

그러니 초보 단계의 전략은 분명합니다. **확신이 설 때까지는 합치지 말고 두세요.** 코드가 충분히 여러 번 반복되고, 그 반복들이 정말로 같은 규칙을 공유한다는 게 눈에 보일 때 — 그때 합쳐도 전혀 늦지 않습니다. 오히려 그때가 돼야 "어떻게 합쳐야 하는지"가 비로소 보입니다.

## 3. 그래서 언제 빼는가 — 세 번의 법칙, 그리고 더 중요한 신호들

리팩토링에 흔히 쓰는 경험칙이 **세 번의 법칙(Rule of Three)** 입니다. 같은 코드가 두 번 나오면 그냥 두고, **세 번째**로 같은 걸 복사하려는 순간 멈춰서 공통으로 뺄지 고민하라는 규칙입니다. 두 번까지는 우연일 수 있지만, 세 번이면 패턴이라고 봐도 좋다는 거죠. 외우기 쉬워서 좋은 출발점입니다.

다만 횟수보다 더 좋은 신호가 몇 가지 있습니다. 세 번이 안 됐어도 빼는 게 맞을 때가 있고, 다섯 번이 넘었어도 두는 게 나을 때가 있거든요. 다음을 스스로 물어보세요.

- **이게 하나의 "지식"인가?** — 통화 표기 방식, 카테고리 라벨, 상태값 같은 건 본질적으로 하나의 약속입니다. 이런 건 두 번만 나와도 한 곳에 모으는 게 좋습니다. 갈라지면 곧 버그니까요. (앞의 "방 렌탈 / 방 렌트"가 정확히 이 경우입니다.)
- **이미 갈라져서 어긋나 있는가?** — 복사본들을 나란히 놓고 봤을 때 색깔이나 글자가 미묘하게 다르다면, 그건 이미 사고가 난 겁니다. 합쳐야 할 강력한 근거입니다.
- **이 둘은 함께 바뀌는가, 따로 바뀌는가?** — 1번에서 말한 그 질문입니다. 함께 바뀌면 합치고, 따로 바뀌면 두세요.
- **빼는 비용이 얼마나 드는가?** — 의존성 없는 순수 함수(날짜·가격 포매터 같은)는 빼는 비용이 거의 0입니다. 이런 건 망설일 이유가 없습니다. 반대로 화면 레이아웃이 얽힌 UI는 빼는 비용이 크니 더 신중해야 합니다.

결국 횟수는 출발점일 뿐이고, 진짜 기준은 "**이게 함께 바뀌어야 하는 하나의 지식인가**"입니다.

## 4. 무엇부터 빼면 본전을 뽑는가 — 리팩토링 우선순위

막상 정리하려고 마음먹으면 어디부터 손대야 할지 막막합니다. 효과는 큰데 위험은 작은 순서로 정리해 봤습니다. 위에서부터 손대세요.

### 1순위 — 표시(포매팅) 로직, 사실상 공짜고 효과는 바로 보인다

가격에 콤마 찍고 "원" 붙이기, 날짜를 "6월 25일"로 바꾸기, "3시간 전" 같은 상대 시간 표시. 이런 함수들은 입력만 받아 출력을 내는 순수 함수라 어디에도 얽혀 있지 않습니다. 빼는 비용이 사실상 0인데, 효과는 사용자에게 바로 보입니다. 한 곳에 모아두면 통화 표기를 "원"에서 "₩"로 바꾸거나, null일 때 "가격 협의"로 보여주는 규칙을 추가할 때 한 곳만 고치면 됩니다. **리팩토링 투자 대비 효과가 가장 좋은 1순위**입니다. 세 번의 법칙을 따질 것도 없이 그냥 빼세요.

```ts
// lib/format.ts — 표시 로직은 전부 여기로 모은다
export const formatPrice = (won?: number | null) =>
  won == null ? "가격 협의" : `${won.toLocaleString()}원`;

export const formatDate = (iso: string) =>
  new Intl.DateTimeFormat("ko-KR", {
    year: "numeric", month: "2-digit", day: "2-digit",
  }).format(new Date(iso));

export const formatRelativeTime = (iso: string) => {
  const hours = Math.floor((Date.now() - new Date(iso).getTime()) / 3_600_000);
  if (hours < 1) return "방금 전";
  if (hours < 24) return `${hours}시간 전`;
  return `${Math.floor(hours / 24)}일 전`;
};
```

이미 수십 군데에 인라인으로 `{price.toLocaleString()}원`이 박혀 있어도 괜찮습니다. 한 번에 다 바꾸려고 무리하지 말고, **새로 쓰는 코드부터 `formatPrice`를 쓰고**, 기존 것은 그 파일을 손볼 일이 생길 때마다 슬쩍 바꿔 나가면 됩니다.

### 2순위 — 상수·라벨·상태값: 갈라지면 버그 (단일 진실 원천)

카테고리 라벨 맵, 상태 배지 색깔, `"결제완료"`·`"배송중"` 같은 상태 문자열. 이런 것들은 본질적으로 "하나의 약속"이라 한 곳에만 있어야 합니다. 이걸 **단일 진실 원천(Single Source of Truth)** 이라고 부릅니다. 진실은 한 군데에만 적어 두고, 나머지는 전부 거기서 가져다 쓴다는 뜻입니다.

```ts
// lib/constants/post.ts
export const POST_TYPE_LABEL = {
  used_goods: "중고거래",
  rental: "방 렌탈",
  jobs: "구인구직",
} as const;

// 상태 문자열도 객체로 모아두면 오타를 컴파일러가 잡아준다
export const ORDER_STATUS = {
  PAID: "결제완료",
  SHIPPING: "배송중",
  DONE: "배송완료",
} as const;
```

상태 문자열을 이렇게 상수로 모아두면, `if (status === "배송 중")` 처럼 띄어쓰기 하나 틀린 오타를 타입스크립트가 잡아줍니다. 문자열을 직접 비교하면 오타가 나도 조용히 false가 되어 버그가 숨어버리는데, 상수를 쓰면 `ORDER_STATUS.SHPPING` 같은 오타에서 에디터가 바로 빨간 줄을 그어 줍니다.

### 3순위 — 타입: 생성된 타입에서 파생하라

Supabase를 쓰면 보통 DB 스키마에서 타입을 자동으로 뽑아냅니다(`database.types.ts` 같은 파일). 그런데 많은 분이 이 생성된 타입을 두고도, 화면마다 같은 테이블의 타입을 손으로 다시 적습니다.

```ts
// 이렇게 손으로 적으면, DB에 컬럼이 추가돼도 이 타입은 모른다
interface Pet {
  id: string;
  name: string;
  animal_type: string;
}
```

문제는 DB 스키마가 바뀌었을 때입니다. 손으로 적은 타입은 스키마 변경을 전혀 모르고, 그래서 코드와 실제 데이터가 조용히 어긋납니다. 어떤 파일에선 `type`이라 적고 다른 파일에선 `animal_type`이라 적는 식으로 갈라지기도 하고요. 생성된 타입에서 필요한 만큼만 **파생**해서 쓰면 이런 일이 없습니다.

```ts
import type { Database } from "@/types/database.types";

type PetRow = Database["public"]["Tables"]["pets"]["Row"];

// 화면에서 필요한 필드만 골라 쓰고 싶으면 Pick으로
type PetCard = Pick<PetRow, "id" | "name" | "animal_type">;
```

이렇게 해두면 DB에서 컬럼 이름이 바뀌는 순간 타입스크립트가 그 컬럼을 쓰는 모든 곳에 에러를 띄워 줍니다. 컴파일러가 "여기 고쳐야 해요"라고 알려주는 거죠. 손으로 적은 타입은 이 안전망을 통째로 포기하는 셈입니다.

### 4순위 — 로직: 커스텀 훅으로

입력에 디바운스를 거는 로직, 스크롤 이벤트를 구독했다가 정리하는 로직, ESC 키로 모달을 닫는 로직, 카운트다운 타이머. 이런 "동작"의 반복은 **커스텀 훅**으로 빼기 좋습니다. UI는 그대로 두고 로직만 뽑아내는 거라, 화면이 흔들릴 위험 없이 정리됩니다.

```ts
// hooks/useDebounce.ts — 0.4초 동안 입력이 멈추면 그때서야 값을 반영
export function useDebounce<T>(value: T, delay = 400) {
  const [debounced, setDebounced] = useState(value);
  useEffect(() => {
    const id = setTimeout(() => setDebounced(value), delay);
    return () => clearTimeout(id); // 다음 입력이 오면 이전 타이머 취소
  }, [value, delay]);
  return debounced;
}
```

검색창마다 `useState` 두 개와 `setTimeout`, `clearTimeout`을 손으로 적던 걸, `const debouncedSearch = useDebounce(search)` 한 줄로 줄일 수 있습니다. 4순위에 둔 이유는, 훅을 잘 만들려면 의존성 배열과 정리 함수(cleanup)를 정확히 이해해야 해서 1~3순위보다 약간 손이 가기 때문입니다. 하지만 한 번 익혀두면 React에서 가장 강력한 재사용 도구입니다.

### 5순위 — UI 컴포넌트는 가장 신중하게

마지막이 화면 마크업입니다. 똑같이 생긴 카드, 빈 상태 안내, 모달 껍데기 같은 게 복사돼 있는 경우죠. 효과는 크지만 위험도 가장 큽니다. 디자인이 아직 자주 바뀌는 단계에서 컴포넌트를 너무 일찍 못박으면, 화면마다 미묘하게 다른 요구를 props로 받아내느라 컴포넌트가 누더기가 됩니다. **디자인이 어느 정도 안정된 뒤에** 빼는 게 좋고, 뺄 때도 "정말 똑같은 것"부터 조심스럽게 빼세요. (이 부분은 다음 절에서 더 다룹니다.)

## 5. React·Next.js에서 중복을 빼는 다섯 가지 연장

우선순위를 알았으니, 이제 실제로 빼는 도구들을 봅시다. 상황에 맞는 연장을 고르는 게 중요합니다.

### (1) 유틸 함수 — 한 곳에 모은 순수 함수

앞서 본 `lib/format.ts`가 대표적입니다. 입력을 받아 출력을 내는, 화면이나 상태와 무관한 순수 로직은 전부 `lib/` 아래 유틸 파일로 모읍니다. 가장 단순하고 가장 안전한 연장입니다.

### (2) 커스텀 훅 — 상태와 생명주기가 얽힌 로직

`useState`, `useEffect`, `useRef`가 함께 움직이는 로직은 유틸 함수로는 못 뺍니다. 훅으로 빼야 합니다. 디바운스, 데이터 페칭, 모달 토글, 외부 클릭 감지 같은 게 여기 해당합니다. 데이터를 불러오는 반복도 이렇게 묶을 수 있습니다.

```ts
// hooks/useFetch.ts — 로딩·에러·데이터 3종 세트를 한 번에
export function useFetch<T>(fetcher: () => Promise<T>, deps: unknown[] = []) {
  const [data, setData] = useState<T | null>(null);
  const [loading, setLoading] = useState(true);
  const [error, setError] = useState<Error | null>(null);

  useEffect(() => {
    let cancelled = false;            // 언마운트 후 setState 방지
    setLoading(true);
    fetcher()
      .then((d) => !cancelled && setData(d))
      .catch((e) => !cancelled && setError(e))
      .finally(() => !cancelled && setLoading(false));
    return () => { cancelled = true; };
    // eslint-disable-next-line react-hooks/exhaustive-deps
  }, deps);

  return { data, loading, error };
}
```

여기서 `cancelled` 플래그를 눈여겨보세요. 컴포넌트가 사라진 뒤에 응답이 도착해서 `setState`를 부르면 경고가 뜨는데, 정리 함수에서 플래그를 올려 이걸 막습니다. 이런 자잘하지만 중요한 처리를 한 곳에 모아두면, 화면마다 직접 적던 try-catch 보일러플레이트가 사라지면서 실수할 여지도 함께 사라집니다. (다만 데이터 페칭이 복잡해지면 TanStack Query 같은 라이브러리를 쓰는 게 정석입니다. 직접 만든 `useFetch`는 그 전 단계의 좋은 연습입니다.)

### (3) 컴포넌트 합성 — props 폭증 대신 children과 슬롯

UI를 컴포넌트로 뺄 때 가장 흔한 실수가 **모든 차이를 props로 받으려는 것**입니다. 카드 컴포넌트 하나에 `showHeart`, `imageRatio`, `hasDescription`, `badgeColor`... 이렇게 옵션이 자꾸 늘어나면, 그게 바로 4절에서 말한 "괴물"이 되어 가는 신호입니다.

이럴 땐 props로 욱여넣는 대신 **합성(composition)** 을 쓰세요. 바깥 껍데기만 컴포넌트로 만들고, 안에 들어갈 내용은 `children`이나 슬롯으로 받는 겁니다.

```tsx
// 페이지 껍데기는 공통, 본문은 페이지마다 자유롭게
function PageShell({ children }: { children: React.ReactNode }) {
  return (
    <main className="min-h-screen bg-surface flex flex-col">
      <Navbar />
      <section className="max-w-screen-lg mx-auto px-4 py-16 flex-1">
        {children}
      </section>
      <Footer />
    </main>
  );
}

// 각 페이지
export default function MaterialsPage() {
  return (
    <PageShell>
      <PageHeader title="MATERIALS" desc="..." />
      {/* 이 아래 본문은 페이지마다 완전히 다르게 */}
    </PageShell>
  );
}
```

이렇게 하면 공통인 레이아웃·네비게이션·푸터는 한 곳에 모이고, 페이지마다 다른 본문은 그대로 자유를 누립니다. props로 모든 차이를 표현하려 들지 않으니 컴포넌트가 단순하게 유지됩니다. "공통 부분은 껍데기로, 다른 부분은 children으로" — 이 감각이 UI 중복을 다루는 핵심입니다.

### (4) 상수 모듈과 생성 타입 파생

4절의 2·3순위에서 본 그대로입니다. 라벨·상태·색깔은 `constants/` 아래 상수로, 타입은 생성된 DB 타입에서 `Pick`이나 교차 타입(`&`)으로 파생합니다. 화면용으로 합산 필드가 더 필요하면 이렇게 확장하면 됩니다.

```ts
import type { Tables } from "@/types/database";

// DB 행 + 화면에서 계산해 붙이는 필드
type EventListItem = Tables<"event"> & {
  totalOrders: number;
  remainingSeats: number;
};
```

### (5) 서버 액션·API 라우트의 가드 래퍼

Next.js로 서버 액션이나 API 라우트를 만들다 보면, 함수 첫머리마다 "로그인했는지 확인하고, 안 했으면 막는" 코드가 똑같이 반복됩니다. 이건 단순한 중복을 넘어 **보안 문제**라서 더 중요합니다. 한 군데서 이 가드를 깜빡 빠뜨리면 그게 곧 인가 누락이 되거든요. 한 곳에 모아두면 빠뜨릴 일이 줄어듭니다.

```ts
// lib/auth.ts
export async function getAuthUser() {
  const supabase = await createClient();
  const { data: { user } } = await supabase.auth.getUser();
  return user;
}

// 인증 + 가드를 한 번에
export async function withUser<T>(fn: (user: User) => Promise<T>) {
  const user = await getAuthUser();
  if (!user) return { error: { code: "UNAUTHORIZED", message: "로그인이 필요합니다." } };
  return fn(user);
}
```

```ts
// 서버 액션에서 — 가드를 잊을 수가 없는 구조가 된다
export async function createPet(input: PetInput) {
  return withUser(async (user) => {
    const supabase = await createClient();
    return supabase.from("pets").insert({ ...input, user_id: user.id });
  });
}
```

"로그인이 필요합니다" 같은 메시지도 한 곳에서 관리되니, 문구를 바꿀 때 서른 군데를 뒤지지 않아도 됩니다. 보안 가드는 흩어져 있을수록 위험하다는 걸 꼭 기억하세요.

## 6. DRY의 진짜 장벽은 추출이 아니라 발견이다

여기까지 읽었으면 "그래, 공통 함수로 빼면 되는구나"라고 생각하실 텐데, 사실 팀 프로젝트에서 더 어려운 건 따로 있습니다. 이번에 여러 조의 코드를 보면서 거의 모든 조에서 똑같은 장면을 발견했어요. **공통 유틸이 이미 만들어져 있는데도, 그 존재를 몰라서 팀원들이 같은 함수를 각자 또 만들고 있던** 겁니다.

페이지 번호를 계산하는 함수가 `lib/utils.ts`에 멀쩡히 있는데 다섯 개 페이지가 똑같은 함수를 각자 다시 적어 둔 조도 있었고, 날짜 포매터가 `lib/format.ts`에 있는데 검색 폴더에서 똑같은 걸 또 만들어 둔 조도 있었습니다. 이건 추출을 못 해서가 아닙니다. 추출은 이미 누군가 해놨어요. **그게 있다는 걸 팀이 발견하지 못한** 게 문제입니다.

진실을 말하자면, DRY의 진짜 장벽은 "어떻게 공통 함수를 만드느냐"가 아니라 "**만들어 둔 공통 함수를 팀원들이 어떻게 발견하느냐**"입니다. 이건 코드 기술이 아니라 협업의 문제고, 그래서 다음 같은 습관들이 효과가 큽니다.

- **둘 자리를 정해두기.** "포매팅은 `lib/format.ts`, 상수는 `lib/constants/`, 훅은 `hooks/`" 처럼 팀이 미리 약속해두면, 새 유틸이 필요할 때 "혹시 거기 이미 있나?" 하고 한 곳만 보면 됩니다. 자리가 정해져 있지 않으면 찾는 비용이 만드는 비용보다 커져서, 다들 그냥 새로 만들어 버립니다.
- **PR에서 "이거 이미 있어요" 한마디.** 코드 리뷰에서 "이 함수 `lib/format.ts`에 비슷한 거 있어요"라고 짚어주는 문화가 쌓이면, 팀 전체가 공통 자산을 자연스럽게 익히게 됩니다. 이 한마디가 중복을 막는 가장 싸고 강력한 방법입니다.
- **이름을 검색 가능하게.** `formatPrice`, `formatDate` 처럼 누구나 떠올릴 이름을 쓰면, 새로 만들기 전에 `format`으로 한 번 검색해보는 것만으로 발견됩니다.

추출 잘하는 사람보다 **팀이 가진 공통 자산을 잘 아는 사람**이 결국 중복을 더 잘 막습니다. 혼자 하는 프로젝트가 아니라면 이 점을 꼭 기억하세요.

## 7. 안전하게 리팩토링하는 법

빼기로 마음먹었다면, 동작을 망가뜨리지 않고 옮기는 게 관건입니다. 리팩토링의 정의 자체가 "겉으로 드러나는 동작은 그대로 두고 안쪽 구조만 바꾸는 것"이거든요. 몇 가지 안전장치를 지키면 무서울 게 없습니다.

- **한 번에 하나만.** 포매터를 한 곳으로 모으는 작업과 컴포넌트를 쪼개는 작업을 같은 커밋에 섞지 마세요. 하나를 끝내고 잘 도는지 확인한 뒤 다음으로 갑니다. 그래야 뭔가 깨졌을 때 어디서 깨졌는지 바로 압니다.
- **작은 단계로 쪼개기.** "포매터 8군데를 한 번에 바꾼다"보다 "한 파일씩 바꾸고 화면 확인"이 안전합니다. 한 걸음마다 앱이 여전히 도는지 보세요.
- **타입스크립트를 안전망으로.** 타입이 잘 잡힌 코드에서는, 함수를 옮기고 이름을 바꾸면 그 함수를 쓰던 모든 곳에 에러가 떠서 "여기 고쳐야 해"라고 알려줍니다. 리팩토링 후 `npx tsc --noEmit`를 한 번 돌려 빨간 줄이 없는지 확인하는 습관을 들이세요. 이게 초보자가 쓸 수 있는 가장 든든한 안전망입니다.
- **테스트가 있으면 더 좋고.** 핵심 로직에 테스트가 있다면, 리팩토링 전후로 테스트가 똑같이 통과하는지만 보면 됩니다. 아직 테스트가 없어도 괜찮습니다. 다만 자주 만지는 유틸 함수 하나쯤은 작은 테스트를 붙여보는 걸 권합니다.

**Next.js에서 하나 더 조심할 것** — 함수를 공통 파일로 옮길 때 서버/클라이언트 경계를 넘지 않는지 보세요. 브라우저에서만 도는 코드(`window`, `localStorage`, `useState`)와 서버에서 도는 코드(`cookies()`, service-role 키)를 한 파일에 섞으면 곤란합니다. 순수 포매터는 양쪽 어디서나 안전하지만, Supabase 클라이언트를 만드는 코드는 "브라우저용"과 "서버용"을 구분해서 각자 자리에 둬야 합니다. 옮길 코드가 어느 환경에서 도는 코드인지 한 번 생각하고 옮기세요.

## 8. 초보자를 위한 실전 순서

이 글이 길었으니, 실제로 어떻게 움직이면 되는지를 한 흐름으로 정리하겠습니다.

1. **일단 동작하게 만든다.** 첫 구현에서는 복사·붙여넣기를 두려워하지 마세요. 화면이 의도대로 도는 게 먼저입니다. 이 단계에서 추상화를 고민하면 진도가 안 나갑니다.
2. **세 번째 복붙에서 한 번 멈칫한다.** 같은 코드를 세 번째로 복사하려는 순간, 잠깐 멈춰 "이거 함께 바뀌어야 하는 하나의 지식인가?"를 물어보세요. 맞으면 빼고, 아니면 그냥 두고 갑니다.
3. **포매터와 상수부터 모은다.** 빼기로 했다면 가장 쉽고 안전한 1·2순위(표시 로직, 상수·라벨)부터입니다. `lib/format.ts`와 `lib/constants/`를 만들고, 새 코드부터 거길 쓰세요.
4. **반복되는 동작은 훅으로.** 디바운스, 데이터 페칭, 모달처럼 상태가 얽힌 반복이 눈에 익으면 커스텀 훅으로 뽑습니다.
5. **UI는 디자인이 안정된 뒤에.** 화면 마크업 중복은 가장 마지막에, 그리고 "껍데기는 공통, 본문은 children" 원칙으로 신중하게 빼세요.

마지막으로 타이밍에 대해 한마디. 리팩토링은 모든 기능이 끝난 뒤에 날 잡고 하는 큰 행사가 아닙니다. 그렇게 미뤄두면 영원히 안 합니다. **기능과 기능 사이에**, 방금 만든 화면이 잘 도는 걸 확인한 그 순간 — "아, 이 포매터 아까 다른 데서도 봤지" 싶을 때 슬쩍 한 곳으로 모으는 식으로, 작게 자주 하는 게 가장 잘 굴러갑니다.

## 마치며

중복을 무조건 없애는 건 좋은 목표가 아닙니다. 어설픈 추상화는 두 번 복사한 코드보다 훨씬 비싸고, 초보 단계에서는 "일단 만들고 나중에 리팩토링"이 거의 항상 맞습니다. 그러니 코드가 좀 반복돼 보여도 조급해하지 마세요.

대신 딱 하나만 기억하면 됩니다. **"이 둘은 항상 함께 바뀌어야 하는 하나의 지식인가?"** 함께 바뀌어야 하는 지식이라면 — 통화 표기, 카테고리 라벨, 상태값, 인증 가드처럼 — 두 번만 나와도 한 곳에 모으는 게 좋습니다. 갈라지면 버그가 되니까요. 그게 아니라 지금 우연히 닮은 코드라면, 확신이 설 때까지 편하게 두세요. 좋은 추상화는 억지로 만드는 게 아니라, 패턴이 충분히 드러난 뒤에 자연스럽게 보이는 법입니다.

천천히, 그렇지만 꾸준히 갑시다. 화이팅입니다.
