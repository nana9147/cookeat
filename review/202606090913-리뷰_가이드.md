## 3. 권한(authorization) 모델

### 3-1. role 저장
회원 테이블에 role을 둡니다(기본 'user', 관리자만 'admin').
```sql
alter table public.users add column if not exists role text not null default 'user';
-- 특정 계정을 관리자로
update public.users set role = 'admin' where email = '관리자@example.com';
```

### 3-2. 가드에서 role 확인 (클라 + 서버 둘 다)
클라이언트 가드(화면 가림):
```tsx
// AdminAuthGuard.tsx — 이름값 하게 role까지
const role = useAuthStore((s) => s.user?.role);
useEffect(() => {
  if (hydrated && (!token || role !== 'admin')) router.replace('/admin/login');
}, [hydrated, token, role, router]);
if (!token || role !== 'admin') return null;
```
> 단, **클라 가드는 화면만 가립니다.** 진짜 보호는 서버에서:
```ts
// 어드민/셀러 API 라우트 공통 헬퍼
async function requireRole(token: string, role: 'admin' | 'seller') {
  const { data: { user } } = await supabaseAdmin.auth.getUser(token);
  if (!user) return null;
  const { data } = await supabaseAdmin.from('users').select('role').eq('id', user.id).maybeSingle();
  return data?.role === role ? user : null;
}
```

## 4. 다듬기

### 4-1. ProductForm 이미지/설명 연결
```tsx
<ImageUploadField data={form.images} onChange={(v) => handleChange('images', ...)} />
<DescriptionEditor value={form.description} onChange={(v) => setForm(f => ({...f, description: v}))} />
```
(섹션 구조에 맞춰 `ProductFormData`에 images/description 필드를 두고 같은 `handleChange` 패턴으로.)

### 4-2. Cart 선택 상태 끌어올리기
선택 여부를 `Cart`에서 관리해 `CartList`에 내려줍니다.
```tsx
const [selectedIds, setSelectedIds] = useState<string[]>(allIds);
<CartList selectedIds={selectedIds} onToggle={...} />
// "전체 선택 (n/m)"의 9/9도 selectedIds.length / items.length로
```

### 4-3. mock 일원화
어드민 통계/테이블 mock을 `app/admin/_mock/` 또는 `*.mock.ts`로 모으면 API 교체가 쉽습니다.
