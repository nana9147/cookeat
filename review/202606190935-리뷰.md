## 멤버별 리뷰

### 엄인호 (@djsy01) — 주문/장바구니/결제 백엔드, 상품 상세

이번 회차에서 가장 묵직한 코드를 맡으셨고, 결제·주문 백엔드의 안전성이 크게 좋아졌습니다.

**잘한 점 (칭찬)**
- `app/api/order/route.ts`는 지난 회차들 내내 짚었던 "결제 금액 위조"를 정공법으로 닫았습니다. 클라이언트가 보낸 금액을 믿지 않고 `productIds`로 DB에서 실제 `price`·`status`·`stock`을 다시 읽어 `totalAmount`를 서버에서 재계산하고(`:95`), `calcShipping`으로 배송비까지 서버가 정합니다. 입력 검증(수량 1~1000, 50개 상한, 중복 productId 차단)도 촘촘합니다.
- 재고 차감을 `.eq('stock', prevStock)` 조건부 업데이트(CAS)로 처리해 동시 주문 충돌을 감지하고, 실패 시 `restoreStock`으로 이미 차감한 항목을 되돌리며, `order_items` 저장이 실패하면 주문까지 삭제해 고아 주문을 막습니다(`:157-179`). 트랜잭션이 없는 Supabase 환경에서 낙관적 잠금으로 근사한 점, 그리고 그 한계를 주석으로 정직하게 밝혀둔 점이 특히 좋습니다.
- `usePaymentConfirm.ts`에서 `initRef`로 마운트 시 1회만 confirm을 실행해 결제 승인이 중복 호출되지 않게 막은 것도 적절합니다.

**[필수] `/shopping/[id]` 상품 상세가 전부 404로 막혀 있습니다.**
시연에서 사용자가 목록 → 상세 → 장바구니로 이어지는 핵심 동선이 상세에서 끊깁니다. 실제로 판매중인 상품 `/shopping/4`(유기농 당근)에 들어가도 404가 뜹니다. 원인은 `lib/products.ts`의 `getProductDetail` select입니다.

```ts
// lib/products.ts:13-20
.select(
  `product_id, name, brand, price, stock, image, description, origin,
   sellers!inner ( seller_id, store_name, cs_phone ),
   ingredients ( ingredient_id, category )`   // ← 여기
)
```

`products`에는 `ingredients`와의 외래키 관계가 없습니다(실제 컬럼은 `category_id` → `categories`). PostgREST가 관계를 못 찾아 `PGRST200` 에러를 내고, 바로 아래 `if (error || !product) return null`에 걸려 **모든 상품의 상세가 null → notFound()**가 됩니다. 목록 페이지는 별도 쿼리라 멀쩡해서 그동안 안 드러났던 것 같습니다. 가이드에 `categories` 조인으로 바꾸는 예시를 적어뒀습니다.

### 최유종 (@jjong0) — 어드민 주문/정산/상품 실DB

어드민 쪽을 mock에서 실DB로 끌어올린 작업이 안정적입니다. 화면으로도 정산 통계(완료 375만원·수수료 75만원)가 실데이터로 잘 떴습니다.

**잘한 점 (칭찬)**
- `app/api/admin/orders/route.ts`·`settlements/route.ts` 모두 `requireAdmin` 가드 뒤에서 동작하고, status 화이트리스트(`VALID_STATUSES`)로 잘못된 상태값을 거릅니다. 페이지네이션도 `limit`을 1~100으로 클램프해 과도한 조회를 막았습니다.
- 특히 `settlements/[settlementId]/route.ts`의 PATCH는 모범적입니다. `parseInt` 검증 → 본문 파싱 실패 처리 → `status === '완료'`만 허용 → **현재 상태가 '대기'인지 먼저 확인하고 이미 완료면 409**로 거르는 멱등성 처리까지 갖췄습니다. 7차에 짚었던 "확정 단계 없음"에 대한 좋은 대답입니다.

**[제안] 검색 키워드를 PostgREST 필터 문자열에 직접 보간하고 있습니다.**
```ts
// admin/products/route.ts:61
query = query.or(`name.ilike.%${keyword}%,seller_id.in.(${matchingSellerIds.join(',')})`);
// admin/orders/route.ts:38
query = query.or(`order_id.ilike.%${keyword}%,recipient.ilike.%${keyword}%`);
```
이건 7차부터 짚던 항목인데, 이번에 주문 라우트에도 같은 패턴이 하나 더 생겼습니다. `keyword`에 쉼표·점·`ilike.` 같은 PostgREST 필터 문법이 들어가면 의도한 검색을 벗어나 다른 조건을 끼워 넣을 수 있습니다(실측에서 `or=(name.ilike.*,status.eq.숨김)` 같은 필터 탈출을 확인). 다행히 두 라우트 모두 `requireAdmin` 게이트 뒤이고 어차피 관리자가 볼 수 있는 데이터라 실질 피해는 낮습니다. 그래도 "사용자 입력을 쿼리 DSL에 직접 합치지 않는다"는 습관을 들이면 좋아서 [제안]으로 둡니다. 가이드에 `keyword`를 escape하거나 OR 분기를 안 쓰는 방법을 적었습니다.

### 추유나 (@nana9147) — 판매자 상품/정산

판매자 상품 목록의 실DB 연동(`app/api/seller/products/db.ts`)이 깔끔합니다.

**잘한 점 (칭찬)**
- `getSellerProducts`에서 `parentId`로 하위 카테고리를 먼저 조회해 `in`으로 거는 방식, 그리고 상품별 연결 레시피 수를 `recipe_ingredients`를 따로 조회해 `Map<number, Set<number>>`로 중복 없이 세는 처리가 군더더기 없이 정확합니다. 여기서는 `ingredients` 같은 잘못된 조인 없이 실제 관계(`categories`, `recipe_ingredients`)를 정확히 짚었습니다.

**판매자 정산 상세는 이번엔 보류로 둡니다.**
`app/seller/settlement/[id]/page.tsx`는 화면 구성(정산 기본정보·금액상세·입금정보·주문별 내역)이 보기 좋게 잘 잡혀 있습니다. 지금은 판매자 상품 실DB 연동처럼 더 중심이 되는 작업에 집중하시는 것 같아서, 정산 상세는 이번 리뷰에서 손볼 거리로 올리지 않겠습니다. 정산 작업을 다시 시작하실 때 그 시점에 맞춰 같이 보겠습니다.

## 팀 공통

- **빌드 상태 좋습니다.** `npx tsc --noEmit` 0건, `npm run build` 통과. 5차까지 5회 연속 빨간불이던 빌드가 6차 이후 계속 green입니다. CI까지 붙어 있으니 이 흐름을 유지합시다.
- **협업은 "한 사람처럼"에 가까워졌습니다.** 주문은 엄인호, 어드민은 최유종, 판매자는 추유나로 영역이 나뉘어 있긴 한데, 인증은 `serverAuth`/`requireSellerContext`로 공유하고, 정산 완료 멱등성·금액 서버재계산처럼 "안전하게 처리한다"는 감각이 세 영역에서 비슷하게 나타납니다. 다만 같은 검색 인젝션 패턴이 products·orders 두 곳에 복붙된 것처럼, 한 사람이 만든 패턴이 다른 영역으로 옮겨갈 때 문제도 같이 복제되는 경향이 보입니다. 공용 헬퍼로 한 번 감싸두면 고칠 때도 한 곳만 고치면 됩니다.
- **mock과 실DB의 성숙도 격차**가 줄긴 했지만 판매자 정산 화면에 아직 남아 있습니다. 시연 동선에 들어가는 화면은 실DB 우선으로 정리하면 좋겠습니다.

## 이번 회차 우선순위 (시연 전 권장 순서)

1. **[필수] `/shopping/[id]` 상세 404 복구** — `getProductDetail`의 `ingredients` 조인을 `categories`로 교체. 사용자 시연 동선의 핵심.
2. **[필수·보안] `sellers` RLS 차단** — anon 사업자/계좌 노출 막기.
3. **[제안] 검색 키워드 인젝션 차단** — products·orders 두 라우트.

판매자 정산 상세는 지금 다른 개발에 집중하시는 것 같아 이번 우선순위에서는 뺐습니다. 정산 작업을 다시 시작하실 때 그때 같이 보겠습니다.

코드 자체는 지난 회차보다 확실히 단단해졌습니다. 위 두 [필수]만 정리하면 월요일 시연은 무리 없이 보여줄 수 있을 거라 생각합니다. 화이팅입니다.
