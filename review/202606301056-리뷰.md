리뷰 포인트는 지난번에 가장 강하게 짚었던 경합 버그였는데, `update(point + amount)` 패턴을 통째로 걷어내고 `award_review_point` RPC 한 번으로 원자 증가하도록 바꿔주셨네요(`reviews/route.ts:141-149`). 더블클릭으로 적립금이 새던 경로가 막혔습니다. 이미지 검증도 개수 상한에 `storageBase` 도메인 검증까지 같이 넣어주셔서(`:83-86`) 외부 URL이 DB로 들어오는 길도 닫혔고요. 좋은 마무리라 생각합니다.

남은 건 SEO입니다. 목록 콘텐츠는 아직 첫 HTML에 안 실려 있어, 이번에도 `[필수]`로 이어갑니다.

## 3. 이번에 새로 본 것 — [필수]

### [필수/보안] 카카오페이 결제 금액 서버 검증 누락

토스 쪽은 잘 돼 있습니다. `app/api/payment/toss/confirm/route.ts:22`에서 `order.final_amount !== amount`를 비교해 클라이언트가 보낸 금액과 DB 금액이 다르면 400으로 막고, PG에 보낼 때도 `:30`에서 클라이언트 값이 아니라 `order.final_amount`를 그대로 씁니다. 이게 정석이라 그대로 칭찬합니다.

문제는 카카오페이가 이 검증을 안 거칩니다.

`app/api/payment/kakao/ready/route.ts:12`에서 클라이언트가 보낸 `totalAmount`를 받아, `:26-27`에서 그 값을 그대로 KakaoPay `total_amount`/`vat_amount`로 넘깁니다. DB의 `final_amount`를 읽는 곳이 없습니다.

```ts
// ready/route.ts:12,26 — 클라이언트 totalAmount를 그대로 PG에 전달
const { orderId, itemName, quantity, totalAmount } = await req.json();
// ...
total_amount: totalAmount,
```

`approve/route.ts`도 마찬가지입니다. `:14-18`에서 orders를 조회하지만 `select`가 `user_id, status`뿐이라 `final_amount`를 안 읽고, KakaoPay 승인 응답의 `amount.total`을 DB 금액과 대조하지 않습니다.

이러면 클라이언트가 `totalAmount`를 임의로 낮춰 보냈을 때 그 금액 그대로 결제가 승인됩니다. 1000원짜리 장바구니를 10원에 결제할 수 있다는 뜻입니다. 토스에는 막혀 있는 구멍이 카카오에는 열려 있는 셈이라 같은 방식으로 메워줘야 하는데, ready에서 DB `final_amount`로 금액을 만들고 approve에서 승인 응답의 `amount.total === final_amount`를 확인하는 흐름입니다(가이드에 코드).

### [필수] `fetchInquiries` 잔류 — unmount 후 setState 누수

`app/admin/support/inquiry/page.tsx`를 보면 데이터 로딩이 두 벌입니다. `:59-76`의 `useEffect` 안 `load()`는 `cancelled` 플래그로 unmount 정리가 잘 돼 있습니다. 그런데 `:47-57`에 예전 `fetchInquiries` 함수가 그대로 남아 있고, `:99`의 `sendReply`에서 이걸 호출합니다.

```ts
// :47-57 — cancellation 없는 옛 버전이 남아 있음
const fetchInquiries = async () => {
  try {
    const { data } = await api.get('/admin/inquiries', { params: { limit: 100 } });
    setInquiries(data.inquiries ?? []);   // 컴포넌트가 사라진 뒤에도 호출될 수 있음
  } ...
};
```

답변을 보낸 직후 사용자가 페이지를 빠르게 떠나면, 응답이 늦게 도착했을 때 이미 사라진 컴포넌트에 `setInquiries`가 실행되면서 누수가 납니다. `load`로 통일하면 깔끔합니다(가이드 참고).
