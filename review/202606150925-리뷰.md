## 2. [필수] 빌드 — 다섯 번째 빨간불, 원인은 한 줄 그대로

`tsc`와 `build` 모두 **딱 1건**에서 멈춥니다. 4차와 같은 자리입니다.

```
hooks/user/useKakaoPostcode.ts:7 - error TS2687:
All declarations of 'kakao' must have identical modifiers.
```

원인은 5-12줄의 전역 선언입니다.

```ts
// useKakaoPostcode.ts (5-12줄 부근)
interface Window {
  kakao: {            // 필수(required)로 선언
    maps: { ... };
  };
}
```

전역으로 병합되는 `Window.kakao`와 optionality(필수/선택)가 충돌합니다. 프로젝트에서 `kakao` 전역 선언은 이 파일이 유일하고(grep 확인), Kakao SDK는 `@types`가 없으며 `skipLibCheck:true`라 라이브러리 쪽은 안 봅니다. 즉 이 한 줄만 `kakao?:`로 고치면 빌드가 풀립니다. 해결은 가이드 1번에 적었습니다.

5회째 같은 자리에서 멈춘다는 건 **머지 전에 빌드를 한 번 돌려보는 사람이 없다**는 뜻이라 생각합니다. CI에 빌드 게이트 하나만 걸면 이 패턴은 정리됩니다(가이드 5번).

## 3. [필수·보안] 결제 — 인증은 닫았습니다, 그러나 금액은 아직 클라이언트가 정합니다

닫힌 것부터 보겠습니다.

- `0b6074c`로 `toss/confirm`·`kakao/ready`·`kakao/approve` **세 라우트 모두** `requireAuth(req)`를 겁니다(`route.ts:7-9`). 4차에 만들어 둔 `serverAuth`를 결제에 실제로 갖다 쓴 셈이라 좋습니다.
- `partner_user_id`도 클라가 보낸 값이 아니라 `String(authed.userId)` — 서버 토큰에서 꺼낸 값을 씁니다(`kakao/ready:23`). 이게 정답입니다.
- 시크릿 키는 여전히 서버 라우트에만 격리돼 있습니다.

남은 [필수] 한 가지입니다.

```ts
// useCheckoutPayment.ts:5 (클라이언트)
const FINAL_AMOUNT = 35410;   // 금액이 여전히 클라이언트에 하드코딩
// :35 — userId: 'test_user' 잔존
```
```ts
// payment/toss/confirm/route.ts:18 (서버)
const { paymentKey, orderId, amount } = await req.json();
// 받은 amount를 그대로 토스 confirm에 전달. 서버가 주문 금액을 다시 계산하지 않음
```

인증을 막아 둔 덕에 "누가" 결제하는지는 이제 서버가 압니다. 다만 "얼마를" 결제하는지는 아직 클라이언트 손에 있습니다. 사용자가 요청 본문의 `amount`를 100원으로 바꿔도 서버 어디에서도 진짜 주문 금액과 대조하지 않습니다. **주문을 DB에 영속화하고, 결제 확정 때 서버가 그 주문에서 금액을 읽어 대조하는 단계**가 필요합니다. 해결 흐름은 가이드 3번에 예시 코드로 적었습니다.

`userId: 'test_user'`(`useCheckoutPayment.ts:35`, `Complete.tsx:30`)는 서버가 무시하긴 하나, 읽는 사람을 헷갈리게 합니다. 인증 토큰으로 일원화됐으니 이제 지워도 됩니다.[제안]

## 5. 세 명이 "한 사람처럼" 가고 있나 — 협업 진단

이번 판정은 **절반의 통합**이라 생각합니다. **인프라·백엔드 계층은 한 사람처럼**, **UI·표현 계층은 아직 따로** 갑니다.

**한 사람처럼 가는 증거(통합).**
- `lib/serverAuth.ts`(엄인호)의 `requireAuth`/`requireSeller`/`requireAdmin`을 결제 4라우트 + admin 대시보드가 **다 같이** 씁니다. 인증 패턴이 한 손에서 나온 것처럼 일관됩니다.
- `lib/api.ts` axios·`store/authStore`도 공용입니다.
- 폴더 평행성이 좋습니다. `admin/components/{Header,Sidebar}` ↔ `seller/components/{Header,Sidebar}`가 같은 형태입니다.
- 머지 레벨 협업이 돕니다(서로 PR 머지 #108, #105, #101).

**아직 따로 가는 증거(분리).**
- (결정적) **`StatusBadge`가 두 벌입니다.** 추유나는 재사용 컴포넌트 `app/seller/components/StatusBadge.tsx`(union + Record 스타일맵)로 통일했는데, 최유종의 admin은 페이지마다 인라인으로 다시 만듭니다(`app/admin/members/page.tsx:76-84`의 `gradeBadge`/`statusBadge` 리터럴). **같은 문제를 둘이 다른 방식으로** 풀고 있습니다.
- **공통 훅이 seller 사일로 안에만** 있습니다. `usePagination`/`useFilter`/`StatusCards`의 소비처가 전부 `app/seller/*`입니다. admin은 안 쓰고 `members/page.tsx:87-108`처럼 `useState` + 인라인 `.filter()`를 직접 굴립니다.
- 데이터 성숙도 격차. 대시보드는 실 DB인데 admin members/orders/products/sellers는 여전히 mock입니다.
- 이번 라운드 cross-area 편집이 0건입니다. 각자 자기 폴더만 손댔습니다.
- 커밋 컨벤션이 미세하게 갈립니다. nana/djsy01은 `feat: `(콜론 뒤 공백 O), jjong0은 `feat:`(공백 X).

여기서 단정은 피하겠습니다. **영역을 셋으로 나눠 맡기로 팀에서 합의하신 건가요?** 그렇다면 분담 자체는 합리적입니다. 다만 합의된 분담이라도 **공통 자산은 합치는 게** 좋다고 봅니다. 같은 배지를 두 벌 유지하면 색을 한 번 바꿀 때 두 곳을 고쳐야 하고, 한 곳을 빠뜨리면 화면마다 배지가 달라 보입니다.

**코드에 녹일 처방 네 가지.**
1. **`StatusBadge`·`useFilter`·`usePagination`·`StatusCards`를 `components/common`(또는 `ui`)으로 끌어올리고**, admin이 seller 것을 import하게 합니다. admin의 인라인 badge/filter를 지웁니다. 이게 가장 눈에 보이는 "한 사람처럼" 신호입니다.
2. **admin mock → 실 DB.** 대시보드에서 패턴이 이미 확립됐으니 옆 페이지에 복제하면 됩니다.
3. **커밋 콜론 뒤 공백**을 README나 PR 템플릿에 한 줄로 못 박습니다.[사소]
4. 서로의 PR에 한 번이라도 실제 코멘트를 달거나 소소한 개선 커밋으로 **cross-review**해 보세요. 자기 폴더만 보던 시야가 풀립니다.

토대(serverAuth·axios·authStore)는 이미 한 사람처럼 잘 맞춰져 있습니다. 여기에 **표현 계층의 공통 자산까지 합치면** 진짜 한 손에서 나온 코드가 됩니다.
