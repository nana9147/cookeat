## 2. 이번 핵심 — 공개 페이지가 크롤러 눈에 어떻게 보이나

SEO는 어려운 기술이 아니라 "**자바스크립트를 실행하기 전의 HTML**에 내 콘텐츠가 들어 있느냐"가 거의 전부라고 보면 됩니다. 크롤러(특히 가격 비교·일부 SNS 봇)는 JS를 안 돌리거나 늦게 돌리기 때문에, 첫 HTML이 비어 있으면 "내용 없는 페이지"로 취급됩니다. 이 기준으로 공개 라우트 5개를 점검했습니다.

### 페이지별 SSR/CSR 점검

| 공개 페이지 | 렌더링 방식 | 첫 HTML에 콘텐츠 | 메타데이터 | 위험도 |
| --- | --- | --- | --- | --- |
| `/` 메인 | **CSR** (홈 섹션이 `'use client'` + `useEffect` fetch) | 비어 있음(추천/오늘의 레시피/카테고리 모두 클라에서 채움) | 루트 정적 'Cookeat' | **높음** |
| `/recipes` 레시피 목록 | **CSR** (`RecipeClient.tsx` `'use client'`, `useEffect`에서 axios) | 그리드 비어 있음 | 'Cookeat' (고정) | **높음** |
| `/shopping` 상품 목록 | **CSR** (`ShoppingClient` → `useShoppingFilter` `useEffect` fetch) | 그리드 비어 있음 | 'Cookeat' (고정) | **높음** |
| `/recipes/[id]` 레시피 상세 | **SSR** (async 서버 컴포넌트, `fetchRecipeDetail` 서버 호출) | 들어 있음 | 'Cookeat' (동적 없음) | 중간(메타만) |
| `/shopping/[id]` 상품 상세 | **SSR** (async 서버 컴포넌트, `getProductDetail` 서버 호출) | 들어 있음 | 'Cookeat' (동적 없음) | 중간(메타만) |

### 잘 된 부분부터

상세 페이지 두 개는 이미 **SSR 모범 사례**입니다. `app/(main)/recipes/[id]/page.tsx:16`이 `async function` 서버 컴포넌트로 `:19`에서 `await fetchRecipeDetail(idNum)`을 서버에서 부르고, `app/(main)/shopping/[id]/page.tsx:14`도 같은 방식으로 `await getProductDetail(productId)`을 부릅니다. 잘못된 id면 `notFound()`로 404까지 정확히 내려갑니다. **이 패턴이 정답이라, 목록 페이지를 고칠 때 이걸 본보기로 삼으면 됩니다.**

루트 레이아웃도 `app/layout.tsx:29`에서 `lang="ko"`를 제대로 박아놨습니다(한국어 사이트인데 `lang="en"`이면 검색·접근성 양쪽에서 손해라, 이건 잘 챙긴 부분입니다).

### 고쳐야 할 부분 — 목록·메인이 CSR이라 크롤러에 빈 화면

가장 큰 SEO 문제는 **사람들이 검색으로 가장 먼저 들어올 메인과 목록 페이지가 정작 CSR**이라는 점입니다.

- **[필수/SEO]** `/recipes` — `RecipeClient.tsx:1`이 `'use client'`이고, 레시피 데이터를 `:44` `await api.get('/recipes', ...)`로 `useEffect` 안에서 가져옵니다(`:47` `res.data.data.recipes`). 즉 서버가 내려주는 HTML엔 레시피 카드가 **하나도 없고**, 브라우저가 JS를 돌려야 채워집니다. 크롤러는 빈 그리드를 봅니다.
- **[필수/SEO]** `/shopping` — `useShoppingFilter.ts:100`의 `fetch('/api/products?...')`도 `useEffect`(`:76`) 안이라 동일하게 빈 HTML입니다.
- **[필수/SEO]** `/` 메인 — `components/home/PopularRecipeSection.tsx:1`, `TodayRecipeCard.tsx:1`, `CategorySection.tsx:1` 모두 `'use client'`에 `useHomeData` 훅으로 채웁니다. 메인 첫 화면 콘텐츠가 통째로 클라에서 그려집니다.

해결 방향은 상세 페이지처럼 **목록 페이지를 async 서버 컴포넌트로 바꿔, URL의 `searchParams`(카테고리·정렬·페이지)를 읽어 서버에서 데이터를 가져오고**, 정렬 셀렉트·카테고리 탭 같은 인터랙션만 작은 클라이언트 자식으로 떼어내는 겁니다. 단계별 코드는 같이 드린 가이드에 정리했습니다.

### 메타데이터 — 지금은 모든 페이지가 'Cookeat'

- **[필수/SEO]** `app/layout.tsx:17-20`의 메타데이터가 정적입니다. `title: 'Cookeat'`에 description은 영문(`'A recipe sharing platform built with Next.js and Tailwind CSS.'`)이고, `metadataBase`가 없습니다. 그래서 김치찌개 레시피 상세든 어떤 상품 상세든 검색결과·SNS 카드 제목이 전부 똑같이 "Cookeat"으로 나옵니다. description도 한국어 서비스인데 영문 한 줄로 고정돼 있습니다.
- **[필수/SEO]** 어디에도 `generateMetadata`가 없습니다. 다행히 상세 페이지는 이미 `recipe.title`/`product.name`을 **서버에서 들고 있으니**, `generateMetadata`만 붙이면 페이지마다 제목·설명을 동적으로 만들 수 있습니다(거의 추가 비용 없음).
- **[필수/SEO 기반]** `app/sitemap.ts`, `app/robots.ts`가 없습니다. 크롤러에게 "이런 페이지들이 있다"고 목록을 건네는 sitemap과 출입 규칙(robots)을 깔아두면 색인 속도와 범위가 달라집니다.

SEO 개념 전반(크롤러가 무엇을 보는지, Metadata API, Open Graph, JSON-LD, sitemap/robots)은 별도로 정리해 둔 SEO 가이드를 함께 보면 좋습니다. 이번 가이드 문서에서 그 문서를 참조하도록 연결해 뒀습니다.

## 4. 팀 공통

- **SEO를 팀 차원에서 한 번 정해두면 좋겠습니다.** 목록·메인 CSR→SSR 전환은 한 사람 일이 아니라 라우트 구조에 걸쳐 있어서, 누가 어느 페이지를 맡을지 가볍게 나눠두면 흩어지지 않습니다. 상세 페이지가 이미 SSR로 잘 돼 있으니 그걸 공통 본보기로 공유하면 손이 빠를 겁니다.
- **응답 래퍼 모양을 팀에서 하나로 통일**해 두면 좋겠습니다. 지금 어떤 라우트는 `{ success, data: {...} }`, 어떤 라우트는 `{ recipes, pagination }`처럼 평평하게 내려와, 클라에서 `res.data.data`냐 `res.data`냐가 컴포넌트마다 달라지고 있습니다(LikesList vs RecipeClient). 응답 봉투 규칙을 문서 한 줄로 정해두면 위 같은 `undefined` 사고가 줄어듭니다.
- **`as` 타입 단언은 검증이 아니다**라는 감각을 팀이 공유하면 좋겠습니다. tab(refunds)·role(login) 모두 `as`로 "이 타입"이라고 우기지만 런타임에선 아무거나 들어올 수 있어서, 외부 입력은 화이트리스트/스키마로 걸러야 합니다.

## 5. 이번에 가장 먼저 손볼 Top 3

1. **[필수/SEO] 레시피·상품 목록 페이지와 메인을 SSR로 전환** — `/recipes`, `/shopping`, `/`가 크롤러에 빈 화면입니다. 상세 페이지를 본보기로 async 서버 컴포넌트 + `searchParams`로 바꿉니다.
2. **[필수] 리뷰 포인트 적립 원자적 증가 + 리뷰 이미지 검증** — `products/[productId]/reviews/route.ts:135-152`(race)와 `:128-131`(이미지 미검증). recipes 리뷰 라우트도 동일.
3. **[필수/SEO 기반] `generateMetadata` + `metadataBase` + `sitemap.ts` + `robots.ts`** — 상세 페이지는 이미 데이터를 서버에서 들고 있어 메타만 붙이면 됩니다.

천천히, 그렇지만 하나씩 확실히 갑시다. 해결 단계는 가이드 문서에 코드와 함께 정리해 뒀습니다. 화이팅입니다.
