# Cookeat 7차 코드 리뷰 (2026-06-17)

6차 이후 develop에 들어온 변경이 꽤 많습니다(관리자 페이지 실 API 연동, 판매자 정산 관리·상세, 상품 상세 페이지, proxy 보안 강화 등). 한꺼번에 다 보면 정확도가 떨어져서, 이번엔 **신규 라우트와 API route, 핵심 컴포넌트 위주**로 봤습니다. 오늘 초점은 **응집도·결합도**와 **React·Next·TypeScript를 제대로 쓰고 있는지**입니다. 관리자/판매자 API라 **인증·인가**도 함께 봤습니다.

먼저 좋은 소식 둘입니다. **빌드가 깨끗합니다**(`npx tsc --noEmit` 0건). 그리고 **3라운드 끌어온 판매자 화면 가드가 드디어 닫혔습니다.** E2E로 비로그인 상태에서 `/seller/*`에 들어가 봤더니 전부 `/login`으로 리다이렉트됩니다. 이건 뒤에서 다시 짚을게요.

## 2. 신규 라우트 / API — 인증·인가 점검 (보안)

이번에 admin·seller API가 많이 늘어서 권한 가드부터 봤습니다. **결론부터 말하면 데이터 보호선은 잘 닫혀 있습니다.**

- **admin API 전부 가드 정상**: `GET /api/admin/products`(`route.ts:9`), `PATCH/DELETE .../[productId]`(`:13`,`:73`), `GET /api/admin/sellers`(`route.ts:16`), `PATCH .../[sellerId]/approve`(`:13`) — 모두 첫 줄에서 `requireAdmin(req)` 호출 후 `instanceof NextResponse` 가드. 입력검증도 양호(status enum/price/stock 범위, commissionRate 0~100, `parseInt`+`isNaN` 가드).
- **auth API**: `/api/auth/refresh`·`/api/auth/sync-session` 모두 토큰 검증 후 쿠키 동기화. 정상.

### [필수/주의] proxy.ts의 주석과 구현이 어긋납니다 (`proxy.ts:74`)
주석엔 "JWT 서명으로 보호된 app_metadata에서 role을 읽는다"고 적혀 있습니다. 그런데 실제 `decodeJwtPayload`(`:13-20`)는 **서명 검증 없이 base64 디코드만** 합니다. 토큰에 role이 없으면 `sb-role` **쿠키 fallback**(`:76`)을 쓰는데, 이 쿠키는 클라이언트가 위조할 수 있고요. 그러니 **proxy 가드는 신뢰 경계가 아니라 "낙관적 체크"** 인 셈입니다.

다행히 실제 데이터 보호선인 admin/seller **API는 전부 `requireAdmin`/`requireSeller`로 서버에서 `getUser`를 다시 검증**하니 실질 보안은 닫혀 있습니다(Next 공식 문서도 proxy를 full authorization으로 쓰지 말라고 명시합니다). 문제는 **주석 문구가 과장**돼 있다는 점입니다. 다음 사람이 "여기서 다 막혔다"고 오해해 API 가드를 빼먹으면 그때 진짜 구멍이 생기거든요. 주석을 실제에 맞게 고치고, "진짜 게이트는 API의 serverAuth"라는 걸 한 줄 남겨주세요. 보안 자체는 닫혀 있으니 [필수]라기보다 "주석/오해 방지" 차원입니다.

### [제안] admin 검색 keyword를 PostgREST 필터 문자열에 직접 보간 (`products/route.ts:61`, `sellers/route.ts:41`)
`.or(\`name.ilike.%${keyword}%,seller_id.in.(...)\`)` 처럼 사용자 keyword를 필터 DSL에 그대로 끼워 넣습니다. SQL 인젝션은 아닙니다. 다만 `keyword`에 `,`·`)`·`.`이 들어오면 필터가 깨지거나(500) 의도치 않은 OR 조건이 될 수 있습니다(PostgREST filter injection). admin 전용이라 위험도는 낮지만, 콤마/괄호를 이스케이프하거나 `%`만 허용하는 sanitize 한 줄을 권합니다.

## 4. 응집도·결합도 핵심 (오늘 주제)

1. **[칭찬] 응집도 — 상품 상세/정산의 컴포넌트 분할이 모범적입니다.** 거대 페이지를 책임 단위(`_components/`, props 인터페이스)로 쪼갰고, 공통 자산(ReviewCard·StarRating·Pagination·StatusBadge)을 `components/common`·`ui`로 끌어올려 영역 간 재사용을 늘렸습니다. 6차에 짚은 "표현 계층 통합"이 한층 진행된 모습입니다.

2. **[제안] 결합 — proxy(낙관적)와 serverAuth(실 게이트)의 역할 경계를 명확히.** 지금은 proxy 주석이 과장된 탓에 "어디가 진짜 보안선인지" 흐릿합니다. 둘의 책임을 한 줄로 정리해두면, 다음 사람이 proxy만 믿고 API 가드를 빠뜨리는 실수를 막을 수 있습니다.

3. **[제안] mock과 실 DB 혼재가 "영역별 성숙도 격차"로 굳어지는 중입니다.** admin은 실 API로 갔는데 seller(settlement/reviews)는 mock에 머물러 있습니다. 특히 settlement/[id]가 params를 안 읽는 건 결합을 따지기 전에 라우팅 미연결이라 우선순위가 높습니다.

4. **[제안] 페이지네이션 계산 로직 중복** — admin/products의 `getPageNumbers`가 페이지마다 인라인으로 들어가 있습니다. 6차에 공통화한 Pagination에 이 계산까지 넣으면 결합이 줄어듭니다.

5. **[제안] 사용자 입력과 쿼리 DSL의 결합** — PostgREST 필터 문자열에 keyword 직접 보간(2장). sanitize 레이어로 분리.
