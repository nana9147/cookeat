## 1. [필수] `/shopping/[id]` 상품 상세 404 복구

### 무엇이 문제인가
`lib/products.ts`의 `getProductDetail`이 `products`에 존재하지 않는 `ingredients` 관계를 조인하고 있습니다. PostgREST는 관계를 못 찾으면 쿼리 전체를 에러로 처리하는데, 코드가 `if (error || !product) return null`로 받아 **모든 상세 페이지가 null → notFound()** 가 됩니다. 목록은 다른 쿼리라 멀쩡해서 그동안 안 보였던 것뿐입니다.

직접 확인해 보면 에러 메시지가 친절하게 정답을 알려줍니다.
```
PGRST200: Could not find a relationship between 'products' and 'ingredients'
hint: Perhaps you meant 'recipe_ingredients' instead of 'ingredients'.
```

`products`의 실제 관계는 `category_id → categories` 입니다(`recipe_ingredients`는 레시피-상품 연결용이라 상세 카테고리 표시 용도와는 다릅니다).

### 어떻게 고치나
`ingredients` 조인을 `categories`로 바꾸고, 매핑부도 그에 맞춥니다.

```ts
// lib/products.ts
const { data: product, error } = await supabaseAdmin
  .from('products')
  .select(
    `product_id, name, brand, price, stock, image, description, origin,
     sellers!inner ( seller_id, store_name, cs_phone ),
     categories ( category_id, name )`   // ingredients → categories
  )
  .eq('product_id', id)
  .eq('status', '판매중')
  .maybeSingle();

if (error || !product) return null;
// ...
const category = product.categories as unknown as { category_id: number; name: string } | null;
// ...
return {
  // ...
  category: category?.name ?? '',   // 기존 ingredient?.category 자리
  // ingredientId가 타입에 꼭 필요하면 우선 null로 두거나 타입에서 제거
};
```

`ProductDetail` 타입에 `ingredientId`가 있다면, 지금 화면에서 실제로 쓰지 않는다면 타입에서 빼는 게 깔끔합니다. 쓰고 있다면 별도 쿼리로 분리하세요(판매자 상품 목록에서 추유나님이 `recipe_ingredients`를 따로 조회한 패턴이 좋은 참고입니다).

### 확인 방법
```bash
curl -s -o /dev/null -w "%{http_code}\n" http://localhost:3000/shopping/4   # 200 이어야 정상
```
실제 판매중 상품 id(예: 4, 5, 7, 8, 9)로 들어가 상세가 뜨면 됩니다.

## 2. [필수·보안] `sellers` 테이블 anon 노출 차단

### 무엇이 문제인가
로그인 없이(anon publishable 키만으로) 아래가 그대로 읽힙니다.
```bash
curl -s "$SUPABASE_URL/rest/v1/sellers?select=*" -H "apikey: $ANON" -H "Authorization: Bearer $ANON"
# → business_number, bank_account, representative_name, cs_phone 까지 노출
```
`users`·`orders`는 잘 막혀 있는데 `sellers`만 열려 있습니다. 사업자등록번호·계좌번호는 민감정보라 공개되면 안 됩니다.

### 어떻게 고치나
두 가지 방향이 있습니다. 시연 전이라면 **(A) 차단**이 가장 빠르고 안전합니다.

**(A) sellers를 anon에서 전면 차단하고 서버 라우트로만 읽기 (권장)**
서버 API는 `supabaseAdmin`(service-role)로 읽으니 차단해도 동작에 지장이 없습니다.
```sql
-- Supabase SQL editor
alter table sellers enable row level security;

-- 기존에 anon에 열어준 정책이 있으면 삭제
-- (정책 이름은 대시보드 Authentication > Policies에서 확인)
drop policy if exists "sellers anon read" on sellers;

-- 본인 판매자 정보만 읽기 허용 (선택)
create policy "own seller read"
on sellers for select
to authenticated
using ( user_id = (select user_id from users where auth_id = auth.uid()) );
```

**(B) 가게명 등 공개 가능한 컬럼만 별도 뷰로 분리**
상품 상세에 가게명을 보여줘야 한다면 민감 컬럼을 뺀 뷰를 만들어 그것만 공개합니다.
```sql
create view public_sellers as
  select seller_id, store_name from sellers;
-- 클라이언트는 public_sellers만 읽고, sellers 원본은 차단
```

### 확인 방법
고친 뒤 anon으로 다시 GET 했을 때 `[]`(또는 본인 행만)이 나와야 합니다. 동시에 로그인한 판매자가 `/seller/info`에서 자기 정보를 정상적으로 보는지(서버 라우트 경유) 같이 확인하세요.

## 4. [제안] 어드민 검색 키워드 인젝션 차단

### 무엇이 문제인가
```ts
query = query.or(`name.ilike.%${keyword}%,seller_id.in.(${matchingSellerIds.join(',')})`);
```
`keyword`에 `*,status.eq.숨김` 같은 PostgREST 필터 문법이 들어가면 의도한 검색을 벗어납니다. admin 게이트 뒤라 실해는 낮지만, 사용자 입력을 쿼리 DSL 문자열에 직접 합치지 않는 습관을 들이면 좋습니다.

### 어떻게 고치나
PostgREST에서 `or()` 안의 값에 들어가면 곤란한 문자는 `,` `.` `(` `)` `*` 등입니다. 가장 간단한 건 위험 문자를 제거하거나, OR 분기를 없애는 것입니다.

**(A) keyword를 정제(권장)**
```ts
// 필터 문법에 쓰이는 문자 제거 — 검색어로는 사실상 불필요한 문자들
const safeKeyword = keyword.replace(/[,.()*]/g, '').trim();
```
그리고 `safeKeyword`만 보간에 사용합니다.

**(B) OR이 꼭 필요할 때는 두 쿼리로 나눠 합치기**
이름 검색 결과와 seller_id 검색 결과를 각각 조회해 애플리케이션에서 합치면 필터 문자열에 keyword를 넣지 않아도 됩니다. 코드가 길어지니 시연 전이라면 (A)로 충분합니다.

`admin/products/route.ts:61`과 `admin/orders/route.ts:38` 두 곳 모두 같은 패턴이니 함께 고치세요. 검색 정제를 작은 헬퍼(`sanitizeKeyword`)로 빼두면 다음에 또 복붙해도 안전합니다.
