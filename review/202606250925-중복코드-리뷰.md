# Cookeat 중복 코드 리뷰 (2026-06-25)

이번엔 **반복·복붙된 코드**를 한 번 훑었습니다. 같이 드린 「리팩토링 가이드」가 원칙이라면, 이 문서는 그걸 Cookeat 코드에 대입한 결과입니다. 무조건 다 합치라는 얘기가 아니라, 지금 빼면 이득인 것과 둬도 되는 것을 나눠 정리했습니다.

먼저 칭찬부터. **인프라 레이어는 거의 흠잡을 데가 없습니다.** Supabase 클라이언트(`lib/supabase.ts`·`lib/supabaseAdmin.ts`), 인증 가드(`lib/serverAuth.ts`의 `requireAuth/requireSeller/requireAdmin`), axios 인스턴스(`lib/api.ts`의 토큰 인터셉터 + 401 자동갱신 큐), 배송비 상수(`lib/shipping.ts`)까지 잘 추상화돼 있어요. 초보 단계 프로젝트치고 인프라 중복이 거의 없습니다. 그래서 아래 중복은 대부분 그 위의 페이지·컴포넌트·타입 레이어에 몰려 있습니다.

---

## 1. [필수] 유틸이 이미 있는데 5곳이 똑같이 다시 만들었습니다 — `getPageNumbers`

가장 명확한 사례입니다. `lib/utils.ts:14`에 **이미 `getPageNumbers(currentPage, totalPages)`가 export돼 있는데**, 다섯 개 admin 페이지가 똑같은 함수를 각자 로컬로 재선언했어요.

- `lib/utils.ts:14` (정식 유틸)
- `app/admin/products/page.tsx`, `app/admin/members/page.tsx`, `app/admin/orders/page.tsx`, `app/admin/sellers/page.tsx`, `app/admin/reviews/page.tsx` (각자 로컬 재선언)

```ts
function getPageNumbers(): (number | string)[] {
  if (totalPages <= 7) return Array.from({ length: totalPages }, (_, i) => i + 1);
  if (page <= 4) return [1, 2, 3, 4, 5, '...', totalPages];
  // ... 문자 단위까지 동일
}
```

이건 추출을 못 해서가 아닙니다. 누군가 이미 `lib/utils.ts`에 만들어 뒀는데, 팀이 그게 있다는 걸 발견하지 못해서 다들 새로 짠 겁니다. `import { getPageNumbers } from '@/lib/utils'` 한 줄로 바꾸고 인자만 넘기면 끝납니다. 망설일 것 없이 지금 제거하세요.

## 2. [필수] DB 행 타입을 admin이 또 정의했습니다 — `types/seller`와 중복

`types/seller/{product,order,user,settlement}.ts`에 정식 인터페이스가 있는데, admin 페이지들이 같은 DB 행을 자기 파일 안에서 다시 선언합니다.

- `types/seller/product.ts:21 Product` vs `app/admin/products/page.tsx:46 Product`
- `types/seller/order.ts:19 Order` vs `app/admin/orders/page.tsx:41 Order`
- `types/seller/settlement.ts:9 Settlement` vs `app/admin/settlements/page.tsx:13 Settlement`
- `Recipe`가 `app/admin/recipes/page.tsx:19`와 `recipes/data/mockRecipes.ts:22` 양쪽에

같은 응답 타입이 두 군데서 따로 진화하면, 필드를 추가할 때 한쪽만 고쳐서 런타임 불일치가 납니다. **DB 행 타입은 한 곳에서만** 정의하도록 `types/`로 모으세요. 다만 admin 화면 전용 가공 타입(표시용 파생 필드)까지 억지로 합칠 필요는 없습니다 — DB 행 타입만 단일 출처로 두면 됩니다.

## 3. [제안] `{x.toLocaleString()}원`이 32개 파일에 흩어져 있습니다

`{값.toLocaleString()}원` 패턴이 최소 32개 파일에 인라인으로 박혀 있습니다(seller 정산·주문·배송에 집중).

- `SettlementTable.tsx:46`, `SettlementHeroCard.tsx:28`, `OrderPaymentSection.tsx:16`, `ProductTable.tsx:90` 등

단순해 보여도, 나중에 "원" → "₩" 표기나 음수·null 처리 규칙이 바뀌면 32곳을 다 손대야 합니다. `lib/format.ts`에 `formatWon(n)` 하나로 모으세요. 빈도가 워낙 높아 지금 빼는 게 이득입니다.

```ts
export const formatWon = (n?: number | null) =>
  n == null ? "-" : `${n.toLocaleString()}원`;
```

날짜 포매팅도 방식이 제각각입니다 — `toLocaleDateString('ko-KR')`(admin), `toLocaleDateString()`(로케일 없음, seller), 직접 `getFullYear/padStart`(`support/inquiry/page.tsx:99`에 `formatDate`를 두 파일에 각자 정의), `split('T')[0].replace(/-/g,'.')`(`sellers/page.tsx:69`)가 섞여 있어요. 화면마다 날짜 표기가 미묘하게 달라질 위험이 있으니 `formatDate(iso)` 하나로 통일하세요.

## 4. [제안] 입력 필드 스타일과 FormField를 만들어 놓고 안 씁니다

같은 Tailwind 클래스 문자열 `inputCls`가 8개 폼 파일에 각각 선언돼 있습니다.

```tsx
const inputCls = 'h-11 px-4 rounded-lg border border-border text-sm outline-none focus:border-primary transition-colors';
```

- `login/LoginFields.tsx:3`, `RegisterFormFields.tsx:5`, `ProfileInfoFields.tsx:3`, `seller/FormField.tsx:3` 등

이미 `components/ui/input.tsx`(shadcn)가 있는데 폼들이 이를 안 쓰고 매번 클래스를 손으로 씁니다. 게다가 `components/(auth)/mypage/seller/FormField.tsx`에 재사용 가능한 `FormField`가 있는데 거의 안 쓰이고, 다른 폼들은 `label + input + error` 구조를 매번 인라인으로 짜요. `FormField`를 `components/ui/`로 승격해 로그인·회원가입·프로필 폼이 공유하게 하면, 포커스 색·높이 같은 디자인 토큰이 한 곳에 모입니다.

## 5. [제안] 상태 문자열과 StatusBadge가 갈라졌습니다

`'판매중'`, `'배송중'`, `'결제완료'`, `'정산완료'` 같은 한글 상태값이 30+ 파일에 흩어져 있고, `StatusBadge`가 두 벌(`components/common/StatusBadge.tsx` vs `app/seller/components/StatusBadge.tsx`)인데 같은 상태에 서로 다른 색(`bg-green-100` vs `bg-emerald-50`)을 매핑합니다.

오타 하나(`'배송중'` vs `'배송 중'`)면 조건 분기가 조용히 깨집니다. `lib/constants/status.ts`에 `ORDER_STATUS`/`PRODUCT_STATUS` 상수와 색상 맵을 모아 단일 출처로 두고, StatusBadge는 하나로 합쳐 필요하면 `variant` prop으로 분기하세요.

## 6. [제안] admin 목록 페이지가 통째로 복제됐습니다 — 단, 두 조각만

`app/admin/{members,products,orders,sellers}/page.tsx`가 거의 같은 골격입니다. 검색 debounce useEffect가 동일하고, 데이터 페칭 블록(`cancelled` 플래그 + `setLoading` + `api.get` + `finally`)이 8곳 이상 반복됩니다.

```ts
const timerRef = useRef<ReturnType<typeof setTimeout> | null>(null);
timerRef.current = setTimeout(() => {
  setDebouncedSearch(search);
  setPage(1);
}, 400);
```

여기서 주의. **한 번에 전부 제네릭 `<AdminListPage>`로 추상화하는 건 초보 단계에선 과합니다**(디버깅이 어려워져요). 대신 두 조각만 먼저 빼세요 — `useDebounce(value, 400)` 훅과 데이터 페칭 훅. 이 둘만 빼도 각 페이지에서 30~40줄이 사라지고, 나머지 테이블 마크업은 페이지마다 다르니 남겨둬도 무방합니다.

## 7. [사소] 지금은 둬도 되는 것

- **API 라우트의 `instanceof NextResponse` 가드**(44개 라우트): 가드 두 줄을 `withSeller(handler)` 고차함수로 묶을 수도 있지만, 미들웨어 추상화는 디버깅 난이도를 올립니다. 그보다 **응답 형식 통일이 먼저**입니다 — 지금 `{error}`/`{errors}`/`{success:false,error}`로 섞여 있는데, `ok(data)`/`fail(message, status)` 헬퍼 두 개로 형식만 맞추는 것부터 하세요. HOF 래퍼는 그다음이라도 늦지 않습니다.
- **`isLoading + try/finally` 미니 반복**(`useAvatarUpload.ts`, `useNicknameCheck.ts`): Rule of Three에 막 닿은 수준이고 맥락이 조금씩 다릅니다. 더 쌓이면 `useAsync`로 정리하고, 지금은 두세요.
- **Pagination 컴포넌트 두 벌**: `components/ui/Pagination.tsx`로 통일하면 좋지만 우선순위는 낮습니다.

## 한 줄 정리

**인프라는 잘 추상화해 놓으셨는데, 그 위에서 "공통 유틸·타입·컴포넌트를 이미 만들어 두고도 팀 내 공유가 안 돼 각자 다시 짜는" 패턴**이 핵심입니다 — `getPageNumbers`·`FormField`·`types/seller`가 멀쩡히 있는데 admin/auth 화면들이 복붙했어요. 코드량 문제라기보다 "팀 공유·발견 가능성" 문제에 가깝습니다. 우선순위는 ① `getPageNumbers` import로 교체(1번) → ② DB 행 타입 단일화(2번) → ③ `formatWon`/`formatDate`(3번) → ④ FormField·상태 상수(4·5번) 순서를 권합니다.
