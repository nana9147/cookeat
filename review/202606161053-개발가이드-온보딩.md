# Cookeat 새 조원 온보딩 — 개발 가이드 (2026-06-16)

새로 합류한 분이 이 문서 하나로 우리 프로젝트의 화면·API·데이터 흐름을 잡을 수 있게 정리했습니다. 기존 조원도 흐름을 다시 맞추는 데 쓰면 됩니다. 로그인·헤더·공통 UI는 이미 있으니, 여기서는 그 위에 새 페이지와 API를 얹는 법만 다룹니다.

읽는 순서는 0번부터 차례대로가 좋습니다. 마지막 6번 체크리스트는 실제로 기능 하나를 만들 때 옆에 띄워 두고 따라가면 됩니다.

---

## 0. 프로젝트 한눈에

먼저 폴더가 어떻게 나뉘는지 봅니다. Cookeat은 `src/`를 쓰지 않고 **루트 바로 아래**에 폴더가 펼쳐져 있습니다.

```
app/            라우트(화면) + API 라우트
  (main)/       일반 사용자 화면 — shopping, board, mypage, recipes, cart
  (auth)/       로그인/회원가입 — login, register
  admin/        어드민 화면 — members, products, orders, sellers ...
  seller/       판매자 화면 — products, orders, shipping ...
  api/          ★ 서버 API (Next API Route) — auth, admin, seller, order, payment, user
components/
  ui/           shadcn/ui 기반 공통 UI (Button, Input, Table, Pagination ...)
  common/       프로젝트 공통 컴포넌트 (StatusBadge 등)
  Header/ Footer/ ...
hooks/          공통 훅 (usePagination, useFilter) + user/ auth/
lib/            api(axios), serverAuth, supabase, supabaseAdmin, validators ...
store/          Zustand 스토어 (authStore, headerStore)
services/       도메인별 호출 묶음 (authService 등)
types/          타입 정의
```

`app/(main)`, `app/(auth)`처럼 괄호로 묶인 폴더는 **라우트 그룹**입니다. URL에는 `(main)`이 안 들어갑니다. `/shopping`, `/login`처럼 됩니다. 화면을 같은 레이아웃·성격끼리 묶어 정리하려고 쓰는 것이지 경로를 만들지는 않습니다.

데이터가 도는 큰 줄기는 이렇게 한 방향입니다.

```
[화면 컴포넌트]
   │  api.get('/admin/users')        ← lib/api.ts (axios). 토큰 자동 첨부
   ▼
[API Route]  app/api/.../route.ts
   │  requireAuth / requireSeller / requireAdmin   ← lib/serverAuth.ts 로 권한 확인
   ▼
[Supabase]   supabaseAdmin (service role)          ← lib/supabaseAdmin.ts 로 DB 읽기/쓰기
   │
   ▼  JSON 응답
[화면으로 다시]
```

핵심은 **화면은 DB를 직접 만지지 않는다**는 점입니다. 화면은 항상 `lib/api`를 통해 우리 API Route를 부르고, DB 접근(특히 service role 키)은 서버에서만 일어납니다. 왜 이렇게 나누냐면, service role 키는 RLS(행 수준 보안)를 통과하는 강력한 키라 브라우저에 절대 노출되면 안 되기 때문입니다. 이 한 방향만 지키면 보안과 구조가 같이 정리됩니다.

## 1. 새 API Route 만드는 법

API는 `app/api/<경로>/route.ts` 파일 하나가 하나의 엔드포인트입니다. 그 파일에서 `GET`, `POST`, `PATCH` 같은 **HTTP 메서드 이름의 함수를 export**하면 그게 곧 핸들러가 됩니다. 예를 들어 `app/api/seller/apply/route.ts`의 `GET`은 `GET /api/seller/apply`가 됩니다.

### 1-1. 가장 작은 형태

```ts
// app/api/example/route.ts
import { NextRequest, NextResponse } from 'next/server';
import { supabaseAdmin } from '@/lib/supabaseAdmin';
import { requireAuth } from '@/lib/serverAuth';

export async function GET(req: NextRequest) {
  // 1) 인증 게이트
  const authed = await requireAuth(req);
  if (authed instanceof NextResponse) return authed; // 통과 못 하면 401/403 응답을 그대로 반환

  // 2) DB 접근 (서버 전용 클라이언트)
  const { data, error } = await supabaseAdmin
    .from('products')
    .select('id, name, price')
    .eq('seller_id', authed.userId);

  // 3) 에러 처리
  if (error) return NextResponse.json({ error: error.message }, { status: 500 });

  // 4) 응답
  return NextResponse.json({ data });
}
```

### 1-2. 인증 게이트 — `requireAuth` / `requireSeller` / `requireAdmin`

이 세 함수가 `lib/serverAuth.ts`에 있습니다. 셋 다 같은 약속을 따릅니다. **통과하면 `AuthedUser`를, 막히면 `NextResponse`(401/403)를 돌려줍니다.** 그래서 호출한 쪽은 항상 이 한 줄로 분기합니다.

```ts
const authed = await requireAdmin(req);
if (authed instanceof NextResponse) return authed;
// 여기서부터 authed.userId, authed.role 사용 가능
```

차이는 통과 기준뿐입니다.

- `requireAuth` — 로그인만 되어 있으면 통과
- `requireSeller` — role이 `seller` 또는 `admin`이면 통과 (admin은 셀러 화면도 볼 수 있게 열어 둠)
- `requireAdmin` — role이 `admin`만 통과

이 함수들은 요청 헤더의 `Authorization: Bearer <토큰>`을 읽어 Supabase로 토큰을 검증하고, `users` 테이블에서 `role`과 내부 `user_id`까지 채워 줍니다. 그래서 핸들러 안에서는 `authed.userId`(우리 DB의 정수 id)와 `authed.role`을 바로 쓸 수 있습니다. **사용자가 누구인지는 토큰에서 서버가 알아내므로, 클라이언트가 `userId`를 본문에 담아 보낼 필요가 없습니다.** 본문으로 받은 id를 그대로 믿으면 남의 데이터를 건드릴 수 있어 위험합니다.

### 1-3. 모범으로 볼 라우트 — `app/api/admin/dashboard/route.ts`

대시보드 집계 라우트가 이 패턴을 잘 보여 줍니다. 맨 위에서 `requireAdmin`으로 막고, 서로 독립적인 집계 쿼리 여러 개를 `Promise.all`로 **병렬**로 돌립니다.

```ts
const authed = await requireAdmin(req);
if (authed instanceof NextResponse) return authed;

const [orderCountResult, cancelCountResult, newMembersResult /* ... */] = await Promise.all([
  supabaseAdmin.from('orders').select('*', { count: 'exact', head: true }).gte('created_at', todayStr),
  supabaseAdmin.from('orders').select('*', { count: 'exact', head: true }).eq('status', '취소'),
  supabaseAdmin.from('users').select('*', { count: 'exact', head: true }).gte('created_at', todayStr),
  // ...
]);
```

왜 `Promise.all`이냐면, 서로 결과가 필요 없는 쿼리를 순서대로 `await`하면 (요청 수 × 왕복 시간)만큼 느려집니다. 병렬로 묶으면 가장 느린 한 번만큼만 걸립니다. **앞 결과가 뒤 쿼리의 입력이 되는 경우에만** 순서대로 기다리면 됩니다.

### 1-4. 응답 형태 — 느슨하지만 일관되게

지금 코드의 관행은 이렇습니다. 새로 만들 때 여기에 맞춰 주세요.

- 목록/단건: `NextResponse.json({ data })` 또는 목적이 분명한 키(`{ users, pagination }`)
- 성공 동작: `NextResponse.json({ success: true, data })`, 생성이면 `{ status: 201 }`
- 에러: `NextResponse.json({ error: '메시지' }, { status: 4xx | 500 })`
- 필드 검증 실패: `NextResponse.json({ errors: fieldErrors }, { status: 400 })` (`app/api/seller/apply/route.ts` 참고)

에러 키를 `error`로 통일해 두면, 클라이언트의 axios 인터셉터가 그 메시지를 자동으로 꺼내 줍니다(2번 참고). 그래서 화면에서 `err.message`만 봐도 서버가 보낸 한국어 메시지가 들어옵니다.

### 1-5. 한 파일에 메서드 여러 개

같은 자원에 대한 조회/수정은 한 `route.ts`에서 `GET`과 `POST`(또는 `PATCH`)를 함께 export합니다. `app/api/auth/profile/route.ts`가 `GET`(내 정보 조회)과 `PATCH`(수정)를 한 파일에 둔 예입니다. 로직이 길어지면 `seller/apply`처럼 옆에 `db.ts`, `validate.ts`를 두고 라우트는 흐름만 담게 쪼개도 좋습니다.

## 2. 화면에서 API 부르기

화면에서는 `fetch`나 `axios`를 직접 만들지 말고 **반드시 `lib/api`(axios 인스턴스)** 를 씁니다.

```ts
import api from '@/lib/api';

const { data } = await api.get('/admin/users?page=1&limit=20');
await api.patch(`/admin/users/${userId}/status`, { status: 'suspended' });
await api.post('/seller/apply', fields);
```

`baseURL`이 `/api`로 잡혀 있으니 경로는 `/admin/users`처럼 `/api`를 뺀 부분만 적습니다. 실제 호출 예는 `hooks/user/useSellerApply.ts`, `app/admin/members/page.tsx`, `services/auth/authService.ts`에 많습니다.

### 2-1. 토큰은 자동으로 붙는다

`lib/api.ts`의 **요청 인터셉터**가 매 호출마다 `authStore`에서 accessToken을 꺼내 `Authorization` 헤더에 넣어 줍니다. 그래서 화면 코드에서 토큰을 직접 챙길 일이 없습니다. 로그인 상태(토큰·사용자)는 `store/authStore.ts`(Zustand)가 들고 있고, "로그인 유지" 여부에 따라 localStorage / sessionStorage에 보관합니다.

### 2-2. 401이 나면 자동으로 토큰을 갱신해 준다

`lib/api.ts`의 **응답 인터셉터**가 하는 일이 핵심입니다.

- 응답이 `401`이면, refresh token으로 Supabase 세션을 다시 받아(`refreshSession`) 스토어를 갱신하고 **원래 요청을 한 번 자동 재시도**합니다. 그래서 화면 입장에서는 토큰 만료를 거의 신경 쓰지 않아도 됩니다.
- 갱신 중에 다른 요청이 또 401을 만나면 큐에 모았다가 갱신이 끝나면 한꺼번에 풀어 줍니다(같은 갱신을 여러 번 돌리지 않으려는 것).
- 갱신마저 실패하면 스토어를 비우고 `/login`으로 보냅니다.

화면에서 알아 둘 점은 두 가지입니다. 첫째, 토큰 만료 재시도 로직을 화면마다 다시 짜지 마세요. 이미 한곳에서 처리됩니다. 둘째, `catch`로 들어오는 에러의 `err.message`에는 서버가 보낸 `error` 메시지가 담겨 있으니, 사용자에게 그대로 보여 주면 됩니다.

```ts
try {
  await api.patch(`/admin/users/${userId}/status`, { status });
} catch (e) {
  alert(e instanceof Error ? e.message : '상태 변경에 실패했습니다.');
}
```

## 3. 목록 페이지 만들기

목록 화면(표 + 검색 + 필터 + 페이지네이션)은 거의 같은 부품으로 조립합니다. 부품을 먼저 외워 두면 새 목록 페이지를 빠르게 찍어낼 수 있습니다.

- 표: `components/ui/table.tsx`의 `Table / TableHeader / TableBody / TableRow / TableHead / TableCell`
- 상태 배지: `components/common/StatusBadge.tsx` — `<StatusBadge status="배송중" />`처럼 한글 상태 문자열만 넘기면 색을 알아서 입혀 줍니다. 색 매핑이 이 파일 한 곳에 모여 있으니, 새 상태가 생기면 여기에 한 줄 추가하면 모든 화면이 같이 바뀝니다.
- 페이지네이션 UI: `components/ui/Pagination.tsx`
- 페이지 계산 훅: `hooks/usePagination.ts` — 배열을 넘기면 `currentPage / paginated / totalPages / getPageNumbers`를 돌려줍니다(클라이언트에서 잘라 보여 줄 때).
- 검색 필터 훅: `hooks/useFilter.ts` — 배열과 필터 함수를 넘기면 `search / setSearch / filtered`를 줍니다.
- 카테고리/상태 탭: `app/seller/components/FilterTabs.tsx` — 제네릭이라 어떤 문자열 옵션 배열이든 받습니다.

### 3-1. 실제로 어떻게 조립되어 있나 — `app/admin/members/page.tsx`

회원 관리 페이지가 "서버 페이지네이션 + 공통 부품" 조합의 좋은 예입니다. 흐름만 추리면 이렇습니다.

```tsx
'use client';
import api from '@/lib/api';
import StatusBadge from '@/components/common/StatusBadge';
import Pagination from '@/components/ui/Pagination';
import { Table, TableBody, TableCell, TableHead, TableHeader, TableRow } from '@/components/ui/table';

const PAGE_SIZE = 20;

export default function MembersPage() {
  const [members, setMembers] = useState<Member[]>([]);
  const [total, setTotal] = useState(0);
  const [page, setPage] = useState(1);

  useEffect(() => {
    let cancelled = false;
    async function load() {
      const params = new URLSearchParams();
      params.set('page', String(page));
      params.set('limit', String(PAGE_SIZE));
      const { data } = await api.get(`/admin/users?${params}`);
      if (!cancelled) {
        setMembers(data.users);
        setTotal(data.pagination.total); // 서버가 전체 개수를 알려 줌
      }
    }
    load();
    return () => { cancelled = true; }; // 정리 함수로 경합 방지
  }, [page]);

  // ... 표 렌더링: <StatusBadge status={...} />, 마지막에 <Pagination ... />
}
```

여기서 두 가지를 꼭 봐 두세요.

첫째, **페이지네이션을 서버에서 합니다.** 화면은 `page`·`limit`만 쿼리스트링으로 보내고, 서버(`app/api/admin/users/route.ts`)가 `.range(from, to)`로 잘라서 그 페이지 데이터와 전체 개수(`count`)를 돌려줍니다. 데이터가 많아질 수 있는 목록은 이렇게 서버 페이지네이션이 맞습니다. 전부 받아 와서 클라이언트에서 자르면 데이터가 늘수록 느려지고 무거워집니다. 반대로 데이터가 적고 이미 다 받아온 배열을 보여 줄 뿐이라면 `usePagination` 훅으로 클라이언트에서 잘라도 됩니다. 상황에 맞게 고르면 됩니다.

둘째, `useEffect` 안의 `let cancelled = false` + 정리 함수 패턴입니다. 사용자가 빠르게 페이지를 바꾸면 이전 요청 응답이 늦게 도착해 화면을 덮어쓸 수 있습니다(경합). 정리 함수에서 `cancelled = true`로 두고 응답 처리 전에 검사하면 이런 뒤집힘을 막습니다. 비동기로 데이터를 받는 화면에서는 이 패턴을 기본으로 쓰면 좋습니다.

> 참고: 같은 목록을 어드민과 셀러가 비슷하게 만듭니다. 배지·필터·페이지네이션 부품을 공통으로 모아 쓰면 색이나 정책을 한 곳만 고쳐도 양쪽이 같이 바뀝니다. 새로 만들 때는 인라인으로 다시 짜기보다 `components/common`·`components/ui`·`hooks`의 기존 부품을 먼저 찾아보세요.

## 4. 인증·권한 — 두 겹으로 막는다

우리 앱은 보호가 **두 겹**입니다. 둘은 역할이 다르니 둘 다 필요합니다.

1. **화면 가드** — 어드민 영역은 `app/admin/layout.tsx`가 `AdminAuthGuard`(파일은 `app/admin/components/AdminAuthGuard.tsx`)로 감싸, 로그인 안 됐거나 role이 admin이 아니면 화면 자체를 안 보여 주고 로그인/홈으로 보냅니다.
2. **API 가드** — 서버의 각 `route.ts`가 `requireAuth`/`requireSeller`/`requireAdmin`으로 다시 확인합니다(1번 참고).

왜 둘 다 필요할까요. **화면 가드는 편의를 위한 것이고, 데이터를 진짜로 지키는 건 API 가드입니다.** 화면 가드는 브라우저에서 도는 코드라 마음먹으면 우회할 수 있고, 누가 API 주소(`/api/admin/...`)를 직접 호출하면 화면 가드는 아무 역할도 못 합니다. 그래서 데이터를 지키는 마지막 선은 항상 서버의 `requireAdmin`입니다.

반대로 API 가드만 있고 화면 가드가 없으면 경험이 나빠집니다. 권한 없는 사용자에게 화면은 열려 버리고, 데이터만 안 와서 빈 표나 에러만 보입니다. 예를 들어 셀러 화면에 화면 가드가 빠져 있으면, 비로그인 사용자가 상품 등록 폼에 들어와 다 채운 뒤 **저장 단계에서야** 서버에 막혀 입력이 날아갑니다. 못 들어오게 막는 게 화면 가드, 데이터를 못 건드리게 막는 게 API 가드라, 둘을 같이 둡니다.

새 보호 영역(예: 셀러 전용 화면)을 만들 때는 `AdminAuthGuard`를 본떠 확인하는 role만 바꾼 가드를 그 영역 `layout.tsx`에 씌우고, 그 영역의 API에는 `requireSeller`를 거는 식으로 짝을 맞춰 주세요.

## 5. 폴더·커밋 컨벤션, 그리고 RLS 한 단락

### 폴더·파일

- `src/` 없이 루트 아래 `app / components / hooks / lib / store / services / types`를 씁니다.
- 화면은 `app/(그룹)/<경로>/page.tsx`, API는 `app/api/<경로>/route.ts`.
- 임포트는 절대경로 별칭 `@/`를 씁니다 (`@/lib/api`, `@/components/common/StatusBadge`). 상대경로(`../../`)보다 읽기 쉽고 파일을 옮겨도 잘 안 깨집니다.
- 여러 화면이 함께 쓸 부품은 `components/common`·`components/ui`·`hooks`로 올립니다. 한 영역에서만 쓰는 건 그 영역 폴더(`app/<영역>/components`)에 둡니다.

### 커밋 메시지

`타입: 내용` 형태로, **콜론 뒤에 공백 한 칸**을 통일해 주세요. 지금 히스토리도 이 모양입니다.

```
feat: 재료 쇼핑 페이지 구현
fix: 페이지네이션 공통 컴포넌트로 교체
refactor: 어드민 인라인 badge → 공통 StatusBadge 컴포넌트로 통합
docs: 6/15 데일리로그
```

작은 일이지만 `feat:내용`처럼 공백이 들쭉날쭉하면 히스토리가 한 사람이 쓴 것처럼 안 읽힙니다. 작업은 짧은 기능 브랜치(`feat/...`)에서 하고 PR로 `develop`에 합칩니다.

### RLS와 service client (한 단락)

Supabase 테이블은 RLS(행 수준 보안)로 보호한다는 전제로 설계되어 있습니다. 브라우저용 클라이언트(`lib/supabase.ts`)는 **publishable(anon) 키**를 쓰고, 이 키로는 RLS 정책에 막혀 민감 데이터에 함부로 접근하지 못합니다. 반대로 서버용 클라이언트(`lib/supabaseAdmin.ts`)는 **service role 키**를 쓰며 RLS를 통과합니다. 그래서 DB 읽기/쓰기는 `supabaseAdmin`을 통해 **API Route 안에서만** 하고, 그 앞에 `requireAuth`/`requireSeller`/`requireAdmin` 게이트를 둬서 "누가 무엇을 할 수 있는지"를 서버 코드로 직접 통제합니다. service role 키는 RLS를 우회하는 강력한 키라 **절대 브라우저로 내보내지 않습니다**(그래서 `supabaseAdmin`은 클라이언트 컴포넌트에서 import 금지). 키 값은 `.env.local`에 두고 깃에 올리지 않습니다.

## 6. 새 기능 추가 체크리스트 — API + 화면 한 바퀴

기능 하나(예: "셀러 주문 목록")를 만들 때 이 순서로 가면 빠짐없이 돕니다.

1. **데이터 모양 정하기** — 어떤 테이블에서 무엇을 읽고/쓰는지, 응답 JSON 모양은 무엇인지 먼저 적어 둡니다. 타입은 `types/`에 둡니다.
2. **API Route 만들기** — `app/api/<경로>/route.ts`에 `GET`/`POST` 등을 export. 맨 위에 알맞은 게이트(`requireAuth`/`requireSeller`/`requireAdmin`) → `if (authed instanceof NextResponse) return authed;` → `supabaseAdmin`으로 DB → 에러는 `{ error }`로, 성공은 `{ data }` 또는 `{ success: true }`로 응답.
3. **권한 확인** — 이 엔드포인트는 누가 호출해야 하나? 게이트를 그에 맞게 골랐는지 다시 봅니다. 사용자 식별은 본문이 아니라 `authed.userId`로.
4. **화면에서 호출** — `lib/api`로 부르기. 목록이면 서버 페이지네이션(`page`/`limit` 쿼리)을 쓰고, 표·배지·페이지네이션은 `components/ui`·`components/common`·`hooks`의 공통 부품으로 조립. 비동기 `useEffect`에는 `cancelled` 정리 패턴.
5. **화면 가드** — 보호 영역이면 그 영역 `layout.tsx`에 role 가드가 씌워져 있는지 확인. 없으면 `AdminAuthGuard`를 본떠 만들어 씌웁니다.
6. **에러·로딩·빈 상태** — 불러오는 중/데이터 없음/실패를 각각 화면에 표시. 실패 메시지는 `err.message`(서버가 보낸 `error`)를 그대로 보여 주면 됩니다.
7. **빌드·커밋** — 올리기 전에 `npx tsc --noEmit`와 `npm run build`로 타입·빌드가 통과하는지 확인하고, `타입: 내용`(콜론 뒤 공백) 형식으로 커밋해 짧은 기능 브랜치에서 PR을 올립니다.

처음엔 `app/api/admin/dashboard/route.ts`(API 모범)와 `app/admin/members/page.tsx`(화면 모범) 두 파일을 나란히 열어 두고, 새 기능을 그 구조에 끼워 맞추는 식으로 시작하면 가장 빠르게 손에 익습니다. 막히는 부분은 편하게 물어봐 주세요.
