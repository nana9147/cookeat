## 1. eslint `react-hooks/set-state-in-effect` 4건 정리 (추유나)

대상: `app/seller/inquiries/page.tsx:54,90,94`, `app/seller/components/Inquiry/InquiryReplyModal.tsx:36`

이 규칙은 "effect 안에서 곧바로 `setState`를 부르면, 렌더 → effect → setState → 다시 렌더가 연쇄로
일어날 수 있다"는 경고입니다. 대개 이런 모양일 때 납니다.

```tsx
// 문제가 되는 형태 — effect가 실행되자마자 동기로 setState
useEffect(() => {
  setSomething(props.initialValue); // 렌더 직후 또 렌더를 유발
}, [props.initialValue]);
```

경우에 따라 처방이 다릅니다.

**(a) props에서 파생된 값을 상태로 복사하는 경우** — 애초에 상태로 둘 필요가 없을 때가 많습니다.
그냥 렌더 중에 계산하거나, 정말 편집 가능한 로컬 상태가 필요하면 `key`로 리마운트하는 방식이 깔끔합니다.

```tsx
// 편집용 로컬 상태가 꼭 필요하면, 부모에서 key를 바꿔 컴포넌트를 새로 마운트
<InquiryReplyModal key={inquiry.inquiryId} inquiry={inquiry} />
// → 모달 내부는 useState 초기값으로만 세팅하면 되고 effect가 필요 없어집니다
const [content, setContent] = useState(inquiry.reply?.content ?? '');
```

**(b) 데이터를 불러와 목록에 넣는 경우**(`page.tsx`의 fetch류) — effect에서 `await` 후 `setState`하는 것 자체는
정상 패턴이라, 이 규칙은 "동기로 곧장 setState"를 특히 문제 삼습니다. 비동기 fetch를 함수로 감싸고 effect는
그 함수를 호출만 하도록 두면 대부분 사라집니다.

```tsx
const [items, setItems] = useState<Inquiry[]>([]);
const [loading, setLoading] = useState(true);

const load = useCallback(async () => {
  setLoading(true);
  try {
    const res = await api.get('/seller/inquiries', { params: { page } });
    setItems(res.data.inquiries); // await 뒤의 setState는 연쇄 렌더가 아님
  } finally {
    setLoading(false);
  }
}, [page]);

useEffect(() => {
  load();
}, [load]);
```

정 급하면 해당 줄만 `// eslint-disable-next-line react-hooks/set-state-in-effect` 로 사유를 적어 끄는 방법도
있지만, 위 (a)/(b)로 구조를 바꾸는 편이 리렌더도 줄고 더 깔끔합니다.

## 2. eslint를 CI 게이트로 (팀 공통)

`.github/workflows`의 CI YAML에 한 스텝만 추가하면, error가 있는 PR이 머지되는 걸 막을 수 있습니다.

```yaml
# 기존 build 스텝 앞이나 뒤에
- name: Lint
  run: npx next lint --max-warnings=0
  # 지금은 warning이 많으니 우선 error만 막고 싶으면 --max-warnings 옵션은 빼세요.
  # (그러면 error가 있을 때만 CI가 빨간불이 됩니다.)
```

지금 상태에서 error 4건만 위 1번으로 정리하면 곧바로 초록불이 됩니다. warning(35건)까지 0으로 강제하는 건
발표 후 천천히 줄여 나가면 됩니다.

## 3. 로그인 라우트 깨진 바디 방어 (엄인호) — [사소]

`app/api/auth/login/route.ts:7`

```ts
// 현재: 바디가 JSON이 아니면 여기서 예외 → 500
const { email, password } = await req.json();

// 제안: 파싱 실패를 400으로
const body = await req.json().catch(() => null);
const email = body?.email;
const password = body?.password;
if (!email?.trim() || !password) {
  return NextResponse.json({ error: '이메일과 비밀번호를 입력해주세요.' }, { status: 400 });
}
```

이미 아래에 빈 값 검증이 있으니, `req.json()`만 `catch`로 감싸면 깨진 요청도 400으로 일관되게 응답합니다.

## 4. 문의 이미지 insert 실패 처리 (엄인호) — [제안]

`app/api/inquiries/route.ts:132`

문의 본문이 저장된 뒤 이미지 insert가 뒤따르는데, 지금은 결과를 확인하지 않습니다. 이미지야 부차적이니
전체를 롤백할 정도는 아니지만, 실패라도 한 줄 남겨 두면 나중에 "사진이 안 올라간 문의"의 원인을 추적하기 쉬워집니다.

```ts
if (imageUrls.length > 0) {
  const { error: imgError } = await supabaseAdmin
    .from('inquiry_images')
    .insert(imageUrls.map((url) => ({ inquiry_id: data.inquiry_id, url })));
  if (imgError) {
    // 문의 자체는 이미 성공했으므로 201은 유지하되, 서버 로그로 원인 추적 가능하게
    console.error('inquiry_images insert 실패', { inquiryId: data.inquiry_id, imgError });
  }
}
```

더 엄격히 가려면 문의 본문 insert와 이미지 insert를 하나의 RPC 트랜잭션으로 묶는 방법도 있는데,
레시피 수정을 `update_recipe_tx`로 묶었던 것과 같은 패턴이라 필요하면 그쪽을 참고하시면 됩니다.
