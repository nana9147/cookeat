## 2. [필수·보안] 결제 — 시크릿 키 격리는 훌륭, 그러나 "금액"을 클라이언트가 정합니다

먼저 잘한 것부터 보겠습니다. 토스·카카오 모두 **시크릿 키를 서버 라우트에만** 두고(`TOSS_SECRET_KEY`, `KAKAO_SECRET_KEY`), 클라이언트는 공개용 클라이언트 키(`NEXT_PUBLIC_TOSS_CLIENT_KEY`)만 씁니다. 1차 때 칭찬했던 service-role 격리 습관이 결제까지 이어진 셈이라 좋습니다.

문제는 **결제 금액의 출처**입니다.

```ts
// useCheckoutPayment.ts (클라이언트)
const FINAL_AMOUNT = 35410;                 // 금액이 클라이언트에 하드코딩
await api.post('/payment/kakao/ready', {
  orderId, userId: 'test_user', totalAmount: FINAL_AMOUNT, ...  // userId도 가짜
});
```
```ts
// payment/toss/confirm/route.ts (서버)
const { paymentKey, orderId, amount } = await req.json();   // 클라가 보낸 amount를
// ...그대로 토스 confirm에 전달. 서버가 주문 금액을 다시 계산하지 않음
```

지금은 목업 단계라 이렇게 둔 걸로 이해합니다. 다만 결제에서 가장 위험한 패턴이고 **실데이터를 붙이기 전 반드시** 고쳐야 하는 부분이라 미리 표시해 둡니다.

- **금액은 서버가 장바구니/주문 테이블에서 다시 계산해야 합니다.** 클라가 보낸 35,410원을 그대로 믿으면 사용자가 요청 본문의 amount를 100원으로 바꿔 결제 confirm을 통과시킬 수 있습니다. 토스/카카오는 "결제 시작 때 요청한 금액"과 confirm 금액이 맞는지만 보는데, 그 시작 금액마저 지금은 클라가 정합니다. **서버 어디에도 "이 주문의 진짜 금액"이 없는 셈입니다.**
- **결제 라우트에 인증이 없습니다.** `userId: 'test_user'` 하드코딩 대신, `serverAuth`의 `requireAuth(req)`로 로그인 사용자를 확인하고 그 사용자 기준으로 주문을 만들어야 합니다. (가드는 이미 만들어 두셨으니 갖다 쓰기만 하면 됩니다.)
- **결제 성공 후 주문을 "결제완료"로 확정하는 서버 단계**가 필요합니다(이상적으로는 웹훅으로, 중복 승인 방지 idempotent하게). 지금은 PG 승인만 받고 끝납니다.

해결 흐름은 가이드에 순서대로 적었습니다.

## 3. [필수] 셀러 화면 가드 — 3차에 이어 여전히 없음

`app/seller/layout.tsx`는 헤더·사이드바만 그릴 뿐 인증/role 체크가 없습니다. 비로그인 사용자도 `/seller/products/new`, `/seller/shipping` 등에 그대로 들어옵니다. API는 `requireSeller`로 막혀 있어도 화면이 열리면 사용자에겐 "되는 것처럼" 보이고, 빈 폼이나 에러가 노출됩니다. 레이아웃에서 서버 가드를 한 번 거는 게 가장 깔끔합니다(가이드).

## 5. 세 명이 "한 사람처럼" 가고 있나 — 협업 진단

**좋은 신호가 많습니다.** 셋이 따로 작업하는데도 공통 자산을 공유하고 있습니다. `serverAuth`(엄인호) 덕에 API들이 같은 인증 패턴을 따르고, `usePagination/useFilter/StatusCards`(추유나)로 어드민·셀러가 같은 표/필터를 쓰며, `lib/api` axios 통일로 요청 방식이 일원화됐습니다. 이게 "한 사람처럼"의 핵심이라 생각합니다. 잘하고 있습니다.

**반대 신호는 한 곳에 모입니다 — 공유 자산을 각자 건드릴 때입니다.**

- 빌드가 1차부터 4차까지 계속 깨진 원인을 보면 대부분 **공유 타입(상태 유니온: `ProductStatus`/`OrderStatus`)이나 전역 선언을 한 사람이 바꾸고 다른 사람이 안 맞춘** 경우였습니다. 이번 빌드 실패(`useKakaoPostcode`의 전역 `kakao` 선언 충돌)도 전역 타입을 두 곳에서 다르게 선언한, 같은 결의 문제입니다.
- 네이밍도 작은 신호로 볼 수 있습니다. `seller/` 폴더에 `SellerApplyForm.tsx`·`SellerApplyStatus.tsx`(PascalCase) 사이로 `sellerapply.tsx`(컴포넌트도 `Sellerapply`)가 섞여 있습니다.

**처방 세 가지.**
1. **공유 자산(타입·전역 선언·공통 컴포넌트)은 "한 곳에서, 합의하고" 바꾸기.** 상태 유니온을 늘려야 한다면 그 타입을 쓰는 모든 곳(다른 사람 코드 포함)을 같은 PR에서 맞춥니다. 혼자 늘리고 머지하면 다음 사람 빌드가 깨집니다.
2. **머지 전 `npm run build`(또는 tsc) 한 번을 팀 규칙으로.** CI에 거는 게 가장 확실합니다. 빌드 빨간불이 4회째인 건 이거 하나로 정리됩니다.
3. **컨벤션 한 장.** 폴더/파일 네이밍(PascalCase 컴포넌트), 상태값 정의 위치, 공통 훅 사용 규칙을 README나 짧은 문서로 남겨두세요. 셋이 같은 규칙을 보면 코드가 한 사람 손에서 나온 것처럼 읽힙니다.

지금도 토대(serverAuth·공통 훅·axios)는 한 사람처럼 잘 맞춰져 있습니다. 여기에 **"공유 자산을 바꿀 때만" 같이 보는 습관**을 더하면 됩니다.
