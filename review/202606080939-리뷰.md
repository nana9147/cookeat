# Cookeat 1차 코드 리뷰

- **날짜**: 2026-06-08 (월)
- **대상 브랜치**: `develop`
- **리뷰 범위**: 첫 리뷰. 전체 구조를 한 번 훑고, 보안에 직접 닿는 인증/API 라우트(`lib/`, `app/api/auth/*`, `store/`, `services/`, `hooks/auth/`)와 각자 맡은 화면(엄인호=인증·헤더·푸터·마이페이지, 최유종=어드민·shadcn ui, 추유나=판매자)을 봤습니다.

---

Cookeat 첫 리뷰입니다. 팀장이 빠져 셋이 진행 중인 상황은 전제하고, 코드만 담담히 보겠습니다.

도메인이 레시피·식단·마켓(상품·주문·판매자·어드민)까지 넓은데도 폴더 구조가 흐트러지지 않았고, 인증 흐름은 꽤 단단하게 짜여 있습니다. 기반이 잘 잡혀 있다고 생각합니다.

이번 리뷰는 틀린 곳을 지적하려는 게 아니라, 지금 잘 잡아 둔 토대 위에 무엇을 더 얹으면 좋을지 같이 보는 자리라 생각해 주세요.

## 팀 공통

### 잘하고 있는 점부터

- **폴더 구조가 역할별로 잘 나뉘어 있습니다.** `app/`(라우트) / `components/`(UI) / `hooks/`(상태 로직) / `services/`(API 호출) / `store/`(전역 상태) / `lib/`(인프라) / `types/`. 이렇게 층(layer)을 나눠 두면 나중에 기능이 늘어나도 "이건 어디에 둬야 하지?"로 헤매지 않습니다. 초보 단계에서 이걸 스스로 지키고 있는 점이 좋습니다.
- **문서화가 모범적입니다.** `docs/api/01~08-*.md`로 도메인별 API 명세를 미리 정리해 둔 점, `docs/daily_log/`로 누가 무엇을 했는지 매일 남기는 점이 좋습니다. 특히 엄인호 님의 데일리 로그는 커밋 해시까지 붙어 있어서, 리뷰할 때 "왜 이렇게 바꿨는지"를 코드만 보고 추측하지 않아도 됐습니다. 협업 도구로서 잘 쓰고 있습니다.
- **shadcn/ui를 도입해 UI 기본 컴포넌트(`components/ui/`)를 공유**하고 있어서, 화면마다 버튼·인풋이 제각각으로 노는 일이 없습니다. 이것도 좋은 선택입니다.

### [제안] 워크로드 분포 — 먼저 배경을 확인하고 싶습니다

커밋 수만 보면 엄인호 님이 인증 전체 + 공통 인프라(`lib/api.ts`, `supabase.ts`, `store`, 헤더/푸터/마이페이지)까지 많이 작성한 것으로 보입니다. 다만 커밋 수만으로는 실제 분담을 알 수 없으니, 단정하지 않고 먼저 묻고 싶습니다.

- 공통 토대(인증·인프라)는 팀에서 함께 설계를 정하고 엄인호 님이 작성·커밋한 건가요? 아니면 진행하다 보니 자연스럽게 한쪽으로 모인 걸까요?
- 최유종 님(어드민·ui)·추유나 님(판매자)은 각자 화면을 탄탄히 만들고 있는데, 공통 코드(`lib/`, `store/`, `types/`)에는 얼마나 함께 관여하고 있나요?

이 답에 따라 다음 제안의 무게가 달라집니다. 만약 한쪽으로 모인 상황이라면, 공통 코드는 한 사람의 코드가 아니라 팀의 코드가 되어야 나중에 그 사람이 바쁠 때 다른 사람이 손댈 수 있다는 점에서, 아래 두 가지를 권합니다. 이미 함께 정하며 진행 중이라면 그대로 가도 좋습니다.

1. `lib/api.ts`, `store/authStore.ts` 같은 핵심 공통 파일은 셋이 한 번씩 같이 읽어 보는 시간을 짧게라도 가져 보세요. 코드 리뷰 겸 지식 공유가 됩니다.
2. 다음 작업 단위부터 마켓의 데이터 연동 부분(상품 목록 API, 주문 API)을 최유종·추유나 님이 나눠 맡으면, 자연스럽게 백엔드(API 라우트) 경험도 분산됩니다. 지금은 두 분 화면이 거의 mock 데이터(하드코딩 배열)라서, 그걸 실제 API로 바꾸는 작업이 좋은 분담 포인트입니다.

### [제안] 코드 포맷이 파일마다 조금씩 다릅니다 (세미콜론)

같은 사람이 쓴 코드인데도 어떤 파일은 세미콜론(`;`)이 있고(`register/route.ts`, `useNicknameCheck.ts`) 어떤 파일은 없습니다(`login/route.ts`, `useLoginForm.ts`). 동작에는 문제가 없지만, 팀 프로젝트에서는 이런 게 쌓이면 git diff가 지저분해집니다(실제 변경이 아닌 포맷 변경이 섞임).

`.prettierrc`가 이미 있으니, 저장 시 자동 포맷(format on save)만 셋 다 VSCode에 켜 두면 이 문제는 정리됩니다. 해결 가이드에 설정 방법을 적어 두겠습니다.

### [필수] 어드민/판매자 페이지에 접근 제한이 아직 없습니다

`app/admin/layout.tsx`와 `app/seller/layout.tsx` 어디에도 "로그인했는지 / 관리자인지 / 판매자인지" 확인하는 코드가 없습니다. 지금은 화면이 전부 mock 데이터라 당장 새는 정보는 없지만, 여기에 실제 회원/주문 데이터를 연결하는 순간 누구나 `/admin`을 주소창에 쳐서 들어올 수 있는 구멍이 됩니다.

지금 당장 완벽히 막을 필요는 없지만, "데이터 연동 전에 반드시 권한 게이트를 단다"를 팀 약속으로 정해 두면 좋겠습니다. 가이드에 패턴을 적어 두겠습니다.

### 엄인호 (@djsy01) — 인증 전반 · 공통 인프라 · 헤더/푸터/마이페이지

**담당 요약**: 로그인·회원가입·소셜 로그인(OAuth) 전체 흐름, Supabase 연동(`lib/supabase.ts`, `lib/supabaseAdmin.ts`, `lib/api.ts`), Zustand 인증 스토어, 인증 API 라우트(`app/api/auth/*`), 헤더/사이드바/푸터, 마이페이지 초안까지. 사실상 이 프로젝트의 기반을 거의 다 깔았습니다.

**잘한 점**

- **[보안 칭찬] service role 키를 서버 전용으로 정확히 격리했습니다.** `lib/supabaseAdmin.ts`는 `SUPABASE_SERVICE_ROLE_KEY`(DB를 통째로 만질 수 있는 만능 키)를 쓰는데, 이걸 클라이언트(`'use client'`) 컴포넌트에서는 단 한 군데도 import하지 않고 오직 `app/api/auth/*` 서버 라우트에서만 씁니다. 클라이언트용은 `lib/supabase.ts`의 publishable key로 따로 분리했고요. 초보자가 가장 많이 실수하는 부분("관리자 키가 브라우저 번들에 섞여 나가서 누구나 DB를 통째로 조작")인데, 정확히 막아 두셨습니다. 파일 첫 줄 주석(`API 라우트에서만 임포트할 것`)까지 남긴 것도 좋습니다.
- **토큰 검증을 제대로 합니다.** `app/api/auth/profile/route.ts`에서 헤더의 Bearer 토큰을 `supabaseAdmin.auth.getUser(token)`로 서버에서 검증한 뒤, 그 결과의 `user.email`로만 데이터를 수정합니다. 즉 클라이언트가 보낸 email을 그대로 믿지 않고 토큰으로 확인된 본인 것만 건드린다는 것, 인가(authorization)의 핵심을 지키고 있습니다.
- **OAuth 콜백(`app/auth/callback/page.tsx`)의 비동기 처리가 꼼꼼합니다.** `settled` 플래그로 중복 실행을 막고, 5초 타임아웃 + `unsubscribe`로 정리(cleanup)까지 했습니다. 데일리 로그를 보니 예전엔 토큰을 `console.log`로 찍던 걸 보안상 제거하셨더군요. 스스로 위험을 인지하고 고친 흔적이라 좋습니다.

**[필수] 서버 라우트에 입력 검증이 거의 없습니다 — `app/api/auth/register/route.ts:4~11`, `login/route.ts:5`**

회원가입 라우트가 `email, password, nickname, phone`을 받아 그대로 `supabase.auth.signUp`에 넘깁니다. 문제는 서버에서 형식을 한 번도 확인하지 않는다는 점입니다. 지금은 화면(`useRegisterFields.ts`)에서 비밀번호 강도·전화번호 입력을 막아 주지만, 그건 브라우저에서만 막는 것입니다. 누군가 Postman이나 fetch로 라우트를 직접 호출하면 빈 닉네임, 형식이 안 맞는 전화번호, 짧은 비밀번호도 그냥 통과합니다.

제가 실제로 `POST /api/auth/register`에 잘못된 입력을 보내 봤더니, 라우트가 자체 검증 없이 그대로 Supabase로 넘기고 **Supabase의 응답만 되돌려줬습니다**(중간 검증 계층이 없음을 확인). Supabase가 자체적으로 막아 주는 건 "비밀번호 6자 미만", "이메일 형식" 정도뿐이라, 여러분 문서 규칙인 **8자 이상·전화번호 형식·닉네임 필수는 서버에서 안 걸립니다.**

`docs/api/01-auth.md`에도 "password 8자 이상, phone은 010-0000-0000 형식"이라고 적어 두셨습니다. 그 규칙을 서버에서도 한 번 더 확인해 주세요. 클라이언트 검증은 사용자 편의용이고, 진짜 방어선은 서버라 생각합니다. (가이드에 간단한 검증 예시를 적어 두겠습니다.)

**[제안] 닉네임 중복확인이 "확인 시점"과 "가입 시점" 사이에 빈틈이 있습니다 — `check-nickname/route.ts` + `register/route.ts`**

지금 흐름은 ① 닉네임 중복확인 API로 "사용 가능" 받고 → ② 회원가입 API 호출, 이렇게 두 번에 나뉘어 있습니다. 그런데 ①과 ② 사이에 다른 사람이 같은 닉네임으로 먼저 가입하면, 검증을 통과했는데 중복이 생깁니다(이른바 race condition). 또 `profile` PATCH에서도 닉네임을 바꿀 때 중복 재확인을 하지 않습니다.

완벽히 막으려면 결국 DB의 `nickname` 컬럼에 UNIQUE 제약을 걸어 DB가 최종 심판을 하게 하는 게 정석입니다. 초보 단계에선 어려운 개념이니, 지금은 "클라이언트 확인만으로는 100% 보장이 안 된다"는 것만 인지하고, 가입 라우트에서 insert 실패(중복 에러) 시 친절한 메시지를 돌려주는 정도까지만 해 두셔도 충분합니다.

**[제안] `check-nickname`은 누구나 호출 가능한 공개 API입니다 — `check-nickname/route.ts:4`**

인증 없이 닉네임 존재 여부를 알려 주기 때문에, 이론상 닉네임을 하나씩 넣어 보며 "이 닉네임 쓰는 사람이 있나"를 긁어 갈 수 있습니다(enumeration). 회원가입에 꼭 필요한 기능이라 공개 자체는 자연스럽지만, 나중에 여유가 생기면 호출 횟수 제한(rate limit)을 떠올려 보세요. 지금 당장 고칠 필요는 없는 [제안] 수준입니다.

**[사소] 같은 "닉네임 중복확인" 로직이 두 군데에 복붙돼 있습니다 — `hooks/auth/useNicknameCheck.ts` vs `app/auth/profile/page.tsx:29~40`**

`useNicknameCheck` 훅에 이미 중복확인 로직이 있는데, 소셜 프로필 페이지에서 거의 같은 코드를 또 손으로 짰습니다. 나중에 메시지 문구 하나 바꾸려면 두 곳을 다 고쳐야 합니다. 프로필 페이지에서도 `useNicknameCheck` 훅을 재사용하면 깔끔해집니다. 동작엔 문제없으니 여유 될 때 정리하면 됩니다.

### 추유나 (@nana9147) — 판매자(셀러) 페이지 · 상품 등록 폼

**담당 요약**: 판매자 대시보드(`app/seller/`) 레이아웃·사이드바, 상품 목록/필터/검색, 그리고 가장 공들인 상품 등록 폼(기본정보·가격·이미지 업로드)과 `types/seller/*` 타입 정의.

- **이미지 업로드 컴포넌트(`ImageUploadField.tsx`)가 이번 리뷰에서 가장 인상적이었습니다.** 드래그앤드롭 + 클릭 업로드, 최대 장수·용량 제한, `dnd-kit`으로 순서 변경(대표 이미지 고정), 그리고 삭제 시 `URL.revokeObjectURL(target.preview)`로 메모리까지 정리합니다. 마지막 그 한 줄(메모리 누수 방지)은 경력자도 자주 빠뜨리는 부분인데 챙기셨습니다. 잘했습니다.
- **타입을 도메인별로 잘 분리(`types/seller/`)** 해 두었고, 카테고리 코드/이름을 유니온 타입으로 정의(`CategoryCode`, `CategoryName`)한 것도 좋습니다. 폼 데이터(`ProductFormData`)와 초기값(`initialProductForm`)을 타입 곁에 둔 구성도 깔끔합니다.
- 상품 검색의 자동완성 제안(suggestion) + `onBlur` 타이밍 처리(`setTimeout 150ms`)도 세심합니다.

**[필수] 타입과 실제 데이터가 어긋나 있습니다 — `types/seller/product.ts:24` vs `app/seller/products/page.tsx:13~58`**

`Product` 인터페이스에 `store: string`이 필수 필드로 들어가 있는데(24번째 줄), 정작 `products/page.tsx`의 mock 데이터에는 `store` 항목이 하나도 없습니다. **제가 `npx tsc --noEmit`을 직접 돌려 보니 실제로 에러가 납니다** — `products/page.tsx:14, 25, 36, 47`에서 모두 `error TS2741: Property 'store' is missing ... but required in type 'Product'` 4건입니다. 즉 지금 상태로는 타입체크/프로덕션 빌드가 막힙니다. 둘 중 하나로 맞춰 주세요.

- 판매처 정보가 아직 필요 없다면 → `store`를 빼거나 `store?: string`(선택)으로,
- 필요하다면 → mock 데이터 4개에 `store: '...'` 추가.

지금처럼 "타입엔 있는데 데이터엔 없는" 상태는 빌드를 깨거나, 나중에 `product.store`를 썼을 때 `undefined`로 조용히 터지는 원인이 됩니다.

**[제안] 상품 등록 폼에 입력 검증이 얕습니다 — `app/seller/components/ProductForm/index.tsx:18~25`**

`handleSubmit`이 값이 비었는지(`!form.name` 등)만 보고 `alert` 후 `console.log`로 끝납니다. 화면 먼저 만드는 단계라 괜찮지만, 다음 단계에서 가격·재고가 음수인지, 숫자가 맞는지 같은 검증이 필요합니다. 특히 가격/재고는 지금 `string`으로 받고 있어서(`ProductFormData`), API로 보낼 땐 숫자 변환·검증이 꼭 들어가야 합니다.

**[사소] 이미지 미리보기 색상이 팀 토큰과 다릅니다 — `ImageUploadField.tsx:39, 201~209`**

업로드 영역과 대표 배지에 `bg-green-700`, `border-green-500`처럼 기본 Tailwind 초록을 직접 썼는데, 팀은 `bg-primary` 같은 디자인 토큰을 쓰고 있습니다(최유종 님 코드도 비슷한 점이 있었습니다). 토큰으로 통일하면 나중에 메인 컬러를 바꿀 때 한 번에 반영됩니다.

## 정합성 메모 (문서 vs 실제 구현)

이건 누구의 잘못이 아니라 계획과 구현이 자연스럽게 달라진 부분이라, 가볍게만 짚습니다.

`docs/api/01-auth.md`에는 자체 백엔드 스펙처럼 적혀 있습니다. Base `/api/v1`, JWT Bearer, 공통 응답 `{ success, data, message }`, `userId`는 `int`, `/auth/refresh`·`/auth/withdraw` 같은 엔드포인트까지요. 그런데 실제 구현(`app/api/auth/*`)은 Supabase를 얇게 감싼 형태라서 경로가 `/api/auth/*`이고, 응답도 `{ user, session }` 그대로이며, `userId`는 Supabase의 UUID(문자열)입니다.

지금 구현 방향(Supabase 래핑)이 5주짜리 부트캠프엔 훨씬 현실적이고 좋은 선택이라 생각합니다. 다만 문서가 실제와 멀어지면 나중에 본인들이 헷갈립니다. 여유가 될 때 `docs/api/01-auth.md`를 "실제 우리가 만든 것" 기준으로 한 번 업데이트해 두면, 마켓 API를 만들 때 이 문서를 믿고 작업할 수 있습니다. 다른 도메인 문서는 아직 미구현이라 그대로 둬도 괜찮습니다.

## E2E 테스트 결과

이번엔 코드만 읽지 않고 실제로 앱을 띄워서 확인했습니다. 리뷰 전용으로 Playwright E2E를 따로 작성했고(`review/e2e/`, 여러분 테스트와는 별개입니다), 캡처는 `review/images/`에 있습니다. 이번엔 `.env.local`을 넣어주셔서 **여러분의 실제 Supabase에 붙여** 로그인까지 확인했습니다. 이메일 확인이 켜져 있어 보여서, 로그인 자체는 확인된 테스트 계정으로 진행했습니다(계정은 `review/e2e/test-account.md`). 구글 로그인은 실제 구글 동의 화면이 필요해 자동화에서 제외했습니다.
