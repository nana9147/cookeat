# Cookeat 10차 코드 리뷰 (2026-06-22) — 시연 대비 집중 점검

오늘 오후 시연을 앞두고 있어서, 9차(2026-06-19) 이후 들어온 변경 가운데 **시연 동선에 직결되는 것 위주로** 빠르게 봤습니다. 먼저 9차에서 막아야 한다고 했던 두 가지([필수] 상품 상세 404, [필수·보안] sellers anon 노출)가 어떻게 됐는지 확인했고, 그 사이 새로 들어온 레시피 목록·상세와 상품 등록 API, 관리자 페이지까지 훑었습니다. 늘 하던 대로 앱부터 띄워 동선을 돌려 보고(아래 "E2E 결과") 그 결과를 근거로 코드를 봤습니다.

결론부터 말씀드리면, 구매 동선을 막던 상품 상세 404는 깔끔하게 풀렸습니다. 다만 **sellers 보안 건은 아직 그대로**라, 시연 전에 이것 하나만 닫고 가면 좋겠습니다.

---

## 1. 지난(9차) 지적 반영 현황

| 9차 지적 | 상태 | 확인 |
| -------- | ---- | ---- |
| [필수] `/shopping/[id]` 상품 상세 전부 404 (구매 동선 차단) | **해결** | `lib/products.ts`의 `getProductDetail` select가 `ingredients` 조인 대신 `categories ( category_id, name )`로 바뀌었습니다(16행). E2E에서 판매중 상품 `/shopping/4` → **200 정상 렌더** 확인(`cookeat-C2c-shopping-detail.png`). 없는 상품 `/shopping/999999`은 그대로 404(`cookeat-C2d`). 동선이 살아났습니다. |
| [필수·보안] `sellers` 테이블 anon 노출 (사업자번호·계좌·대표자명·CS전화) | **미해결** | anon(publishable) 키로 `GET /rest/v1/sellers`를 직접 호출하니 여전히 2행이 그대로 읽힙니다(아래 2장). 9차와 동일합니다. |

상품 상세를 막던 조인 한 줄을 정확히 찾아 고친 게 좋습니다(엄인호). 같은 커밋에서 `CategoryName`까지 DB 실값으로 반영한 걸 보면, "왜 404가 났는지"를 데이터 구조까지 내려가서 짚은 수정이라 생각합니다.

## 2. 시연 전 막아야 할 [필수] — sellers anon 노출

9차에 이어 다시 확인했습니다. 로그인 없이 anon 키만으로 판매자 민감 정보가 그대로 읽힙니다.

```
GET /rest/v1/sellers?select=seller_id,store_name,business_number,bank_account,representative_name,cs_phone
→ [{"seller_id":13,"store_name":"33333","business_number":"333-33-33333",
    "bank_account":"333333333333333333","representative_name":"이규화","cs_phone":"333-3333-3333"}, ...]
```

사업자등록번호·은행계좌·대표자 실명·CS 전화는 외부에 그대로 노출되면 안 되는 정보입니다. `users`·`orders`는 anon으로 찔러도 0행으로 잘 막혀 있는데(아래 E2E), 유독 **sellers만 열려 있습니다.**

다행히 고치기 까다롭지는 않습니다. 상품 상세에서 판매자 정보를 읽는 `getProductDetail`은 `supabaseAdmin`(service-role)로 조회합니다(`lib/products.ts:11`). 화면에 필요한 판매자 정보는 이미 서버(service-role)를 거쳐서 나가니, **sellers 테이블의 anon 직접 읽기는 완전히 닫아도 화면이 깨지지 않습니다.** Supabase에서 `sellers`에 대한 anon SELECT 정책만 막으면 됩니다. 구체적인 정책은 가이드 1번에 적어 두었습니다.

> 참고: 9차 때 이 노출을 두고 "42P17 RLS 무한재귀를 풀다가 정책이 느슨해지면서 열린 것 같다"고 봤는데, 이번에도 같은 상태인 걸 보면 그 정책을 아직 다시 조이지 못한 듯합니다. 재귀는 피하면서도 anon은 막는 정책 형태를 가이드에 함께 적었습니다.

## 3. 새로 들어온 것 — 레시피·상품 등록·관리자

9차 이후 도메인이 또 넓어졌습니다. 시연 직결 위주로 짧게 봤습니다.

### 3-1. 엄인호(@djsy01) — 레시피 목록·상세 + 상품 상세 404 수정

상품 상세 404를 푼 데 더해(1장), 레시피 목록(`/recipes`)·상세(`/recipes/[id]`)도 새로 올렸습니다. E2E에서 둘 다 200으로 잘 뜹니다(`cookeat-C6-recipes-list.png`, `cookeat-C6b-recipe-detail.png`). 상품 도메인에서 보여 준 "서버 컴포넌트 + 컴포넌트 분할" 패턴이 레시피에도 이어지는지는 다음 회차에 본문까지 들여다보겠습니다. 일단 시연 동선에서 목록 → 상세가 끊김 없이 도는 건 확인했습니다.

### 3-2. 최유종(@jjong0) — 레시피/포인트 실 DB 연동, 회원/판매자 관리 페이지네이션

mock으로 돌던 레시피·포인트를 실 DB로 붙였고, 관리자 회원/판매자 관리에서 "필터 적용 시 페이지네이션이 어긋나던" 문제를 고쳤습니다(`c0e23ad`). 필터와 페이지네이션이 상태를 따로 들고 있으면 흔히 어긋나는 부분인데, 그 결을 잡아 준 건 좋은 수정이라 생각합니다.

### 3-3. 추유나(@nana9147) — 상품 등록 폼·API 연동

판매자 상품 등록을 카테고리 2단 선택 + 배송/반품 정책 템플릿 선택 UI로 다듬고 등록 API까지 연동했습니다. 배송/반품 정책을 자유 입력이 아니라 템플릿에서 고르게 한 건, 판매자가 매번 같은 문구를 다시 쓸 일도 줄이고 데이터도 일관되게 모이게 하는 좋은 선택이라 생각합니다.

## 4. E2E 결과 (2026-06-22)

dev 서버(`PORT=3200`)로 사용자 동선을 돌렸습니다. 캡처는 `review/images/2026-06-22/`. 계정은 `cookeat-review@example.com` 재사용.

**사용자 여정 — 통과**
- 메인(데스크톱/모바일) ✓ — `cookeat-C1-main-desktop.png`
- 상품 목록 `/shopping` ✓ → **상품 상세 `/shopping/4` 200(9차 404 해결)** ✓ — `cookeat-C2c-shopping-detail.png`
- 없는 상품 `/shopping/999999` → 404(정상) ✓
- **신규 레시피 목록 `/recipes`·상세 `/recipes/1` 200** ✓ — `cookeat-C6-recipes-list.png`, `cookeat-C6b-recipe-detail.png`
- 장바구니(게스트/로그인)·체크아웃·마이페이지·주문내역 렌더 ✓

**보안 실측 (anon publishable 키 직접 GET)**
- `users` → 0행, `orders` → 0행 (잘 막힘)
- **`sellers` → 2행 노출** (사업자번호·계좌·대표자명·CS전화) — 2장 [필수]

> 관리자 여정(로그인 후 `/admin/*` 연속 순회)은 긴 단일 시나리오라 이번에도 150초 타임아웃이 났는데, 이건 한 테스트에서 여러 화면을 직렬로 도는 리뷰 측 하네스 문제이지 제품 버그는 아닙니다. 관리자 화면 자체는 `/admin`·`/admin/sellers`·`/admin/recipes` 모두 개별 200 렌더를 확인했고, 대시보드·회원·상품 관리는 캡처도 남겨 뒀습니다(`cookeat-A1`~`A3`).

## 5. 시연까지 우선순위

1. **[필수·보안] `sellers` anon SELECT 차단** — 사업자번호·계좌·대표자명·CS전화 노출 막기. 상세 페이지는 service-role로 읽으니 anon은 완전히 닫아도 됩니다(2장·가이드 1번). **시연 전 최우선.**
2. (해결됨) 상품 상세 404 — 확인 완료, 추가 작업 없음.

상품 상세가 살아나면서 "목록 → 상세 → 장바구니 → 결제"로 이어지는 핵심 동선이 끊김 없이 돌고, 레시피 도메인까지 화면이 붙어 시연에서 보여 줄 게 한층 풍성해졌습니다. 남은 건 sellers 한 테이블의 anon 접근을 닫는 것 하나뿐이라, 그것만 정리하면 시연은 자신 있게 보여줄 수 있는 상태라 봅니다. 화이팅입니다.
