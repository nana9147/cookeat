## A. 목록·메인 페이지를 SSR로 전환 [필수/SEO]

핵심은 한 줄입니다. **검색에 걸려야 하는 페이지에서 `'use client'` + `useEffect` fetch를 걷어내고, 서버 컴포넌트에서 데이터를 받아 HTML에 담아 내려보낸다.** 상세 페이지(`recipes/[id]`, `shopping/[id]`)가 이미 이렇게 잘 돼 있으니 그걸 본보기로 삼으면 됩니다.

### A-1. 레시피 목록 `/recipes`

지금은 `app/(main)/recipes/page.tsx`가 껍데기만 서버 컴포넌트고, 실제 데이터는 `RecipeClient.tsx`(`'use client'`)가 `useEffect`에서 가져옵니다. 이걸 이렇게 바꿉니다.

먼저 서버에서 부를 fetch 함수를 둡니다(상세에서 쓰는 `lib/serverRecipes.ts`와 같은 자리).

```ts
// lib/serverRecipes.ts (추가)
import { cache } from 'react';

export const fetchRecipeList = cache(async (opts: {
  recipeCategoryId?: number;
  sort: string;
  page: number;
  limit: number;
}) => {
  // 상세에서 쓰던 것과 같은 서버용 supabaseAdmin / 쿼리를 재사용
  // 반환: { recipes, pagination: { page, limit, total, hasNext } }
});
```

> `cache()`로 감싸면 같은 요청 안에서 두 번 불러도 한 번만 DB에 갑니다(`seo-가이드.md` 9장 참고).

그다음 페이지를 **async 서버 컴포넌트**로 만들고, 카테고리·정렬·페이지를 **URL `searchParams`에서 읽습니다**. 이게 SSR 목록의 핵심인데, 상태를 `useState`가 아니라 URL이 들고 있게 해야 서버가 첫 요청에서 바로 올바른 데이터를 그릴 수 있기 때문입니다.

```tsx
// app/(main)/recipes/page.tsx
import { fetchRecipeList } from '@/lib/serverRecipes';
import RecipeGrid from './_components/RecipeGrid';
import RecipeToolbar from './_components/RecipeToolbar'; // 탭·정렬만 떼어낸 작은 클라 컴포넌트
import Pagination from '@/components/ui/Pagination';

const PAGE_SIZE = 12;

interface Props {
  searchParams: Promise<{ category?: string; sort?: string; page?: string }>;
}

export default async function RecipesPage({ searchParams }: Props) {
  const sp = await searchParams;
  const categoryId = sp.category ? Number(sp.category) : undefined;
  const sort = sp.sort ?? 'latest';
  const page = Math.max(1, Number(sp.page ?? '1'));

  // 서버에서 직접 fetch → HTML에 카드가 박혀서 내려간다
  const { recipes, pagination } = await fetchRecipeList({
    recipeCategoryId: categoryId, sort, page, limit: PAGE_SIZE,
  });

  return (
    <div className="max-w-360 mx-auto px-4 tablet:px-6 desktop:px-10 py-6">
      {/* 브레드크럼·헤더는 그대로 */}
      <RecipeToolbar selectedCategory={categoryId} sort={sort} />
      <RecipeGrid recipes={recipes} />
      <Pagination
        currentPage={page}
        totalPages={Math.ceil(pagination.total / PAGE_SIZE)}
      />
    </div>
  );
}
```

인터랙션(탭·정렬 셀렉트)만 작은 클라이언트 컴포넌트로 떼어내고, **상태를 바꾸는 대신 URL을 바꿉니다**.

```tsx
// app/(main)/recipes/_components/RecipeToolbar.tsx
'use client';
import { useRouter, useSearchParams } from 'next/navigation';

export default function RecipeToolbar({ selectedCategory, sort }: {
  selectedCategory?: number; sort: string;
}) {
  const router = useRouter();
  const params = useSearchParams();

  const setParam = (key: string, value: string) => {
    const next = new URLSearchParams(params.toString());
    next.set(key, value);
    next.set('page', '1'); // 필터 바꾸면 1페이지로
    router.push(`/recipes?${next.toString()}`);
  };

  // 카테고리 탭 클릭 → setParam('category', String(id))
  // 정렬 변경 → setParam('sort', value)
}
```

이렇게 하면 카테고리/정렬/페이지가 전부 URL에 남아서, 공유한 링크도 그대로 같은 화면이 뜨고, 크롤러도 첫 HTML에서 카드를 봅니다. `Pagination`도 `<Link href={...?page=2}>`로 바꿔주면 페이지 이동도 SSR로 동작합니다.

### A-2. 상품 목록 `/shopping`

똑같습니다. `ShoppingClient` + `useShoppingFilter`(`useEffect` fetch)를 걷어내고, `app/(main)/shopping/page.tsx`를 async 서버 컴포넌트로 만들어 `searchParams`(카테고리·정렬·가격대·셀러·페이지)를 읽어 서버에서 `fetch('/api/products')` 대신 **서버용 데이터 함수**(상세에서 쓰는 `getProductDetail`이 있는 `lib/products.ts`에 `getProductList`를 추가)를 직접 부릅니다. 가격 슬라이더·필터 시트 같은 인터랙션만 클라 자식으로 남깁니다.

### A-3. 메인 `/`

`components/home/PopularRecipeSection.tsx`, `TodayRecipeCard.tsx`, `CategorySection.tsx`가 각각 `useHomeData` 훅으로 채우는데, 이 데이터들을 `app/(main)/page.tsx`(서버 컴포넌트)에서 한 번에 받아 **props로 내려주면** 됩니다.

```tsx

    fetchPopularRecipes(),

    fetchRecipeCategories(),
```

`UserPointCard`처럼 로그인한 개인에게만 보이는 위젯은 SEO 대상이 아니니 `'use client'`로 둬도 괜찮습니다. SEO가 필요한 건 누구에게나 똑같이 보이는 공개 콘텐츠(추천 레시피·카테고리)입니다.

## B. generateMetadata + metadataBase [필수/SEO]

### B-1. 루트 레이아웃에 metadataBase + 한국어 기본값

```tsx
// app/layout.tsx
export const metadata: Metadata = {
  metadataBase: new URL('https://cookeat.example.com'), // 실제 배포 도메인으로
  title: { default: 'Cookeat — 레시피 공유와 식재료 마켓', template: '%s | Cookeat' },
  description: '맛있는 레시피를 찾고, 필요한 식재료를 한 번에 주문하세요.',
};
```

`template: '%s | Cookeat'`을 쓰면 하위 페이지가 제목을 `'김치찌개'`로만 줘도 `'김치찌개 | Cookeat'`으로 완성됩니다. `metadataBase`가 있어야 OG 이미지 상대경로가 절대 URL로 변환됩니다(`seo-가이드.md` 1장).

### B-2. 상세 페이지에 generateMetadata

상세 페이지는 이미 데이터를 서버에서 들고 있어서(`fetchRecipeDetail`, `getProductDetail`), `generateMetadata`만 붙이면 됩니다. 같은 파일에 export 하나 추가하면 끝입니다.

```tsx
// app/(main)/recipes/[id]/page.tsx
import type { Metadata } from 'next';

export async function generateMetadata({ params }: Props): Promise<Metadata> {
  const { id } = await params;
  const recipe = await fetchRecipeDetail(Number(id)); // cache()라 본문과 중복 호출 안 됨
  if (!recipe) return { title: '레시피를 찾을 수 없습니다' };
  return {
    title: recipe.title,
    description: recipe.description?.slice(0, 120),
    openGraph: {
      title: recipe.title,
      description: recipe.description?.slice(0, 120),
      images: recipe.thumbnail ? [recipe.thumbnail] : [],
    },
  };
}
```

상품 상세도 동일하게 `product.name`/설명/대표이미지로 만들면 됩니다. `fetchRecipeDetail`을 `react`의 `cache()`로 감싸두면 `generateMetadata`와 페이지 본문이 같은 데이터를 두 번 fetch하지 않습니다.

## D. 리뷰 포인트 적립 — 원자적 증가 [필수]

지금 `app/api/products/[productId]/reviews/route.ts:135-152`는 "잔액 읽고 → +10 해서 덮어쓰기" 구조라, 동시에 두 번 들어오면 한 번치가 사라집니다. **읽지 말고 DB가 직접 더하게** 만들면 경쟁이 사라집니다. Supabase에선 RPC가 깔끔합니다.

```sql
-- supabase SQL: 원자적 적립 (포인트 증가 + 이력 기록을 한 트랜잭션으로)
create or replace function award_point(
  p_user_id int, p_amount int, p_description text
) returns void language plpgsql as $$
begin
  update users set point = point + p_amount where user_id = p_user_id;
  insert into point_history(user_id, type, amount, description)
  values (p_user_id, '적립', p_amount, p_description);
end;
$$;
```

```ts
// route.ts — :135-152 자리를 이렇게 교체
const { error: awardError } = await supabaseAdmin.rpc('award_point', {
  p_user_id: authed.userId,
  p_amount: pointAmount,
  p_description: '상품 리뷰 작성',
});

// 실패하면 응답에 정직하게 반영 (C-제안 항목)
return NextResponse.json(
  { reviewId: inserted.review_id, pointAwarded: awardError ? 0 : pointAmount },
  { status: 201 },
);
```

RPC가 부담되면 최소한 `update({ point: ... })`를 **읽은 값에 더하는 대신** raw 증가식(`point = point + 10`)으로 바꾸세요. supabase-js에서는 위 RPC가 가장 안전합니다. `recipes/[recipeId]/reviews/route.ts:98,111`도 똑같이 바꿉니다.

## E. 리뷰 이미지 검증 [필수]

`route.ts:74`에서 `images`를 받을 때, 그리고 `:128-131`에서 insert 하기 전에 검증을 넣습니다.

```ts
const STORAGE_HOST = 'xxxx.supabase.co'; // 우리 스토리지 도메인
const MAX_IMAGES = 5;
const MAX_LEN = 500;

const rawImages = Array.isArray(body.images) ? body.images : [];
const images = rawImages
  .filter((u: unknown): u is string => typeof u === 'string')
  .slice(0, MAX_IMAGES)
  .filter((u) => {
    if (u.length > MAX_LEN) return false;
    try {
      const url = new URL(u);
      return url.protocol === 'https:' && url.hostname.endsWith(STORAGE_HOST);
    } catch {
      return false;
    }
  });
```

이렇게 하면 문자열이 아닌 값, 개수 초과, 너무 긴 문자열, 우리 스토리지가 아닌 외부 URL이 전부 걸러집니다. 외부 입력은 "일단 의심하고 형식을 강제한다"가 기본입니다.

## F. 환불 목록 DB 페이지네이션 [제안]

`app/api/seller/orders/refunds/db.ts`가 전부 메모리에 올려놓고 `slice`로 자르고 있는데, 데이터가 쌓이면 매 요청이 전체 스캔이 됩니다. 가능한 필터(상태·기간)는 supabase 쿼리에 `WHERE`로 내리고, 페이지네이션은 `.range(from, to)`로 DB에서 자르세요.

```ts
const from = (page - 1) * limit;
let q = supabaseAdmin
  .from('refund_requests')
  .select('...', { count: 'exact' })
  .in('item_id', sellerItemIds)
  .order('requested_at', { ascending: false })
  .range(from, from + limit - 1);   // DB에서 페이지 단위로만 가져옴

if (startDate) q = q.gte('requested_at', `${startDate}T00:00:00`);
if (endDate) q = q.lte('requested_at', `${endDate}T23:59:59`);
```

"최신 1건만"(`latestByItem`)이나 키워드 같은 복잡한 조건은 한 번에 SQL로 옮기기 어려울 수 있으니, 우선 **기간·상태·페이지네이션부터** DB로 내리고 나머지는 단계적으로 줄여가면 됩니다.

## H. 응답 래퍼 통일 & 에러 안 삼키기

- **LikesList의 404 의심** — `LikesList.tsx:31`이 `/users/me/bookmarks`를 부르는데, `baseURL`이 `/api`라 실제 경로는 `/api/users/me/bookmarks`입니다. `app/api`에는 `users`가 아니라 `user`만 있으니, 라우트 경로(`user` vs `users`)와 응답 모양(`{ recipes }` vs `{ success, data: { recipes } }`)부터 맞춰 주세요. RecipeClient는 `res.data.data.recipes`인데 여기는 `data.recipes`라 둘 중 하나는 `undefined`가 됩니다.
- **catch 빈 처리** — `PointHistory.tsx:17`의 `.catch(() => {})`, `LikesList.tsx:33`·`CancelList.tsx:31`의 `.catch(() => setRecipes([]))`는 네트워크 실패를 "데이터 없음"으로 둔갑시킵니다. 에러 상태를 따로 두고 `toast.error('불러오지 못했습니다')`라도 띄워 주세요. 위 LikesList 404도 지금 이 catch 때문에 화면엔 "빈 목록"으로만 보여 발견이 늦어집니다.
