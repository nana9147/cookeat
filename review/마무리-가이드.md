# 남은 이틀, 개발을 어떻게 마무리할까 — D-2 안정화 가이드

> 오늘이 월요일, 내일이 화요일, 발표가 수요일입니다. 실제로 코드를 만질 수 있는 날은 오늘과 내일 이틀뿐이라 생각하고 읽어주세요.
> 발표를 어떻게 보여줄지(시연 가이드)와 채용담당자에게 레포가 어떻게 보이는지(취업 레포 가이드)는 앞서 따로 정리했으니, 이 글은 그 사이에 있는 "지금부터 발표 전까지 코드를 어떻게 다뤄야 하는가"만 다룹니다.

지난 6개월의 결과물을 마지막에 스스로 깨뜨리는 조가 매 기수 꼭 나옵니다. 이유는 거의 똑같습니다. 발표 전날 밤에 "하나만 더" 하고 새 기능을 붙이다가, 잘 되던 것까지 같이 망가뜨리는 거죠. 남은 이틀은 무언가를 더 만드는 시간이 아니라, **이미 만든 것이 발표장에서 안 깨지게 만드는 시간**이라 생각하고 접근하면 좋겠습니다.

---

## 1. 오늘부로 기능 동결(feature freeze)입니다

가장 먼저, 그리고 가장 중요한 이야기입니다. **오늘부터 새 기능은 시작하지 않습니다.** 지금 절반쯤 만들다 만 기능이 있다면, 완성하려 애쓰기보다 **깔끔하게 감추거나(비활성화) 떼어내는 쪽**을 먼저 고민해주세요. 반쯤 된 기능은 데모에서 십중팔구 그 반쪽이 터집니다.

동결의 기준을 이렇게 잡으면 판단이 쉬워집니다.

- **된다 → 손대지 않는다.** 동작하는 코드를 "더 예쁘게" 리팩토링하고 싶은 욕구는 발표가 끝난 뒤로 미룹니다. 지금의 리팩토링은 개선이 아니라 새로운 버그의 입구입니다.
- **거의 됐다 → 오늘 안에 끝내거나, 오늘 안에 숨긴다.** 내일까지 끌지 않습니다. 내일은 검증하는 날이지 만드는 날이 아닙니다.
- **안 된다 → 데모 동선에서 뺀다.** 없는 척이 아니라, 아예 클릭할 수 없게 링크를 감추거나 "준비 중" 상태로 둡니다.

"이건 넣고 싶었는데" 하는 미련은 발표 때 "다음 스프린트 계획"으로 말하면 됩니다. 미완성 기능을 억지로 밀어 넣어 데모를 불안하게 만드는 것보다, 되는 것만 자신 있게 보여주는 편이 훨씬 완성도 있게 보입니다.

---

## 2. 데모 경로부터 거꾸로 잠급니다

발표 시연 대본에 나오는 클릭 순서 — 메인 진입 → 목록 → 상세 → 로그인 → 핵심 행동 → 마이페이지 확인, 그리고 관리자/판매자 흐름까지 — 이 **한 줄기 경로가 이번 이틀의 1순위**입니다. 이 경로 위에 있는 화면과 버튼은 한 번의 예외도 없이 매끄럽게 돌아가야 하고, 이 경로 밖에 있는 것들은 전부 2순위로 미뤄도 좋습니다.

방법은 단순합니다. **하루에 최소 한 번, 발표 대본 그대로 처음부터 끝까지 직접 클릭해보세요.** 기능 단위로 여기저기 눌러보는 게 아니라, 실제 사용자가 서비스를 처음 쓰듯 한 호흡으로 따라가는 겁니다. 중간에 한 번이라도 멈칫하거나, 에러 토스트가 뜨거나, 빈 화면이 나오면 그게 오늘의 최우선 수정 대상입니다.

이때 반드시 **배포된 실제 주소에서** 걸어보세요. `localhost`에서만 되는 건 발표장에서 안 되는 것과 같습니다(3번에서 이어집니다).

---

## 3. 발표 대상은 localhost가 아니라 "배포본"입니다

발표는 여러분의 노트북이 아니라 실제 배포된 사이트에서 이뤄집니다. 그런데 로컬에서만 개발하다 보면 배포본이 며칠 전 상태로 멈춰 있거나, 환경변수가 빠져 특정 화면만 500이 나는 경우가 흔합니다. 오늘 안에 아래를 꼭 확인해주세요.

- **프로덕션 빌드가 진짜 통과하는가.** `npm run build`를 로컬에서 한 번 돌려보세요. dev 서버(`npm run dev`)에서는 잘 뜨던 화면이 프로덕션 빌드에서는 타입 오류나 `use client` 누락, 프리렌더 실패로 막히는 일이 자주 있습니다. dev만 믿지 마세요.
- **배포본이 최신인가.** 배포 사이트를 열어 오늘 만든 변경이 실제로 반영돼 있는지 봅니다. 자동 배포(Vercel 등)가 걸려 있는지, 아니면 수동으로 다시 올려야 하는지 지금 확인해두세요.
- **프로덕션 환경변수가 다 들어가 있는가.** Supabase URL·키, 결제 키, 사이트 URL 같은 값이 배포 환경에도 설정돼 있어야 합니다. 로컬 `.env.local`에만 있고 배포 대시보드에는 빠진 값이 있으면 그 기능이 배포본에서만 조용히 죽습니다.
- **콜드 스타트와 새 창(시크릿 모드)에서도 되는가.** 로그인 캐시가 남은 내 브라우저 말고, 아무 상태도 없는 새 시크릿 창으로 처음부터 걸어보세요. 발표장의 브라우저는 여러분의 로그인 세션을 기억하지 못합니다.

---

## 4. 이월된 필수 항목부터 닫습니다

그동안 리뷰에서 `[필수]`로 짚어둔 것들 중 아직 안 닫힌 게 남아 있을 겁니다. 특히 **보안 구멍**은 발표 후 레포를 공개하는 순간 "누구나 따라 할 수 있는 재현 레시피"가 되기 때문에, 지금이 마지막 정리 타이밍입니다. 남의 개인정보가 비인증으로 읽히거나, 결제 금액을 클라이언트가 정하도록 열려 있거나, 인가 없이 관리자 기능이 열려 있는 종류는 기능 예쁘게 만드는 것보다 우선입니다.

각 조가 지금 닫아야 할 구체적인 항목은 이번 회차 리뷰 본문의 "우리 조 마무리 우선순위" 부분에 정리해뒀으니, 그걸 오늘의 할 일 목록 맨 위에 올려주세요.

---

## 5. 빈 화면이 아니라 살아 있는 화면을 만듭니다

데모에서 제일 초라해 보이는 순간은 목록이 텅 비어 있거나, 이미지가 깨진 회색 네모로 뜨는 때입니다. 실제 서비스처럼 보이려면 **그럴듯한 시드 데이터**가 필요합니다.

- 목록·상세·검색 결과에 실제로 볼 만한 데이터가 몇 개는 들어가 있게 채워주세요. 제목이 "테스트", "asdf"인 글만 있으면 아무리 기능이 좋아도 미완성처럼 보입니다.
- 이미지가 필요한 서비스라면 대표 이미지 몇 장은 진짜(또는 그럴듯한 플레이스홀더)로 넣어주세요.
- **발표용 계정을 미리 만들어두세요.** 발표장에서 회원가입부터 하는 건 리스크가 큽니다(이메일 인증 대기, 네트워크 지연). 로그인된 상태에서 시작하거나, 로그인이 즉시 되는 준비된 계정으로 여세요.

---

## 6. Git은 지금부터 "조심 모드"입니다

막판에 여러 명이 급하게 커밋을 몰아치면 충돌이 방치되거나, 어제 고친 걸 오늘 누군가 되살리는 사고가 납니다. 남은 이틀은 아래를 지켜주세요.

- **열려 있는 PR을 오늘 정리합니다.** 리뷰 대기 중인 PR, 충돌난 PR을 발표 전날까지 끌지 마세요. 머지하거나 닫거나, 오늘 결정합니다.
- **발표 직전에는 머지하지 않습니다.** 늦어도 화요일 저녁에는 통합 브랜치를 얼리고, 그 시점의 안정된 커밋 하나를 "발표 버전"으로 기억해두세요(태그를 달아두면 더 좋습니다). 발표 당일 아침의 급한 커밋이 제일 위험합니다.
- **고친 건 다시 안 깨지게.** 오늘 필수 항목을 고쳤다면, 그 부분이 포함된 데모 경로를 다시 한 번 걸어 회귀가 없는지 확인하고 넘어갑니다. 고치자마자 다른 커밋이 되살리는 일을 막는 최소한의 습관입니다.

---

## 7. 남은 이틀, 역할을 겹치지 않게 나눕니다

인원이 있는 조라면 오늘 아침 10분만 모여 이틀치 역할을 나눠주세요. 다 같이 같은 화면을 만지면 충돌만 나고 진도는 안 나갑니다. 예를 들면 이런 식입니다.

- **한 명은 데모 경로 QA.** 하루 두 번 대본대로 걸으며 깨진 곳을 이슈로 올립니다.
- **한 명은 배포/환경.** 배포본 최신화, 프로덕션 빌드·환경변수·시드 데이터를 책임집니다.
- **한 명은 이월된 필수 항목.** 리뷰에서 남은 `[필수]`를 닫습니다.
- **한 명은 발표/문서.** README 데모 링크·스크린샷·조원별 기여 정리와 발표 대본을 맡습니다.

1인 조라면 순서를 이렇게 잡으면 됩니다 — 배포본 살리기 → 데모 경로 한 바퀴 → 남은 필수 → 발표 자료. 혼자일수록 "다 잘하기"보다 "제일 잘 보이는 한 장면을 확실히"가 이깁니다.

## 8. 오늘·내일 체크리스트 (한눈에)

**오늘(월)**
- [ ] 기능 동결 선언 — 새 기능 시작 금지, 반쯤 된 기능은 끝내거나 감추기
- [ ] 발표 대본대로 데모 경로 1회 완주(배포본·시크릿 창)
- [ ] `npm run build` 통과 확인, 배포본 최신화
- [ ] 이번 리뷰의 "우리 조 마무리 우선순위"([필수]) 착수
- [ ] 시드 데이터·발표용 계정 준비
- [ ] 역할 분담(팀)

**내일(화)**
- [ ] 남은 [필수] 마무리 + 데모 경로 재완주(회귀 확인)
- [ ] 열린 PR 정리, 통합 브랜치 동결(저녁)
- [ ] "발표 버전" 커밋 확정(가능하면 태그)
- [ ] 배포본에서 최종 데모 완주 1회 + 예비(동영상) 준비

---

마지막으로 하나만 더 말하자면, 이 이틀의 목표는 새 기능이 아니라 **"발표장에서 내가 손을 떨지 않을 자신"** 입니다. 되는 걸 확실히 되게 만들어 두면, 발표는 훨씬 편해집니다. 천천히, 그렇지만 침착하게 마무리해 갑시다. 화이팅입니다.
