# 열자마자 빛나는 레포 — 발표 다음, 이제 채용담당자를 위해

발표가 수요일이니 실제 개발은 화요일까지입니다. 지금부터는 관점을 하나 더 얹어야 합니다. 이 프로젝트는 발표로 끝나는 게 아니라 **GitHub에 공개돼 여러분 이력서에 링크로 걸릴** 물건입니다. 채용담당자나 현업 개발자가 그 링크를 눌렀을 때, 6개월을 갈아 넣은 결과물이 그만큼 빛나 보여야 합니다. 그런데 냉정하게 말하면, 잘 만든 프로젝트도 "레포를 열었을 때 보이는 것"을 신경 쓰지 않으면 그 노력의 절반도 전달되지 않습니다.

이 글은 다섯 조 코드를 쭉 훑고 나서, 조마다 공통으로 아쉬웠던 지점을 모아 정리한 겁니다. 마감까지 사흘, 코드는 이미 웬만큼 완성돼 있으니 이제 **적은 시간으로 인상을 크게 바꾸는** 작업 위주로 골랐습니다.

---

## 1. 라이브 데모 링크가 없으면, 대부분은 코드를 안 봅니다

이번에 제일 크게 눈에 걸린 공통점입니다. 다섯 조 README 어디에도 "지금 눌러볼 수 있는 링크"가 없습니다. 심지어 배포는 해놓고 링크만 안 건 조도 있었습니다.

채용담당자 입장에서 생각해봅시다. 하루에 이력서 수십 개를 봅니다. 링크를 눌렀는데 GitHub 코드부터 나오면, `git clone` 하고 `.env` 채우고 `npm install` 돌려서 실행해볼 사람은 거의 없습니다. 반대로 **눌렀더니 바로 돌아가는 서비스가 뜨면**, 그 순간 "아 이 사람들 진짜 만들었네"가 됩니다. 이 한 줄 차이가 생각보다 큽니다.

```md
## 🔗 라이브 데모
https://your-project.vercel.app

> 테스트 계정: guest@demo.com / demo1234
```

- Vercel 무료 배포면 충분합니다. Supabase 붙은 조는 환경변수만 Vercel 프로젝트 세팅에 넣으면 됩니다.
- 배포했다면 **README 최상단**에 링크를 답니다. 스크롤 안 내려도 보이게.
- 로그인이 필요한 서비스면 **데모용 계정 하나를 README에 같이** 적어주세요. 방문자가 회원가입부터 하게 만들면 절반은 거기서 나갑니다.
- 배포 후에는 실제로 눌러서 **최신 코드가 반영됐는지** 꼭 확인하세요. 링크는 걸어놨는데 옛날 버전이 뜨거나 깨져 있으면 안 하느니만 못합니다.

정적 사이트(백엔드 의존이 없는 조)라면 배포 리스크가 사실상 0이라 지금 당장 올려도 됩니다.

## 2. 스크린샷 한 장이 문단 열 개를 이깁니다

README를 열었는데 글자만 빽빽하면, 완성도가 아무리 높아도 그게 안 보입니다. **주요 화면 캡처 두세 장, 핵심 기능은 짧은 GIF 하나**면 첫인상이 완전히 달라집니다.

- 홈 화면, 핵심 기능 화면(예약·채팅·결제·상세 등) 정도면 됩니다.
- 실시간 채팅이나 3D 인터랙션처럼 **움직임이 핵심인 기능은 GIF**로. 정지 이미지로는 그 매력이 안 삽니다.
- 캡처는 이미 데이터가 그럴듯하게 차 있는 상태에서 찍으세요(다음 항목 참고).
- `public/screenshots/` 같은 폴더에 넣고 README에서 상대경로로 부르면 됩니다.

> 참고로 이번 리뷰에서 각 조 `review/images/2026-07-04/`에 오늘자 화면을 캡처해 뒀습니다. 그대로 README에 쓰긴 리뷰용이라 안 맞지만, "이 화면을 이렇게 찍으면 되겠구나" 정도 참고는 됩니다.

## 3. "누가 뭘 했는지"가 안 보이면, 팀 프로젝트는 오히려 손해입니다

지금 대부분 조의 README 팀원 표는 `이름 · 역할 · GitHub` 세 칸뿐입니다. 이러면 채용담당자가 **"이 지원자가 이 프로젝트에서 정확히 뭘 만들었는지"** 를 알 방법이 없습니다. 팀 프로젝트에서 개인 기여가 안 드러나면, 잘한 사람일수록 손해입니다.

담당 영역 한 칸만 더 붙이면 됩니다.

```md
## 조원별 기여

| 이름 | 담당 영역 | 대표 작업 |
|------|----------|-----------|
| 김OO | 결제·주문 | 서버 금액 재검증, 토스/카카오 결제 연동, 주문 원자 트랜잭션 |
| 이OO | 실시간 채팅 | Supabase Realtime 1:1 채팅, 읽음 처리, 차단 |
| 박OO | 관리자 | 대시보드, 신고·회원 관리, 서버측 권한 검증 |
```

- 이미 각자 만든 게 뚜렷하게 나뉘어 있습니다(커밋 로그를 보면 파일 소유권이 거의 안 겹칩니다). **없는 걸 지어내는 게 아니라, 있는 걸 README로 끌어올리는** 작업입니다.
- 이력서 자기소개에 "저는 이 프로젝트에서 결제 서버 검증을 맡아…"라고 쓸 때, 레포 README가 그걸 뒷받침해주면 신뢰도가 올라갑니다.
- **git author 이름을 하나로 통일**해두세요. 한 사람이 `gyuhwa9922` / `gyuhwa`처럼 두세 개 이름으로 커밋하면, GitHub Contributors 그래프에서 기여가 쪼개져 실제보다 적어 보입니다.

## 4. 기술 스택은 "무엇을"이 아니라 "왜"까지

지금 기술 스택 표는 대체로 로고 나열입니다. 채용담당자가 보고 싶은 건 "Next.js를 썼다"가 아니라 **"왜 이걸 골랐고, 어떤 문제를 이걸로 풀었나"** 입니다. 한 줄씩만 붙여도 사고 과정이 드러납니다.

```md
- **Zustand** — Redux보다 보일러플레이트가 적어 5주 일정에 맞았고, 장바구니처럼 전역이지만 단순한 상태에 적합
- **Supabase Realtime** — 채팅 실시간성을 위해 별도 소켓 서버 없이 postgres 변경 구독으로 구현
```

여유가 되면 **인증 흐름이나 결제 흐름 다이어그램 한 장**을 넣어주세요. 텍스트로 열 줄 설명할 걸 그림 한 장이 대신합니다. 이미 `docs/` 폴더에 좋은 설계 문서를 쌓아둔 조들이 있는데, README에서 그 문서로 가는 링크조차 없어서 아무도 안 들어갑니다. **레포의 진짜 강점이 README에서 안 보이면 없는 것과 같습니다.**

## 5. 레포를 열어 `npm run lint`를 돌려보는 사람도 있습니다

다섯 조 전부 `npm run build`는 통과하는데, `npx eslint .`를 돌리면 에러가 적게는 8개, 많게는 60~70개씩 나옵니다. 대부분 최신 Next.js가 새로 도입한 `react-hooks/set-state-in-effect` 규칙에 걸리는 것들이라 **당장 버그는 아닙니다.** 다만 채용담당자가 레포를 열어 lint를 한 번 돌려보면 빨간 줄이 그대로 보입니다.

- CI(`.github/workflows`)가 `tsc`와 `build`만 검사하고 **lint는 게이트에서 빠져 있어서**, 이 에러들이 계속 쌓인 채 머지돼 왔습니다.
- 시간이 있으면 `npm run lint -- --fix`로 정리하거나, 의도적으로 짠 패턴이면 왜 그런지 주석 한 줄 + `eslint-disable-next-line`로 사유를 남겨두세요.
- 최소한 **CI에 lint 스텝 하나**만 추가해두면 "관리되는 레포"라는 인상을 줍니다. "빌드/타입/린트 다 초록불"인 레포는 그 자체로 신뢰를 삽니다.

## 6. 데모 데이터가 비어 있으면 서비스가 초라해 보입니다

캡처를 찍든 라이브 데모를 걸든, **화면에 그럴듯한 데이터가 차 있어야** 합니다. 방금 만든 빈 계정으로 들어가면 목록도 비고 후기도 없어서, 잘 만든 서비스가 껍데기처럼 보입니다.

- 이번에 한 조는 홈 화면 상품·레시피 썸네일이 전부 `600 × 400` 회색 플레이스홀더로 떠 있었습니다. 기능은 다 되는데 이미지가 비어 있으니 완성도가 확 깎여 보입니다. 데모용 시드 데이터에 실제 이미지 몇 장만 채워도 인상이 달라집니다.
- 상품 몇 개, 대화 몇 개, 후기 몇 개가 이미 있는 상태를 만들어두세요. 발표 시연에서도 그대로 씁니다.

## 7. 오픈소스로 공개하면, 보안 구멍은 "재현 레시피"가 됩니다

이건 취업 레포에서 특히 무서운 부분입니다. 지금까지 리뷰에서 몇 번 짚은 보안 이슈들(anon 키로 개인정보가 읽히거나, 결제 금액을 클라이언트가 정하거나)은, **레포가 비공개일 때는 그나마 덜 위험하지만 공개하는 순간 성격이 바뀝니다.** 코드가 다 보이니까, 취약점이 있으면 그게 곧 "이렇게 하면 뚫린다"는 설명서가 됩니다.

- **anon(공개) 키로 민감 테이블이 읽히면 안 됩니다.** 실제 팀원 이름·전화번호·주소가 브라우저 네트워크 탭이나 공개 RPC로 새는 조가 있었습니다. 관심 있는 리뷰어는 devtools만 열어도 봅니다. RLS로 막고, 필요한 필드만 뷰로 노출하세요.
- **결제 금액은 반드시 서버가 다시 계산**합니다. 클라이언트가 보낸 금액을 믿으면 1원 결제가 됩니다. 이건 코드 한 파일만 열어도 보이는 구조적 결함이라, 공개 레포에서는 특히 눈에 띕니다.
- 세부 처방은 각 조 리뷰에 파일·줄 번호까지 적어뒀으니, 공개 전에 이것만은 꼭 닫읍시다.

## 8. 마지막 손질 체크리스트

레포를 공개 상태로 정리하기 전에 이 정도만 훑어도 인상이 크게 정돈됩니다.

- [ ] README 최상단에 **라이브 데모 링크**(+ 데모 계정)
- [ ] 홈·핵심 기능 **스크린샷/GIF**
- [ ] **조원별 담당 영역** 표
- [ ] 기술 선택 이유 한 줄씩 + (가능하면) 아키텍처 다이어그램 1장
- [ ] `docs/` 좋은 문서로 가는 **링크**
- [ ] **LICENSE** 파일 (MIT 등 — 대부분 조가 없습니다)
- [ ] README **클론 URL이 조직 저장소**인지 (개인 fork URL로 적힌 조 있었음)
- [ ] 중도 하차·역할 변경이 팀 표에 **어떻게 반영될지 팀 합의**
- [ ] `.env.local`이 git에 **안 올라갔는지** (`git ls-files | grep env`로 확인)
- [ ] anon 키로 **개인정보·민감 테이블이 안 읽히는지**
- [ ] `npm run build` 초록불 + 가능하면 **lint까지 초록불**
- [ ] 커밋 author 이름 **하나로 통일**

코드는 이미 충분히 만들었습니다. 이제 남은 사흘은, 그 코드를 처음 보는 사람에게 **어떻게 보여줄지**에 조금만 더 쓰면 됩니다. 같은 결과물이라도 이 손질을 거치면 훨씬 멀리 갑니다. 화이팅입니다.
