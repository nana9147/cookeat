## 1. [필수] 카카오페이 금액 서버검증

토스 쪽에 이미 잘 잡아 둔 패턴이 있으니, 그걸 카카오에 그대로 옮겨오면 됩니다.

### (1) `ready`에서 클라 금액을 무시하고 서버 값 강제

```ts
// payment/kakao/ready/route.ts
export async function POST(req: Request) {
  const { orderId } = await req.json(); // ← totalAmount는 클라에서 받아도 쓰지 않는다

  // 주문 실금액을 서버에서 조회 (본인 주문인지도 확인)
  const { data: order, error } = await supabase
    .from("orders")
    .select("final_amount, user_id, status")
    .eq("id", orderId)
    .single();

  if (error || !order) return NextResponse.json({ error: "주문 없음" }, { status: 404 });
  if (order.user_id !== user.id) return NextResponse.json({ error: "권한 없음" }, { status: 403 });

  // KakaoPay ready에 넘기는 금액은 반드시 서버의 final_amount
  const kakaoRes = await fetch("https://open-api.kakaopay.com/online/v1/payment/ready", {
    // ...
    body: JSON.stringify({
      // ...
      total_amount: order.final_amount, // ← 클라 totalAmount 아님
    }),
  });
}
```

### (2) `approve`에서 실제 승인 금액을 주문 금액과 대조

```ts
// payment/kakao/approve/route.ts
const approveRes = await fetch(".../payment/approve", { /* ... */ });
const approved = await approveRes.json();

// 카카오가 실제 승인한 금액 vs 주문 실금액
if (approved.amount.total !== order.final_amount) {
  // 금액 불일치 → 결제 완료로 마킹하지 말고 취소/보류
  return NextResponse.json({ error: "결제 금액 불일치" }, { status: 400 });
}
// 여기까지 왔을 때만 status='paid'
```

토스의 `toss/confirm/route.ts:22`에서 하는 `order.final_amount !== amount` 검사와 원리가 같은데, 결국 금액은 클라이언트가 정하는 게 아니라 서버가 정한다는 게 핵심입니다.

## 2. [필수] 쿠폰 `'%'` → `'rate'`

세 곳만 고치면 되는데, DB와 상품 폼이 이미 `'rate'`를 쓰고 있으니 나머지를 여기에 맞춰 주는 거라 보면 됩니다.

```ts
// order/route.ts:142
- if (coupon.discount_type === '%') {
+ if (coupon.discount_type === 'rate') {
    discount = Math.floor(amount * coupon.discount_value / 100);
  } else {
    discount = coupon.discount_value; // 정액
  }
```

`coupons/validate/route.ts:46`, `DiscountSection.tsx:151`도 똑같이 `'%'` → `'rate'`로 바꿔 주세요.

### 재발 방지 — 도메인 상수로 모으기

```ts
// lib/constants.ts
export const DISCOUNT_TYPE = { rate: "rate", amount: "amount", none: "none" } as const;
export type DiscountType = (typeof DISCOUNT_TYPE)[keyof typeof DISCOUNT_TYPE];
```

이렇게 두고 `discount_type === DISCOUNT_TYPE.rate`처럼 쓰면 `'%'` 같은 오타를 내는 순간 tsc가 바로 잡아 주고, 상품과 쿠폰 양쪽이 같은 상수를 바라보게 되니 도메인 간에 값이 어긋나는 일도 없어집니다.

### 확인

```bash
curl "localhost:3130/api/coupons/validate?code=WELCOME10&amount=100000"
# 고치기 전: discountAmount: 10  (정액 10원 — 버그)
# 고친 후:   discountAmount: 10000 (10% — 정상)
```

## 3. [제안] coupons 테이블 anon 차단

```sql
alter table public.coupons enable row level security;
drop policy if exists "coupons readable by everyone" on public.coupons;
-- anon select 정책을 두지 않으면 REST로 못 읽음.
-- 쿠폰 유효성 검증은 서버 라우트가 service-role로 수행하므로 화면 기능엔 지장 없음.
```

검증 라우트(`coupons/validate`)가 서버에서 service-role로 조회하기 때문에, anon read를 막아도 사용자 쿠폰 적용은 그대로 동작합니다.

## 4. [사소] `req.json()` 감싸기

```ts
let body: unknown;
try {
  body = await req.json();
} catch {
  return NextResponse.json({ error: "잘못된 요청 형식입니다." }, { status: 400 });
}
```

라우트 진입부에서 이렇게 한 번 감싸 두면 깨진 바디가 들어와도 500이 아니라 400으로 처리됩니다. 반복되면 공통 헬퍼로 빼도 좋습니다.

## 5. [제안] 정산 성능 (여유 있을 때)

지금은 `ensureSettlements`가 매 요청마다 전건을 JS로 합산하고 있는데, 마감이 급하면 이건 후순위로 미뤄도 괜찮습니다. 방향만 적어 두면, 주차별 합계는 아래처럼 SQL로 넘기는 편이 훨씬 빠릅니다.

```sql
select date_trunc('week', confirmed_at) as week,
       sum(price * quantity) as total
from order_items
where seller_id = $1 and status = 'confirmed'
group by 1;
```

아니면 정산을 확정하는 시점에 그 주차 합계를 `settlements` 행으로 굳혀 두는 스냅샷 방식도 있는데, 이러면 대시보드는 그 행만 읽으면 됩니다. 대시보드에서 `getSellerOrderIds`를 두 함수가 따로따로 부르고 있는 것도 상위에서 한 번만 구해 인자로 내려주면 order_items 조회가 절반으로 줄어듭니다.
