## 2. 이번 핵심 지적

### [필수] `npm run build` / `npx tsc --noEmit` 실패 — 타입 6건 (실측)

2차에 이어 빌드가 다시 막혀 있습니다. 직접 돌린 결과입니다.

```

app/seller/page.tsx:21,29,37,45           Type 'number' is not assignable to type 'string'   (Order.price)

```

`next build` 도 같은 StatusBadge 에서 `Failed to type check` 로 멈춥니다. 원인이 3가지로 갈립니다.

1. **StatusBadge `취소`/`환불`** — `f49620a`에서 OrderStatus 배지를 추가하면서 `취소`/`환불` 스타일을 넣었는데, `types/seller/order.ts`의 `OrderStatus`는 `'배송준비중' | '결제완료' | '배송중' | '배송완료'` 4개뿐입니다. `Record<...>` 키에 타입에 없는 값을 넣어 에러가 납니다. → `OrderStatus`에 `'취소' | '환불'`을 추가하거나, 배지에서 두 키를 빼야 합니다.
2. **`app/seller/page.tsx`의 mock `price`** — `Order.price`는 `string`인데 mock은 `price: 15900`처럼 숫자입니다. 또 mock엔 `orderDate`가 있는데 `Order` 타입엔 없습니다. 타입과 데이터가 어긋났어요.
3. **`edit/page.tsx`의 `deliveryMethod`** — `d1b6b25`에서 배송을 택배 단일화하며 `ShippingData`에서 `deliveryMethod`를 뺐는데, 상품 수정 페이지 mock(`MOCK_PRODUCT.shippingInfo`)엔 아직 `deliveryMethod: '택배'`가 남아 있습니다.

세 곳 다 "타입을 바꾼 쪽"과 "그 타입을 쓰는 데이터"가 같이 안 움직여서 생긴 일입니다. **빌드가 막히면 Vercel 배포가 안 되니 이번 회차 1순위**로 맞춰주세요. 해결법은 가이드 문서에 코드로 적었습니다.

> 제안: 이런 회귀를 막으려면 PR 머지 전에 `npm run build`(또는 `tsc --noEmit`)를 한 번 돌리는 걸 팀 규칙으로 두면 좋습니다. CI(GitHub Actions)에 build 체크 한 줄만 걸어도 "타입 깨진 PR"이 자동으로 걸립니다.

### [필수] 셀러 화면(`/seller/*`) 인가 게이트 부재 — E2E로 재확인

2차에 이어 이번에도 **비로그인 상태로 `/seller/products/new`가 그대로 풀 렌더**됩니다(E2E 캡처 `3rd-01`). `app/seller/layout.tsx`에 가드가 없어서요(=어드민은 `AdminAuthGuard`가 있는데 셀러는 비어 있음).

지금은 mock이라 데이터가 새진 않지만, 상품 등록 API를 붙이는 순간 "판매자가 아닌 사람이 판매자 화면에 들어와 등록을 시도"하는 길이 열립니다. `serverAuth`에 이미 `requireSeller`를 만들어 두셨으니, **화면 가드(셀러 레이아웃)**와 **API 가드(`requireSeller`)** 양쪽을 함께 거는 형태로 가는 걸 권합니다. 어드민 레이아웃 가드를 셀러용으로 복제하면 금방입니다(가이드 참고).

### [제안] `RoleGuard`(구 AdminAuthGuard)는 아직 토큰만 본다

`app/admin/components/AdminAuthGuard.tsx`는 컴포넌트 이름을 `RoleGuard`로 바꿨지만 내부는 여전히 `accessToken` 유무만 확인합니다. 즉 **로그인한 일반 회원도 `/admin`에 들어옵니다**(토큰만 있으면 통과). 이름이 RoleGuard인데 role을 안 보는 상태라, 다음 단계에서 `requireAdmin`처럼 role까지 확인하도록 채워주면 이름과 동작이 맞습니다.

다만 화면 가드는 어디까지나 UX(엉뚱한 화면을 안 보여주기)이고, **진짜 방어선은 서버**입니다. 어드민 데이터 API를 붙일 때 각 라우트에서 `requireAdmin`을 호출하는 걸 최우선으로 보세요. 화면 가드만 있고 API가 무방비면, API를 직접 때리는 요청은 막히지 않습니다.

## 3. 멤버별 리뷰

### 엄인호 (@djsy01) — serverAuth / 판매자 신청 / profile auth_id 전환

이번 변경의 보안 골격을 만든 부분이라 먼저 칭찬부터 하겠습니다.

- **[칭찬] `lib/serverAuth.ts` 설계가 좋습니다.** `requireAuth` / `requireSeller` / `requireAdmin`을 단계적으로 쌓고, `NextResponse` 반환을 `instanceof`로 구분하는 패턴이 깔끔합니다. 토큰으로 `auth.getUser` → `users` 테이블에서 role 조회 → role 검사 흐름이 교과서적이에요.
- **[칭찬] 판매자 신청 API가 제대로 짜였습니다.** `route.ts`(흐름) / `validate.ts`(검증) / `db.ts`(쿼리) 3분할이 읽기 좋고, `pending`/`approved`/`rejected` 상태에 따라 409 분기 → 재신청(`reapplySeller`) 처리까지 들어가 있습니다. 실측으로 **비로그인 POST는 401**, 잘못된 입력은 **400 + 필드별 메시지**가 정확히 나왔습니다.
- **[칭찬] `0814350` profile 조회 기준을 email → auth_id로 바꾼 건 옳은 방향입니다.** 이메일은 바뀔 수 있는 값이고 auth_id(Supabase user id)가 불변 식별자라, 조인/조회 키로 auth_id를 쓰는 게 맞습니다.

- **[제안] `useSellerApply`의 fetch 에러 처리.** `useSellerApply.ts`의 GET은 `.then(r => r.json())`만 있고 `res.ok`/네트워크 실패를 안 봅니다. 401(토큰 만료)이나 500이 와도 `json.data`가 없으면 그냥 `null` 처리돼, 사용자는 "신청 안 한 상태"와 "에러"를 구분 못 합니다. `if (!r.ok)` 분기로 에러 상태를 따로 두면 화면에서 안내할 수 있습니다.
- **[사소] `submit` 후 낙관적 갱신.** 신청 성공 후 로컬 state를 만들어 채우는데(`created_at: new Date()...`), 서버가 준 값과 미세하게 다를 수 있습니다. 지금 단계에선 괜찮지만, 가능하면 성공 후 GET 재호출로 서버 값을 다시 받아오는 쪽이 안전합니다.

## 6. 보안 실측 (RLS)

프로젝트 실제 Supabase에 anon(publishable) 키로 직접 조회해 봤습니다(service role로 행 수 대조).

| 테이블 | service role 행 수 | anon 조회 결과 | 판정 |
| ------ | ----------------- | ------------- | ---- |
| `users` | 7 | `[]` | **RLS 차단 정상** |
| `sellers` | 1 | `[]` | **RLS 차단 정상** (신규 sellers 테이블도 보호됨) |
| `products` | 0 | `[]` | 빈 테이블 — 판정 보류 |
| `orders` | 0 | `[]` | 빈 테이블 — 판정 보류 |

`users`에 7행이 있는데 anon으론 안 읽힙니다 → RLS가 살아 있습니다. 새로 생긴 `sellers`도 1행 있는데 anon 차단 → 좋습니다. `products`/`orders`는 아직 데이터가 없어 RLS가 켜졌는지 단정은 못 합니다. 상품/주문 데이터를 넣기 시작하면 같은 방식으로 다시 확인해주세요(anon으로 남의 행이 읽히면 RLS 구멍).
